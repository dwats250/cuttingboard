# PRD-164 Claude Review

VERDICT
ACCEPT WITH CHANGES

SUMMARY
PRD-164 hardens the PRD lifecycle tooling so closeout is a single commit
(`prd_close.sh` flips the existing registry row + both status markers + active
pointer), Stage-0 rows land in the main registry table (`prd_open.sh`), and the
validator detects commit-hash drift. The lane (STANDARD / INFRA), FILES set,
and the eight requirements are well-formed and the change is genuinely
high-leverage. Four requirements under-specify behavior in ways that pass the
new tests while breaking real-repo validation or PRD-doc formatting: R6 (git
root + multi-hash), R3 (exact marker outputs), R2 (missing-row path), and a
metacharacter-safety gap in R5.

REQUIRED EDITS

1. **R6 must run git against the validation root and skip non-git / docless
   trees.** `PRD-164.md:137`. The validator is invoked with an explicit root
   (`validate_prd_registry.py` `main`, `Path(args[0])`), and
   `tests/test_prd_open.py::test_resulting_tree_passes_validator` runs it
   against a non-git temp tree whose COMPLETE rows use synthetic hashes
   (`aaaaaaa`) and `File = —` (`tests/test_prd_open.py:32-52,161`). The git
   resolvability check must target that root and be skipped when the root is
   not a git work tree.
   FAIL: an otherwise-consistent non-git tree, or a tree whose COMPLETE rows use
   `File = —`, fails validation solely because of the new check.

2. **R6 must define multi-hash handling.** `PRD-164.md:137`. The real registry
   records comma-separated hashes for COMPLETE PRDs (`docs/PRD_REGISTRY.md:178`,
   PRD-158 = 7 hashes). The resolvability check must split the commit cell and
   verify each hash, and the doc/registry-agreement check must compare against
   the full hash set.
   FAIL: a `COMPLETE` row with `h1, h2` is treated as a single unresolved
   string (false-fail) or passes without each hash being checked.

3. **R3 must pin exact marker outputs.** `PRD-164.md:116`. The scaffold carries
   two markers with different completed forms (cf. PRD-163: header
   `Status: COMPLETE`, trailing `STATUS: COMPLETE @ b57e1a4`). A uniform
   case-insensitive substitution would corrupt the header into
   `STATUS: COMPLETE @ <hash>`. The requirement must state the header becomes
   exactly `Status: COMPLETE` and the trailing line exactly
   `STATUS: COMPLETE @ <hash>`, with exactly one of each.
   FAIL: after closeout the header is not exactly `Status: COMPLETE`, or the
   trailing line is not exactly `STATUS: COMPLETE @ <hash>`, or a duplicate
   trailing marker exists — even with no `IN PROGRESS` remaining.

4. **R2 must state the missing-row policy.** `PRD-164.md:108`. R2 changes the
   existing-row case from skip → flip; the PRD must also say what happens when
   the row is absent (`scripts/prd_close.sh:111`), reconciled against the
   closeout-skill Registry-row invariant
   (`.claude/skills/prd-closeout-verified/SKILL.md`), which has the preflight
   refuse a missing row before the script runs.
   FAIL: the PRD leaves the missing-row behavior unspecified, so an
   implementation may silently append a row that the documented invariant
   forbids.

RECOMMENDED EDITS

1. R5 `--next` should insert via a literal/callable replacement, not an `re.sub`
   template — mirroring the PRD-147 hardening already applied to `--title` /
   `--summary` in `scripts/prd_close.sh`. FAIL: `--next 'foo \d bar'` raises
   `re.error` or inserts corrupted text.

2. The R1 regression test should assert the new row's *position* relative to a
   trailing `## Audit Reports` table, not mere containment
   (`tests/test_prd_open.py:90` currently checks containment only); the
   `_make_tree` fixture must be extended with an Audit Reports table to
   reproduce the bug.

RATIONALE
- R6 is the highest-risk surface: the validator runs both in CI (real git repo)
  and against synthetic fixture trees, and the real registry's hash shapes
  (multi-hash, `File = —`) are not uniform. Under-specifying git scope or
  multi-hash handling yields a check that is simultaneously too strict (fixtures)
  and too loose (multi-hash) — CLAUDE.md § PRD-author disciplines, realizability.
- R3 maps to the actual scaffold/closeout convention (two markers, two forms);
  a binary FAIL on the exact output prevents the obvious uniform-substitution
  defect — CLAUDE.md § FAIL lines must be observable.
- R2's missing-row policy is a dead-branch enumeration item: the retired skip
  branch has a sibling (append-when-absent) whose fate must be declared —
  CLAUDE.md § dead-branch enumeration.
