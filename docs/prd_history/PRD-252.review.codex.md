review already caught and fixed):** No test in the reviewed snapshot

continuation-mode result, so nothing would fail if the

`CONTINUATION_MAX_RISK_PCT_PER_TRADE` were removed or bypassed.
Sol's independent search converged on the exact same site the

`cuttingboard/options.py`'s correlation-modifier recompute (then at
line ~224, now ~232-237 post-fix). **This is now fixed** — a

`tests/test_phase5.py`) was added after the Claude review found the same
gap; it landed before this Codex disposition ran but was intentionally

**CONFIRMED — genuinely new finding, NOT caught by Gate A or the
fresh-context Claude review, PRE-EXISTING (predates PRD-252; `contract.py`
and `audit.py` are not in PRD-252's diff):** `cuttingboard/contract.py`'s
`_build_trade_candidates` and `cuttingboard/audit.py`'s qualified-trade
audit-record builder both source `position_size`/`contracts` and
`dollar_risk` from the ORIGINAL `QualificationResult` (pre-correlation-
modifier), not from the correlation-adjusted `OptionSetup` that
`build_option_setups` produces — even though both functions DO pull
`strategy`/`structure`/`dte`/`estimated_debit` from `OptionSetup` for the
same candidate. Verified directly against the current worktree (not
trusted from Sol's prose):
- `cuttingboard/contract.py:359-364` — `"position_size": ... result.max_contracts`,
  `"dollar_risk": ... result.dollar_risk` (comment at :336-339 explicitly
  documents this as a PRD-157 "sizing passthrough").
- `cuttingboard/audit.py:132-133` — `"contracts": r.max_contracts,
  "dollar_risk": r.dollar_risk` (same `r` = `QualificationResult`).
- By contrast, `cuttingboard/output.py`'s `render_report` (the
  human-facing premarket report) takes `option_setups: list[OptionSetup]`
  directly as a parameter and is NOT affected.

Practical impact, worked through and verified (not merely asserted):
- The human-facing premarket report is unaffected and correctly reflects
  PRD-252's continuation-vs-direct budget split (it reads `OptionSetup`
  directly).
- PRD-252's specific $150-vs-$400 continuation split DOES still reach the
  canonical `contract.trade_candidates` JSON and the audit log correctly
  -- because `QualificationResult.max_contracts`/`dollar_risk` for a
  continuation candidate are already computed against the correctly
  decoupled `CONTINUATION_MAX_RISK_PCT_PER_TRADE` at
  `qualification.py:729`, before `build_option_setups` ever runs. The
  qualification-layer decouple is, in effect, the one that actually
  matters for this specific surface.
- What IS broken, and appears to predate PRD-157's original correlation-
  modifier feature entirely: the correlation `risk_modifier` reduction
  itself (the entire point of `build_option_setups`'s recompute, for ANY
  candidate -- direct or continuation) never reaches
  `contract.trade_candidates`'s `position_size`/`dollar_risk` or the audit
  log's `contracts`/`dollar_risk`. A correlation-flagged trade
  (`risk_modifier < 1.0`) would show its full, un-reduced Gate-8 size in
  the JSON contract and audit trail, while the human-readable report
  correctly shows it reduced -- a real discrepancy between what a trader
  reads and what downstream automation (dashboards, notification
  formatters, audit consumers) would see for that field.

Severity: real and worth a dedicated investigation, but OUT OF SCOPE for
PRD-252 (does not touch PRD-252's FILES; predates this PRD; PRD-252's own
