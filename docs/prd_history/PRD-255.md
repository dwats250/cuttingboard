GOAL
Bring `prd-review-claude` to the 2026-07-07 review-artifact spec
(SHA-pinning, merge-base naming, fresh-context attestation) and let it
assert implementation pass/fail, since REVIEW is defined
(`audits/EXECUTION_DOCTRINE.md` sec 1) as covering the implementation
against the PRD, not the PRD document alone.

REQUIREMENTS

R1 — REVIEWED STATE header block
The Review structure gains a `REVIEWED STATE` block above VERDICT:
reviewed SHA, merge base, and the fresh-context/different-model/
same-context independence line (subsuming `PRD_REVIEW_TEMPLATE.md`'s
existing checkbox, not duplicating it).

FAIL: a review the skill's structure defines can omit reviewed SHA or
merge base and still be well-formed.

R2 — Verify the header, don't just require it
Phase 2 gains two checks: the reviewed SHA resolves to a real commit
(`git cat-file -e`), and the stated merge base equals `git merge-base
HEAD origin/main` computed at review time, not asserted from memory.

FAIL: either check is absent from the Phase 2 table, or present as a
manual-attestation line with no described verification step.

R3 — Remove V9; implementation gets a verdict
Delete V9 ("does not assert implementation passes/fails"). The slot
becomes a requirement that the review states, per REQUIREMENT, whether
the diff satisfies its FAIL line, evidenced by real command/test output
— the REVIEW stage covers implementation-against-PRD by doctrine, and a
skill forbidding that is reviewing a world PRD-242 retired.

FAIL: the skill file still contains a check forbidding an
implementation-pass/fail assertion, or the Review structure has no field
for it.

R4 — Second-model delegation pattern, as prose, in PRD_PROCESS.md
`## Second-Model Disposition` gains a subsection describing the pattern:
a git-free pre-review snapshot of the reviewed commit; the reviewer never
shown the Claude-review (Leg-1) findings; an orchestrator model that
forms falsifiable hypotheses and issues exact search directives, and a
retriever model that executes them exhaustively with no risk judgment of
its own — `PRD-252.review.codex.md`'s Sol/Luna split cited as the worked
example, not mandated as required-forever-named roles.

FAIL: the section doesn't name the three binding conditions
(git-free snapshot, findings blind, persona split), or a new skill file
is created instead.

R5 — DECISIONS.md calibration entry
Dated entry: PRD-252's review found mutation-style review catches
in-diff leaks and exhaustive out-of-diff sweeps catch consumer-side
interactions — both were needed; neither framing alone would have found
what the other did.

FAIL: no entry citing PRD-252 makes this claim.

R6 — DECISIONS.md Pass-3 resolution line
One line: `.claude/skills/` is NOT INFRA HIGH-RISK by default; a skill is
governance-tier iff it gates a merge; today only `prd-review-claude`
qualifies.

FAIL: no such line exists, or it contradicts CLAUDE.md's named
governance-guardrail list.

R7 — Cross-PRD Lane Mixing subsection (rule-vs-practice divergence)
Add, immediately after `### Lane Downgrade Prohibition (PRD-121 R11)`, a
new `### Cross-PRD Lane Mixing (PRD-255)` subsection, verbatim:

> Multiple PRDs of different lanes MAY share one PR — existing practice
> (PR #99's Block-1 batch, 2026-07-04) permits it; R11 governs only each
> PRD's own declared lane, not a PR's aggregate ceremony, and was silent
> on the cross-PRD case until now. **The PR's merge ceremony is the
> STRICTEST lane present among its constituent PRDs** — a PR carrying
> any HIGH-RISK or governance-guardrail PRD is human-held regardless of
> what else rides along. This does not relax the separate, categorical
> 2026-07-07 rule that agents never initiate or queue a merge for any
> lane — that rule already makes every PR human-held at the merge action
> itself; this subsection states the ceremony floor for what happens
> before that action (which review legs are owed, whether auto-merge
> may even be queued once a human does act).

Disposition chosen: CODIFY-PERMITTED, not forbid-one-lane-per-PR.
Rationale: matches actual practice (PR #99, and PR #135 itself);
forbidding it would force every cosmetic MICRO fix into its own PR
forever, working against the anti-ceremony-bloat philosophy PRD-229/230
already established; and FILES-level scope lock already bounds each
PRD's own blast radius independently of how many PRDs share a PR.

FAIL: the subsection is absent, is not placed adjacent to R11, or omits
the "strictest lane present" rule.

R8 — Allocated-but-Unlanded PRD Numbers subsection
Add, to `## Registry Maintenance`, a new `### Allocated-but-Unlanded PRD
Numbers (PRD-255)` subsection, verbatim:

> A PRD number is allocated at Stage-0 authoring time
> (`scripts/prd_open.sh`), which can happen on any branch or worktree —
> not only one that lands promptly. The registry validator requires
> every allocated number up to `latest_complete` to have a doc on
> `main`; a Stage-0 PRD drafted and left parked on an unmerged branch
> therefore blocks the closeout of any later-numbered PRD, even one
> unrelated to it in content or authorship (PRD-254's closeout was
> blocked by exactly this gap — PRD-253, allocated 2026-07-10, parked
> unmerged until PR #136 landed it — 2026-07-11). **Rule: before closing
> out PRD-N, every allocated number < N MUST already have a landed doc
> on `main`** (any status suffices — IN PROGRESS/PROPOSED is fine,
> COMPLETE is not required). If a lower number is still parked, land its
> Stage-0 scaffold first, unmodified, with its original authorship
> preserved, as its own PR — a landing-order constraint on the
> DOCUMENT, not an implementation-order constraint on the WORK; the
> parked PRD's actual build stays wherever it was queued.

FAIL: the subsection is absent, is not placed under Registry
Maintenance, or omits the "landed doc on main" rule.
