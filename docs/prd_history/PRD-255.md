R7 — Cross-PRD Lane Mixing subsection (rule-vs-practice divergence)
Add, immediately after `### Lane Downgrade Prohibition (PRD-121 R11)`, a
new `### Cross-PRD Lane Mixing (PRD-255)` subsection, verbatim:

> Multiple PRDs of different lanes MAY share one PR — existing practice
> (PR #99's Block-1 batch, 2026-07-04) permits it; R11 governs only each
> PRD's own declared lane, not a PR's aggregate ceremony, and was silent
> on the cross-PRD case until now. **The PR's merge ceremony is the
> STRICTEST lane present among its constituent PRDs** — a PR carrying
> any HIGH-RISK or governance-guardrail PRD is human-held regardless of
> what else rides along. This does not relax the separate, categorical
> 2026-07-07 rule that agents never initiate or queue a merge for any
> lane — that rule already makes every PR human-held at the merge action
> itself; this subsection states the ceremony floor for what happens
> before that action (which review legs are owed, whether auto-merge
> may even be queued once a human does act).

Disposition chosen: CODIFY-PERMITTED, not forbid-one-lane-per-PR.
Rationale: matches actual practice (PR #99, and PR #135 itself);
forbidding it would force every cosmetic MICRO fix into its own PR
forever, working against the anti-ceremony-bloat philosophy PRD-229/230
already established; and FILES-level scope lock already bounds each
PRD's own blast radius independently of how many PRDs share a PR.

FAIL: the subsection is absent, is not placed adjacent to R11, or omits
the "strictest lane present" rule.

R8 — Allocated-but-Unlanded PRD Numbers subsection
Add, to `### Registry Maintenance`, a new `### Allocated-but-Unlanded PRD
Numbers (PRD-255)` subsection, verbatim:

> A PRD number is allocated at Stage-0 authoring time
> (`scripts/prd_open.sh`), which can happen on any branch or worktree —
> not only one that lands promptly. The registry validator requires
> every allocated number up to `latest_complete` to have a doc on
> `main`; a Stage-0 PRD drafted and left parked on an unmerged branch
> therefore blocks the closeout of any later-numbered PRD, even one
> unrelated to it in content or authorship (PRD-254's closeout was
> blocked by exactly this gap — PRD-253, allocated 2026-07-10, parked
> unmerged until PR #136 landed it — 2026-07-11). **Rule: before closing
> out PRD-N, every allocated number < N MUST already have a landed doc
> on `main`** (any status suffices — IN PROGRESS/PROPOSED is fine,
> COMPLETE is not required). If a lower number is still parked, land its
> Stage-0 scaffold first, unmodified, with its original authorship
> preserved, as its own PR — a landing-order constraint on the
> DOCUMENT, not an implementation-order constraint on the WORK; the
> parked PRD's actual build stays wherever it was queued.

FAIL: the subsection is absent, is not placed under Registry

NOTES — CHECK-ID POLICY (append-only, retire-never-reuse)
Verification check IDs (V1, V2, ... in `prd-review-claude/SKILL.md`'s
Phase 2 table) are append-only once any committed review artifact could
plausibly cite one by number. When a check is removed, retire its
number permanently (mark it RETIRED in the table, state why) and give
its replacement, if any, the next unused number. Do not reuse a retired
number for a check of different or opposite meaning — a past review
artifact's citation is immutable evidence of what it asserted at the
time, and reuse silently reverses that assertion out from under it.
(Origin: Leg-2 second-model disposition on this PRD, 2026-07-11, found
the original R3 draft reused V9's number and collided with 15 existing
citations across `docs/prd_history/*.review.claude.md`.)

1. **FILES has no schema, and R11's intersection test is unverified.**
   R11 (Lane Downgrade Prohibition) turns on whether a PRD's FILES list
   "intersects" a CLASS matrix HIGH-RISK FILES entry. But FILES has no
   defined format anywhere in the repo — file paths vs. directory
   prefixes, repo-relative vs. absolute, trailing slashes, globs,
   whether the git status letter (M/A/D) is part of the entry. Every
   consumer answered those questions independently. This is the same
   X1/X2 pattern Fable named as the audit's most interesting finding,
   and it is the third instance: `protect_files.sh`'s H1 (regex FILES
   parser that captured only the first entry) and H2 (three-way string
   comparison so loose that a `.env` entry authorized any path ending
   in `.env`) were both this same root cause. Both were amputated in
   PRD-254 without the root being named. Evidence R11 may be
   decorative: PRD-255 and #141 were both classified HIGH-RISK by
   CLAUDE.md's named-guardrail clause — human judgment reading AROUND
   the matrix — not by R11's mechanical intersection firing. Fable
   found the matrix carried an impossible MICRO example practice had
   ignored for weeks. Nothing anywhere demonstrates R11 has ever
   actually matched anything. Note: #141 just added two paths to the
   GOVERNANCE HIGH-RISK FILES row; if they don't match the format the
   intersection check expects, they are two more decorative entries.
   Three deliverables when it's built: (a) define and document the
   FILES entry format; (b) one parser, one path-comparison function,
   used by every consumer; (c) a test proving R11 FIRES on a known
   HIGH-RISK path and does NOT on a known-safe one — this is the
   important one, a guard you believe in is more dangerous than one you
   know is broken. FIRST priority: not tooling hygiene, a guard that may
   be silently not-guarding, in the layer that decides whether
   HIGH-RISK review legs happen at all. Found while ruling on #141,
