Status: COMPLETE

WHY NOW
PRD-177 exposed the gap: a finished visual-layer change cannot be verified against
fresh data anywhere except production — the only fresh-data render path is the hourly
CI cycle (render to ui/, commit to main, Pages deploy), and pages.yml additionally
deploys on every push to main, so the first content-correct render of any renderer
change happens on the live site.

MAX EXPECTED DELTA
2 new non-test files (~100 lines of YAML + shell, zero production Python LOC) plus
~40 test LOC appended to one existing test file. Exceeding stops implementation and
amends this PRD.

GOAL
Any visual-layer change can be verified against genuinely fresh data, with the real
publish gates exercised, before it reaches main — via an on-demand CI preview
workflow that can never send, commit, or deploy, plus a local preview script that can
never write under ui/.

SCOPE
- New .github/workflows/dashboard_preview.yml ("Dashboard Preview"):
  workflow_dispatch-only, dispatchable on any ref. Steps: checkout; Python 3.11;
  pip install -e .; record start timestamp; python -m cuttingboard.alert_runner
  --force-slot with NO Telegram secrets in env (send_telegram degrades to an audited
  status=skipped / reason=not_configured row and never raises — verified at
  cuttingboard/output.py:522); hard-fail if logs/latest_hourly_payload.json is
  missing or its mtime predates job start (the hourly workflow's freshcheck pattern,
  inverted from skip-publish to fail-job); render with the exact hourly command
  (--payload logs/latest_hourly_payload.json --run logs/latest_hourly_run.json
  --market-map-path logs/latest_hourly_market_map.json --output ui/dashboard.html)
  so the PRD-118 coherence and PRD-119 freshness gates execute on the real publish
  path against fresh data, in the ephemeral runner workspace; cp ui/dashboard.html
  ui/index.html; cmp byte-identity; log checked-in size, rendered size, and percent
  delta (the evidence the dashboard-publish-refresh size_delta_override requires);
  upload ui/ as artifact dashboard-preview-<run_number>, retention 7 days.
- New scripts/preview_dashboard.sh: local fresh preview. Fetch via
  env -u TELEGRAM_BOT_TOKEN -u TELEGRAM_CHAT_ID python3 -m cuttingboard.alert_runner
  --force-slot (zero outbound sends), then render the hourly artifact trio to
  reports/output/preview_dashboard.html — a non-ui path, exempt from publish gates
  by PRD-119 R12 design (no gate weakened; nothing under ui/ is written).
  SKIP_FETCH=1 skips the fetch for pure layout iteration.
- Workflow-content guard tests appended to tests/test_ci_artifact_hygiene.py
  pinning the preview workflow's safety anchors (R8).

OUT OF SCOPE
- Any change to cuttingboard/delivery/dashboard_renderer.py, hourly_alert.yml,
  pages.yml, cuttingboard.yml, or any gate logic. Gates are exercised, never
  modified.
- Any preview deploy target (no preview Pages environment, no staging branch).
- Any change to the dashboard-publish-refresh skill or its size-delta guard
  semantics.
- scripts/check_readiness.py and the regime_history aggregation step (see NOTES —
  hourly runs the aggregator after render, so omitting it preserves render parity).
- POLYGON_API_KEY plumbing (dead per the 2026-05-22 cleanup; not mirrored).

REQUIREMENTS

R1 — Dispatch-only trigger
dashboard_preview.yml's `on:` block contains workflow_dispatch and nothing else.
FAIL: `schedule:`, `push:`, or `workflow_run:` appears in dashboard_preview.yml.

R2 — Zero send capability
No Telegram credential is available to any step of the preview workflow.
FAIL: the literal `TELEGRAM` appears anywhere in dashboard_preview.yml.

R3 — Zero publish capability
The workflow declares `permissions: contents: read`; it contains no commit, push, or
deploy step; pages.yml and hourly_alert.yml are byte-unchanged by this PRD.
FAIL: `git commit`, `git push`, or `deploy-pages` appears in dashboard_preview.yml;
or its permissions block grants any write; or `git diff` shows pages.yml or
hourly_alert.yml modified.

R4 — Publish-gate exercise with fresh data
The render step targets `--output ui/dashboard.html` in the runner workspace using
exactly the hourly artifact trio (latest_hourly_payload / latest_hourly_run /
latest_hourly_market_map), so validate_coherent_publish
(cuttingboard/delivery/dashboard_renderer.py:394) runs its PRD-118 and PRD-119
checks on the publish path (_output_under_ui, dashboard_renderer.py:353, is True).
FAIL: the render step's --output is not ui/dashboard.html, or any of the three
artifact arguments differs from the corresponding argument in hourly_alert.yml's
render step.

R5 — Stale-fetch hard fail
The job records a start timestamp before invoking alert_runner and exits nonzero if
logs/latest_hourly_payload.json is missing or its mtime predates that timestamp; the
render step runs only after this check passes.
FAIL: the render step executes in a run where the payload's mtime predates job
start, or a missing payload yields a green job.

R6 — Artifact evidence
On a green run the workflow asserts `cmp ui/dashboard.html ui/index.html`, logs three
labeled lines (checked-in byte size, rendered byte size, percent delta), and uploads
ui/ as artifact dashboard-preview-<run_number> with 7-day retention.
FAIL: a green run lacks the uploaded artifact, the cmp step, or any of the three
labeled size lines in the job log.

R7 — Local script never touches ui/
scripts/preview_dashboard.sh writes its render output only to
reports/output/preview_dashboard.html; its alert_runner invocation strips both
Telegram variables via `env -u TELEGRAM_BOT_TOKEN -u TELEGRAM_CHAT_ID`; SKIP_FETCH=1
bypasses the fetch step only.
FAIL: any write target under ui/ appears in the script, or the alert_runner line
lacks either `env -u` flag.

R8 — Workflow-content guard tests
tests/test_ci_artifact_hygiene.py gains assertions that read dashboard_preview.yml
and pin: (a) workflow_dispatch-only trigger (R1), (b) zero TELEGRAM occurrences
(R2), (c) absence of git commit / git push / deploy-pages literals (R3), (d) render
anchor `--output ui/dashboard.html` preceded by the three hourly artifact arguments
(R4), (e) a permissions block granting exactly `contents: read` and no other
permission key (R3). Each assertion fails — not smoke-passes — when its anchor is
violated.
FAIL: any of the five anchors can be violated in dashboard_preview.yml while
`python3 -m pytest tests/test_ci_artifact_hygiene.py -q` stays green.

DATA FLOW
CI: dispatch(ref) -> alert_runner --force-slot (no Telegram env; sends audited as
skipped/not_configured) -> fresh logs/latest_hourly_* in runner workspace -> mtime
freshness check (fail-hard) -> dashboard_renderer --output ui/dashboard.html
(PRD-118 + PRD-119 gates execute) -> cp to ui/index.html -> cmp -> size-delta log ->
upload-artifact. Nothing committed, pushed, or deployed.
Local: preview_dashboard.sh -> env-stripped alert_runner --force-slot -> render
hourly trio to reports/output/preview_dashboard.html (non-ui; PRD-119 R12 exemption)
-> open in browser.

FAIL CONDITIONS
- schedule/push/workflow_run trigger present in dashboard_preview.yml.
- TELEGRAM literal present in dashboard_preview.yml.
- Commit/push/deploy step or write permission in dashboard_preview.yml; pages.yml or
  hourly_alert.yml modified.
- Render step output not ui/dashboard.html, or artifact trio diverges from hourly.
- Render executes after a stale or missing payload; or stale fetch yields green job.
- Green run missing artifact, cmp step, or size-delta log lines.
- preview_dashboard.sh writes under ui/ or invokes alert_runner without both env -u
  flags.
- Any R8 anchor violated while the hygiene test stays green.

VALIDATION
Manual:
- ruff clean (test file only); python3 -m pytest tests/test_ci_artifact_hygiene.py -q
  passes; full suite green.
- Local: run scripts/preview_dashboard.sh; expect zero Telegram messages received,
  logs/audit.jsonl gains skipped/not_configured rows only,
  reports/output/preview_dashboard.html exists with payload timestamp within the
  last few minutes, ui/ untouched (git status shows no new ui/ modification beyond
  pre-existing state).
- Local: SKIP_FETCH=1 scripts/preview_dashboard.sh re-renders without invoking
  alert_runner (no new fetch; payload mtime unchanged).
- CI: push branch; dispatch "Dashboard Preview" on the branch; expect green run,
  downloadable dashboard-preview-<N> artifact whose dashboard.html shows fresh data,
  three size lines in the log, no new commit on the branch, no Pages deployment
  triggered (pages.yml run list unchanged).
- CI negative: confirm in the run log that the freshness check step ran before the
  render step.

NOTES
- Numbering: PROJECT_STATE.md earmarked PRD-178 as the startable slot for the
  runtime.py-split PRD-B track; this PRD takes 178, shifting that track to PRD-179+.
  Pure slot allocation, no dependency between the tracks.
- Tier-1 caveat, stated plainly: the local fetch mutates tracked logs/* (payload,
  run, contract, market_map, audit.jsonl). This is already routine local dirt; the
  hourly CI force-add overwrites these on its next commit. The script must NOT be
  followed by committing logs/* (CLAUDE.md artifact discipline unchanged).
- Render parity: the hourly workflow runs the regime_history aggregation AFTER the
  render step, so production renders consume the checked-in regime_history.jsonl —
  the preview omitting the aggregator therefore matches production exactly.
- scripts/check_readiness.py is not run in the preview job; if it later proves
  useful as preview evidence, add it in a follow-up amendment.
- tests/test_dashboard_renderer.py:105
  (test_ci_workflows_publish_dashboard_with_same_render_copy_contract) enumerates
  PUBLISHING workflows (cuttingboard.yml, hourly_alert.yml). The preview workflow
  does not publish, so it is intentionally NOT added to that list; its render-copy
  parity is pinned by R8(d) instead. Downstream-consumer audit found no other test
  or module enumerating .github/workflows/ contents (no closed allowlist).
- Realizability: send_telegram's not-configured branch (output.py:522, skip path)
  is exercised on every preview run; alert_runner --force-slot always writes fresh
  hourly artifacts and exits 0 (alert_runner.py:42-117), so the freshness check is
  the load-bearing failure detector — hence R5's fail-hard inversion of the hourly
  freshcheck's skip-publish semantics.
- The workflow definition must exist on the dispatched ref; after this PRD merges to
  main, every subsequent branch carries it.
- The verification loop this creates for any future visual PRD: implement on a
  branch -> push branch -> dispatch Dashboard Preview on the branch -> inspect
  artifact (fresh-data render, gate pass, byte delta) -> merge to main. Production
  render is no longer the first fresh-data render.
