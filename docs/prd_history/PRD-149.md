# PRD-149 — PT-Anchored Hourly Alert Window (6:00 AM – 1:00 PM PT)

STATUS: COMPLETE
LANE: STANDARD
COMMIT: c562259
CLASS: runtime / notifications
DEPENDS ON: PRD-141 (canonical slot + cross-run idempotency), PRD-144 (premarket
redundancy — superseded by `routine_pt_slot` allowed-slot resolver)

## GOAL

Restrict routine Telegram hourly alerts to a fixed, PT-anchored slot set
spanning the NY trading session (06:00 AM – 1:00 PM PT), and title every
alert by its **canonical PT slot** rather than GitHub Actions wall-clock
run time. This eliminates the observed failure mode where a 21:30 UTC
cron fire delayed by GitHub Actions produces a "STAY FLAT — 3:00 PM PT"
alert that arrives nearly two hours after its useful decision window.

Allowed routine PT slots (Mon–Fri, America/Vancouver):

```
06:00, 06:30, 07:00, 08:00, 09:00, 10:00, 11:00, 12:00, 13:00
```

Anything outside this set MUST be suppressed for routine (non-force-slot)
runs.

## SCOPE

- Add an allowed-slot resolver `routine_pt_slot(now_utc, max_lag_minutes=25)`
  in `cuttingboard/notifications/hourly_slot.py`, plus an
  `ALLOWED_PT_SLOTS` constant.
- Gate the routine path in `cuttingboard.alert_runner.main` on the
  resolver: `None` → audit `suppressed/outside_routine_window`, exit 0,
  do not touch the dedup file.
- Thread the resolver's slot as `slot_utc` into `_execute_notify_run`,
  so `format_hourly_notification` titles using the canonical PT slot
  (existing wiring at `cuttingboard/runtime.py:545–550` requires no
  change).
- Replace `canonical_slot_utc(now_utc)` with `routine_pt_slot(now_utc)`
  on the routine path so 06:00 PT and 06:30 PT are *distinct* dedup
  slots.
- Preserve `--force-slot` (workflow_dispatch / `CUTTINGBOARD_FORCE_SLOT=1`)
  semantics: bypass the resolver gate, keep `canonical_slot_utc(now_utc)`
  as the slot for both dedup and title.
- Drop `is_premarket_slot` usage from `alert_runner.main`. The helper
  stays in the module for compatibility.
- Trim `.github/workflows/hourly_alert.yml` intraday cron to a
  DST-resilient superset of the allowed PT slots, with a single 06:05
  PT redundancy backup for the 06:00 PT slot only.

## OUT OF SCOPE

- Decision logic, qualification gates, regime engine, sizing, contract.
- Dashboard renderer and payload schema.
- Notification body formatting; only the title's PT clock value
  changes, and only because `slot_utc` is now PT-anchored.
- Holiday calendar (NYSE half-days). 1:00 PM PT cutoff is later than
  10:00 AM PT early close, so no special handling is required.
- Backfill / catch-up of missed slots. A missed slot is a missed slot.

## REQUIREMENTS

R1. `cuttingboard/notifications/hourly_slot.py` defines:

    ```python
    ALLOWED_PT_SLOTS: tuple[tuple[int, int], ...] = (
        (6, 0), (6, 30), (7, 0), (8, 0), (9, 0),
        (10, 0), (11, 0), (12, 0), (13, 0),
    )
    ```

    All slots are interpreted in `America/Vancouver` (DST-correct,
    matches the existing `_PT_TZ` in this module).

    FAIL: `ALLOWED_PT_SLOTS` is missing, contains different entries, or
    is interpreted in any timezone other than `America/Vancouver`.

R2. `routine_pt_slot(now_utc: datetime, max_lag_minutes: int = 25) -> Optional[datetime]`:

    - Requires tz-aware `now_utc`. Naive datetimes raise `ValueError`.
    - Converts to `America/Vancouver`.
    - Returns the **largest** allowed PT slot `slot_pt` such that
      `slot_pt ≤ now_pt` and `(now_pt − slot_pt) ≤ max_lag_minutes`.
    - The returned datetime is tz-aware UTC.
    - Returns `None` if no allowed slot satisfies the constraint.

    FAIL: Naive datetime does not raise `ValueError`; the returned
    slot is greater than `now_pt`; the slot's lag exceeds
    `max_lag_minutes`; the returned datetime is not tz-aware UTC; the
    PT (hour, minute) pair is not in `ALLOWED_PT_SLOTS`.

R3. `alert_runner.main` routine path (`force_slot=False`):

    - Compute `slot_utc = routine_pt_slot(now_utc)`.
    - If `slot_utc is None`: call

      ```python
      now_pt = now_utc.astimezone(_PT_TZ)
      state_key = f"outside:{now_pt.strftime('%Y-%m-%dT%H:%M%z')}"
      write_notification_audit(
          transport="telegram",
          status="suppressed",
          alert_title="hourly",
          attempted=False,
          success=False,
          reason="outside_routine_window",
          state_key=state_key,
      )
      ```

      `state_key` MUST be a deterministic PT-window grouping key
      (here: `outside:` prefix + PT wall-clock minute) so two
      outside-window suppressions within the same PT minute share a
      key. Raw `now_utc.isoformat()` is NOT permitted because it
      defeats audit grouping and diverges from the PRD-141
      `slot_utc.isoformat()` convention. Then return 0.
      `logs/last_hourly_slot.json` MUST NOT be written or rewritten
      in this branch.
    - Otherwise, run the existing PRD-141 same-slot dedup against
      `slot_utc`, then call
      `_execute_notify_run(mode=MODE_LIVE, run_date=now_utc.date(),
      notify_mode=NOTIFY_HOURLY, slot_utc=slot_utc)`.

    FAIL: A 14:00 PT routine run produces a Telegram send; a 13:30 PT
    routine run produces a Telegram send; `last_hourly_slot.json` is
    modified during after-window suppression; audit reason missing or
    not equal to `"outside_routine_window"`; the audit `state_key`
    equals `now_utc.isoformat()` (i.e. uses the raw UTC instant
    rather than the PT-window grouping key); two outside-window
    suppressions occurring inside the same PT minute produce
    different `state_key` values.

R4. `alert_runner.main` force-slot path (`force_slot=True` or
    `CUTTINGBOARD_FORCE_SLOT=1`):

    - Bypass the resolver gate entirely.
    - Compute `slot_utc = canonical_slot_utc(now_utc)` (existing
      behaviour from PRD-141).
    - Proceed to `_execute_notify_run` with that `slot_utc`.

    FAIL: A `--force-slot` run at 15:00 PT is suppressed; force-slot
    dedup behaviour diverges from the PRD-141 baseline.

R5. Title canonicalisation: because `cuttingboard/runtime.py:545–550`
    already uses the threaded `slot_utc` as `header_asof` for
    `_HOURLY_MODES`, the title automatically reflects the PT slot for
    both 06:00 and 06:30 routine fires. No formatter change is
    required.

    FAIL: A `now_utc` 17 minutes past the 13:00 PT slot produces a
    title other than `"STAY FLAT — 1:00 PM PT"` (assuming STAY_FLAT
    outcome).

R6. `alert_runner.main` MUST NOT call or import `is_premarket_slot`
    after this PRD. The helper remains importable from
    `cuttingboard.notifications.hourly_slot` for compatibility, with
    its module attribution preserved. Existing
    `tests/test_hourly_slot.py` cases for `is_premarket_slot`
    continue to pass unchanged.

    FAIL: `cuttingboard/alert_runner.py` still references
    `is_premarket_slot`; `from cuttingboard.notifications.hourly_slot
    import is_premarket_slot` raises `ImportError`;
    `is_premarket_slot.__module__ !=
    "cuttingboard.notifications.hourly_slot"`; any prior
    `tests/test_hourly_slot.py` `is_premarket_slot` test starts
    failing.

R7. `.github/workflows/hourly_alert.yml` intraday `schedule:` block is
    replaced with the following deduplicated, DST-resilient superset.
    Each unique UTC minute is listed **exactly once**. The 06:00 PT
    slot — and only the 06:00 PT slot — has a +5-minute redundancy
    backup in both PDT (`5 13`) and PST (`5 14`). The 06:30 PT, 07:00
    PT, 08:00 – 13:00 PT slots have **no redundancy backup**; their
    sole resilience is the runner-gate's idempotency.

    ```yaml
    schedule:
      - cron: '0 13 * * 1-5'    # 06:00 PDT (PST: 05:00 → gate-suppressed)
      - cron: '5 13 * * 1-5'    # 06:05 PDT — 06:00 redundancy
      - cron: '30 13 * * 1-5'   # 06:30 PDT (PST: 05:30 → gate-suppressed)
      - cron: '0 14 * * 1-5'    # 06:00 PST  /  07:00 PDT
      - cron: '5 14 * * 1-5'    # 06:05 PST — 06:00 redundancy
      - cron: '30 14 * * 1-5'   # 06:30 PST  /  07:30 PDT → gate-suppressed
      - cron: '0 15-21 * * 1-5' # 08:00–14:00 PDT (14:00 → gate-suppressed)
                                # 07:00–13:00 PST (all in-window)
    ```

    The runner gate at R3 (`routine_pt_slot`) remains authoritative.
    Some scheduled fires are intentional DST-resilience artifacts and
    will be suppressed with `reason="outside_routine_window"` — this
    is by design, not a bug:

    - PDT: `0 21 * * 1-5` fires at 14:00 PT (outside window) →
      suppressed.
    - PDT: `30 14 * * 1-5` fires at 07:30 PT (not in allowed set) →
      suppressed.
    - PST: `0 13 * * 1-5` fires at 05:00 PT (before earliest allowed
      slot) → suppressed.
    - PST: `30 13 * * 1-5` fires at 05:30 PT (before earliest allowed
      slot) → suppressed.

    FAIL: any cron line strictly after `0 21 * * 1-5` is present
    (would correspond to ≥ 14:30 PT PDT or ≥ 14:00 PT PST); any
    `*/30` range remains; any of the legacy PRD-144 premarket warm-up
    lines `50 12`, `55 12`, or `50 13` remain; the realised
    `.github/workflows/hourly_alert.yml` `schedule:` block contains
    any UTC cron line not in the seven lines above (`0 13`, `5 13`,
    `30 13`, `0 14`, `5 14`, `30 14`, `0 15-21`), or contains any
    duplicate of those lines. The `5 13` / `5 14` lines are RETAINED
    as the 06:05 PT redundancy backup and are NOT a fail.

R8. Dedup file behaviour on after-window suppression:
    `logs/last_hourly_slot.json` MUST NOT be advanced or rewritten in
    the `outside_routine_window` branch. A subsequent **earlier-in-day**
    in-window slot must not exist (chronologically impossible), and a
    next-day in-window slot must still be deliverable.

    FAIL: After a suppressed 14:00 PT run, `last_hourly_slot.json`
    mtime or contents differ from their pre-run state.

## DATA FLOW

```
GitHub Actions cron (UTC; DST-resilient superset per R7)
  → workflow runs `python -m cuttingboard.alert_runner [--force-slot]`
  → alert_runner.main computes now_utc = datetime.now(timezone.utc)
      ├── force_slot=True  → slot_utc = canonical_slot_utc(now_utc)   (R4)
      └── force_slot=False → slot_utc = routine_pt_slot(now_utc)       (R3)
                              ├── None → write_notification_audit(
                              │          status="suppressed",
                              │          reason="outside_routine_window")
                              │           → exit 0, dedup file untouched
                              └── dt   → PRD-141 dedup against dt
                                          → _execute_notify_run(slot_utc=dt)
  → runtime.py:545–550 uses slot_utc as header_asof
  → format_hourly_notification renders title from slot_utc
  → Telegram send + last_hourly_slot.json update (PRD-141 path)
```

  in `ALLOWED_PT_SLOTS`.
- F2. Routine alert at 13:30 PT or later is sent.
- F3. `--force-slot` after 13:00 PT is suppressed.
- F4. Title for a delayed 13:00 PT slot run shows anything other than
  `"1:00 PM PT"`.
- F5. `last_hourly_slot.json` is rewritten on after-window suppression.
- F6. Dedup collides 06:00 and 06:30 routine fires.

- F8. Workflow cron contains an intraday line outside the R7 schedule,
