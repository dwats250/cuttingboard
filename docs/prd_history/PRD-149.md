GOAL
GitHub Actions' scheduled-cron event delivery is degraded enough on this repo to drop entire days of firings (2026-05-19: zero scheduled firings of either workflow over ~18+ hours, with manual `workflow_dispatch` working normally). The PRD-141 slot canonicalization and PRD-144 backup-cron redundancy both presume some scheduled firing reaches the runner. Add a Cloudflare Workers cron sidecar that fires `workflow_dispatch` against `hourly_alert.yml` on the same canonical UTC times the GH schedule targets. The existing PRD-141 slot dedup gate guarantees one-alert-per-PT-hour regardless of whether GH cron, the Worker, or both fire — provided the workflow no longer unconditionally passes `--force-slot` for `workflow_dispatch`. Include the minimum workflow YAML edit needed to differentiate operator manual dispatch (keep `--force-slot`) from external-scheduler dispatch (drop `--force-slot`, route through dedup).

SCOPE
- New Cloudflare Worker under `tools/external_trigger/` that POSTs to `POST /repos/dwats250/cuttingboard/actions/workflows/hourly_alert.yml/dispatches` with `{ref: "main", inputs: {scheduler: "external_cron"}}` on each scheduled tick.
- Worker cron schedule is a coarse 15-minute heartbeat across the weekday UTC window covering premarket through close (`*/15 12-21 * * 1-5`). Exact-slot mirroring of `hourly_alert.yml` is not required because the PRD-141 slot dedup gate collapses overlapping firings; Cloudflare's free plan caps a Worker at 5 cron triggers, which forces the coarse-heartbeat shape. Amended 2026-05-19 PATCH (see end of file).
- `.github/workflows/hourly_alert.yml`: add `workflow_dispatch.inputs.scheduler` (enum: `operator | external_cron`, default `operator`) and update the `Run hourly alert` step to pass `--force-slot` only when `github.event_name == 'workflow_dispatch' AND inputs.scheduler != 'external_cron'`.
- README under `tools/external_trigger/` documenting deploy (`wrangler deploy`) and secret setup (`wrangler secret put GITHUB_TOKEN`, fine-grained PAT scoped to this repo with Actions Read and write).

- Monitoring/alerting on Worker failures.
- Any new Python module under `cuttingboard/`.

PATCH 2026-05-19 — Cloudflare cron-trigger cap

Initial `wrangler deploy` failed at `/workers/scripts/cuttingboard-external-trigger/schedules` with `You have exceeded the limit of 5 cron triggers.` — Cloudflare's free plan caps a Worker at 5 cron triggers and the original schedule declared six. `wrangler tail` connected but the schedules never installed.

Fix: reduced `tools/external_trigger/wrangler.toml` `[triggers].crons` to a single coarse heartbeat `*/15 12-21 * * 1-5` (Mon–Fri, every 15 minutes across the weekday UTC window covering premarket through close). The PRD-141 slot dedup gate already collapses overlapping firings to one alert per PT hour, so exact-slot mirroring of `hourly_alert.yml` is not needed. SCOPE line 17 and R2/R2-FAIL above were amended in place to reflect the coarse heartbeat. No change to `worker.js`, `hourly_alert.yml`, runtime, notifications, dashboard, or any generated artifact. Deploy should now fit under the Cloudflare limit; operator one-time `wrangler deploy` step from PROJECT_STATE remains the next operational step.
