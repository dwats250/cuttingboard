## Findings

### 1. REQUIRED CHANGE — Renderer precedence conflict: `MARKET_CLOSED` is unreachable as drafted

**Severity:** blocking. The PRD as written cannot satisfy its own R5 success
criterion under current code.

**Root cause.** `cuttingboard/delivery/dashboard_renderer.py:683` reads:

```python
if inactive_session:
    return "INACTIVE_SESSION"
```

This branch sits **before** the freshness check (line 687) and before the
existing `usable_count == 0 → FALLBACK` return (line 693). R5 of PRD-123
preserves "existing returns in their current precedence" and slots the new
`MARKET_CLOSED` / `AWAITING_DATA` branch at the position where `FALLBACK`
used to live — i.e. **after** the `INACTIVE_SESSION` early-return.

Consequence: whenever `inactive_session == True`, the function returns
`INACTIVE_SESSION` at line 684 and the new `MARKET_CLOSED` branch never
fires. R5's success criterion `MARKET_CLOSED when … inactive_session is
True` is unreachable.

**Required change.** R5 must explicitly redesign the precedence to allow
`MARKET_CLOSED` to fire under `inactive_session`. Two acceptable shapes:

- **Option A (recommended):** Move the `usable_count == 0` branch **above**
  the `INACTIVE_SESSION` branch. Semantics: "no usable rows under inactive
  session" specializes to `MARKET_CLOSED`; "inactive session with usable
  rows" (cached intraday after-hours, e.g.) still returns
  `INACTIVE_SESSION`. This keeps `INACTIVE_SESSION` meaningful and gives
  `MARKET_CLOSED` first refusal when there's truly no data.

- **Option B:** Remove the `INACTIVE_SESSION` early-return from this
  function entirely and make `MARKET_CLOSED` the trend-structure-specific
  inactive-session label. Cleaner semantically but a wider behavior change
  — any existing test asserting `SOURCE: INACTIVE_SESSION` on the trend
  panel would need updating, and the umbrella INACTIVE_SESSION semantic
  for other panels is preserved (PRD-117 used it in system-state and
  market-map, not trend-structure-as-primary).

Either option requires R5 text to be rewritten with the chosen new
precedence order spelled out explicitly. Recommend Option A — minimal
existing-behavior impact, preserves `INACTIVE_SESSION` as a fallback for
the "cached data during inactive window" edge case.

### 2. REQUIRED CHANGE — Pre-edit audit miss: removing `FALLBACK` breaks an existing PRD-120 test

**Severity:** blocking before first edit. Will fire on the same scope-lock
pattern that surfaced in PRD-122 (mid-implementation discovery of test
files asserting against the prior shape).

**Root cause.** `tests/test_dashboard_renderer.py:2243–2268`
(`test_prd120_trend_structure_source_fallback_zero_usable`, R14-7) asserts:

```python
assert "SOURCE: FALLBACK" in section
assert "TREND SYMBOLS: 0/6" in section
```

The test builds a snapshot where every symbol record is `{"symbol": sym}`
(no required fields). Under PRD-123 R5, this case will return either
`MARKET_CLOSED` (if inactive) or `AWAITING_DATA` (if active), never
`FALLBACK`. The string `"SOURCE: FALLBACK"` will no longer appear → test
fails.

The file is already in PRD-123 FILES (`tests/test_dashboard_renderer.py`),
so no scope-lock amendment is needed. But the PRD currently does not
acknowledge that PRD-120's R14-7 test must be **updated**, not just
**augmented**, by PRD-123.

**Required change.** PRD R8 must explicitly state that the R14-7 test at
line 2243 is rewritten to assert the new `MARKET_CLOSED` or
`AWAITING_DATA` label (under whichever session-default the existing
`_prd120_coherent_render` produces). Add an explicit FAIL line: "The
string `SOURCE: FALLBACK` must not appear in any trend-structure source
diagnostic after this PRD merges." This makes the FALLBACK removal an
auditable invariant rather than an implicit consequence.

Also recommend the PRD's `# NOTES` or a `# MIGRATION` section call out:
- `FALLBACK` removal from `_trend_structure_source_health` return surface
- The R14-7 test update is in-scope
- Any other test grepping for `SOURCE: FALLBACK` in trend-structure
  context must be enumerated. Pre-edit audit recommended:
  `grep -n "SOURCE: FALLBACK\\|FALLBACK" tests/ cuttingboard/ |
   grep -i trend`.

### 3. ACCEPT — Runtime placement is sound

`runtime.py:332`). `_execute_notify_run` (line 437) does its own pipeline
assembly and is reached via `alert_runner.py`, not via `_run_pipeline`.
The `MODE_LIVE` gate inside `_refresh_trend_structure_sidecar` plus the
gate inside `_write_trend_structure_snapshot`'s caller correctly excludes
fixture/sunday writes. No double-write risk under current architecture.

One minor reinforcement: R2 should specify that the call site within
`_run_pipeline` be placed **inside the MODE_LIVE-only branch** rather
than at a top-level point that relies solely on the helper's gate.
Defense in depth — both the call site and the helper check `mode`. This
is mentioned in the FAIL line ("call placed before `ohlcv` is populated")
but should be added as an explicit positive criterion.

## Implementation hazards

- **Sunday/Fixture branches in `_run_pipeline`.** The function has
  separate code paths for MODE_LIVE / MODE_SUNDAY / MODE_FIXTURE (see
  `runtime.py:701`, `757`, `771`, `794`, `825`). The PRD says "place the
  call in the MODE_LIVE branch" but the actual MODE_LIVE branch boundary
  is non-trivial in this function. Implementer should grep for the
  existing `ohlcv = {…}` assignment at line ~794 and place the helper
  call in the same scope.

- **`run_at_utc` naming.** PRD R2 uses `run_at_utc`. Inside
  `_run_pipeline` the local variable name appears to be `run_at_utc`
  (line 692: `run_at_utc = _deterministic_run_at(...) if mode == MODE_FIXTURE
  else datetime.now(timezone.utc)`). Confirm at implementation time —
  if the variable is renamed in `_run_pipeline`, the helper call must
  use the local name.

- **`pd` import in helper signature.** The helper type-hints
  `dict[str, pd.DataFrame]`. `cuttingboard/runtime.py` already imports
  `pandas as pd` (verified by `grep -n "import pandas" runtime.py`
  before implementation), so no new import is required, but the
  implementer should confirm before adding the helper.

- **Mock target for runtime test.** When mocking
  `_write_trend_structure_snapshot` from
  `tests/test_runtime_trend_structure_refresh.py`, ensure the mock
  patches the symbol at its definition site (`cuttingboard.runtime.
  _write_trend_structure_snapshot`) so that the call from inside
  `_refresh_trend_structure_sidecar` resolves to the mock. Patching at
  `cuttingboard.trend_structure.build_trend_structure_snapshot` would
  miss the runtime-level wrapper.

- **Live verification step 5 is non-deterministic.** The live render
  branch outcome (`OK` vs `MARKET_CLOSED` / `AWAITING_DATA`) depends on
  whether US markets are open at the time of the validation run. The
  PRD should accept either outcome as a pass for that step, and the
  implementer should document which case was observed in the
  validation report.
