PRD-113 Codex Review

1. [schema ambiguity] The seven-class taxonomy is close but not minimal because `PATCH` is a lifecycle/status modifier, not an architectural class. `GOVERNANCE` is not redundant with `PATCH` because process changes can be net-new rather than corrective, and `INFRA` is not redundant with `CONSUMER` because hooks, CI, deployment, and artifact-push plumbing are not presentation consumers; recommend six base classes (`GOVERNANCE | SIDECAR | CONSUMER | EXECUTION | CONTRACT | INFRA`) plus `PATCH` as an overlay with root-cause metadata.

2. [scope creep] `CHANGE SURFACE` is distinct from `FILES`: `FILES` says which paths may change, while `CHANGE SURFACE` says what depth of mutation is allowed inside those paths. It is justified for high-risk files and broad docs/process edits, but making it universal would duplicate ordinary `SCOPE`/`OUT OF SCOPE`; keep it mandatory only for T0/T1 and optional for T2/T3 unless a file has mixed ownership.

3. [factual drift] `STABILITY TIER` is useful but premature as a manually assigned mandatory field for every PRD because only 112 prior PRDs is enough to reveal drift patterns, not enough to prove stable tier boundaries. Keep the tier model in `PRD_PROCESS.md`, but infer or reviewer-confirm it from class and FILES during review until enough post-PRD-113 evidence exists to make manual tier declarations binding.

4. [non-determinism] `MAX EXPECTED DELTA` should be binding, not advisory, because advisory ceilings will not stop scope growth when implementation pressure appears. Binding should mean "exceeding this requires PRD amendment/re-review," not "the implementation must contort to fit a bad estimate"; this makes drift visible without blocking legitimate discoveries.

5. [derived semantics] Of the six mandatory header fields, `REQUIRED REVIEWERS` should remain informal or derived because it should fall out of class, stability tier, and the CLAUDE.md cross-review gate. `ARCHITECTURAL OWNER` can also be derived for many classes from FILES and canonical owner docs, but `CLASS`, `WHY NOW`, and `MAX EXPECTED DELTA` are valuable author assertions; `STABILITY TIER` should be reviewer-confirmed rather than mandatory author input.

6. [scope creep] The minimal subset that captures at least 80% of drift reduction is R1 with `PATCH` converted to an overlay, R4 as conditional for T0/T1 or mixed-owner files, R5 forbidden-pattern catalog, R6 reviewer matrix constrained by CLAUDE.md, R7 review taxonomy, and R8 governance-only guarantee. R2 should be reduced to `CLASS`, `WHY NOW`, and `MAX EXPECTED DELTA`; R3 should define defaults in process but not require repeated manual fields everywhere.

7. [ownership violation] The highest reviewer-fatigue risk is mandatory repetition of `STABILITY TIER`, `ARCHITECTURAL OWNER`, `REQUIRED REVIEWERS`, and `CHANGE SURFACE` on low-risk T2/T3 docs, dashboard, and process PRDs. Make `CHANGE SURFACE` mandatory for T0/T1 and mixed-owner files only, make reviewer fields derived, and make owner explicit only when FILES cross architectural ownership or touch contract/runtime/sidecar boundaries.

8. [hidden coupling] Required reviewers, validation depth, forbidden path defaults, max stability tier, and default leakage guards should be inferred from PRD classification rather than manually repeated in every PRD. Convention is superior here because the process can define one class matrix and reviewers can check deviations; manual repetition invites stale boilerplate where the header says one thing and the class matrix or CLAUDE.md gate says another.
