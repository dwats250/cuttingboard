Fast-follow proposal — continuation-path HOLD-confirmation gate defect

Status: PROPOSAL — not started, no Gate A yet. Not a registered PRD (this
is a `.proposal.md` sidecar, allowlisted from the registry-gap check per
PRD-248; it deliberately carries no `docs/PRD_REGISTRY.md` row and claims
no PRD number until Dustin gives it a Gate A direction ruling).

ORIGIN
Surfaced during PRD-256 Phase 1/R1's real-data validation (2026-07-13),
replaying the real, unmodified `_qualify_continuation_candidate` against
real cached OHLCV/VIX data to find a real accepted continuation candidate
to walk through. None existed to find — this proposal documents why.

THE FINDING
`cuttingboard/qualification.py::detect_continuation_breakout` (lines
623-639) computes `breakout_level` as the max High over the 5 bars *up
to and including* yesterday (`lookback = df.iloc[-(n+1):-1]`, `n=5`, on
the frame ending today). `_qualify_continuation_candidate`'s HOLD check
(lines 669-673) then asks whether *yesterday's own close*
(`hold_close = df.iloc[-(hold_candles+1)]`, `hold_candles=1`, i.e.
`df.iloc[-2]`) already exceeded that same window's max — but position -2
sits *inside* the lookback window (`-6` to `-2` inclusive) that computed
`breakout_level` in the first place. This is a self-referential
comparison: it is satisfiable only in the degenerate case where
yesterday closed exactly at its own high and that high was also the
window's maximum.

This appears inconsistent with the documented spec
(`docs/trade_qualification.md:323`, gate 4: "Close 1 completed bar ago
also held above the breakout level," constant `CONTINUATION_HOLD_CANDLES
= 1`) — the prose implies an *earlier* breakout level (established
before the hold candle) than the one the current bar's breakout check
just set, not a same-window self-comparison.

EVIDENCE — stated at the strength the data actually supports, not
overstated
The load-bearing evidence is a full historical replay of the real,
unmodified `_qualify_continuation_candidate`/`detect_continuation_breakout`
against real cached OHLCV for all 16 real tradable symbols (`SPY QQQ IWM`
+ `GLD SLV GDX PAAS USO XLE` + `NVDA TSLA AAPL META AMZN COIN MSTR`, 255
trading days each — 3,984 real symbol-days total), with the VIX-timing
gate held favorable to isolate the HOLD question specifically. Result:
breakout fired 680 times; HOLD passed **zero** times, across every
symbol, every day, with no exception.

Two corroborating (not confirming) sources were also checked and do not
contradict this:
- **95 captured production runs** (`logs/run_*.json`, 2026-04-12 to
  2026-07-10). Only 1 of the 95 carries a non-null `continuation_audit`
  block (`total_candidates=2, accepted=0`); the other 94 are **silent**
  on the question — they carry no continuation_audit data at all, which
  is absence of data, not a confirming zero. Do not read this as "95/95
  runs confirm"; that framing overstates what 94 of the 95 files
  actually show.
- **`logs/audit.jsonl`** (640 total records: 635 notification-transport,
  5 genuine pipeline records per `docs/audit_doctrine.md`'s two-family
  structure). None of the 5 pipeline records carries a non-null
  `continuation_audit` block either — also silent, not confirming.

**Does the conclusion hold on this honest evidence?** Yes. The
conclusion — the continuation path has functionally never qualified a
candidate, consistent with this gate defect — rests entirely on the
3,984-symbol-day replay, which is independent of and does not need the
log-count framing to stand. The overstated "95/95 confirm" phrasing was
never load-bearing for this conclusion; removing it changes nothing
about whether the finding is real.

WHY THIS IS NOT A DROP-IN FIX
Correcting the self-reference requires deciding what "held above the
breakout level" should actually compare — e.g. re-deriving
`breakout_level` from a window that excludes the hold candle(s), or
re-anchoring the comparison to a differently-shifted lookback. That is a
modeling decision about the intended confirmation semantics, not a
one-line arithmetic fix, and needs its own Gate A before any build (same
posture PRD-251's continuation-path proposal took toward the width
question, and PRD-256 R1's own text toward the FIX-approach choice).

OPEN DESIGN QUESTION (needs its own Gate A before any build)
What should "held above the breakout level" mean when
`CONTINUATION_HOLD_CANDLES = 1`? Not evaluated, not ranked:
1. Compute `breakout_level` from a window that ends *before* the hold
   candle(s) (e.g. shift the lookback back by `hold_candles`), so the
   hold check compares against a level the candidate could plausibly
   have already cleared.
2. Redefine "hold" as confirming today's breakout persisted through
   yesterday's own high/close relative to the *prior* (pre-breakout-day)
   structure, decoupling it from `detect_continuation_breakout`'s window
   entirely.
3. Drop the HOLD gate's dependency on `detect_continuation_breakout`'s
   window and give it an independently-derived reference level.

RELATIONSHIP TO PRD-256
Same function (`_qualify_continuation_candidate`), a different gate
(HOLD, gate 4 — a qualification precondition) than PRD-256's ATR-proxy
finding (sizing, the final gate before acceptance). Independent defects
in the same code path: this gate defect is why PRD-256's sizing
understatement has had zero real-world trades to date, but the sizing
bug would fire on literally the first candidate that ever clears HOLD,
including under today's decoupled $150 cap. Not folded into PRD-256's
R3 (FIX) scope — needs its own PRD number and its own Gate A.
