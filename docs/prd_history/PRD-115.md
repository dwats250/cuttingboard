# PRD-115 — Dashboard Artifact Lineage Visibility

STATUS: COMPLETE

GOAL:
Expose deterministic artifact lineage/coherence visibility in the dashboard so operators can immediately identify artifact consistency state without inspecting raw files or mutating runtime behavior.

SCOPE:
Add a deterministic dashboard diagnostics section that:

- Displays artifact lineage/coherence status using fixed classifications
- Displays participating artifact generation identifiers
- Displays artifact timestamp visibility already available from runtime artifacts
- Detects and surfaces generation mismatch conditions
- Detects and surfaces missing artifact conditions
- Reuses existing runtime/generated artifact data only
- Preserves current dashboard rendering architecture

The implementation must remain read-only and observational.

OUT OF SCOPE:

- Governance/process changes
- Notification changes
- Runtime scheduling changes
- Decision logic changes
- Qualification logic changes
- Policy layer changes
- Market map grading changes
- Trade evaluation changes
- Artifact self-healing/remediation
- Historical persistence
- Cross-run analytics
- UI redesign/theme/layout overhaul
- New runtime producers
- New scheduler dependencies
- Automatic repair behavior
- New artifact generation flows
- Any mutation of contract outcomes
- Any change to ALLOW_TRADE/BLOCK_TRADE/HALT semantics

CLASS:
SMALL

MAX EXPECTED DELTA:
120 production LOC

FILES:

- tests/test_dashboard_renderer.py

REQUIREMENTS:

R1.
Dashboard must expose a deterministic artifact lineage state using ONLY the following exact classifications:

- COHERENT
- MIXED
- STALE
- MISSING

No additional lineage states are permitted.

Precedence when multiple conditions apply:
MISSING > MIXED > STALE > COHERENT.

Exactly one classification renders per dashboard run.

R2.
COHERENT must render only when all participating artifacts:

- exist
- contain generation identifiers
- share the same generation identifier
- are not classified stale by existing renderer/runtime stale logic

R3.
MIXED must render when participating artifacts exist but generation identifiers differ.

Artifacts whose generation identifier is absent are treated as MISSING-id and do not trigger MIXED on their own. MIXED requires at least two present, non-equal generation identifiers.

R4.
MISSING must render when one or more required dashboard artifacts (payload, run, market_map) are unavailable.

R5.
STALE renders only when market_map_stale_for_run is True under the existing baseline timestamp comparison logic already present in dashboard_renderer.py. No new stale predicates, heuristics, or thresholds may be introduced.

R6.
Dashboard diagnostics must expose the participating artifact generation identifiers for:

- run artifact (logs/latest_run.json)
- payload artifact (logs/latest_payload.json)
- market_map artifact (existing market_map source)

If an artifact lacks a generation identifier, the diagnostics layer must render the literal placeholder text unavailable, matching the existing renderer convention.

R7.
The renderer must remain observational-only:

- no artifact mutation
- no repair behavior
- no runtime writes
- no new artifact creation
- no scheduler interaction

No writes to logs/ are permitted. Existing reports/ HTML renderer output behavior, paths, and shape remain unchanged.

R8.
No runtime decision fields or contract outcomes may change.

R9.
No files outside the FILES allowlist may be modified.

FAIL CONDITIONS:

- Any governance/process mutation
- Any new lineage classification outside the approved set
- Any mutation of runtime decisions/contracts
- Any new stale-threshold heuristics
- Any scheduler/runtime dependency introduction
- Any notification-layer modification
- Any artifact write/repair behavior
- Any modification outside FILES boundaries
- Any renderer crash caused by missing lineage fields
