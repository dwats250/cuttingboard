## GOAL

Add a structured, deterministic `explanation` block to each trade_candidate that documents blocking reasons, macro alignment, and required changes using fixed template strings.

---

## SCOPE

- Create `trade_explanation.py` to build per-candidate explanation dicts
- Inject after visibility layer in `runtime.py`
- Extend contract to include `explanation` per trade_candidate
- Render `explanation` in the PRD-065 expansion panel in `ui/app.js`

---

## OUT OF SCOPE

- No modification to trade decisions or execution policy
- No alerts
- No new signals
- No natural language generation â€” all output strings are fixed templates
- No duplication of `visibility_status`, `visibility_reason`, or `enable_conditions` (those live in PRD-064 fields)

## REQUIREMENTS

### R1 â€” Explanation structure

```python
{
    "block_reasons": list[str],       # empty if ACTIVE
    "macro_alignment": str | None,    # "ALIGNED" | "MISALIGNED" | None
    "required_changes": list[str],    # empty if ACTIVE; populated for NEAR_MISS
}
```

`explanation` does NOT duplicate `visibility_status`, `grade`, or `direction` â€” those are already present on the candidate.

FAIL: explanation contains fields already present on the candidate (status, grade, direction).

### R2 â€” block_reasons derivation

`block_reasons` is derived from `policy_reason` on the `TradeDecision`. Split only if multiple reasons are present (currently policy produces a single string â€” emit as a single-element list).

If `policy_allowed=True`, `block_reasons` must be empty.

FAIL: block_reasons non-empty on an ACTIVE decision.
FAIL: block_reasons empty on a BLOCKED or NEAR_MISS decision.

### R3 â€” macro_alignment derivation

`macro_alignment` is derived from the `overall_pressure` field now present on trade_candidates (added by PRD-063 wiring, exposed in contract) and the candidate's `direction`.

| overall_pressure | direction | macro_alignment |
|---|---|---|
| `RISK_ON` | LONG | `ALIGNED` |
| `RISK_ON` | SHORT | `MISALIGNED` |
| `RISK_OFF` | SHORT | `ALIGNED` |
| `RISK_OFF` | LONG | `MISALIGNED` |
| `MIXED` | any | `None` |
| `NEUTRAL` | any | `None` |
| `UNKNOWN` | any | `None` |

FAIL: macro_alignment is non-null when overall_pressure is MIXED, NEUTRAL, or UNKNOWN.
FAIL: any table row produces a different alignment than specified.

### R4 â€” required_changes derivation

`required_changes` is populated only for NEAR_MISS candidates (per PRD-064 classification). For ACTIVE and BLOCKED, it is an empty list.

Fixed template mappings:

| policy_reason | required_change string |
|---|---|
| `macro_pressure_conflict` | `"macro_pressure must align with trade direction"` |
| `orb_inside_range` | `"price must break ORB range"` |
| `cooldown` | `"cooldown period must expire"` |
| `low_confidence` | `"regime confidence must reach 0.60"` |
| other | `"blocking condition must resolve: {policy_reason}"` |

FAIL: required_changes populated for ACTIVE decision.
FAIL: required_changes populated for BLOCKED (non-NEAR_MISS) decision.
FAIL: any string deviates from the template for the given policy_reason.

### R5 â€” Function signature

```python
def build_explanation_map(
    decisions: list[TradeDecision],
    visibility_map: dict[str, dict],
    overall_pressure: str,
) -> dict[str, dict]:
    ...
```

Returns `{symbol: {block_reasons, macro_alignment, required_changes}}`.

`overall_pressure` is the same value computed in PRD-063 and passed through runtime.

FAIL: function mutates any TradeDecision.
FAIL: function mutates visibility_map.

### R6 â€” Runtime injection

In `runtime.py`, call `build_explanation_map(trade_decisions, visibility_map, overall_pressure)` after `build_visibility_map`. Store result in `PipelineResult` as `explanation_map: dict[str, dict]`.

In `build_pipeline_output_contract`, extract `explanation_map = getattr(pr, 'explanation_map', {})` and pass it to `_build_trade_candidates` as an additional parameter after `visibility_map`.

FAIL: explanation computed before visibility_map is available.

### R7 â€” Contract integration

In `contract.py:_build_trade_candidates`, merge explanation fields as a nested `"explanation"` key per candidate from `pipeline_result.explanation_map.get(symbol, {})`.

Missing explanation_map must produce `explanation={}` â€” not raise.

FAIL: contract builder raises when explanation_map is absent.

### R8 â€” UI rendering

In `ui/app.js`, render the `explanation` block inside the PRD-065 expansion panel (below visibility fields). Show:

- `block_reasons` as a bulleted list (hidden if empty)
- `macro_alignment` as a badge (hidden if null)
- `required_changes` as a bulleted list (hidden if empty)

FAIL: explanation section renders when all fields are empty/null.
FAIL: JS error when explanation key is absent from candidate.
