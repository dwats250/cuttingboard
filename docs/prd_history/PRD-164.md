# PRD-164: Harden PRD lifecycle tooling (single-commit closeout, correct-table Stage-0, hash-drift detection)

Lane: STANDARD / INFRA
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-05

## WHY NOW

Every PRD in the last five (159, 160, 161, 162, 163) paid the same avoidable
tooling tax: `prd_close.sh` leaves three gaps that force a second
"registry status COMPLETE + reset active pointer" bookkeeping commit, and
`prd_open.sh` mis-targets the registry table so Stage-0 needs a manual
row-move. PRD-160 additionally spent a whole commit reconciling a recorded
hash that a rebase had invalidated. These are deterministic, scriptable
defects in the governance tooling itself — fixing them removes a manual step
from both ends of every future PRD.

## MAX EXPECTED DELTA

~160 production LOC across `scripts/prd_open.sh`, `scripts/prd_close.sh`, and
`tools/validate_prd_registry.py` (excluding tests and the skill doc).
Exceeding this stops implementation and re-triggers review.

## GOAL

Make `prd_close.sh` produce a complete closeout in a single commit, make
`prd_open.sh` insert the Stage-0 row into the main PRD table, and make
`validate_prd_registry.py` detect commit-hash drift — so the recurring manual
fixup commits and row-moves disappear.

## SCOPE

- **`prd_close.sh`** — when a registry row for the PRD already exists (the
  normal case, since `prd_open.sh` created it at Stage-0), flip that row in
  place to `COMPLETE` with the implementation hash instead of taking the
  current skip branch (`scripts/prd_close.sh:111-112`).
- **`prd_close.sh`** — flip *both* status markers in `PRD-NNN.md`: the
  capital-S `Status:` header line and the trailing all-caps `STATUS:` line.
  The current regex (`scripts/prd_close.sh:99`, `^STATUS:.*$`) matches only
  the trailing line and leaves the header at `IN PROGRESS`.
- **`prd_close.sh`** — reset `PROJECT_STATE.md` `**Active PRD:**` to `none`
  (`docs/PROJECT_STATE.md:41`). Add an optional `--next "<text>"` argument
  that, when supplied, rewrites the `**Next step` line
  (`docs/PROJECT_STATE.md:43`); when omitted, the `**Next step` line is left
  byte-for-byte unchanged (it carries meaningful author-written content and
  must not be clobbered with a canned string).
- **`prd_open.sh`** — insert the new `IN PROGRESS` row as the last row of the
  *main* PRD table, not after the trailing `## Audit Reports` table. The
  current scan (`scripts/prd_open.sh:129-136`) picks the last
  `| PRD-\d{3} |` line anywhere in the file, which is the `| PRD-016 |` row
  inside Audit Reports.
- **`validate_prd_registry.py`** — add a check that every `COMPLETE` PRD's
  recorded commit hash resolves to a real git commit (catches post-rebase
  drift), gated on git availability so non-git validation trees still pass;
  and that the trailing `STATUS: COMPLETE @ <hash>` line in `PRD-NNN.md`
  agrees with the registry/index hash for that PRD.
- Update `.claude/skills/prd-closeout-verified/SKILL.md` to remove the now
  obsolete "patch the gaps the script leaves" manual steps.
- Tests for every behavior above.

## OUT OF SCOPE

- The `#4` scope-churn observation (PRD-162 opened broad, narrowed before

  the narrowing happened *before* the review gate — i.e. the gate working as
  designed. No new rule is added; no CLAUDE.md edit.
- Any change to the PRD authoring or review skills/scripts beyond the

- Retroactively reconciling existing registry/doc hashes; the validator

- Auto-generating `**Next step` text. The script either sets it from
  `--next` or leaves it unchanged.

## REQUIREMENTS

R1 — Stage-0 row lands in the main table.
`prd_open.sh` inserts the new `IN PROGRESS` row as the final row of the main
PRD table, contiguous with the existing `| PRD-\d{3} |` rows, regardless of a
trailing `## Audit Reports` table.

FAIL: running `prd_open.sh --prd 200 ...` against a registry fixture whose
main table is followed by an `## Audit Reports` section containing a
`| PRD-016 |` row places the new `| PRD-200 |` row after the
`## Audit Reports` header line (i.e. not contiguous with the main table).

R2 — Existing registry row flipped in place at close.
When a `| PRD-NNN |` row already exists, `prd_close.sh` rewrites that row to
`COMPLETE` with the supplied hash rather than skipping it.

FAIL: after `prd_close.sh --prd NNN --hash H ...` runs against a registry
already containing an `IN PROGRESS` row for PRD-NNN, that row still reads
`IN PROGRESS`, or its commit column is not `H`.

R3 — Both status markers flipped in the PRD doc.
`prd_close.sh` sets both the `Status:` header line and the trailing `STATUS:`
line in `PRD-NNN.md` to the completed state.

FAIL: after `prd_close.sh` runs, `PRD-NNN.md` contains any line matching the
regex `^[Ss][Tt][Aa][Tt][Uu][Ss]:` whose value still contains `IN PROGRESS`.

R4 — Active PRD pointer reset at close.
`prd_close.sh` rewrites the `**Active PRD:**` line in `PROJECT_STATE.md` to
`none`.

FAIL: after `prd_close.sh` runs without a successor argument, the
`**Active PRD:**` line's value is not `none`.

R5 — Next-step line set only when requested.
`prd_close.sh` accepts an optional `--next "<text>"`; when supplied it rewrites
the `**Next step` line, and when omitted that line is unchanged.

FAIL: omitting `--next` changes the `**Next step` line, OR supplying
`--next "X"` leaves the `**Next step` line not containing `X`.

R6 — Validator detects hash drift.
`validate_prd_registry.py` reports an error when a `COMPLETE` PRD's recorded
commit hash does not resolve to a real commit (when run inside a git work
tree) or when the trailing `STATUS: COMPLETE @ <hash>` line in `PRD-NNN.md`
disagrees with the registry/index hash for that PRD. When run against a
non-git tree, the git-resolvability check is skipped and validation still
passes on an otherwise-consistent tree.

FAIL: a `COMPLETE` entry whose recorded hash is a syntactically valid but
non-existent commit passes validation inside a git work tree; OR a
`PRD-NNN.md` `STATUS` hash that differs from the registry hash passes
validation; OR an otherwise-consistent non-git tree fails validation solely
because of the new git-resolvability check.

R7 — Closeout is a single commit.
Closing a PRD through the documented `prd-closeout-verified` path requires no
second commit to flip the registry row, the `Status:` header, or the
`**Active PRD:**` pointer.

FAIL: the documented closeout path leaves any of {registry row `IN PROGRESS`,
`Status:` header `IN PROGRESS`, `**Active PRD:**` naming the just-closed PRD}
after `prd_close.sh --commit`, forcing a follow-up fixup commit.

R8 — Closeout skill doc reflects the new behavior.
`prd-closeout-verified/SKILL.md` no longer instructs patching the registry
row, `Status:` header, or `**Active PRD:**` pointer as manual steps after
`prd_close.sh`.

FAIL: `SKILL.md` still contains a manual "patch the gaps" instruction to flip
the registry row / `Status:` header / `**Active PRD:**` pointer after running
the script.

## DATA FLOW

`prd_open.sh` → insert IN PROGRESS row at end of main table → `PRD_REGISTRY.md`
`prd_close.sh --commit [--next]` → flip row + both STATUS markers + Active PRD
  + (optional) Next step → single bookkeeping commit
`validate_prd_registry.py` → parse registry + index + PRD-doc STATUS + git →
  error list (drift detected)

## FAIL CONDITIONS

- Stage-0 row inserted outside the main table (R1).
- Existing registry row left `IN PROGRESS` after close (R2).
- A `Status`/`STATUS` line left `IN PROGRESS` after close (R3).
- `**Active PRD:**` not reset to `none` after close (R4).
- `**Next step` clobbered when `--next` omitted, or not set when supplied (R5).
- Non-existent COMPLETE hash or doc/registry hash mismatch passes the
  validator inside a git tree; or non-git tree fails on the git check (R6).
- Closeout requires a second fixup commit (R7).
- Closeout skill doc still documents the obsolete manual gap-patch (R8).
