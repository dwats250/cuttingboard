# PRD-164: Harden PRD lifecycle tooling (single-commit closeout, correct-table Stage-0, hash-drift detection)

Lane: STANDARD / INFRA
Status: COMPLETE
Author: Dustin
Filed: 2026-06-05

## WHY NOW

Every PRD in the last five (159, 160, 161, 162, 163) paid the same avoidable
tooling tax: `prd_close.sh` leaves three gaps that force a second
"registry status COMPLETE + reset active pointer" bookkeeping commit, and
`prd_open.sh` mis-targets the registry table so Stage-0 needs a manual
row-move. PRD-160 additionally spent a whole commit reconciling a recorded
hash that a rebase had invalidated. These are deterministic, scriptable
defects in the governance tooling itself ‚Äî fixing them removes a manual step
from both ends of every future PRD.

## MAX EXPECTED DELTA

~190 production LOC across `scripts/prd_open.sh`, `scripts/prd_close.sh`, and
`tools/validate_prd_registry.py` (excluding tests and the skill doc). The
upper bound reflects multi-hash splitting, git-root discovery, and doc/STATUS
parsing in the validator surfaced during review. Exceeding this stops
implementation and re-triggers review.

## GOAL

Make `prd_close.sh` produce a complete closeout in a single commit, make
`prd_open.sh` insert the Stage-0 row into the main PRD table, and make
`validate_prd_registry.py` detect commit-hash drift ‚Äî so the recurring manual
fixup commits and row-moves disappear.

## SCOPE

- **`prd_close.sh`** ‚Äî when a registry row for the PRD already exists (the
  normal case, since `prd_open.sh` created it at Stage-0), flip that row in
  place to `COMPLETE` with the implementation hash instead of taking the
  current skip branch (`scripts/prd_close.sh:111-112`).
- **`prd_close.sh`** ‚Äî flip *both* status markers in `PRD-NNN.md`: the
  capital-S `Status:` header line and the trailing all-caps `STATUS:` line.
  The current regex (`scripts/prd_close.sh:99`, `^STATUS:.*$`) matches only
  the trailing line and leaves the header at `IN PROGRESS`.
- **`prd_close.sh`** ‚Äî reset `PROJECT_STATE.md` `**Active PRD:**` to `none`
  (`docs/PROJECT_STATE.md:41`). Add an optional `--next "<text>"` argument
  that, when supplied, rewrites the `**Next step` line
  (`docs/PROJECT_STATE.md:43`); when omitted, the `**Next step` line is left
  byte-for-byte unchanged (it carries meaningful author-written content and
  must not be clobbered with a canned string).
- **`prd_open.sh`** ‚Äî insert the new `IN PROGRESS` row as the last row of the
  *main* PRD table, not after the trailing `## Audit Reports` table. The
  current scan (`scripts/prd_open.sh:129-136`) picks the last
  `| PRD-\d{3} |` line anywhere in the file, which is the `| PRD-016 |` row
  inside Audit Reports.
- **`validate_prd_registry.py`** ‚Äî add two drift checks. (a) Every `COMPLETE`
  PRD's recorded commit hash(es) resolve to real git commits ‚Äî splitting
  comma-separated multi-hash cells (e.g. PRD-158's seven hashes,
  `docs/PRD_REGISTRY.md:178`) and verifying each ‚Äî run against the *validation
  root* (`Path(args[0])`), skipped entirely when that root is not a git work
  tree so non-git fixture trees still pass. (b) When a `PRD-NNN.md` doc exists
  (registry File column not `‚Äî`), its trailing `STATUS: COMPLETE @ <hash>` hash
  is a member of that PRD's registry commit set.
- Update `.claude/skills/prd-closeout-verified/SKILL.md` to remove the now
  obsolete "patch the gaps the script leaves" manual steps.
- Tests for every behavior above.

## REQUIREMENTS

R1 ‚Äî Stage-0 row lands in the main table.
`prd_open.sh` inserts the new `IN PROGRESS` row as the final row of the main
PRD table, contiguous with the existing `| PRD-\d{3} |` rows, regardless of a
trailing `## Audit Reports` table.

FAIL: running `prd_open.sh --prd 200 ...` against a registry fixture whose
main table is followed by an `## Audit Reports` section containing a
`| PRD-016 |` row places the new `| PRD-200 |` row after the
`## Audit Reports` header line (i.e. not contiguous with the main table).

R2 ‚Äî Existing registry row flipped in place at close.
When a `| PRD-NNN |` row already exists, `prd_close.sh` rewrites that row to
`COMPLETE` with the supplied hash rather than skipping it. The missing-row case
is unchanged by this PRD: `prd_close.sh` retains its existing append-when-absent
fallback (`scripts/prd_close.sh:111-130`), and the `prd-closeout-verified`
preflight still refuses a missing row before the script runs (its Registry-row
invariant), so append-when-absent stays a defensive fallback, not a primary
path.

FAIL: after `prd_close.sh --prd NNN --hash H ...` runs against a registry
already containing an `IN PROGRESS` row for PRD-NNN, that row still reads
`IN PROGRESS`, or its commit column is not `H`; OR the missing-row case behaves
differently from the current append-when-absent fallback.

R3 ‚Äî Both status markers flipped, to their distinct completed forms.
`prd_close.sh` sets the `Status:` header line to exactly `Status: COMPLETE` and
the trailing `STATUS:` line to exactly `STATUS: COMPLETE @ <hash>` (the two
markers take different forms, per the PRD-163 convention ‚Äî header without hash,
trailing with hash). Exactly one header marker and one trailing marker remain.

FAIL: after `prd_close.sh` runs, the header line is not exactly
`Status: COMPLETE`, OR the trailing line is not exactly
`STATUS: COMPLETE @ <hash>`, OR any `^[Ss]tatus:`-anchored line still contains
`IN PROGRESS`, OR more than one trailing `STATUS:` marker exists.

R4 ‚Äî Active PRD pointer reset at close.
`prd_close.sh` rewrites the `**Active PRD:**` line in `PROJECT_STATE.md` to
`none`.

FAIL: after `prd_close.sh` runs without a successor argument, the
`**Active PRD:**` line's value is not `none`.

R5 ‚Äî Next-step line set only when requested, metacharacter-safe.
`prd_close.sh` accepts an optional `--next "<text>"`; when supplied it rewrites
the `**Next step` line, and when omitted that line is unchanged. The text is
inserted via a literal/callable replacement, not an `re.sub` template ‚Äî the
same PRD-147 hardening already applied to `--title` / `--summary`
(`scripts/prd_close.sh` callable replacements).

FAIL: omitting `--next` changes the `**Next step` line, OR supplying
`--next "X"` leaves the `**Next step` line not containing `X`, OR
`--next 'foo \d bar'` raises `re.error` / inserts corrupted text.

R6 ‚Äî Validator detects hash drift.
`validate_prd_registry.py` gains two checks, both scoped to PRD numbers
‚â• `TRACKING_START`:
(a) **Commit resolvability** ‚Äî for each `COMPLETE` registry row, split the
commit cell on commas and verify every hash resolves to a real commit, running
git against the *validation root* (`Path(args[0])` in `main`), not the process
cwd. Skipped entirely when the root is not a git work tree, so non-git fixture
trees (e.g. `test_resulting_tree_passes_validator`, synthetic hash `aaaaaaa`)
still pass.
(b) **Doc/registry hash agreement** ‚Äî when a `PRD-NNN.md` doc exists (registry
File column not `‚Äî`), its trailing `STATUS: COMPLETE @ <hash>` hash must be a
member of that PRD's registry commit set. Skipped when File is `‚Äî` or the doc
has no `STATUS: COMPLETE @` line.

FAIL (any of):
- a `COMPLETE` row whose commit cell is a syntactically valid but non-existent
  hash passes validation when run with a git work-tree root;
- a multi-hash cell (`h1, h2`) is treated as one unresolved string (false-fail)
  or left unchecked;
- a `PRD-NNN.md` `STATUS` hash absent from its registry commit set passes
  validation;
- an otherwise-consistent non-git tree, or a tree whose COMPLETE rows use
  File `‚Äî`, fails validation solely because of these checks.

R7 ‚Äî Closeout is a single commit.
Closing a PRD through the documented `prd-closeout-verified` path requires no
second commit to flip the registry row, the `Status:` header, or the
`**Active PRD:**` pointer.

FAIL: the documented closeout path leaves any of {registry row `IN PROGRESS`,
`Status:` header `IN PROGRESS`, `**Active PRD:**` naming the just-closed PRD}
after `prd_close.sh --commit`, forcing a follow-up fixup commit.

R8 ‚Äî Closeout skill doc reflects the new behavior.
`prd-closeout-verified/SKILL.md` no longer instructs patching the registry
row, `Status:` header, or `**Active PRD:**` pointer as manual steps after
`prd_close.sh`.

FAIL: `SKILL.md` still contains a manual "patch the gaps" instruction to flip
the registry row / `Status:` header / `**Active PRD:**` pointer after running
the script.

## DATA FLOW

`prd_open.sh` ‚Üí insert IN PROGRESS row at end of main table ‚Üí `PRD_REGISTRY.md`
`prd_close.sh --commit [--next]` ‚Üí flip row + both STATUS markers + Active PRD
  + (optional) Next step ‚Üí single bookkeeping commit
`validate_prd_registry.py` ‚Üí parse registry + index + PRD-doc STATUS + git ‚Üí
  error list (drift detected)

## FAIL CONDITIONS

- Stage-0 row inserted outside the main table (R1).
- Existing registry row left `IN PROGRESS` after close (R2).
- A `Status`/`STATUS` line left `IN PROGRESS` after close (R3).
- `**Active PRD:**` not reset to `none` after close (R4).
- `**Next step` clobbered when `--next` omitted, not set when supplied, or
  `--next` corrupted by a regex metacharacter (R5).
- Non-existent / unchecked-multi-hash COMPLETE hash or doc/registry hash
  mismatch passes the validator inside a git tree; or a non-git / File-`‚Äî` tree
  fails solely on the new checks (R6).
- Closeout requires a second fixup commit (R7).
- Closeout skill doc still documents the obsolete manual gap-patch (R8).
