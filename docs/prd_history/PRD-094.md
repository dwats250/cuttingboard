# PRD-094 — Public Dashboard Artifact Contamination Guard

**STATUS:** COMPLETE

---

## GOAL

Prevent public dashboard artifacts from being committed or published when they contain pytest temporary paths or local machine paths.

`ui/dashboard.html` and `ui/index.html` must not contain artifact diagnostic sources pointing to `pytest-of-`, `/tmp/pytest`, or `/tmp/`. This protects the GitHub Pages dashboard from exposing local/test artifact paths, as currently observed in `ui/index.html` (artifact diagnostic spans, lines 88–91, contain `/tmp/pytest-of-dustin/...`).

PRD-094 must leave the repository fully green after implementation.

---

## SCOPE

1. Add contamination pattern validation to `tests/test_dashboard_renderer.py`.
2. Add PA-style published-artifact contamination tests for both `ui/dashboard.html` and `ui/index.html` (skip if absent — consistent with existing PA1/PA2/PA3 convention).
3. Add a positive contamination test using synthetic HTML string literals to prove detection works.
4. If either public artifact is contaminated at implementation time, clean it using the preferred order defined in REQUIREMENTS R7.
5. All contamination fixes to public artifacts must be validated: no `pytest-of-`, `/tmp/pytest`, or `/tmp/` paths remaining after cleanup.
6. Do not add a separate script or helper module — inline in the test file.

---

## OUT OF SCOPE

1. Do not modify dashboard visual layout.
2. Do not modify artifact diagnostics rendering behavior.
3. Do not modify runtime logic, trade decision logic, candidate grading, lifecycle logic, market_map generation, data fetching, notifications, workflows, or schemas.
4. Do not modify PRD registry or PROJECT_STATE during implementation (separate bookkeeping step).

---

## FILES

- `tests/test_dashboard_renderer.py` — add PA contamination tests and positive contamination unit test
- `ui/index.html` — cleanup or regeneration only, if contamination is confirmed present
- `ui/dashboard.html` — cleanup or regeneration only, if contamination is confirmed present

No other files.

## REQUIREMENTS

**R1 — Contamination patterns are defined in one place.**

The test file must define a single collection used by all contamination checks:

```python
_FORBIDDEN_ARTIFACT_PATTERNS = ("pytest-of-", "/tmp/pytest", "/tmp/")
```

FAIL: Patterns are scattered across multiple test functions without a shared definition.

---

**R2 — Both public artifact files are checked when present.**

Add PA-style tests that read `ui/dashboard.html` and `ui/index.html` from disk and assert none of `_FORBIDDEN_ARTIFACT_PATTERNS` appear. If a file is absent, the test skips via `pytest.skip` (matching PA1/PA2/PA3 convention).

FAIL: Only one file is checked while the other exists and is not tested.

---

**R3 — Failure output identifies file and pattern.**

Assertion messages must be of the form:

```
f"{path}: contains forbidden pattern {pattern!r}"
```

FAIL: A contamination failure is raised without naming the file or pattern.

---

**R4 — Validation is local and deterministic.**

Tests must not call network services, fetch market data, regenerate dashboard HTML, or depend on market hours or current time.

FAIL: Any contamination test triggers network I/O or artifact regeneration.

---

**R5 — Clean synthetic HTML passes without false positives.**

A synthetic HTML string containing no forbidden patterns must pass the contamination check without raising or failing.

FAIL: A synthetic string with no forbidden patterns causes validation to fail.

---

**R6 — Positive contamination cases are tested.**

Add parametrized or individual tests that construct minimal synthetic HTML strings each containing exactly one forbidden pattern and assert the check fails for each.

Required positive cases:
- String containing `pytest-of-`
- String containing `/tmp/pytest`
- String containing `/tmp/`

FAIL: No test exercises a contaminated synthetic HTML string through the validation logic.

---

**R7 — Public artifacts are clean after implementation.**

If `ui/index.html` or `ui/dashboard.html` contains any pattern from `_FORBIDDEN_ARTIFACT_PATTERNS` at implementation time, that contamination must be removed before commit using this preferred order:

1. **Prefer regeneration.** If real repo artifacts (`logs/latest_hourly_payload.json`, `logs/latest_hourly_run.json`, `logs/market_map.json`, `logs/latest_hourly_contract.json`) are present and not themselves contaminated, regenerate the dashboard HTML using the normal renderer command. Validate the regenerated output contains no forbidden patterns.
2. **Fallback: minimal blank.** Only if regeneration is not possible (artifacts absent, stale, or also contaminated), minimally blank or remove the contaminated artifact diagnostic source path values within the existing `artifact-diagnostics` block. Do not manually edit unrelated HTML.

After cleanup by either method, confirm:

```bash
grep -r "pytest-of-\|/tmp/pytest\|/tmp/" ui/
```

Returns no output.

FAIL: Either public artifact retains a forbidden pattern after the commit.

---

**R8 — Existing renderer tests remain green.**

All existing tests in `tests/test_dashboard_renderer.py` must pass. `test_normal_render_no_pytest_paths` (line 678) is preserved unchanged — it covers the renderer function output, distinct from public-artifact scanning.

FAIL: Any existing test in `tests/test_dashboard_renderer.py` fails after implementation.

---

**R9 — Dashboard renderer test suite is green after implementation.**

`python3 -m pytest tests/test_dashboard_renderer.py -q` passes with zero failures, zero errors.

FAIL: Any test fails or errors after implementation.

## DATA FLOW

No dashboard data flow changes.

Validation-only flow:

```
public artifact path (ui/dashboard.html or ui/index.html)
  -> check file exists (skip via pytest.skip if absent)
  -> read static HTML text
  -> scan each pattern in _FORBIDDEN_ARTIFACT_PATTERNS
  -> pass if none found
  -> pytest.fail(f"{path}: contains forbidden pattern {pattern!r}") if found
```

Positive contamination test flow:

```
synthetic HTML string literal (contains one forbidden pattern)
  -> same scan logic
  -> assert failure is raised
```

Artifact cleanup flow (if required):

```
ui/index.html or ui/dashboard.html
  -> confirm contaminated lines (grep)
  -> preferred: regenerate from real repo artifacts via normal renderer command
  -> fallback: minimally blank contaminated artifact diagnostic source path values
  -> re-confirm: grep -r "pytest-of-\|/tmp/pytest\|/tmp/" ui/ returns no output
  -> stage only confirmed-clean artifacts
```

- F9: No test exercises a contaminated synthetic HTML string.
- F10: Either public artifact retains a forbidden pattern after commit.
