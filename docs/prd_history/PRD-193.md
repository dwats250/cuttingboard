PRD-193 — OHLCV cache trading-day freshness + publish-safe prefetch persistence

Status: COMPLETE

REVISION NOTE (2026-06-24)
Re-scoped from the original "publish-safe prefetch with OHLCV cache persistence"
problem statement after a Stage-0 realizability check found the original
mechanism INERT. The plan was: persist the 12:50 prefetch warm-up cache so the
13:00 live run reads it warm. But `_is_fresh_ohlcv_cache` (ingestion.py) judged
freshness by the age of the last DAILY bar against a 12h TTL
(`OHLCV_STALE_HOURS`). Daily bars are always >= 1 day old, and both the prefetch
(12:50 UTC) and live (13:00 UTC) slots are pre-market (US open 13:30 UTC), so the
most recent bar is the prior session — > 12h old at both slots. The live run
therefore discarded the prefetched cache as stale and re-fetched regardless of
persistence: zero benefit. Adding `actions/cache` alone would have been dead code
with extra steps.

The fix that makes persistence pay off is a corrected freshness model. Dustin
chose the trading-day-aware model (2026-06-24): the cache is fresh when its last
bar IS the most recent completed trading session — exactly the data a re-fetch
would return that day — rather than measuring an always-stale age-vs-TTL. With
that, the prefetched cache is correctly reused by the live run, and the original
persistence + publish-safety + cron re-enable finally deliver real latency /
rate-limit relief.

PROBLEM
Two coupled defects keep the 12:50 prefetch slot worthless:

1. FRESHNESS (the load-bearing defect). `_is_fresh_ohlcv_cache`
   (ingestion.py:146-158) returns fresh iff `now - last_bar < OHLCV_STALE_HOURS`
   (12h). For DAILY OHLCV the last bar is always the prior session (the fetch
   uses `end=<today UTC>`, exclusive, so today's bar is never returned), which is
   always > 12h old at any run time. The cache is therefore judged stale on every
   read and never serves a hit — pre-market or intraday. The parquet cache is a
   latent dead optimization.

2. PERSISTENCE + PUBLISH SAFETY (the original PRD-193 problems, still valid).
   `data/cache` is written to the ephemeral runner and discarded: the
   Commit-artifacts step stages only `reports/ logs/ ui/`, and no `actions/cache`
   exists. So even a fresh cache would not survive to the next run. And the
   prefetch step sets `PUBLISH_READY=true`, which would render/publish a stale
   dashboard and trip the PRD-119 freshness gate — which is exactly why PRD-189
   dropped the 12:50 cron to `noop` (resolve_run_mode.py). The slot is currently
   dead (noop), not failing.

SCOPE
R1 — Trading-day freshness model. Add
`time_utils.most_recent_completed_session_date(now_utc) -> date` (the latest
weekday strictly before today's UTC date; holiday-unaware by design). Rewrite
`_is_fresh_ohlcv_cache` to return fresh iff `last_bar_date >=
most_recent_completed_session_date(now)`. Stateless — no cache-format change, no
stored fetch timestamp. Self-heals every weekday (the key advances); a
post-holiday day falls through to a safe, redundant re-fetch (never serves
stale).

R2 — Retire `OHLCV_STALE_HOURS`. Its only reader was the old freshness check
(grep-swept: config.py def, ingestion.py:157, and two test_derived.py fixtures).
Remove the constant and all readers (dead-branch enumeration: no other reader).

R3 — Persist the cache via `actions/cache` (cuttingboard.yml). Restore `data/cache`
keyed on the UTC trading day (prefix-fallback restore-keys; safe because R1
re-validates) for the warming modes; SAVE under the day key on the `prefetch`
run. The 13:00 live run restores the 12:50 prefetch warm cache and, by R1, reuses
it instead of re-fetching.

R4 — Publish-safe prefetch. Drop `echo "PUBLISH_READY=true"` from the prefetch
step so prefetch is cache-warm-only: it renders/publishes nothing and cannot trip
the PRD-119 gate. (execute_prefetch already writes no payload and sends no
notification.)

R5 — Re-enable the cron. Add `"50 12 * * 1-5": "prefetch"` to
`resolve_run_mode._DEDICATED` and update its header comment. The existing
resolver guards (`test_every_scheduled_slot_is_wired_to_a_valid_dispatch_step`,
`test_every_workflow_invocation_is_parser_valid`) then assert prefetch is wired to
a parser-valid dispatch step; flip `test_dropped_crons_resolve_to_noop` to drop
`50 12` and add a positive prefetch-resolution test.

R6 — Reconcile `docs/prd_history/INTERFACE_LOCK.md`: the prefetch row (currently
"12:50 UTC M-F live scheduled invocation", stale since PRD-189) becomes the
re-enabled cache-warm-only, publish-safe scheduled prefetch this PRD lands.

DESIGN NOTES
- Correctness: R1 serves the cache only when it equals what a same-slot re-fetch
  would return, so all six `fetch_ohlcv` consumers see identical data — fewer
  yfinance calls, never different values. The PRD-190/204 "stale cache self-heals
  within <= 1 session" safety property is preserved and strengthened (re-fetch
  now triggers every new weekday, not on a 12h clock).
- Accepted edge: `auto_adjust=True` means a corporate action could retroactively
  change older bars; a same-day cache hit serves pre-adjustment bars until the
  next weekday refresh. Rare (splits/dividends) and self-correcting next session.
- actions/cache producer/consumer split (prefetch saves, live restores) avoids
  empty-cache key poisoning: only the cache-warming `prefetch` run writes the day
  key.

NON-GOALS
- No `actions/cache` in hourly_alert.yml (intraday cross-run reuse is a possible
  follow-up; R1 makes it correct, but persistence there is out of scope here).
- No holiday calendar (weekday heuristic errs toward re-fetch — safe).
- No change to the live / sunday cron resolution (PRD-189) or the
  intraday/notify routing (PRD-192).
- No change to OHLCV_FETCH_MONTHS (12, PRD-190) or the PRD-119 publish gate.

RISKS
- Serving stale data if R1 is wrong. Mitigated: the predicate serves the cache
  only when its last bar equals the most-recent-completed-session a re-fetch would
  return; older falls through to fetch. Red tests cover fresh / stale / weekend /
  post-holiday.
- Cache persistence only manifests in CI. Mitigated: VERIFICATION below.
