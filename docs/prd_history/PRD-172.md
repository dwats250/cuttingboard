# PRD-172 — prd_close.sh baseline-bullet regex tolerates the N-xfailed suffix

Lane: MICRO / INFRA

Filed: 2026-06-09

GOAL
`scripts/prd_close.sh` updates the PROJECT_STATE test-baseline bullet
with the match pattern at line 198:
`r"^- \*\*\d[\d,]* passing\*\*[^\n]*$"`. The `passing\*\*` anchor requires
the closing `**` immediately after `passing`, so a bullet written in the
common `- **NNNN passing, M xfailed** (...)` form never matches — the
script prints `WARN: test baseline bullet not found` and leaves the
bullet stale. This is why `docs/PROJECT_STATE.md` line 104 has read
`- **2499 passing, 1 xfailed** ...` since PRD-153 while the live baseline
is 2533 (observed as a WARN at both PRD-170 and PRD-171 closeout).
Loosen the regex so any non-newline content between `passing` and the
closing `**` is tolerated, and add a regression test.

SCOPE
- In `scripts/prd_close.sh` line 198: change the match pattern from
  `r"^- \*\*\d[\d,]* passing\*\*[^\n]*$"` to
  `r"^- \*\*\d[\d,]* passing[^\n]*?\*\*[^\n]*$"`. The replacement string
  (line 199) is unchanged.
- In `tests/test_prd_close.py`: add a regression test that rewrites the
  fixture's baseline bullet to the `, 1 xfailed` form, runs the script,
  and asserts the bullet is updated to the new passing count with the
  `xfailed`-suffix form removed.
- Track this PRD in registry/state per existing convention.

OUT OF SCOPE
- The replacement format itself (it intentionally normalizes to
  `**N passing** (as of …)` and drops the xfailed count) — unchanged.
- Any other `prd_close.sh` logic or regex.
- No `cuttingboard/` engine code.
