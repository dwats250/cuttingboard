# PRD-197: Codex cross-review via GitHub Actions (host-independent gate satisfier)

Lane: HIGH-RISK / INFRA
Status: IN PROGRESS
Author: Claude Code (scoped per 2026-06-17 charge; held for Dustin)
Filed: 2026-06-17

## Problem

The HIGH-RISK review gate requires a Claude review artifact **plus** a Codex
cross-review (CLAUDE.md "Review and commit discipline"). Today the Codex half is
produced by `codex exec` running on a single host (asher). That host is a single
point of failure: a host/IP outage, an egress-allowlist gap (observed
2026-06-17 — `api.openai.com` not in the sandbox egress allowlist, 403
`host_not_allowed`), or a local-toolchain drift blocks the Codex half and
therefore **blocks every HIGH-RISK merge**. The gate's availability should not
depend on one machine.

Move the Codex half to CI: a reproducible, host-independent GitHub Actions
workflow that runs `codex exec` read-only against a target PR/PRD at a pinned
SHA and emits the durable `.review.codex.md` artifact. This implements the
**mechanism** that produces the artifact the PRD-#21 gate-rule already defines
(in-tree, SHA-pinned, verified-real Codex, read-only, fresh-context).

## Lane classification (per CLAUDE.md / docs/PRD_PROCESS.md)

**HIGH-RISK / INFRA.** It adds a CI workflow, handles an API-key secret on a
**public** repo, and writes review artifacts back to branches — three
decision-/security-surface properties that put it in the HIGH-RISK lane. Per
CLAUDE.md this PRD itself requires the Claude + Codex review gate before merge
(bootstrapping note below).

## Dependency

- **Depends on PRD-#21 (merged):** PRD-#21 defines *what artifact satisfies* the
  Codex gate (the five properties). PRD-197 builds the *mechanism* that produces
  exactly that artifact via CI. PRD-197 must not redefine the gate; it conforms
  to it.

## Scope

### Trigger
- `workflow_dispatch` **ONLY**, taking the target PR number (and/or PRD number)
  and the head SHA as inputs.
- **Explicitly NOT `pull_request` / `pull_request_target`.** A `pull_request`
  trigger from a fork would expose `OPENAI_API_KEY` to untrusted fork code;
  `pull_request_target` runs with secrets against fork content and is equally
  unsafe here. Dispatch-only keeps the secret on trusted, maintainer-initiated
  runs.

### Auth — via the official `openai/codex-action` proxy (NOT a hand-rolled key env)
- `OPENAI_API_KEY` sourced from the **repo Actions secret**, consumed **only** as
  the `openai-api-key` *input* of the official `openai/codex-action`, which
  proxies the key so the Codex process that reads repo code never sees the raw
  secret. The repo is public: the key is **never echoed**, never written to the
  artifact, never printed in logs, and is **never set as a job- or step-level
  `env:`** in any job that runs repo code (exfil risk). No hand-rolled
  `npm i -g @openai/codex` + key-as-env step.

### Execution — official action, READ-ONLY, two-job split (OpenAI secure pattern)
- Uses **`openai/codex-action`** (pinned), `sandbox: read-only` — **never**
  `workspace-write` (it re-persists `trust_level=trusted` for the cwd and breaks
  read-only durability; called out in the PRD-#21 gate-rule).
- **Fresh-context:** a clean checkout of the **target branch at the pinned input
  SHA**, no authoring-conversation state, so the artifact is tied to exactly the
  commit reviewed.
- **Two-job split** (the documented secure architecture):
  - **`review` job** — `permissions: contents: read` ONLY; runs the action
    (holding the key via its proxy) read-only; writes the Codex final message to
    `output-file` and **uploads** the assembled `PRD-NNN.review.codex.md` as a
    workflow artifact. **No write access**, so even a compromised review step
    cannot push or exfil via the repo token.
  - **`commit` job** — `permissions: contents: write`; **NO `OPENAI_API_KEY`
    anywhere**; downloads the artifact and commits the `.review.codex.md` to the
    target PRD branch, SHA-tied. The job that can write never holds the key; the
    job that holds the key cannot write.

### Output
- A durable in-tree **`docs/prd_history/PRD-NNN.review.codex.md`** committed to
  the **target PRD branch**, tied to the reviewed SHA — the parallel to
  `.review.claude.md` and the exact artifact the PRD-#21 gate requires.
- Artifact records: reviewed SHA, target PR/PRD, verdict, the DRIFT CHECK section
  (PRD-186), and the real-Codex provenance (below).

### Permissions (least privilege)
- Top-level `permissions: {}` (no scopes by default); each job opts in:
  - `review`: `contents: read` only (holds the key, cannot write).
  - `commit`: `contents: write` only (writes the artifact, no key).
- No `pull-requests: write` unless a status comment is explicitly added later
  (out of scope here).

### Provenance
- The artifact records the **real Codex model id/version** used — the resolved
  `model` input plus the pinned `codex-version` (queried via `codex --version`)
  and the pinned `openai/codex-action` ref — so a routed/cheaper substitute model
  cannot masquerade as Codex without a visible workflow edit. Satisfies PRD-#21
  property 3 ("verified-real Codex").

## FILES

- `.github/workflows/codex-review.yml` — the dispatch-only, two-job workflow.
- `tests/test_codex_review.py` — hermetic assertions over the workflow contract
  (see FAIL-lines): dispatch-only trigger; `read-only` sandbox + no
  `workspace-write`; the secret appears only as the action `openai-api-key`
  input and in no `env:`/`echo`; two-job permission split; provenance recorded.
- `docs/prd_history/PRD-197.md` — this doc (design reconcile + bootstrapping
  waiver).

A standalone `scripts/codex_review.sh` helper is **not** used: the official
action runs Codex, so the workflow needs only thin inline shell for artifact
assembly and the commit. (Dropped from the original scope; reconciled here.)

## Acceptance criteria

- A `workflow_dispatch` run against a target PR/PRD + SHA produces a
  gate-satisfying `.review.codex.md` (all five PRD-#21 properties) and commits it
  to the target PRD branch tied to that SHA.
- `codex exec` runs **read-only**; `-s workspace-write` never appears.
- The artifact records real-Codex model/version provenance.
- **Zero secret material in logs** — `OPENAI_API_KEY` never echoed or written to
  the artifact.
- **Dispatch-only enforced** — the workflow has no `pull_request` /
  `pull_request_target` trigger; a test asserts this so a future edit adding one
  fails CI.

## FAIL-line observability

- FAIL: `tests/test_codex_review.py` asserts `.github/workflows/codex-review.yml`
  declares `on: workflow_dispatch` and **no** `pull_request` /
  `pull_request_target` key (guards secret exposure).
- FAIL: same test asserts the action runs with `sandbox: read-only` and that
  `workspace-write` appears nowhere in the workflow.
- FAIL: test asserts `secrets.OPENAI_API_KEY` appears **only** as the
  `openai-api-key` input of `openai/codex-action`, never under any `env:` and
  never inside a `run:` step (no echo/print); and that the `commit` job does not
  reference the secret at all.
- FAIL: test asserts the two-job permission split — `review` has
  `contents: read` (not write), `commit` has `contents: write`.
- FAIL: test asserts the artifact-writing step records a model/version
  provenance field (non-empty, sourced from the run).

## Non-goals

- Not changing the PRD-#21 gate definition (this conforms to it).
- Not adding a `pull_request` auto-review path (secret-exposure risk).
- Not retiring the local `codex exec` path; CI is an additional,
  host-independent producer of the same artifact.
- Not posting PR status comments (possible later PRD; would need
  `pull-requests: write`).

## Bootstrapping waiver (one-time, explicit exception — scoped to PRD-197 only)

PRD-197 is HIGH-RISK and so requires a Codex cross-review — but the mechanism
that produces a gate-satisfying `.review.codex.md` is the very thing this PRD
builds, and the local `codex exec` path is egress-blocked (2026-06-17:
`api.openai.com` not in the sandbox allowlist). This is a genuine chicken-and-egg.

**Dustin waives the Codex half of the gate for PRD-197 — ONCE, scoped to PRD-197
only — as an explicit documented exception (not a silent skip), on the condition
that PRD-197 self-reviews as this workflow's first run.** That is: after merge,
the first `workflow_dispatch` of `codex-review.yml` targets PRD-197 itself and
produces `docs/prd_history/PRD-197.review.codex.md`. Producing that artifact is
the **close condition** for PRD-197.

This waiver does not generalize: every other HIGH-RISK PRD still requires its
Codex cross-review per CLAUDE.md and the PRD-#21 gate-rule. Record the waiver and
the first-run self-review in `docs/DECISIONS.md` at closeout.
