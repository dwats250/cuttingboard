# PRD-167: RUN SNAPSHOT relative-freshness token

Lane: HIGH-RISK / CONSUMER
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-05

## WHY NOW

The dashboard RUN SNAPSHOT field renders an absolute Pacific timestamp
(`dashboard_renderer.py:1864-1869`) via the generic `.value` class — e.g.
`2026-06-04 10:16:35 PT`. PRD-158 §4.2 specifies the translation
"Run snapshot timestamp → relative freshness," and PRD-158 §10's
publish-verification hook (criterion 2) requires the field to render
relative freshness (`N minutes old` / `STALE (>5 min)`), not an absolute
timestamp. The 2026-06-04 real-run inspection confirmed criterion 2 **FAILED**
against the published `ui/dashboard.html`. This was R3 of the PRD-165 candidate,
split out by Dustin on 2026-06-05 so the contract fix ships independently of
the visual-hierarchy work (PRD-165, now COMPLETE). The renderer already owns
the freshness primitives (`_compute_timestamp_freshness`, `_utcnow`,
`DASHBOARD_STALE_AFTER_SECONDS = 300`) — this PRD wires them into the one field
that still leaks an absolute timestamp.

## GOAL

Render the RUN SNAPSHOT field as a relative-freshness token derived from the
snapshot timestamp and `_utcnow()`, replacing the absolute PT string, so the
field tells the operator *how old* the data is at a glance and surfaces
staleness past the 5-minute threshold.

## SCOPE

- **R1 — RUN SNAPSHOT renders relative freshness.** Replace the absolute PT
  string at `dashboard_renderer.py:1868-1869` with a relative-age token computed
  from the snapshot timestamp (`payload_timestamp_value or timestamp`) and
  `_utcnow()`. Fresh (≤ `DASHBOARD_STALE_AFTER_SECONDS`) renders an age token
  (`N minutes old` / `<1 min old`); past the threshold renders a staleness token
  (`STALE (>5 min)`). An unparseable/absent timestamp renders `unavailable`
  (current fallback preserved).

a relative-age token from a timestamp string and `_utcnow()`. The helper reuses

## REQUIREMENTS

R1 — RUN SNAPSHOT renders a relative-freshness token, not an absolute timestamp.
In a rendered dashboard, the RUN SNAPSHOT value is a relative-age/staleness token
derived from the snapshot timestamp and `_utcnow()`: a snapshot within
`DASHBOARD_STALE_AFTER_SECONDS` of now renders an age token containing the word
`old` (e.g. `3 minutes old`, `<1 min old`); a snapshot older than
`DASHBOARD_STALE_AFTER_SECONDS` renders a staleness token containing `STALE`; an
absent/unparseable timestamp renders `unavailable`.

FAIL: the RUN SNAPSHOT value renders an absolute clock timestamp (matches a
`HH:MM:SS` or ` PT` pattern), OR a fresh snapshot is marked `STALE`, OR a
snapshot older than the threshold is NOT marked `STALE`, OR an unparseable
timestamp raises instead of rendering `unavailable`.

## DESIGN DECISIONS FOR REVIEW

- **D1 (token wording/granularity).** PRD-158 §10 names `N minutes old` and
  `STALE (>5 min)`. Recommendation: fresh → `<1 min old` (age < 60s) else
  `N minutes old` (floor of age/60); stale (> 300s) → `STALE (>5 min)`. Minutes
  granularity only (no seconds/hours). Confirm wording.
- **D2 (visual prominence).** The PRD-165 candidate also wanted the RUN SNAPSHOT
  visually prominent (it currently shares `.value` with neighbors). Options: (a)
  keep `.value` — freshness token only, this PRD; (b) also bold it via the
  existing `.value-key` class shipped in PRD-165. Recommendation: (a) keep
  scope to the freshness token; (b) is a trivial follow-on. Confirm.

## DATA FLOW

`payload.meta.timestamp` (or fallback chain) → relative-age helper(`_utcnow()`)
→ RUN SNAPSHOT token (`<1 min old` / `N minutes old` / `STALE (>5 min)` /
`unavailable`)

## FAIL CONDITIONS

- RUN SNAPSHOT renders an absolute timestamp / ` PT` string (R1).
- A fresh snapshot is marked STALE, or a stale snapshot is not (R1).
- An unparseable timestamp raises instead of rendering `unavailable` (R1).
