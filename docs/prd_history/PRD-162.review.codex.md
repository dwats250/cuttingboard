# PRD-162 Codex Review
VERDICT
ACCEPT WITH CHANGES

SUMMARY
PRD-162 identifies a real downstream contradiction and is implementable if it treats `top_trades` as the actionable display projection while preserving `trade_candidates` as the broader contract/audit surface. The current draft is not yet binary enough: it over-includes `output.py`, leaves `contract.py` conditional, frames AC1 around `market_map` without editing `market_map.py`, and proposes one `TradeDecision` predicate for both objects and dicts without specifying the bridge. Scope is close, but the production FILES and AC3/PRD-161 interaction need to be resolved before implementation.

REQUIRED EDITS
1. Remove `cuttingboard/output.py` from production FILES or mark it verify-only: PRD-162.md:68-69 asks for a code edit to `output.py`, but `render_report_from_payload` reads the already-built payload at `cuttingboard/output.py:464-465` and sets `TRADE` only from non-empty `sections.top_trades` at `cuttingboard/output.py:479-484`; after `cuttingboard/delivery/payload.py:40` gates `top_trades`, this path is correct unchanged and has candidate dicts, not `TradeDecision` objects. FAIL: PRD-162 production FILES still list `cuttingboard/output.py` with a code-edit instruction.

2. Define the actionable predicate as a representation bridge, not only a `TradeDecision` helper: PRD-162.md:65-67 places one predicate in `trade_decision.py`, but runtime has `TradeDecision.status` / `TradeDecision.size_multiplier` at `cuttingboard/trade_decision.py:27-42` while payload gating reads candidate dict keys emitted by `cuttingboard/contract.py:319-340`. Use one semantic rule with two adapters, or one helper over normalized fields such as `symbol`, `status`, and `size_multiplier`. FAIL: the PRD claims one shared definition without naming how it is applied to both `TradeDecision` objects and candidate dicts.

3. Include the non-tradable exclusion in the runtime outcome definition: PRD-162.md:73-76 excludes `config.NON_TRADABLE_SYMBOLS` from `top_trades`, but `cuttingboard/runtime.py:875-879` currently emits `TRADE` for any `ALLOW_TRADE` decision and does not check `symbol` / `ticker`; `config.NON_TRADABLE_SYMBOLS` is the macro-driver set at `cuttingboard/config.py:177-178`, and `is_tradable_symbol` also rejects caret symbols at `cuttingboard/universe.py:8-11`. FAIL: AC1 can pass payload gating while runtime still emits `outcome=TRADE` for a non-tradable allowed decision.

4. Make AC1 data-layer precise instead of `market_map`-primary: PRD-162.md:122-125 says verdict/`market_map` agreement, but `cuttingboard/market_map.py` is absent from PRD-162.md:64-99 and `market_map` is built after outcome at `cuttingboard/runtime.py:875-893`; the changed surfaces are `trade_decisions`, `contract.trade_candidates`, and payload `sections.top_trades`. FAIL: AC1 still requires `market_map` agreement without either adding `market_map.py` to FILES or restating the binary condition in terms of the actionable decision/top_trades set.

5. Make AC3 and the PRD-161 re-point single-valued: PRD-162.md:129-133 allows either suppressing blocked candidates from `top_trades` or labeling them blocked, while PRD-162.md:113-117 allows re-pointing PRD-161 to either `contract.trade_candidates` or a new observational section. The existing PRD-161 test asserts the candidate exists in payload `top_trades` at `tests/test_prd161_sizing_gate_fixture.py:215-224` and reads sizing fields from that same entry at `tests/test_prd161_sizing_gate_fixture.py:244-250`. FAIL: the PRD still permits more than one `BLOCK_TRADE` rendering behavior or more than one PRD-161 replacement surface.

6. Resolve `contract.py` now and keep gating at the payload projection unless the PRD explicitly expands scope: PRD-162.md:77-80 leaves `contract.py` conditional, but the contract builder preserves blocked and allowed candidates with decision fields at `cuttingboard/contract.py:319-340`, and contract validation permits `BLOCK_TRADE` candidates at `cuttingboard/contract.py:731-736`. The minimal compatible design is to leave `contract.trade_candidates` intact and gate only `sections.top_trades`. FAIL: `cuttingboard/contract.py` remains listed as a conditional production edit instead of being removed from production FILES or explicitly specified as unchanged.

7. Declare `size_multiplier > 0` defensive, not currently necessary for `ALLOW_TRADE`: `size_multiplier_for_confidence` returns zero only below confidence 0.60 at `cuttingboard/execution_policy.py:59-68`, `evaluate_execution_policy` blocks confidence below 0.60 and other zero-size paths at `cuttingboard/execution_policy.py:215-233`, and allowed macro-pressure paths keep a positive fraction of `size` at `cuttingboard/execution_policy.py:239-251`. FAIL: PRD-162.md:66 or PRD-162.md:129-130 presents `ALLOW_TRADE` plus `size_multiplier == 0` as a current reachable state without a repo-backed path.

8. Add an explicit runtime/output outcome-agreement test to FILES: PRD-162.md:82-99 lists predicate, payload, contract, runtime, dashboard, PRD-161, and new reconciliation tests, but no named test asserts that runtime outcome and `render_report_from_payload` outcome agree after the payload `top_trades` projection. FAIL: no FILES entry or AC test asserts runtime-path/output-path outcome equality for the same actionable/non-actionable candidate set.

RATIONALE
- Implementability: yes, if the implementation creates one actionable semantic rule with object/dict adapters, applies non-tradable filtering consistently at runtime outcome and payload `top_trades`, and leaves contract construction unchanged.
- Binary/testable ACs: AC2, AC4, AC5, and AC6 can be made binary. AC1 is not binary while phrased against `market_map` without a `market_map.py` change. AC3 is not binary while it permits either suppression or labeled rendering.
- Scope minimality: payload gating at `cuttingboard/delivery/payload.py:40` plus a runtime outcome predicate at `cuttingboard/runtime.py:875-879` is the minimal production path. Editing `output.py` is unnecessary because `cuttingboard/output.py:479-484` already follows `sections.top_trades`.
- `top_trades` / `trade_candidates` consumer audit: payload `top_trades` is consumed by `render_report_from_payload` at `cuttingboard/output.py:479-484`, CLI transport counts at `cuttingboard/delivery/transport.py:47-57`, and dashboard/test payload helpers; gating it changes display/actionability semantics intentionally. Contract `trade_candidates` is consumed by notifications at `cuttingboard/output.py:769-783` and `cuttingboard/output.py:853-916`, notification suppression at `cuttingboard/notifications/state.py:52-87`, premarket focus list at `cuttingboard/reports/premarket.py:342-350`, dashboard contract context at `cuttingboard/delivery/dashboard_renderer.py:2244-2262`, overnight policy at `cuttingboard/overnight_policy.py:69-80`, audit serialization at `cuttingboard/audit.py:168-229`, and post-trade evaluation through audit `trade_decisions` at `cuttingboard/evaluation.py:114-137`. Leaving `trade_candidates` intact avoids breaking those consumers; changing contract construction would require a much larger audit.
- Postmarket report does not currently read `trade_candidates`; it uses `audit_summary.qualified_count` and rejections at `cuttingboard/reports/postmarket.py:106-153`, so it is not a direct breakage point for payload-only gating.
- Dashboard integrator/renderer exclusion is defensible for this PRD: the integrator consumes market-map-derived symbol data at `cuttingboard/delivery/dashboard_renderer.py:1125-1158` and `cuttingboard/delivery/dashboard_integrator.py:40-110`; PRD-162 should not reopen that layer unless AC1 remains explicitly market-map-based.
