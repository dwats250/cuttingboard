WHY NOW
`pyproject.toml` declares `"ruff>=0.4.0"` — unpinned — and the repo has NO
ruff configuration at all: no `[tool.ruff]` section, no `ruff.toml`, no
`.ruff.toml`. The lint contract has therefore always been "whatever the
installed ruff version happens to default to". On 2026-07-24 ruff 0.16.0
became the resolved `>=0.4.0` and expanded its default rule set; CI went
red on code nobody had touched.

Measured, not inferred — identical trees, opposite verdicts:

| Tree | ruff 0.15.8 | ruff 0.16.0 |
|---|---|---|
| pristine `origin/main` (`15b2819`) | All checks passed | **1112 errors** |
| PR #166 head | All checks passed | **1113 errors** |

`main`'s own last recorded CI run (`15b2819`, 2026-07-23T05:51Z) is GREEN,
as are the 30 most recent push-to-main runs. Nothing in `main` changed —
only the ruff version resolved at `pip install -e ".[dev]"` time. The
newly-firing rules are `UP045`, `UP017`, `DTZ001`, `BLE001`, `FURB162`.

This blocks EVERY merge, not one: `ci.yml` runs on `pull_request` and on
`push: branches: [main]`, and the `test` check is a required gate.

This is PRD-198 invariant 6 — "Pin identities that matter... dependency →
declared AND locked. A movable identity changes behavior with no diff."

SAME CLASS AND SAME FILE AS PRD-264 (Dustin, 2026-07-24). PRD-264 closed the
PYTHONPATH-shadowing trap, which was possible precisely because
`pyproject.toml` had no `[tool.pytest.ini_options]` section — an implicit,
movable tool contract. This is the identical shape one section over. Fix it
at the config layer, not by chasing the version.

MAX EXPECTED DELTA

GOAL
The repo's lint contract is explicit and version-independent: `ruff check`
produces the same verdict regardless of which permitted ruff version is
installed, and that verdict reproduces today's green baseline exactly.

SCOPE
- Pin `ruff` to a known-good version range in `pyproject.toml`'s `dev`
  extra. AND add an explicit `[tool.ruff.lint]` rule set. Both, not either:
  the pin bounds which ruff runs, the rule set makes the contract explicit
  so a future pin widening cannot silently re-expand it.
- Rule set: `select = ["E4", "E7", "E9", "F"]` — ruff's historical default,
  which is exactly what the green baseline was enforcing. Verified to
  produce `All checks passed!` on pristine `origin/main` under BOTH ruff
  0.15.8 and 0.16.0, so it reproduces the baseline without suppressing
  anything that was previously enforced.
- Red-by-construction tests proving each half of the guard is load-bearing:
  removing EITHER of the pin's bounds turns R1 red, deleting the declared
  rule set turns R2 red, and pinned+configured reproduces the known-good
  baseline exactly under every permitted ruff version — where "every" is
  EXHAUSTIVE, not sampled (see R3). (Deliberately NOT claimed: that an
  unconfigured run fails — see R4 for why that is false at the pinned
  version, and why it is the reason both halves are needed.)

REQUIREMENTS

R1 — ruff is pinned to a bounded range
`pyproject.toml`'s `dev` extra constrains ruff to a range that excludes the
versions whose default rule set differs from the pinned baseline — in BOTH
directions. The declared range is `>=0.8.0,<0.16`, and it is the VERIFIED
range rather than a wider guess.

  LOWER BOUND RAISED 0.4.0 -> 0.8.0 (2026-07-24). The original floor was
  never measured. All 121 releases in [0.4.0, 0.16.0) were then tested:
  releases 0.4.0 through 0.7.4 (34 of them) resolve E999 as a rule code, so
  ruff's `linter.rules.enabled` is 60 codes rather than the 59 in
  `BASELINE_RULES` — FLOOR 1 fails on a PRISTINE tree under a version the
  pin permitted. All 87 releases in [0.8.0, 0.16.0) match the baseline
  exactly and lint clean; the passing block is contiguous with no holes.
  Note the shape: `ruff check` was CLEAN on every one of the 121, so the
  defect was invisible to R3 and visible only to the rule-set floor. A
  declared range wider than the verified range claims coverage nobody
  measured — the same defect as an unbounded ceiling, at the other end.
  (Dustin's ruling, 2026-07-24: "If BASELINE_RULES reds under a permitted
  0.4.x, the pin is wrong, not the baseline.")

FAIL: the ruff specifier in `pyproject.toml` has no upper bound, has no
lower bound, or admits any version outside [0.8.0, 0.16.0). Mutation-
verified in both directions, including evasive forms: `>=0.4.0,<0.16` ->
RED; `>0.7.3,<0.16` -> RED; `<=0.16` -> RED; `<1` -> RED;
`>=0.4,<0.17,!=0.16.0,!=0.16.1` -> RED.

R2 — the lint rule set is declared explicitly
`pyproject.toml` contains a `[tool.ruff.lint]` table with an explicit
`select` list. The repo no longer relies on ruff's implicit defaults.
FAIL: no `[tool.ruff.lint]` table exists, or it exists without a `select`
key.

R3 — the declared rule set reproduces the green baseline exactly
Running `ruff check cuttingboard/ tests/` under the declared rule set
produces zero errors on the current tree, and produces zero errors under
every ruff version permitted by R1's range.

  "EVERY PERMITTED VERSION" IS EXHAUSTIVE, NOT SAMPLED. Earlier revisions
  made this universally-quantified claim on the strength of three data
  points (0.15.8, 0.15.22, and 0.16.0 — the last of which is outside the
  range), while the range spanned two years of releases. That is the
  sampling error this PRD rejects, applied to the PRD's own prose. It is
  now discharged by enumeration: every release in the declared range was
  installed and run — 87 of 87 report `All checks passed!` AND resolve a
  rule set equal to `BASELINE_RULES`. The 34 excluded releases were the
  reason the floor moved (see R1).

FAIL: `ruff check cuttingboard/ tests/` reports any error under any
permitted ruff version, or any permitted version resolves a rule set other
than `BASELINE_RULES`.

R5 — MECHANISM-INDEPENDENT FLOOR (Dustin, 2026-07-24)
The enumerate-mechanisms approach produced FIVE holes across THREE review
rounds (chatgpt-codex-connector, PR #168): an unbounded-ceiling check that
`<1` satisfied; a `select`-only check that `ignore` walked past; a
`per-file-ignores` route that left `rules.enabled` untouched; an
`--isolated` substring test that was true with no config at all; and an
exit-code check that `exclude` reduced to zero linted files while still
exiting 0. That is not bad luck — it is the shape. A config surface is
open-ended, so a guard assembled from a list of known bypasses is provably
incomplete, and each fix only proved that ONE hole was closed. Note the
fifth in particular: `exclude` did not weaken the rule set at all, it
emptied the file list — a different dimension the earlier guards did not
even measure.

The exit is a ruling that closes the question, not a better enumeration.
The floor asserts the OUTCOME, so anything that empties the file list,
disables rules, or suppresses per-file fails WITHOUT anyone having
predicted the mechanism. Same move as PRD-269's fail-loud parser floor.
FAIL: a mutation that weakens the gate — disabling a rule, gutting a rule's
behavior, emptying or narrowing the linted file set, re-scoping a subtree,
or changing the command CI runs — leaves every test in
`tests/test_prd273_lint_baseline.py` green. Equivalently: any row of the
MUTATION TABLE below has an empty failing-test set.

FLOOR NUMBERING. There are SIX floors, numbered 0-5, each defined exactly
once. An earlier revision gave the number 3 to two different guards (the
allow-list and the ci.yml pin), which made every cross-reference in this
document ambiguous; the ci.yml guard is now FLOOR 5.
  FLOOR 0 — the gate actually flags known-bad code (canary)
  FLOOR 1 — the effective rule set equals the pinned baseline, exactly
  FLOOR 2 — every file ruff would discover under the targets is linted
  FLOOR 3 — the root config contains ONLY sanctioned keys (allow-list)
  FLOOR 4 — exactly one ruff configuration exists, repo-wide
  FLOOR 5 — the pinned command is what CI actually runs

THE STRUCTURAL EXIT (round 5) — ALLOW-LIST, NOT DENY-LIST
Rounds 1-4 all failed the same way: every guard named the bypasses it knew
about, and each round found one it did not. Round 5 made the theorem
explicit — `pyflakes.allowed-unused-imports` and, before it,
`dummy-variable-rgx`, are two of several HUNDRED ruff settings, with more
each release. A deny-set over that surface is provably incomplete; no amount
of care closes it, which is why five rounds of careful work did not.

The exit is to invert the quantifier:

FLOOR 3 — the root config contains ONLY sanctioned keys (`lint.select`).
An allow-list is complete BY CONSTRUCTION: any unlisted key fails, including
options nobody has heard of. Verified — it catches both `dummy-variable-rgx`
and `pyflakes.allowed-unused-imports` without either being enumerated.

FLOOR 4 — exactly one ruff configuration exists, repo-wide, found by walking
the filesystem for `ruff.toml` / `.ruff.toml` / any other `[tool.ruff]`.
Filesystem enumeration is EXHAUSTIVE where settings-probing is a sample:
probing two files and generalising missed a nested config under `tests/`
AND one under `cuttingboard/delivery/`, both with every test green.

Together these close the configuration surface by construction: config may
come from exactly one file, and that file may contain exactly one key.
FLOOR 0 and FLOOR 2 remain as behavioral backstops for anything that
weakens the gate WITHOUT touching config.

FLOOR 0 — THE GATE ACTUALLY FLAGS KNOWN-BAD CODE (added round 4)
The config-inspection floors below are all proxies, and round 4 broke two of
them in ways no config inspection could catch: `dummy-variable-rgx = ".*"`
leaves F841 in the enabled set while making it ignore every variable
(behavior changed, codes identical), and a nested `tests/ruff.toml`
re-scopes a whole subtree while the probed file's settings are unchanged.
Both verified; all 14 tests passed under each.

Freezing the complete settings dump — the obvious answer, and what the
reviewer suggested — is NOT viable: the dump is not stable across the
pinned range (0.15.8 vs 0.15.22 differ by 7 lines of newly-added options),
so it would go red on benign patch bumps. Neither config-inspection route
closes the question.

So FLOOR 0 stops inspecting configuration entirely. A file containing real
violations (F401, F841, E731, F541) is written INSIDE the lint targets, so
every scoping mechanism applies to it, and ruff must actually report them.
Anything that disables a rule, guts its semantics, suppresses it per-file,
or re-scopes it via a nested config makes the canary go silent — whether or
not anyone predicted the mechanism. It asserts the OUTCOME with no
reference to config at all, which is the end of the "assert the resolved,
not the requested" invariant.

FLOOR 1 — the effective rule set equals the pinned baseline EXACTLY
(59 rule codes, frozen in `BASELINE_RULES`), AND ruff's resolved
`linter.per_file_ignores` is empty. Absolute, not relative: the expectation
is not derived from the config being checked, so narrowing `select` cannot
move both sides together. The per-file half is required because
`rules.enabled` is global and does not reflect per-file suppression.

FLOOR 2 — the run lints EVERY file ruff would discover under the CI
targets: ruff's resolved `--show-files` set equals filesystem discovery. A
run that lints zero files is a FAILURE, not a pass. Set equality (not
"non-empty") also catches partial erosion — excluding one directory fails
identically.

  The expected set is DERIVED FROM RUFF (`file_resolver.include` /
  `.exclude` / `.respect_gitignore`), not from a hand-listed glob tuple.
  That tuple diverged from ruff three times in three rounds — `*.pyi`/
  `*.ipynb`, then gitignored trees, then `**/pyproject.toml` — and every
  divergence was a FALSE POSITIVE: the guard reddening the merge gate on
  correct behavior. Three instances of one shape is the sampling error this
  PRD rejects, one level up; the fix is the same quantifier inversion FLOOR
  3 applies to config keys. Verified: a benign nested
  `tests/_probe_nested/pyproject.toml` (no `[tool.ruff]`) redded the old
  guard and leaves the current one green, while the same path carrying
  `[tool.ruff]` still fails FLOOR 4.
  RESIDUAL: the pattern LIST is derived; glob MATCHING is still
  reimplemented (`rglob` + `fnmatch`). A ruff release changing glob
  semantics rather than its pattern list would still diverge.

FLOOR 5 — the floor measures the command CI actually runs: `LINT_TARGETS`
is asserted against `.github/workflows/ci.yml`'s own parsed structure, so
the floor cannot silently guard an invocation nobody uses. It also asserts
the workflow still TRIGGERS on `pull_request` without a narrowing filter,
and that neither the job nor the step declares `continue-on-error` at all
— absent, not merely "not literally `true`", because GitHub permits
expression forms (`${{ matrix.experimental }}`) that YAML leaves as an

unrelated `ruff format --check` step does not red the gate.
