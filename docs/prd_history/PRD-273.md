WHY NOW
`pyproject.toml` declares `"ruff>=0.4.0"` ŌĆö unpinned ŌĆö and the repo has NO
ruff configuration at all: no `[tool.ruff]` section, no `ruff.toml`, no
`.ruff.toml`. The lint contract has therefore always been "whatever the
installed ruff version happens to default to". On 2026-07-24 ruff 0.16.0
became the resolved `>=0.4.0` and expanded its default rule set; CI went
red on code nobody had touched.

Measured, not inferred ŌĆö identical trees, opposite verdicts:

| Tree | ruff 0.15.8 | ruff 0.16.0 |
|---|---|---|
| pristine `origin/main` (`15b2819`) | All checks passed | **1112 errors** |
| PR #166 head | All checks passed | **1113 errors** |

`main`'s own last recorded CI run (`15b2819`, 2026-07-23T05:51Z) is GREEN,
as are the 30 most recent push-to-main runs. Nothing in `main` changed ŌĆö
only the ruff version resolved at `pip install -e ".[dev]"` time. The
newly-firing rules are `UP045`, `UP017`, `DTZ001`, `BLE001`, `FURB162`.

This blocks EVERY merge, not one: `ci.yml` runs on `pull_request` and on
`push: branches: [main]`, and the `test` check is a required gate.

This is PRD-198 invariant 6 ŌĆö "Pin identities that matter... dependency ŌåÆ
declared AND locked. A movable identity changes behavior with no diff."

GOAL
The repo's lint contract is explicit and version-independent: `ruff check`
produces the same verdict regardless of which permitted ruff version is
installed, and that verdict reproduces today's green baseline exactly.

SCOPE
- Pin `ruff` to a known-good version range in `pyproject.toml`'s `dev`
  extra. AND add an explicit `[tool.ruff.lint]` rule set. Both, not either:
  the pin bounds which ruff runs, the rule set makes the contract explicit
  so a future pin widening cannot silently re-expand it.
- Rule set: `select = ["E4", "E7", "E9", "F"]` ŌĆö ruff's historical default,
  which is exactly what the green baseline was enforcing. Verified to
  produce `All checks passed!` on pristine `origin/main` under BOTH ruff
  0.15.8 and 0.16.0, so it reproduces the baseline without suppressing
  anything that was previously enforced.
- Red-by-construction tests proving each half of the guard is load-bearing:
  removing the pin's upper bound turns R1 red, deleting the declared rule
  set turns R2 red, and pinned+configured reproduces the known-good baseline
  exactly under every permitted ruff version. (Deliberately NOT claimed:
  that an unconfigured run fails ŌĆö see R4 for why that is false at the
  pinned version, and why it is the reason both halves are needed.)

OUT OF SCOPE

R5 ŌĆö MECHANISM-INDEPENDENT FLOOR (Dustin, 2026-07-24)
The enumerate-mechanisms approach produced FIVE holes across THREE review
rounds (chatgpt-codex-connector, PR #168): an unbounded-ceiling check that
`<1` satisfied; a `select`-only check that `ignore` walked past; a
`per-file-ignores` route that left `rules.enabled` untouched; an
`--isolated` substring test that was true with no config at all; and an
exit-code check that `exclude` reduced to zero linted files while still
exiting 0. That is not bad luck ŌĆö it is the shape. A config surface is
open-ended, so a guard assembled from a list of known bypasses is provably
incomplete, and each fix only proved that ONE hole was closed. Note the
fifth in particular: `exclude` did not weaken the rule set at all, it
emptied the file list ŌĆö a different dimension the earlier guards did not
even measure.

The exit is a ruling that closes the question, not a better enumeration.
The floor asserts the OUTCOME, so anything that empties the file list,
disables rules, or suppresses per-file fails WITHOUT anyone having
predicted the mechanism. Same move as PRD-269's fail-loud parser floor.

SOUNDNESS CHECKS ON THE FLOORS (Dustin, 2026-07-24)
Two checks on whether FLOOR 3 and FLOOR 4 are sound AS CONSTRUCTED ŌĆö not a
hunt for another mechanism. The value of the quantifier inversion is that
its assumptions are checkable, so they were checked.

CHECK 1 ŌĆö FLOOR 4's domain. RESULT: it was exhaustive over the WRONG
domain; now narrowed and the gap closed.
  COVERS: ruff configuration FILES ŌĆö `ruff.toml`, `.ruff.toml`, and
  `[tool.ruff]` in any `pyproject.toml`, found by walking the tree. This
  part is genuinely exhaustive; a config file has to exist somewhere.
  DOES NOT COVER: config supplied outside the filesystem. Ruff accepts
  `--config`, `--isolated`, `--select`, `--ignore`, `--extend-select`,
  `--exclude`, `--ignore-noqa` on the command line, none of which appear as
  a file. VERIFIED UNGUARDED at the time of the check: appending
  `--ignore F401` to CI's ruff command left all 17 tests passing.
  ENVIRONMENT: checked and found NOT to be a config channel ŌĆö ruff's only
  documented env vars are `RUFF_CACHE_DIR`, `RUFF_NO_CACHE`,
  `RUFF_OUTPUT_FILE`, `RUFF_OUTPUT_FORMAT`, which affect caching and output
  formatting, not rule selection or file discovery.
  CLOSED BY: pinning CI's ruff invocation to an EXACT line match rather
  than a substring. A substring check was what let the appended flag
  through. Mutation-verified: appended `--ignore F401` -> RED.
  RESIDUAL: this pins the command CI runs. A human running ruff locally
  with flags is not guarded and is not meant to be ŌĆö the gate is CI.

CHECK 2 ŌĆö FLOOR 3's load-bearing claim, that `lint.select` (the one
sanctioned key) cannot itself weaken the gate. RESULT: VERIFIED CLOSURE,
not a limitation.
  Falsified rather than asserted: `select` reduced from
  `["E4","E7","E9","F"]` (59 rules) to `["E4"]` (2 rules) -> caught by
  THREE independent guards (FLOOR 1 rule-set equality, FLOOR 0 canary, and
  the R2 exact-list pin).
  Stronger case, designed so the canary could not catch it: `select`
  reduced to exactly the four rules the canary exercises
  (`["F401","F841","E731","F541"]`) -> still RED, via FLOOR 1's ABSOLUTE
  equality against the 59-code baseline. FLOOR 1 does not derive its
  expectation from the config it checks, so no reduction of `select` can
  move both sides together.
  The one permitted key is therefore guarded, and by a floor independent of
  it.

THE STRUCTURAL EXIT (round 5) ŌĆö ALLOW-LIST, NOT DENY-LIST
Rounds 1-4 all failed the same way: every guard named the bypasses it knew
about, and each round found one it did not. Round 5 made the theorem
explicit ŌĆö `pyflakes.allowed-unused-imports` and, before it,
`dummy-variable-rgx`, are two of several HUNDRED ruff settings, with more
each release. A deny-set over that surface is provably incomplete; no amount
of care closes it, which is why five rounds of careful work did not.

The exit is to invert the quantifier:

FLOOR 3 ŌĆö the root config contains ONLY sanctioned keys (`lint.select`).
An allow-list is complete BY CONSTRUCTION: any unlisted key fails, including
options nobody has heard of. Verified ŌĆö it catches both `dummy-variable-rgx`
and `pyflakes.allowed-unused-imports` without either being enumerated.

FLOOR 4 ŌĆö exactly one ruff configuration exists, repo-wide, found by walking
the filesystem for `ruff.toml` / `.ruff.toml` / any other `[tool.ruff]`.
Filesystem enumeration is EXHAUSTIVE where settings-probing is a sample:
probing two files and generalising missed a nested config under `tests/`
AND one under `cuttingboard/delivery/`, both with every test green.

Together these close the configuration surface by construction: config may
come from exactly one file, and that file may contain exactly one key.
FLOOR 0 and FLOOR 2 remain as behavioral backstops for anything that
weakens the gate WITHOUT touching config.

FLOOR 0 ŌĆö THE GATE ACTUALLY FLAGS KNOWN-BAD CODE (added round 4)
The config-inspection floors below are all proxies, and round 4 broke two of
them in ways no config inspection could catch: `dummy-variable-rgx = ".*"`
leaves F841 in the enabled set while making it ignore every variable
(behavior changed, codes identical), and a nested `tests/ruff.toml`
re-scopes a whole subtree while the probed file's settings are unchanged.
Both verified; all 14 tests passed under each.

Freezing the complete settings dump ŌĆö the obvious answer, and what the
reviewer suggested ŌĆö is NOT viable: the dump is not stable across the
pinned range (0.15.8 vs 0.15.22 differ by 7 lines of newly-added options),
so it would go red on benign patch bumps. Neither config-inspection route
closes the question.

So FLOOR 0 stops inspecting configuration entirely. A file containing real
violations (F401, F841, E731, F541) is written INSIDE the lint targets, so
every scoping mechanism applies to it, and ruff must actually report them.
Anything that disables a rule, guts its semantics, suppresses it per-file,
or re-scopes it via a nested config makes the canary go silent ŌĆö whether or
not anyone predicted the mechanism. It asserts the OUTCOME with no
reference to config at all, which is the end of the "assert the resolved,
not the requested" invariant.

FLOOR 1 ŌĆö the effective rule set equals the pinned baseline EXACTLY
(59 rule codes, frozen in `BASELINE_RULES`), AND ruff's resolved
`linter.per_file_ignores` is empty. Absolute, not relative: the expectation
is not derived from the config being checked, so narrowing `select` cannot
move both sides together. The per-file half is required because
`rules.enabled` is global and does not reflect per-file suppression.

FLOOR 2 ŌĆö the run lints EVERY `.py` file under the CI targets: ruff's
resolved `--show-files` set equals filesystem discovery. A run that lints
zero files is a FAILURE, not a pass. Set equality (not "non-empty") also
catches partial erosion ŌĆö excluding one directory fails identically.

FLOOR 3 ŌĆö the floor measures the command CI actually runs: `LINT_TARGETS`
is asserted against `.github/workflows/ci.yml`'s own text, so the floor
cannot silently guard an invocation nobody uses.

MUTATION-VERIFIED AGAINST BOTH KNOWN BYPASS SHAPES, and against three
mechanisms nobody enumerated:
  rule-set weakening   `ignore = ["F401"]`            -> FLOOR 1 RED
  file-list emptying   `exclude` both targets         -> FLOOR 2 RED
  rule-set WIDENING    `extend-select = ["UP"]`       -> FLOOR 1 RED
  partial erosion      `extend-exclude = ["tests/**"]`-> FLOOR 2 RED (64/173)
  per-file suppression `per-file-ignores = {...}`     -> FLOOR 1 + 0 RED
  BEHAVIOR gutting     `dummy-variable-rgx = ".*"`    -> FLOOR 0 RED (only)
  nested re-scoping    `tests/ruff.toml`              -> FLOOR 0 + 2 RED

Note the two round-4 shapes: `dummy-variable-rgx` is caught by FLOOR 0 and
by NOTHING else, because every other guard compares rule identity rather
than rule behavior. `exclude` is the mirror image ŌĆö FLOOR 0 does not catch
it (invoking ruff on an explicit path bypasses `exclude`), FLOOR 2 does.
The two floors are complementary, not redundant.

ROUND-4 COUNT: eight confirmed holes across four rounds. The floor added in
round 3 was itself built on two proxies ŌĆö one probed file generalised to
all, and rule-code identity treated as rule behavior. That is the same
error one level up, which is why FLOOR 0 refuses to inspect config at all.

WHAT THE FLOOR DOES NOT COVER ŌĆö stated plainly rather than implied:
1. **Source-level `# noqa` comments.** Verified: adding
   `import json  # noqa: F401` to a linted file leaves ruff green and all
   14 tests passing. The floor guards the CONFIGURATION surface, not the
   source-annotation surface. Judged acceptable and NOT patched, on a
   different basis than the others: a `noqa` appears in the source diff at
   the point of use and is visible to ordinary code review, whereas a
   config change silently re-scopes every file at once. Enumerating it
   would be the very move this ruling rejects.
2. **Edits to the ruff step in `ci.yml` beyond its target list** ŌĆö e.g.
   adding `--exit-zero`, or deleting the step. FLOOR 3 pins the command's
   substring, not the absence of extra flags.
3. **A ruff outside the pin being installed regardless of the specifier.**
   Partially covered by the installed-version assertion, which checks the
   resolved interpreter's ruff rather than the declared intent.

WHAT IS **NOT** CLAIMED ŌĆö and why:
This PRD does NOT assert that an unconfigured run fails. Under the pinned
ruff (<0.16) the implicit defaults COINCIDE with the declared set, so
`ruff check --isolated` is also clean. A test asserting "unconfigured
fails" would assert something false at the pinned version; an earlier draft
of R4 made exactly that claim and its test failed, correctly, on first run.
That coincidence is not an inconvenience to work around ŌĆö it is the whole
argument for Dustin's "both, not either" ruling: the pin's protection rests
on a version-specific accident that holds only until a release moves the
defaults, which is precisely what ruff 0.16.0 did. The explicit selection is
what survives that.
