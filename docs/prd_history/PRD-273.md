PRD-273 — Pin ruff and declare the lint rule set explicitly

Status: IN PROGRESS

LANE
HIGH-RISK

CLASS
INFRA

WHY NOW
`pyproject.toml` declares `"ruff>=0.4.0"` — unpinned — and the repo has NO
ruff configuration at all: no `[tool.ruff]` section, no `ruff.toml`, no
`.ruff.toml`. The lint contract has therefore always been "whatever the
installed ruff version happens to default to". On 2026-07-24 ruff 0.16.0
became the resolved `>=0.4.0` and expanded its default rule set; CI went
red on code nobody had touched.

Measured, not inferred — identical trees, opposite verdicts:

| Tree | ruff 0.15.8 | ruff 0.16.0 |
|---|---|---|
| pristine `origin/main` (`15b2819`) | All checks passed | **1112 errors** |
| PR #166 head | All checks passed | **1113 errors** |

`main`'s own last recorded CI run (`15b2819`, 2026-07-23T05:51Z) is GREEN,
as are the 30 most recent push-to-main runs. Nothing in `main` changed —
only the ruff version resolved at `pip install -e ".[dev]"` time. The
newly-firing rules are `UP045`, `UP017`, `DTZ001`, `BLE001`, `FURB162`.

This blocks EVERY merge, not one: `ci.yml` runs on `pull_request` and on
`push: branches: [main]`, and the `test` check is a required gate.

This is PRD-198 invariant 6 — "Pin identities that matter... dependency →
declared AND locked. A movable identity changes behavior with no diff."

SAME CLASS AND SAME FILE AS PRD-264 (Dustin, 2026-07-24). PRD-264 closed the
PYTHONPATH-shadowing trap, which was possible precisely because
`pyproject.toml` had no `[tool.pytest.ini_options]` section — an implicit,
movable tool contract. This is the identical shape one section over. Fix it
at the config layer, not by chasing the version.

MAX EXPECTED DELTA
Zero production-code lines. `pyproject.toml`: one dependency-specifier edit
plus one new `[tool.ruff.lint]` table (~5 lines). One new test file
(~60-80 lines) carrying the red tests.

GOAL
The repo's lint contract is explicit and version-independent: `ruff check`
produces the same verdict regardless of which permitted ruff version is
installed, and that verdict reproduces today's green baseline exactly.

SCOPE
- Pin `ruff` to a known-good version range in `pyproject.toml`'s `dev`
  extra. AND add an explicit `[tool.ruff.lint]` rule set. Both, not either:
  the pin bounds which ruff runs, the rule set makes the contract explicit
  so a future pin widening cannot silently re-expand it.
- Rule set: `select = ["E4", "E7", "E9", "F"]` — ruff's historical default,
  which is exactly what the green baseline was enforcing. Verified to
  produce `All checks passed!` on pristine `origin/main` under BOTH ruff
  0.15.8 and 0.16.0, so it reproduces the baseline without suppressing
  anything that was previously enforced.
- Red-by-construction tests proving each half of the guard is load-bearing:
  removing the pin's upper bound turns R1 red, deleting the declared rule
  set turns R2 red, and pinned+configured reproduces the known-good baseline
  exactly under every permitted ruff version. (Deliberately NOT claimed:
  that an unconfigured run fails — see R4 for why that is false at the
  pinned version, and why it is the reason both halves are needed.)

OUT OF SCOPE
- Fixing the 1112 violations the expanded rule set reports. 889 of them are
  auto-fixable and adopting the wider set is a genuine improvement, but it
  is a large mechanical diff across ~65 files that changes repo-wide
  convention (`timezone.utc` -> `datetime.UTC`). Recorded under FOLLOW-UP
  CANDIDATES; explicitly NOT queued (Dustin, 2026-07-24).
- Any production module, test fixture, or workflow file.
- Widening or narrowing the rule set beyond reproducing today's baseline.

REQUIREMENTS

R1 — ruff is pinned to a bounded range
`pyproject.toml`'s `dev` extra constrains ruff to a range that excludes the
versions whose default rule set differs from the pinned baseline.
FAIL: the ruff specifier in `pyproject.toml` has no upper bound, or admits
a version outside the verified range.

R2 — the lint rule set is declared explicitly
`pyproject.toml` contains a `[tool.ruff.lint]` table with an explicit
`select` list. The repo no longer relies on ruff's implicit defaults.
FAIL: no `[tool.ruff.lint]` table exists, or it exists without a `select`
key.

R3 — the declared rule set reproduces the green baseline exactly
Running `ruff check cuttingboard/ tests/` under the declared rule set
produces zero errors on the current tree, and produces zero errors under
every ruff version permitted by R1's range.
FAIL: `ruff check cuttingboard/ tests/` reports any error under any
permitted ruff version.

R4 — red by construction: each half of the guard fails when removed
Stated as what was actually verified, not as a stronger claim the tests do
not support (the PRD-263 R6 discipline).

WHAT IS VERIFIED:
(a) Removing the upper bound from the ruff specifier turns R1 RED.
(b) Deleting `[tool.ruff.lint] select` turns both R2 tests RED.
(c) Pinned + configured reproduces the known-good baseline EXACTLY:
    `ruff check cuttingboard/ tests/` reports zero errors under every ruff
    version tested — 0.15.8 (last version `main` was green under), 0.15.22
    (what the pin resolves to), and 0.16.0 (the version that broke CI).
(d) The declared selection is consequential, not vacuous: a deliberately
    wider set (`E4,E7,E9,F,UP,DTZ,BLE`) DOES report violations on this same
    tree, so the declared set draws a real boundary.
(e) The config is actually loaded by ruff, not merely present in the file
    (asserted via ruff's own resolved settings — invariant 2).

WHAT IS **NOT** CLAIMED — and why:
This PRD does NOT assert that an unconfigured run fails. Under the pinned
ruff (<0.16) the implicit defaults COINCIDE with the declared set, so
`ruff check --isolated` is also clean. A test asserting "unconfigured
fails" would assert something false at the pinned version; an earlier draft
of R4 made exactly that claim and its test failed, correctly, on first run.
That coincidence is not an inconvenience to work around — it is the whole
argument for Dustin's "both, not either" ruling: the pin's protection rests
on a version-specific accident that holds only until a release moves the
defaults, which is precisely what ruff 0.16.0 did. The explicit selection is
what survives that.

FAIL: any of (a), (b), (d) or (e) leaves its test green when the thing it
guards is removed; or (c)'s baseline reports a violation under any ruff
version permitted by R1.
