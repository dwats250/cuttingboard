GOAL
Suppress SHORT-direction trade candidates intraday during gap-down
opens until a "permission" condition is met. Permission is granted
when the symbol has either failed to reclaim its opening range low
(`failed_reclaim`) or has shown acceptance below it
(`acceptance_below_level`). Suppression is filter-based: blocked
candidates are popped from the candidate dict before qualification,
not flagged as BLOCK_TRADE downstream.

The feature exists because shorting a gap-down open without first
seeing the gap fail to reclaim is a high-loss-rate setup against
reflexive squeezes. Requiring failed-reclaim or acceptance evidence
filters out the worst of the squeeze-prone entries while still
permitting shorts once the downside thesis has been intraday-validated.

SCOPE (as-built)

Module: `cuttingboard/intraday_state_engine.py`

- `classify_gap(open_price, prev_close, threshold=0.0025) -> str`
  (`intraday_state_engine.py:182`). Returns `"DOWN"` if `gap_pct ≤
  -threshold`, `"UP"` if `gap_pct ≥ threshold`, `"FLAT"` otherwise.
  Threshold constant `_GAP_THRESHOLD = 0.0025` (0.25%) at
  `intraday_state_engine.py:42`.
- `downside_short_permission(context, intraday_state) -> bool`
  (`intraday_state_engine.py:242`). Pure function. Returns True if
  any of:
  - `context.gap_type != "DOWN"` (gate is gap-down-only; no
    restriction on flat or gap-up opens).
  - `intraday_state.failed_reclaim` is True (price tried to reclaim
    ORB low and closed back below).
  - `intraday_state.acceptance_below_level` is True (≥ 2 consecutive
    contiguous closes below ORB low).
  Returns False if `context.gap_type == "DOWN"` AND
  `intraday_state.phase == "OPEN"` (block during the first 5 minutes
  of the session) OR `or_low_broken` without failed-reclaim or
  acceptance evidence.
- `SessionContext` (frozen dataclass; carries `open_price`,
  `prev_close`, `gap_type`).
- `DownsidePermissionState` (frozen dataclass; carries `phase`,
  `or_low_broken`, `failed_reclaim`, `acceptance_below_level`).
- `count_consecutive_closes_below_level` and
  `detect_acceptance_below_level` helpers
  (`intraday_state_engine.py:208–238`); acceptance requires
  `_ACCEPTANCE_CLOSES_MIN = 2` consecutive closes within the expected
  bar interval (1 minute ± 1 minute tolerance via
  `_EXPECTED_BAR_INTERVAL` / `_BAR_INTERVAL_TOLERANCE`).
- `detect_failed_reclaim(high, close, level)`
  (`intraday_state_engine.py:204`). A failed reclaim requires the
  bar's high to reach the level AND its close to fall back below it.

Module: `cuttingboard/runtime.py`

- `_apply_intraday_short_permission(candidates, execution_quotes) ->
  tuple[dict, dict]` (`runtime.py:1205`). Iterates candidates,
  filtering by `direction == "SHORT"` only. For each SHORT candidate:
  fetches intraday bars via `fetch_intraday_bars(symbol)`, reconstructs
  previous close from the normalized quote, computes intraday state
  via `compute_intraday_state`, derives the downside-permission
  decision from the state, and pops the symbol from `filtered` when
  permission is False. Returns the filtered candidate dict plus a
  context dict recording per-symbol state-availability and
  permission decisions. Fails open: if intraday state is unavailable
  (missing bars, computation exception), the candidate is preserved
  and the context records `intraday_state_available: False`.

Call sites in `cuttingboard/runtime.py`:

- `runtime.py:489` — `_QUALIFY_ONLY_MODES` branch. Filters candidates
  before `qualify_all`.
- `runtime.py:518` — `_HOURLY_MODES` branch when posture is
  non-STAY_FLAT. Same filter, same position.
- `runtime.py:805` — main daily / live branch. Gated by
  `if mode != MODE_FIXTURE`, so fixture-mode runs skip the intraday
  filter (deterministic-fixture invariant).

OUT OF SCOPE

- Long-direction candidates are not affected by this gate. The
  filter checks `direction == "SHORT"` only.
- Fixture-mode runs at `runtime.py:805` skip the gate entirely. This
  is intentional — fixtures use canned data that doesn't represent
  real intraday bars.
- The gate operates pre-qualification. Suppressed candidates do not
  appear in `qual.excluded`, `qual.watchlist`, or `rejections[]` at
  any stage. They are silent in the contract artifact.
- No sidecar artifact is produced for gap-down suppression. The
  per-symbol context dict returned by `_apply_intraday_short_permission`
  is consumed by the daily-mode call site at `runtime.py:805`
  (`intraday_state_context`) but is not written to a separate file.

REQUIREMENTS (as-built behavior contract)

R1 — Gap classification is deterministic on `(open_price, prev_close)`.
`classify_gap(open_price, prev_close)` returns one of `{"DOWN", "UP",
"FLAT"}` based on `(open - prev_close) / prev_close` against
`_GAP_THRESHOLD = 0.0025`. Threshold is symmetric: same magnitude
applies to both directions.

R2 — Downside-short permission gates only gap-down opens.
`downside_short_permission` returns True unconditionally when
`context.gap_type != "DOWN"`. The gate is a no-op for flat opens and
for gap-up opens.

R3 — During the OPEN phase of a gap-down session, all shorts are
blocked.
`downside_short_permission` returns False when
`intraday_state.phase == "OPEN"`. The OPEN phase ends 5 minutes
after market open (`classify_phase(minutes_since_open)` returns
"OPEN" for the first 5 minutes; see `intraday_state_engine.py:196`).

R4 — Outside the OPEN phase, shorts require evidence the downside
thesis is intraday-validated.
After the OPEN phase, `downside_short_permission` returns True only
when one of `failed_reclaim` or `acceptance_below_level` is True. A
clean break of ORB low without either condition returns False.

R5 — Acceptance below level requires bar contiguity.
`acceptance_below_level` requires ≥ 2 consecutive closes below the
level AND those closes must be in contiguous bars (gap detection
via `_EXPECTED_BAR_INTERVAL + _BAR_INTERVAL_TOLERANCE`; see
`_acceptance_closes` at `intraday_state_engine.py:220–230`).
Non-contiguous closes do not satisfy acceptance.

R6 — Failed reclaim requires both the high and the close to confirm.
`detect_failed_reclaim(high, close, level)` returns True only when
`high ≥ level` (a reclaim attempt occurred) AND `close < level` (the
attempt failed).

R7 — Runtime suppression is filter-based, not flagging-based.
`_apply_intraday_short_permission` pops suppressed candidates from
the returned dict. Suppressed candidates do not flow into
`qualify_all`, do not appear in qualification outputs, and do not
appear in `rejections[]`. Audit visibility for gap-down suppression
lives in the returned `context` dict and in `logger.info` output
("SUPPRESSED %s: SHORT blocked pending downside permission").

R8 — Fail-open on missing intraday state.
When `fetch_intraday_bars(symbol)` returns None or an empty
DataFrame, when `_intraday_state_bars_from_df` produces no bars, or
when `compute_intraday_state` raises an exception, the candidate is
preserved (not suppressed). The context dict records
`{intraday_state_available: False}` for forensic visibility.

R9 — Fixture mode skips the gate.
At the main daily call site (`runtime.py:805`), the gate is invoked
only when `mode != MODE_FIXTURE`. Fixture-mode runs bypass intraday
filtering entirely.

INVALIDATION CONDITIONS

The gate would be wrong, no longer needed, or due for redesign if
any of the following becomes true:

- **The gap-down-fade thesis stops holding.** The gate assumes that
  shorting a clean break of ORB low without failed-reclaim or
  acceptance evidence is a worse setup than shorting after either
  condition fires. If post-hoc evaluation (PRD-047 / `evaluation.py`)
  shows OPEN-phase or clean-break shorts have equal or better
  performance than gated shorts on gap-down sessions, the gate is
  filtering trades that should run.
- **The gate becomes regularly the binding constraint.** If
  `_apply_intraday_short_permission` suppresses a meaningful fraction
  of intended short entries (say, > 30% over a meaningful window)
  AND the suppressed entries would have been profitable, the
  permission criteria are too strict and need recalibration.
- **The 0.25% gap threshold drifts off market reality.** If price
  moves at 0.25% are no longer distinguishable from intraday noise
  (e.g. an extended low-VIX regime where typical opens are >0.25%
  but not directionally meaningful), the threshold needs adjustment.
- **Acceptance contiguity check produces false negatives.** If the
  bar-interval tolerance (1 minute ± 1 minute) starts rejecting
  legitimate acceptance because of normal data delivery jitter, the
  contiguity rule needs widening or replacement.
- **Long-direction symmetry becomes worth adding.** Currently only
  SHORT candidates are gated. If a parallel argument applies for
  blocking longs during gap-up opens without acceptance-above-level
  evidence, the gate needs a symmetric path — at which point the
  function name and module structure deserve reconsideration.
- **Visibility into suppressed shorts becomes a behavioral
  requirement.** The gate is silent in `rejections[]`. If Dustin
  starts wanting to see "which shorts would have fired but didn't,"
  the suppression channel needs a surface (a new rejection stage, a
  sidecar field, or a notification annotation). The recon at
  `audits/recon-2026-05-22/gap-down-prd150-coupling.md § 4` records
  this as the current observability gap.

NOTES — retrospective discipline

This PRD is the first instance of the retrospective-documentation
pattern in cuttingboard. The pattern is intended for one specific
failure mode: a feature was built and merged without a PRD, and the
governance documents (VISION.md, PROJECT_STATE.md) now disagree
with the code. The resolution is to write the PRD that should have
existed when the feature was built, then update the governance
documents to match the code.

This is distinct from:

- A regular PRD (specifies prospective work).
- A PATCH PRD (corrects a defect in a prior PRD).
- A DEPRECATED PRD (proposal that won't be implemented).

The retrospective form has STATUS = COMPLETE on first write, no
COMMIT PLAN section (the work is already committed across many
commits), and an explicit NOTES section flagging the retrospective
nature so future readers don't mistake it for prospective work.

`docs/DECISIONS.md` 2026-05-22 entry records this as the precedent
for handling similar drift in the future: retrospective
documentation, not silent acknowledgment.

UPDATE — PRD-181: open-window fail-closed interaction (2026-06-14)

The 2026-06-12 gate recon found that this gate was inert from the
09:30 open through 09:45 ET. `cuttingboard/intraday_state_engine.py`
emits no intraday state before `_NOISE_END` (09:45 ET):
`compute_intraday_state` returns `None` for any bar whose ET time is
`< _NOISE_END`. `_apply_intraday_short_permission` historically fails
OPEN when state is unavailable, so PRD-151's OPEN-phase SHORT blocking
did nothing during exactly the window it was written to cover.

PRD-181 (ratified D-Q7) fixes this filter-side only. The gate now
takes the run's wall-clock ET (`now_et`) and, when intraday state is
unavailable AND `now_et` is within the opening window
`[_ORB_START 09:30, _NOISE_END 09:45) ET`, fails CLOSED for SHORT
candidates (removes them) instead of fail-open. Window bounds are read
from `intraday_state_engine` (`_ORB_START`, `_NOISE_END`), not
redefined. Outside the window the historical fail-open default is
unchanged; LONG candidates are never affected; and once state is
available (at/after 09:45) the existing
`_downside_permission_from_state` gating documented above applies
exactly as before. `intraday_state_engine.py` itself is unchanged.
See `docs/prd_history/PRD-181.md`.
