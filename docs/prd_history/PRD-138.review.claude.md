## Findings against each PRD-138 requirement

### R1 — Shared Layout Module Exists and Is Frozen

✓ Module is 69 LOC. `TapeSlot` and `TapeRow` are
`@dataclass(frozen=True)`. Slots use a tuple, not a list. The two
mapping constants are wrapped in `types.MappingProxyType` — stronger
than the PRD required, good defensive choice. `inspect.getmembers`-
based purity test in `test_macro_tape_layout.py:40-62` rejects any
public callable or unexpected export. Module has no I/O, no logging,
no `os.environ`, no imports from `dashboard_renderer` or
`notifications`. R1 satisfied.

One design note worth recording (not a defect): `TapeSlot.payload_key`
is `str | None`, with `None` reserved for tradable slots. This
collapses two slot kinds into one dataclass and introduces an implicit
invariant — only non-None payload_keys may be routed through
`_pct_arrow`. The renderer relies on this invariant by iterating
`MACRO_ROW_1` and `MACRO_ROW_2` (never `TRADABLES_ROW`) when computing
arrows. Acceptable, but the invariant is not asserted in code; a future
themed row that accidentally sets `payload_key=None` would silently
fall back to `_DASH` rather than raise. Out of scope for PRD-138.

### R2 — Ordering Source of Truth

✓ Both consumers import `MACRO_ROW_1`, `MACRO_ROW_2`, `TRADABLES_ROW`
from `macro_tape_layout` and iterate them. Local ordering tuples
`_TAPE_DRIVER_DEFS`, `_TAPE_SPOT_METAL_DEFS`, `_TAPE_MM_SYMBOLS`,
`_MACRO_TAPE_SYMBOLS`, `_TRADABLES_ORDER` are all removed. Manual R2
grep returns zero hits. R2 satisfied.

### R3 — Renderer Layout

✓ HTML emission order verified by reading the diff at
`dashboard_renderer.py:1751-` and by the prior live render run: macro
bias div → MACRO_ROW_1 div (`class="macro-spot-metals-row"`) → MACRO_ROW_2
div (`class="macro-drivers-row"`) → separator → tradables grid. Each
macro row is a single `<div>` with one `<span class="macro-tape-slot
tape-slot {dir}">` per slot.

**Cosmetic finding (advisory, not blocking):** the CSS class
`macro-spot-metals-row` is now applied to the row containing
`XAU, XAG, BTC`. The name was accurate under PRD-136 (XAU+XAG only)
but is mildly misleading after PRD-138 (BTC is not a spot metal). A
future themed-row PRD can rename to `macro-row-1` or similar without
risk; doing it now would expand CHANGE SURFACE into CSS, which is out
of scope for PRD-138's stated goal. Acceptable to defer.

### R4 — XAU/XAG Color Parity

✓ Renderer iterates `MACRO_ROW_1.slots` and `MACRO_ROW_2.slots`
uniformly through `_pct_arrow(block["change_pct"])`. The hardcoded
`tape-slot na` styling at the old `dashboard_renderer.py:1771` is
removed. Live render from the prior pipeline run confirmed XAU and
XAG carry directional CSS classes derived from current-run
`change_pct`. R4 satisfied.

### R5 — Notification Parity

✓ `_macro_tape_block` iterates `MACRO_ROW_1` then `MACRO_ROW_2`,
yielding XAU, XAG, BTC, VIX, DXY, 10Y, OIL when quotes are present.
`_tradables_block` iterates `TRADABLES_ROW.slots`, yielding SPY, QQQ,
GLD, GDX, SLV, XLE. Slots whose quote is missing are dropped (same
behavior as legacy). R5 satisfied.

**Advisory finding (not blocking):** `_fmt_level` in notifications now
dispatches by `label`. The label-based ladder is `10Y → .2f`, `BTC →
K-formatted`, **everything else → .1f**. The renderer's
`_format_tape_value` independently uses `XAG → .2f`. So XAG in the
notification body formats to one decimal (`88.1`) while the dashboard
formats to two decimals (`88.12`). This is a real divergence in
visible output formatting. Strictly, PRD-138 explicitly forbade
formatting helpers in the layout module ("pure data only"), so the
divergence cannot be fixed *via* the shared module without expanding
its purpose. A targeted follow-up PRD (or a small in-PRD-138
amendment) could either (a) align notifications to two decimals for
XAG, or (b) accept the difference (one-decimal is fine for a SMS-class
push body). Recommend deferring to a follow-up; flag for the closeout
PROJECT_STATE entry.

### R6 — Payload-Key vs Quote-Symbol Mapping

✓ Both mappings exist as `MappingProxyType` constants:
`MACRO_LABEL_TO_PAYLOAD_KEY` and `MACRO_PAYLOAD_KEY_TO_QUOTE_SYMBOL`.
Both are exhaustively populated by a comprehension over
`MACRO_ROW_1 ∪ MACRO_ROW_2`, guaranteeing parity with the row
definitions by construction (single source of truth). Tests at
`test_macro_tape_layout.py:27-37` assert exhaustiveness. Neither
consumer translates slot identity locally — both reach payload key
and quote symbol via `slot.payload_key` / `slot.quote_symbol` on the
iterated row. R6 satisfied.

### R7 — Observational-Only Boundary

✓ Diff inspection confirms no edits to runtime.py, regime.py,
qualification.py, sizing, execution_policy, market_map writers,
contract.py, or payload schema. Renderer changes are confined to the
macro-tape block + the macro-bias arithmetic. Macro-bias arithmetic
change is a notable internal cleanup: the legacy `tape_slots[:4]`
positional slice is replaced with a payload-key-based filter
(`frozenset({"volatility", "dollar", "rates", "bitcoin"})`). This is
**more robust** than the prior code (which would have silently
included the wrong slots after the row reorder) and produces
identical output for the legacy 4-driver set. R7 satisfied.

### R8 — Color Resolution Without Pipeline Cycle

✓ XAU/XAG colors derive from the current run's
`macro_drivers["gold"|"silver"]["change_pct"]`. No prior-snapshot
diffing introduced. The prior live render confirmed colors resolved on
the first run after the change, which is the strongest possible
empirical verification of R8.
