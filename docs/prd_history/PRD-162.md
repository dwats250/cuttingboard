# PRD-162: outcome / regime / market_map reconciliation

Lane: STANDARD / LOGIC
Status: COMPLETE
Author: Dustin
Filed: 2026-06-04

## Problem

The dashboard surfaces render mutually-contradictory state honestly but
incoherently — each field is individually faithful to its source, but the
combination presents impossible/misleading situations to a trader. **Scope:
actionable-trade reconciliation only** — the four contradictions below, all on
the verdict-engine outcome → payload `top_trades` actionable path (`market_map` is
**downstream** of this state and is not edited). (A fifth contradiction,
regime/permission *wording* — EXPANSION rendering "Stand down" — is split out to
**candidate PRD-163**; see Non-goals.)

- **Verdict-engine contradiction.** The verdict engine sets `outcome=TRADE`
  (→ "TRADE SETUP ACTIVE" via `_decision_title`) in states where `market_map`
  has no qualifying candidates ("No qualifying long setups" / STALE). Same
  contradiction shape as PRD-160, different code path (verdict engine, not the
  macro-bias arithmetic).
- **D1 — non-tradable symbols as trade-like candidates.** A symbol in
  `config.NON_TRADABLE_SYMBOLS` (e.g. `^VIX`) surfaces in `top_trades` with full
  sizing. Observed live in the PRD-157 fixture-gate report
  (`audits/recon-2026-05-24/prd-157-fixture-gate-report.md`).
- **D2 — sized-but-blocked candidates.** `position_size` presents as actionable
  while `decision_status=BLOCK_TRADE` and/or `size_multiplier=0.0`. A W4 report
  reading `position_size` verbatim would print contracts for a zero-sized,
  blocked trade.
- **D3 — populated top_trades under NO_TRADE.** `outcome=NO_TRADE` renders a
  populated, trade-like `top_trades` state.

Evidence: `audits/recon-2026-05-24/prd-157-fixture-gate-report.md` (D1/D2/D3),
the 2026-06-04 real-run inspection recorded in `docs/PROJECT_STATE.md` (the
verdict contradiction observed live in a published artifact). The regime/permission
wording contradiction from that inspection is split to **candidate PRD-163**.

## Background / not in dispute

Each field renders honestly today; the defect is the *absence of cross-field
gating*, not a single wrong value. This mirrors PRD-160 (which fixed the
macro-bias arithmetic) but on the verdict-engine outcome derivation and the payload
`top_trades` projection (`market_map` is downstream of these and is not edited).
Per the PRD-author disciplines (`CLAUDE.md`), the fix must enumerate every
downstream reader of any changed emission and confirm compatibility.

## FILES (LOCKED — proposed; from the 2026-06-04 pre-implementation grep sweep)

Pinned by the grep sweep and the Claude + Codex cross-review
(`PRD-162.review.claude.md`, `PRD-162.review.codex.md`). The verdict engine has
**two outcome derivations that can disagree**: `runtime.py:875-879`
(`outcome=TRADE` iff any decision has `status == ALLOW_TRADE` — no symbol or size
check) and `output.py:479-484` (`outcome=TRADE` iff payload `sections.top_trades`
is non-empty, where `top_trades` currently copies *all* candidates including
blocked / non-tradable ones).

The fix is **one actionable-trade semantic rule** — `symbol ∉ NON_TRADABLE_SYMBOLS
AND status == ALLOW_TRADE` (with a `size_multiplier > 0` defensive invariant; see
below) — applied through **two representation adapters**:
1. an **object adapter** over `TradeDecision` (`ticker`, `status`,
   `size_multiplier`) for the runtime outcome site, and
2. a **dict adapter** over the candidate dict (`symbol`, `decision_status`,
   `size_multiplier`) for the payload `top_trades` projection.

The rule is applied at exactly two places — `runtime.py:875` (outcome) and
`payload.py:40` (`top_trades` gate) — **both including the NON_TRADABLE
exclusion**, so the two outcome derivations agree. **`output.py` is NOT edited**:
its outcome reads the already-built payload (`output.py:464-465`), so it becomes
correct once `top_trades` is gated. **`contract.trade_candidates` is left intact** —
gating happens only on the payload `top_trades` projection, so the many
contract-candidate consumers (notifications, audit, premarket, overnight policy,
evaluation; see AC6) are unaffected. `market_map` is **downstream** of the
corrected outcome/payload state (built after outcome at `runtime.py:875-893`) and
is **not** an implementation target here.

(The regime/permission contradiction — `_regime_to_permission_verb` at
`renderer:1027` mapping EXPANSION → "Stand down" — is **out of scope**; candidate
PRD-163. `dashboard_renderer.py` and `dashboard_integrator.py` are therefore NOT in
this PRD's production FILES; they consume the now-correct data unchanged.)

Production:
- M `cuttingboard/trade_decision.py` — define the single actionable-trade rule
  (`symbol ∉ NON_TRADABLE_SYMBOLS AND status == ALLOW_TRADE`) plus the **object
  adapter** over `TradeDecision`. The `size_multiplier > 0` term is a **defensive
  invariant only**: under current `execution_policy` routing
  (`execution_policy.py:59-68,215-251`) `status == ALLOW_TRADE ⟹ size_multiplier > 0`,
  so it never changes today's behavior; it guards against future routing that could
  emit a zero-size allow. (AC1, AC3, AC5)
- M `cuttingboard/runtime.py` — outcome site (`runtime.py:875-879`): replace the
  inline `any(decision.status == ALLOW_TRADE ...)` with the object adapter so the
  outcome **excludes NON_TRADABLE-symbol allows** (targeted swap; **no broader
  `runtime.py` refactor**). (AC1)
- M `cuttingboard/delivery/payload.py` — `top_trades` assembly (`payload.py:40`,
  currently a straight copy of `trade_candidates`): gate via the **dict adapter** —
  exclude `symbol ∈ NON_TRADABLE_SYMBOLS` (D1) and `decision_status == BLOCK_TRADE`
  / `size_multiplier == 0.0` (D2); empty ⇒ idle (D3). **Only `sections.top_trades`
  is gated; `contract.trade_candidates` is untouched.** (AC2, AC3, AC4)

`cuttingboard/output.py` and `cuttingboard/contract.py` are **NOT** production
edits: `output.py`'s outcome is transitively correct once `top_trades` is gated
(it reads the built payload at `output.py:464-465`), and `contract.trade_candidates`

below only as a no-regression / schema test target.

### Interaction risk (must resolve in design)

PRD-161's gate test (`tests/test_prd161_sizing_gate_fixture.py`) deliberately
verifies sizing on a **blocked-but-surfaced** candidate (the `^VIX`-shaped path,
substituted with SPY). AC3 **suppresses** blocked candidates from the actionable
payload `top_trades`. Resolution (single-valued, decided): the PRD-161 assertions
(`:215-224` membership, `:244-250` sizing reads) **re-point to
`contract.trade_candidates`** — the surface that retains the blocked candidate with
its sizing. `contract.trade_candidates` is unchanged by this PRD, so the pinned
`position_size=2 / dollar_risk=150.0 / estimated_debit=75.0` values still hold.
`docs/PROJECT_STATE.md`'s "fixture gate CLEARED" note is updated to state the sizing
is verified on `contract.trade_candidates` (not payload `top_trades`). This
reconciliation is in scope for PRD-162.

## Acceptance criteria

- **AC1 — outcome agreement on the actionable set (data-layer).** The `runtime.py`
  outcome and the `render_report_from_payload` outcome are both `TRADE` iff ≥1
  *actionable* candidate exists (`symbol ∉ NON_TRADABLE_SYMBOLS AND
  status == ALLOW_TRADE`), else `NO_TRADE` (or `HALT`). FAIL: any input where the two
  derivations differ, OR where `outcome=TRADE` with zero actionable candidates.
  (`market_map` is downstream of this corrected state, not part of the gate; its
  display follows automatically — `market_map.py` is not edited.)
- **AC2 — D1: no non-tradable in top_trades.** No `top_trades` entry has a
  `symbol` in `config.NON_TRADABLE_SYMBOLS`. (Such symbols may still appear in an
  observational/macro surface, but not as a trade-like candidate.)
- **AC3 — D2: blocked ⇒ suppressed from actionable top_trades.** A candidate with
  `decision_status == BLOCK_TRADE` or `size_multiplier == 0.0` is **suppressed from**
  payload `sections.top_trades` (not relabeled there); its sizing remains available
  on `contract.trade_candidates`. FAIL: any `top_trades` entry has
  `decision_status == BLOCK_TRADE` or `size_multiplier == 0.0`.
- **AC4 — D3: no-actionable ⇒ empty actionable top_trades.** When no actionable
  candidate exists (`outcome=NO_TRADE`), payload `sections.top_trades` is empty and
  the existing renderer shows the idle/no-setup state. FAIL: `outcome=NO_TRADE` with
  a non-empty `sections.top_trades`.
- **AC5 — no regression in honest single-field rendering.** Where state is
  coherent (a genuine qualified ALLOW_TRADE candidate with `size_multiplier > 0`),
  `top_trades`, sizing, and the decision title render exactly as today (no
  suppression of valid trades).
- **AC6 — downstream-consumer audit (surface split).** The only changed emission is
  payload `sections.top_trades` (now actionable-only); `contract.trade_candidates` is
  unchanged. Verify: (a) payload `top_trades` consumers (`render_report_from_payload`
  `output.py:479-484`, transport counts `delivery/transport.py:47-57`, dashboard
  payload helpers) tolerate actionable-only / empty `top_trades`; (b)
  `contract.trade_candidates` consumers (notifications `output.py:769-783,853-916`;
  suppression `notifications/state.py:52-87`; premarket `reports/premarket.py:342-350`;
  overnight `overnight_policy.py:69-80`; audit `audit.py:168-229`; evaluation
  `evaluation.py:114-137`) still receive blocked / watchlist candidates where expected.
  FAIL: any consumer in (a) or (b) is not enumerated and confirmed compatible.

(Former AC5 — regime/permission wording agreement — is removed; it is split to
**candidate PRD-163** "Fix regime permission wording for EXPANSION posture".)
