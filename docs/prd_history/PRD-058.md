# PRD-058: Lifecycle Transition Notifications

## GOAL

Trigger compact notifications from existing `market_map` lifecycle metadata when a symbol has a meaningful lifecycle transition.

This PRD modifies notification builder logic only. It does not change lifecycle computation, dashboard rendering, trade decisions, qualification, policy, runtime persistence, or data ingestion.

---

## SCOPE

- Add notification-builder support for lifecycle transition alerts sourced from `market_map`.
- Emit alerts only for meaningful lifecycle transitions:
  - `UPGRADED` always
  - `NEW` only when current grade is `A+`, `A`, or `B`
  - `DOWNGRADED` only when previous grade was `A+`, `A`, or `B`
  - `REMOVED` only when previous grade was `A+`, `A`, or `B`
- Render compact, watch-friendly alert lines.
- Optionally render one deterministic reason line using existing fields only.
- Add focused tests for lifecycle notification selection, formatting, suppression, and deduplication.
- Update `docs/PRD_REGISTRY.md`.
- Create `docs/prd_history/PRD-058.md`.

---

## OUT OF SCOPE

- `cuttingboard/market_map_lifecycle.py` changes
- Runtime lifecycle logic changes
- Dashboard changes
- Trade decision logic
- Qualification or policy logic
- Execution / paper execution logic
- New data sources
- Price levels, R:R, stops, targets, or invented trade instructions
- AI commentary, subjective reasoning, or inferred explanations
- Timing-based alert randomness
- Cross-run notification persistence or historical alert state

## REQUIREMENTS

### R1 — Lifecycle alert input

Notification builder logic must accept an optional `market_map` dict and read only:

- `market_map["symbols"]`
- `market_map["removed_symbols"]`
- `symbol_entry["lifecycle"]`
- `symbol_entry["grade"]`
- `symbol_entry["direction"]`
- `symbol_entry["setup_state"]`
- `symbol_entry["trade_framing"]["downgrade"]`
- `symbol_entry["reason_for_grade"]`
- removed entry `symbol`
- removed entry `previous_grade`
- removed entry `grade_transition`

The builder must not call lifecycle computation, load files, fetch quotes, or derive new market data.

FAIL: Lifecycle notifications read any source outside the supplied `market_map`.
FAIL: Notification code imports or calls `market_map_lifecycle.py`.
FAIL: Notification code performs file I/O to load lifecycle data.

### R2 — Trigger selection

Emit lifecycle alert lines only for these cases:

- `grade_transition == "UPGRADED"`: always emit.
- `grade_transition == "NEW"`: emit only when current grade is `A+`, `A`, or `B`.
- `grade_transition == "DOWNGRADED"`: emit only when `lifecycle["previous_grade"]` is `A+`, `A`, or `B`.
- removed entry `grade_transition == "REMOVED"`: emit only when `previous_grade` is `A+`, `A`, or `B`.

High grades are exactly `A+`, `A`, and `B`.

FAIL: `UPGRADED` does not emit an alert line.
FAIL: `NEW` with current grade `C`, `D`, or `F` emits an alert line.
FAIL: `DOWNGRADED` from previous grade `C`, `D`, or `F` emits an alert line.
FAIL: `REMOVED` from previous grade `C`, `D`, or `F` emits an alert line.

### R3 — Suppression

Do not emit lifecycle alert lines for:

- `UNCHANGED`
- `UNKNOWN`
- missing lifecycle blocks
- malformed lifecycle blocks
- low-grade `NEW`
- low-grade `DOWNGRADED`
- low-grade `REMOVED`

Malformed per-symbol lifecycle data must suppress only that symbol's lifecycle alert without crashing notification formatting. This does not require suppressing unrelated existing notification-builder validation errors.

FAIL: `UNCHANGED` emits an alert line.
FAIL: `UNKNOWN` emits an alert line.
FAIL: A symbol without `lifecycle` emits an alert line.
FAIL: Malformed lifecycle data crashes notification formatting.

### R4 — Alert line format

Lifecycle alert lines must be ASCII-safe and compact:

```text
09:42 NVDA LONG - UPGRADED TO A+
09:45 TSLA SHORT - DOWNGRADED TO B
10:05 META LONG - NEW (A)
10:20 GDX - REMOVED (prev: A)
```

Rules:

- Timestamp must be `HH:MM` and must use the same timestamp source and timezone convention already used by existing notification formatting. Do not introduce a new timestamp source.
- Symbol is uppercased.
- Direction is included for current symbols when present.
- Direction is omitted for removed symbols.
- Use ASCII hyphen separators only.
- `UPGRADED` format: `{HH:MM} {SYMBOL} {DIRECTION} - UPGRADED TO {current_grade}`
- `DOWNGRADED` format: `{HH:MM} {SYMBOL} {DIRECTION} - DOWNGRADED TO {current_grade}`
- `NEW` format: `{HH:MM} {SYMBOL} {DIRECTION} - NEW ({current_grade})`
- `REMOVED` format: `{HH:MM} {SYMBOL} - REMOVED (prev: {previous_grade})`

FAIL: Lifecycle alert output contains non-ASCII punctuation.
FAIL: Current-symbol alert omits direction when direction is present.
FAIL: Removed-symbol alert includes direction.
FAIL: Any alert line exceeds the compact format above with added commentary.

### R5 — Optional deterministic reason line

An alert may include one second line only when an existing field provides meaningful text. The reason source priority is:

1. `symbol_entry["trade_framing"]["downgrade"]`
2. `symbol_entry["setup_state"]`
3. `symbol_entry["reason_for_grade"]`

Rules:

- Use only one reason line per alert.
- Do not synthesize, paraphrase, score, or interpret the reason.
- Do not add a reason line when all allowed fields are empty, missing, or non-scalar.
- Removed-symbol alerts do not require a reason line unless the removed entry already carries one of the allowed fields.

FAIL: Reason text is invented or paraphrased.
FAIL: More than one reason line is emitted for a single lifecycle alert.
FAIL: Empty or missing reason fields produce placeholder text.

### R6 — Deduplication within one run

Do not emit duplicate lifecycle alert lines for the same transition in the same notification build.

Deduplication key:

```text
{symbol}|{grade_transition}|{previous_grade}|{current_grade}
```

For removed symbols:

```text
{symbol}|REMOVED|{previous_grade}|
```

FAIL: Duplicate symbol entries with the same transition emit duplicate alert lines.
FAIL: Removed duplicates emit duplicate alert lines.

### R7 — Notification integration

Lifecycle alert lines must be integrated in the notification builder without changing dashboard, lifecycle, runtime persistence, trade decision, qualification, or policy behavior.

If the implementation adds a helper, it must remain inside the notification layer and be called only from notification formatting paths.

FAIL: `cuttingboard/market_map_lifecycle.py` changes.
FAIL: `cuttingboard/delivery/dashboard_renderer.py` changes.
FAIL: Trade decision, qualification, or policy tests require fixture rewrites for this PRD.

### R8 — No regression

All existing notification tests must pass without loosening assertions. Existing alert formatting that does not include lifecycle data must remain unchanged.

FAIL: Existing notification output changes when `market_map` is absent.
FAIL: Any previously passing notification test is weakened or removed.
FAIL: Any previously passing test fails after this change.

---

## DATA FLOW

1. Notification caller provides existing `market_map` to notification builder.
2. Builder reads `market_map["symbols"]` and each symbol's existing `lifecycle` block.
3. Builder applies deterministic trigger rules from R2/R3.
4. Builder formats one compact alert line per qualifying transition.
5. Builder reads `market_map["removed_symbols"]` and applies the removed-symbol trigger rule.
6. Builder deduplicates lifecycle alert lines within the same formatting call.
7. Builder includes lifecycle alert lines in notification output without mutating `market_map`.

No lifecycle computation, file I/O, external data fetch, dashboard rendering, or trade-policy evaluation occurs in this flow.

---

## FAIL CONDITIONS

- Lifecycle notifications read any source outside supplied `market_map` → R1 FAIL
- Notification code imports or calls `market_map_lifecycle.py` → R1/R7 FAIL
- `UPGRADED` does not emit → R2 FAIL
- Low-grade `NEW`, `DOWNGRADED`, or `REMOVED` emits → R2/R3 FAIL
- `UNCHANGED`, `UNKNOWN`, missing lifecycle, or malformed lifecycle emits → R3 FAIL
- Malformed lifecycle data crashes formatting → R3 FAIL
- Alert line format differs from R4 → R4 FAIL
- Alert output contains non-ASCII punctuation → R4 FAIL
- Reason line is invented, paraphrased, duplicated, or placeholder text → R5 FAIL
- Duplicate transition emits duplicate lines in one build → R6 FAIL
- Dashboard, lifecycle computation, runtime persistence, trade decision, qualification, or policy behavior changes → R7 FAIL
- Existing notification output changes when `market_map` is absent → R8 FAIL
- Existing tests are weakened or fail → R8 FAIL
