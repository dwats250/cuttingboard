Fast-follow proposal — continuation-path credit-spread max-loss fix

Status: PROPOSAL — not started, no Gate A yet. Not a registered PRD (this
is a `.proposal.md` sidecar, allowlisted from the registry-gap check per
PRD-248; it deliberately carries no `docs/PRD_REGISTRY.md` row and claims
no PRD number until Dustin gives it a Gate A direction ruling).

ORIGIN
Surfaced during PRD-251 (A1a)'s Stage 0 sweep, 2026-07-10. PRD-251 fixes
credit-spread max-loss arithmetic on the standard Gate 8 path
(`cuttingboard/qualification.py::qualify_candidate`, the 11-gate flow).
This proposal is the same conceptual defect, reached through a second,
structurally independent door.

THE FINDING
`cuttingboard/qualification.py::_qualify_continuation_candidate` (lines
630-730) qualifies EXPANSION-regime continuation candidates via its own
sizing block (line 707: `spread_width = max(0.50, dm.atr14 * 0.05)`) —
an ATR-scaled debit estimate added in PRD-008 (commit 30b583a),
structurally unrelated to the strike-distance-based 30%-of-width
convention `cuttingboard/options.py` uses for the standard path.

This candidate's `direction` is hardcoded `"LONG"` (line 726). Its
resulting `QualificationResult` flows into
`cuttingboard/options.py::build_option_setups`, which calls
`_select_strategy(result.direction, sr.iv_environment)` (options.py:183)
the same way it does for every other candidate — so in ELEVATED_IV or
HIGH_IV, a continuation candidate becomes `BULL_PUT_SPREAD`, a CREDIT
strategy. Its qualify/reject decision and its dollar_risk, though, were
computed by the ATR proxy above, under the same latent assumption
PRD-251 is fixing elsewhere: that the estimated premium collected IS the
max loss, rather than max loss = width - premium.

WHY THIS IS NOT A DROP-IN OF PRD-251's FIX
PRD-251's correction (max loss = width - credit, i.e. 70% of width) is
computable directly because `width` (strike_distance: $5 ETF / $2.50
stock, `cuttingboard/options.py::_MAX_STRIKE_DIST_ETF`/`_STK`) is a real,
known quantity at the Gate 8 call site. The continuation path's ATR
proxy has NO such width term — it's a dollar value derived straight from
ATR14, with no strike-distance concept anywhere in its computation or
its PRD-008 history/comments. Applying "width - credit" here requires
first deciding what "width" MEANS for a continuation trade. That is a
modeling decision, not an arithmetic correction, and per
`audits/EXECUTION_DOCTRINE.md`'s HARD CONSTRAINT for this slice ("if the
continuation-path fix needs materially different logic... HALT and
report before building — that's evidence it deserves its own slice"),
Dustin ruled to descope it (2026-07-10, see docs/DECISIONS.md) rather
than fold it into PRD-251.

OPEN DESIGN QUESTION (needs its own Gate A before any build)
What should "width" mean for an ATR-based continuation candidate, so its
credit-strategy outcomes get a real max-loss correction? Candidate
options surfaced so far (not evaluated, not ranked):

1. Borrow options.py's existing strike-distance buckets ($5 ETF / $2.50
   stock) as the implied width for continuation candidates too; treat
   the current ATR-based value as the credit estimate; apply
   max_loss = width - credit the same shape as PRD-251, but now
   cross-referencing a constant from a different subsystem's convention
   that wasn't designed with continuation trades in mind.
2. A continuation-native width derivation (e.g. scale the implied width
   off ATR itself, independent of the ETF/stock buckets) — invents a new
   convention rather than reusing an existing one.
3. Leave the continuation path's proxy as a debit-only estimate and
   force continuation candidates that land on a credit strategy through
   a different, more conservative path (e.g. always MANUAL_CHECK or
   reject) rather than sizing them at all until real economics are
   known — sidesteps the modeling question by refusing to price it.

None of these is picked. This needs Dustin's direction before Stage 0 of
whatever PRD number gets assigned to it.

### 1. Two layers, not one — how max loss is actually estimated

**Layer 1 — qualification (`cuttingboard/qualification.py::
_qualify_continuation_candidate`, lines 707-733):**

```
spread_width = max(0.50, dm.atr14 * 0.05)
spread_cost = spread_width * 100
continuation_budget = config.ACCOUNT_EQUITY * config.CONTINUATION_MAX_RISK_PCT_PER_TRADE
max_contracts = floor(continuation_budget / spread_cost)
dollar_risk = max_contracts * spread_cost
```

`spread_width` is a pure function of `dm.atr14` (5% of the real,
Wilder-RMA ATR14, floored at $0.50/share) — no strike-distance, width,
DTE, or moneyness term anywhere in its computation. `QualificationResult.
max_contracts`/`.dollar_risk` are what this layer computes. Direction is
hardcoded `"LONG"` (line 742, in the `QualificationResult` this function
returns).

**Layer 2 — the options resize (`cuttingboard/options.py::
build_option_setups`, lines 207-268), the layer that actually determines
what gets rendered and audited:**

```
strike_distance = _MAX_STRIKE_DIST_ETF if symbol in _INDEX_ETFS else _MAX_STRIKE_DIST_STK   # $5.00 / $2.50, NOT ATR-derived
spread_width = candidate.spread_width if candidate is not None else _estimated_debit(strike_distance)   # ALWAYS = 0.30 * strike_distance -- see note below
effective_max_loss = (
    candidate.max_loss
    if (candidate is not None and candidate.max_loss is not None and result.entry_mode != ENTRY_MODE_CONTINUATION)
    else spread_width
)
risk_per_contract = effective_max_loss * 100
effective_risk = config.ACCOUNT_EQUITY * budget_pct * risk_modifier   # budget_pct = CONTINUATION_MAX_RISK_PCT_PER_TRADE for continuation results
raw_adjusted = floor(effective_risk / risk_per_contract)
final_contracts = max(1, min(result.max_contracts, raw_adjusted))
final_dollar_risk = final_contracts * risk_per_contract
```

Two facts fall directly out of reading this code, both confirmed by
direct execution (section 2):

1. **For continuation results specifically, `effective_max_loss` is
   NEVER the qualification layer's ATR-based `spread_width`, and NEVER
   PRD-251's correct `_max_loss_for_strategy` value.** The `result.
   entry_mode != ENTRY_MODE_CONTINUATION` guard means continuation
   results always fall to the `else spread_width` branch — and that
   `spread_width` is the OPTIONS-LAYER local variable (line 210), not
   the qualification layer's. Per the in-code comment at lines 240-252,
   this is deliberate: a co-existing direct-path candidate for the same
   symbol must never re-price a continuation result off the wrong
   candidate's economics.
2. **That local `spread_width` is always `_estimated_debit(strike_
   distance) = strike_distance * 0.30`, regardless of whether a
   `candidate` object exists.** `TradeCandidate.spread_width` (built in
   `_build_candidate`, `options.py:385-431`) is itself always
   `_estimated_debit(strike_distance)` — it has no `dm`/ATR term either.
   So whether or not a same-symbol direct candidate happens to be present
   in the `candidates` dict, line 210 resolves to the identical value:
   $0.75/share (stock bucket) or $1.50/share (ETF bucket) — **a fixed
   number that depends only on ETF-vs-stock classification, never on
   ATR14.**

The qualification layer's ATR-scaled `spread_width`/`dollar_risk`
therefore never reaches the trader or the audit record for a continuation
result. Its only surviving downstream effect is `result.max_contracts`,
used as one of two operands in `min(result.max_contracts, raw_adjusted)`
— a contract-count cap, not a dollar figure (quantified in sections 4 and
8). `_select_strategy(result.direction, sr.iv_environment)`
(`options.py:300-316`) still resolves `BULL_CALL_SPREAD` (debit,
LOW_IV/NORMAL_IV) or `BULL_PUT_SPREAD` (credit, ELEVATED_IV/HIGH_IV) the
same way for continuation as for any other candidate.

### 3. DEBIT strategies: no structural gap, at either layer

For `BULL_CALL_SPREAD` / `BEAR_PUT_SPREAD`, real max loss = debit paid.
At layer 1, `spread_width` (ATR-based) is charged as the max loss — no
conflation, gap 0. At layer 2, `effective_max_loss` = `_estimated_debit
(strike_distance)` (fixed $0.75/$1.50 per share) is also charged as the
max loss for a debit strategy — still no conflation, since debit strategy
max loss IS the debit paid by definition, regardless of which number is
used as the debit estimate. Confirmed by direct execution of
`build_option_setups` against real ATR14 for `PAAS`, `IWM`, `MSTR`,
`META` under `LOW_IV` (forcing `BULL_CALL_SPREAD`): `real_max_loss /
final_dollar_risk` = **1.000x** in every case tested, independent of
ATR14. (Whether $0.75/$1.50 per share is an *accurate* debit estimate
against a real option chain is a separate, unverified magnitude
question — no option-chain data exists in this repo to check it — but
there is no *structural* credit-vs-max-loss conflation on this branch,
at either layer, unlike the direct path's pre-PRD-251 bug.)

### 4. CREDIT strategies: quantified understatement, at the layer that matters

Layer 2's `effective_max_loss` for a continuation CREDIT result is
`_estimated_debit(strike_distance) = 0.30 * strike_distance` — the exact
same debit-proxy-as-max-loss conflation PRD-251 fixed on the direct path,
via `options.py::_max_loss_for_strategy`, which the codebase already
computes correctly for CREDIT strategies (`strike_distance -
_estimated_debit(strike_distance) = 0.70 * strike_distance`) but
deliberately never applies to continuation results (section 1, point 1).
This is not a borrowed benchmark from a different subsystem's convention
— it is a direct comparison between what the code charges and what the
code's own PRD-251-fixed formula, for the identical strategy, says the
real number is.

Confirmed by direct execution of `build_option_setups` (real
`QualificationResult` objects, `ELEVATED_IV` forcing `BULL_PUT_SPREAD`,
`risk_modifier=1.0`) against real ATR14 for all 16 tradable symbols:

| PAAS | 2.377  | 3 | 2 | $150.00 | $350.00 | 2.333x |

2.333x exactly — the ratio is also invariant to the correlation modifier.

This supersedes the qualification-layer-only figures the prior version
of this section reported (4x-9x, ATR-dependent) as the answer to "how
much does the system understate," because that layer's dollar figure is
discarded before anything is rendered or audited. The qualification
layer's ATR-based proxy is not irrelevant, though — see section 5 for
what it still controls.

### 5. What drives the error, and which dimension dominates

At layer 2 (the number actually charged and audited):

- **Strategy type is the ONLY driver of the dollar-per-contract error,
  and it is a strict binary switch.** 100% of the understatement is on
  the CREDIT side (2.333x, constant); DEBIT candidates carry zero
  structural gap (section 3, 1.000x, constant). There is no continuous
  driver at this layer — ATR14, ETF-vs-stock bucket, and correlation
  modifier all cancel out of the ratio (section 4).
- **DTE and moneyness are not inputs to either layer's sizing
  arithmetic.** No expiration date or strike-distance/delta term appears
  anywhere in `_qualify_continuation_candidate`'s sizing block or in
  layer 2's `effective_max_loss`/`risk_per_contract` computation;
  `build_option_setups` picks a DTE for contract *construction* only,
  never feeding back into either layer's dollar-risk figure.
- **Spread structure (put-side vs call-side) does not matter
  independently** — `BULL_PUT_SPREAD` and `BEAR_CALL_SPREAD` both resolve
  through identical `_max_loss_for_strategy` arithmetic at both layers;
  only the credit/debit axis (strategy type, above) matters.

ATR14 (layer 1's proxy) is not irrelevant to the system's behavior — it
still determines `result.max_contracts`, one of two operands `min()`ed
against layer 2's fixed contract cap (`raw_adjusted` = 2 for the stock
bucket, 1 for the ETF bucket, at `risk_modifier=1.0`). Table above shows
where this binds: at the ATR floor (`atr14 <= 10`), layer 1 always
computes `max_contracts=3` (independent of ETF/stock bucket, since layer
1 has no width concept at all), which exceeds layer 2's fixed cap in
every case — so layer 2's cap is what actually determines final contract
count for all 11 floor-regime symbols. Only past `atr14 ~ 19-20` (`TSLA`,
`META`, stock bucket) does layer 1's own count (1) drop below layer 2's
cap (2) and become the binding constraint instead — and even there, the
*dollar-per-contract* figure charged is still the fixed layer-2 value,
never layer 1's ATR-scaled figure. So ATR14 can shift which contract
count gets sized (a real, if narrow-banded, effect — see section 8's
`@$150` column for where `final contracts` changes), but it never
changes the ratio between what's charged and what's real for whichever
count is sized.

### 6. Bounded, or arbitrarily large?

At layer 2 — the number that matters — **constant, not a range**: 2.333x
for every CREDIT continuation result, verified across all 16 real
tradable symbols' full real ATR14 range, three `risk_modifier` values,
and both the current and hypothetical uncoupled budget (section 8). There
is no crossover, no narrowing, no breakdown at this layer, because
neither the charged figure nor the real figure depends on ATR14 at all —
both are pure multiples of `strike_distance`.

Layer 1's own ATR-vs-width comparison (what the prior version of this
document characterized as "the" answer) does have the bounded-then-
undefined shape originally described — per-share gap constant at $1.50
(stock)/$4.00 (ETF) while `atr14 <= 10`, narrowing and crossing zero at
`atr14 = 10 x width`, and going mathematically undefined past `atr14 = 20
x width` ($50 stock / $100 ETF; reachable only by `GC=F`/`BTC_USD`, both
`config.NON_TRADABLE_SYMBOLS`, never routed through option qualification)
— but that describes layer 1's internal arithmetic in isolation, not the
dollar figure the system actually charges. It remains true and worth
carrying forward for one reason: it is the shape of `result.max_contracts`,
which still interacts with layer 2's fixed cap to determine final
contract count (section 5).

### 7. Same direction as PRD-251's bug?

Yes — and more precisely than the prior version of this section stated
it. This is not merely "the same class" of issue reached through a
different formula; at layer 2 it is the literal, identical unfixed
arithmetic PRD-251 corrected on the direct path
(`_estimated_debit(strike_distance)` charged as max loss instead of
`_max_loss_for_strategy`'s `strike_distance - _estimated_debit
(strike_distance)`), reached via a code path
(`build_option_setups`'s `result.entry_mode != ENTRY_MODE_CONTINUATION`
guard) that deliberately excludes continuation results from the PRD-251
fix it applies to every other result. DECISIONS.md's 2026-07-10 entry
ruled that this class of issue (understating in the direction that lets
more risk through) warrants the decouple by default; section 4 confirms
this directly — real risk charged against the $150 continuation budget
is 2.333x what's rendered/audited, for any CREDIT-resolving continuation
candidate, at any ATR14. Confirmed, not refuted — same mechanism, not
just the same class.

uncoupled budget where BOTH call sites PRD-252 decoupled

`budget_pct`) are flipped to `MAX_RISK_PCT_PER_TRADE` (~$400.005) — i.e.
what "re-coupling" would actually mean end to end, not just at one site —
for a credit-strategy resolution (the branch that carries the gap):

| QQQ  | 15.836 | 1 | $150.00 | $350.00 | 2 | $300.00 | $700.00 |

The ratio (real / charged) is exactly 2.333x in every cell, at both

`effective_max_loss` would scale real exposure to $525-$875 per sized

decouple's protective effect (holding the continuation path at the

Unlike the picture the prior version of this section presented, there is
no ATR regime (e.g. near `META`'s crossover) where the gap narrows or the
decouple's effect becomes negligible — the 2.333x multiplier is constant
across the entire real observed range.

### 9. Out-of-scope finding surfaced during real-data validation

Section 2's historical replay found the real, unmodified
`_qualify_continuation_candidate` never once clears `NO_HOLD_CONFIRMATION`
across ~4,000 real symbol-days, even with breakout firing 39/249 times on
NVDA alone in isolation. Cause: `detect_continuation_breakout`'s
`breakout_level` is the max High over the 5 bars *up to and including*
yesterday (`df.iloc[-(n+1):-1]` on the frame ending today); the hold check
then asks whether *yesterday's own close* already exceeded that same
window's max (which includes yesterday's own High) — a self-referential
comparison that is satisfiable only in the degenerate case where
yesterday closed exactly at its own high and that high was the window
maximum. This appears inconsistent with the documented spec
(`docs/trade_qualification.md:323`: "Close 1 completed bar ago also held
above the breakout level," implying an *earlier* breakout level than the
one the current bar just set). This is a candidate explanation for why
the one captured production run with usable data and the full real-data
historical replay both show zero accepted continuation candidates ever —
95 captured runs are consistent with (not independent confirmation of)
this finding, since 94 of the 95 carry no continuation_audit data at all
(section 2, point 1); the load-bearing evidence is the ~4,000-symbol-day
replay, not the run-log count.

**Interaction with the layer-2 resize (section 1):** none, structurally.
The HOLD gate sits inside `_qualify_continuation_candidate`, upstream of
`build_option_setups` — a candidate that fails HOLD never becomes a
`qualified=True` result and never reaches the resize step at all. Given
zero real acceptances anywhere in the available data, section 4's
layer-2 finding is, like layer 1's original characterization, necessarily
a description of what the code *would* do if a continuation candidate
ever qualified, not of anything that has happened. Both out-of-scope
findings remain unrelated to each other's fix and unfixed here — flagged
for separate ticketing, not folded into this PRD's scope.
