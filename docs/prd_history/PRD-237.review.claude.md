SUMMARY
PRD-237 (master-plan J1) turns the pipeline output contract's shape into a
single importable artifact — five TypedDicts in a new leaf module
`cuttingboard/contract_types.py` — adopted as annotations in
`cuttingboard/contract.py` and `cuttingboard/delivery/payload.py`, with four
sync guards (`tests/test_contract_types.py`) making the types load-bearing in a
repo with no static type checker. This is a CONTRACT-class PRD whose entire
value is schema exactness, so this review re-derived the schema independently
and adversarially from every producer. The re-derivation found zero
mismatches: every declared field, type, and NotRequired placement corresponds
to what the producers actually write, the guards are non-vacuous and were
re-mutation-verified during this review, and the contract.py/payload.py diffs
are annotation-only as claimed.

## Independent schema re-derivation (R1, PRD-237.md:76-84)

Method: read every producer and every post-build mutation site, then compared
against the TypedDicts field by field.

**Top level — `PipelineContract` (16 required + NotRequired `outcome`).**
Verified against both builders: `build_pipeline_output_contract`
(cuttingboard/contract.py:119-147) and `build_error_contract`
(contract.py:159-200) each write exactly the 16 required keys
unconditionally, as return-dict literals. Value expressions confirm the
declared types: `schema_version`/`generated_at`/`status`/`timezone` are str
(`_iso_str` returns str; status validated against `_VALID_STATUSES`);
`generation_id`/`session_date`/`mode` are Optional (getattr defaults to None;
error path coalesces generation_id to str, compatible with Optional[str]);
`correlation`/`regime` are Optional[dict] (`_build_correlation`/
`_build_regime_block` return dict-or-None); the rest are dict/list builders.
An exhaustive sweep of `contract["<key>"] =` assignments across the package
found exactly three top-level injection sites, all writing declared keys:
`contract["outcome"]` at runtime/__init__.py:770 (finalize), :292 and :587
(error paths, `OUTCOME_HALT`), :1869 (hourly, `OUTCOME_NO_TRADE`) — all str,
matching `NotRequired[str]`. runtime/__init__.py:1870 re-assigns
`generation_id` on the hourly path — a value update to an already-required
key, not a shape injection, so the TypedDict is unaffected. No
`.update()`/`setdefault` writes touch contract keys anywhere in the package.

**`SystemState` (6 required + 5 NotRequired).** `_build_system_state`
(contract.py:256-264) writes exactly the 6 required keys plus `confidence`;
`build_error_contract`'s system_state block (contract.py:167-174) writes
exactly the 6 required keys and omits `confidence` — so `confidence` is
correctly NotRequired (present-in-normal, absent-in-error), and its
`Optional[float]` inner type matches `float(regime.confidence) if regime is
not None else None`. The four runtime injections are exactly the four
remaining NotRequired keys: `outcome` (str), `permission` (str — `_ss_perm`
is always a string literal from `_PERMISSION_LINES.get(..., default)` or the
halt string), `reason` (`.get("stay_flat_reason")` → Optional[str], matching
`NotRequired[Optional[str]]`), at runtime/__init__.py:776-778; and
`session_type` = `"SUNDAY_PREMARKET"` (str) on the MODE_SUNDAY path only, at
runtime/__init__.py:787. The Sunday path's :786 write is a re-assignment of
required `stay_flat_reason` (str ⊂ Optional[str]), not an injection. Every
`_safe_str(...)` producer maps to a declared Optional[str] (contract.py:459-462
confirms `_safe_str` returns `Optional[str]`).

**`ContractCandidate` (27 required + NotRequired `overnight_policy`).**
`_build_trade_candidates` (contract.py:340-373) writes exactly the 27 required
keys per candidate. Adversarial type spot-checks all pass: `position_size` is
`int(...) or None` → Optional[int] ✓; `dollar_risk` `float(...) or None` →
Optional[float] ✓; `entry/stop/target/risk_reward/size_multiplier/
estimated_debit` are unconditional `float(...)` → float ✓;
`enable_conditions` is `vis.get("enable_conditions", [])` and the sole

writes only `[]` or `[enable_str]` → list[str] ✓; `visibility_status`/
`visibility_reason` are Optional[str] (string constants / policy_reason / None,
and None when the symbol is absent from the map) ✓; `decision_status` is str
(TradeDecision validates against VALID_DECISION_STATUSES) ✓; `policy_allowed`
is bool (TradeDecision.__post_init__ coalesces None and rejects non-bool) ✓;
`decision_trace` matches `DecisionTrace` exactly — TradeDecision's
__post_init__ enforces exactly {stage, source, reason}, all non-empty str ✓;
`trigger` is always-None, documented in-line as retained-for-shape-stability
(dead-branch discipline satisfied) ✓. The only post-build candidate mutation
in the package is `candidate["overnight_policy"] = _evaluate_candidate(...)`
(cuttingboard/overnight_policy.py:75, EOD window only), and
`_evaluate_candidate` returns exactly `{"decision": str, "reason": str}`
(overnight_policy.py:147) — matching `OvernightPolicyDecision` and the
NotRequired placement.

**NotRequired placement matches the PRD's spec exactly** (PRD-237.md:78-80):
system_state outcome/permission/reason/session_type/confidence; candidate
overnight_policy; top-level outcome. Nothing more, nothing less. R1 PASS.

**Rot-to-always-green analysis.** Every guard asserts strict set equality
against either a live-built object (guards b/c/d build a real contract through
the real builders) or the runtime-enforced whitelist (guard a; the chain is
closed because `assert_valid_contract` enforces `SYSTEM_STATE_ALLOWED_KEYS`
against every contract at runtime, contract.py:586-590). None can pass
vacuously: the optional-key assertions name non-empty expected sets.

**PEP-563 caveat empirically confirmed** (the contract_types.py:34-36 comment).
I built a scratchpad copy of contract_types.py with
`from __future__ import annotations` re-added: on this repo's Python 3.11.15,
`SystemState.__optional_keys__` becomes empty (all 11 keys classified
required) and `PipelineContract.__optional_keys__` likewise empties. Guards
(a) is unaffected (`__annotations__` keys unchanged) but the required/optional
assertions in guards (b), (c), and the system_state required-keys guard all
fail loudly — the assertions compare against explicit non-empty optional sets,
so the failure mode is a red suite, not a vacuous green. The comment is
accurate and the guard design covers the regression. R2 PASS.
