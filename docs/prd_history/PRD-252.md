PRD-252 — Per-trade risk budget: raise effective cap from $150 to $400

Status: IN PROGRESS

LANE
HIGH-RISK

CLASS
EXECUTION

WHY NOW
PRD-251 corrected credit-spread max-loss arithmetic, shrinking how many credit-spread candidates clear the existing $150 per-trade risk cap. This is the separately-reviewable follow-on: a risk-tolerance decision (not a correctness fix) that raises the cap, decided independently per BUILD_PLAN.md decision 4.

MAX EXPECTED DELTA
<= 40 production lines across cuttingboard/config.py +
cuttingboard/qualification.py + cuttingboard/options.py combined (two
constants, one call-site swap, one entry_mode branch mirroring an
existing pattern in options.py — not a rewrite of any module). No new
files, no new external dependencies, no new modules.

GOAL
Raise the effective per-trade risk budget from $150 to ~$400 by moving
`config.MAX_RISK_PCT_PER_TRADE` (not `ACCOUNT_EQUITY`), while keeping the
EXPANSION-regime continuation path -- whose ATR-based debit proxy still
carries the same max-loss-understatement PRD-251 fixed on the direct path
-- frozen at the pre-raise $150 budget until its own fast-follow corrects
that proxy.

SCOPE
- `config.MAX_RISK_PCT_PER_TRADE`: 0.01 -> 0.026667 (effective budget
  $150.00 -> $400.005). Deliberate loosening, not a pure offset of
  PRD-251's corrected arithmetic (~0.023/$345) -- BUILD_PLAN.md decision 4
  frames this as a preference knob the operator holds well below
  personally, not a restoration target.
- A new `config.CONTINUATION_MAX_RISK_PCT_PER_TRADE = 0.01` constant,
  read at both sites where a continuation-path candidate is sized:
  `qualification.py::_qualify_continuation_candidate` (`continuation_budget`)
  and `options.py::build_option_setups`'s correlation-modifier recompute
  (`effective_risk`, branched on `result.entry_mode ==
  ENTRY_MODE_CONTINUATION`). Both sites must decouple -- the
  qualification.py site alone does not fully close the leak, because
  options.py's `min(result.max_contracts, raw_adjusted)` only masks it
  when the correlation `risk_modifier` stays near 1.0; at a low enough
  modifier, `raw_adjusted` (computed off the wrong, raised constant) can
  undercut the correctly-decoupled qualification-layer value while still
  exceeding what a fully-consistent $150-based figure would allow.
- Test-suite reconciliation for the raised default (see REQUIREMENTS
  R2/R3/R4/R5/R6/R6a for the full breakdown by category).
- A tracked-requirement note appended to
  `docs/prd_history/PRD-251.continuation-path.proposal.md`: whoever builds
  that fast-follow must validate the continuation path's sizing AT the
  raised budget when retiring `CONTINUATION_MAX_RISK_PCT_PER_TRADE`, not
  silently inherit it.

OUT OF SCOPE
- The continuation path's ATR-based proxy itself (still understates true
  max loss the same way PRD-251 fixed on the direct path) -- that
  correction is `docs/prd_history/PRD-251.continuation-path.proposal.md`'s
  fast-follow, not this PRD. This PRD only decouples the BUDGET the proxy
  is measured against; it does not correct the proxy.
- PRD-256/A1b's live chain-priced economics.
- Any change to Gate 8's max-loss arithmetic itself (PRD-251's territory,
  already COMPLETE) -- this PRD only changes the budget constant(s) that
  arithmetic is compared against.

REQUIREMENTS

R1 — Effective per-trade risk budget raised via MAX_RISK_PCT_PER_TRADE, not ACCOUNT_EQUITY
`config.MAX_RISK_PCT_PER_TRADE`: 0.01 -> 0.026667 (effective budget
$150.00 -> $400.005). `config.ACCOUNT_EQUITY` unchanged at 15000.0 --
it is a factual account-size input, not a risk-tolerance dial (verified:
every source consumer of these constants reads only their product, never
`ACCOUNT_EQUITY` alone, so this is a semantic choice, not a
consumer-divergence-avoidance one).

FAIL: `MAX_RISK_PCT_PER_TRADE != 0.026667`, or `ACCOUNT_EQUITY` changed
from 15000.0.

R2 — Default-value test states intent, not the rounding artifact
`tests/test_account_equity_sizing.py`'s product-equals-default assertion
(renamed `test_defaults_yield_400_dollar_effective_budget`, was
`test_defaults_preserve_target_dollar_risk_of_150`) asserts
`pytest.approx(400.0, abs=0.01)`. No exact-$400 constant is introduced
(would require reintroducing a dollar-budget knob PRD-157 retired).

FAIL: assertion pins the literal 400.005, or still asserts 150.0.

R3 — PRD-251-era arithmetic invariants stay budget-independent
`test_credit_bull_put_max_loss_is_width_minus_credit_prd251`,
`test_debit_bull_call_max_loss_unchanged_prd251`,
`test_max_loss_none_falls_back_to_spread_width_prd251`
(`tests/test_qualification.py`), and
`test_credit_strategy_no_candidate_fallback_stays_debit_proxy_prd251`
(`tests/test_phase5.py`) each locally monkeypatch
`config.ACCOUNT_EQUITY=15000.0` / `config.MAX_RISK_PCT_PER_TRADE=0.01`
(pinning the pre-PRD-252 $150 budget) instead of reading the live
default. Each proves a formula-selection invariant (which value drives
sizing, not how large the budget is) and its pass/fail outcome and
numeric assertions do not change -- only where the budget value comes
from changes.

FAIL: any of the four reads `MAX_RISK_PCT_PER_TRADE` / `ACCOUNT_EQUITY`
without monkeypatching -- i.e. is sensitive to the live production
default.

R4 — New PRD-252-owned regression on the real (unmocked) $400 budget
A new test, `test_credit_bull_put_clears_gate8_at_raised_budget_prd252`
(`tests/test_qualification.py`), proves the same $350/contract

at 1 contract / $350 `dollar_risk` under the real, unmocked config
default -- the intended consequence of raising the cap.

FAIL: no test asserts the real $400-budget sizing outcome on unmocked
config for the credit-spread case.

FAIL: any of the three numeric assertions still encodes the $150-budget-
derived value.

R6 — Continuation path derives budget from CONTINUATION_MAX_RISK_PCT_PER_TRADE at every sizing site
Confirmed by explicit verification (not assumed): a continuation-path
`QualificationResult` is sized at `qualification.py:725`
(`_qualify_continuation_candidate`) AND re-sized at `options.py:224`
(`build_option_setups`'s correlation-modifier recompute, which every
`QualificationResult` -- continuation or direct -- passes through). Both
sites must read `CONTINUATION_MAX_RISK_PCT_PER_TRADE`, not the raised
main constant. A new test,
`test_continuation_sizes_against_decoupled_budget_prd252`
(`tests/test_continuation_mode.py`), proves a BULL_PUT-shaped
continuation candidate sizes against $150 on LIVE, unmocked config;
paired with R4's direct-path test proving $400 under the same live
config, the two together prove the split.

FAIL: any continuation sizing path reads the raised main constant; or no
test pins the $150-vs-$400 split.

R6a — test_wide_spread_exceeds_max_fails_gate8 kept on live config, fixture rescaled
This test has no arithmetic invariant of its own to preserve (unlike
R3's group) -- its only content is the gate-rejection outcome, which must
track the REAL production budget, not a frozen one. Rescaled from
`spread_width=3.0` ($300, which clears the raised ~$400 budget at 1
contract and would silently flip the test's meaning if left pinned or
unchanged) to `spread_width=5.0` ($500, `floor(400.005/500)=0`). Stays on
live, unmocked config.

FAIL: pinned to a stale budget via monkeypatch, or the fixture no longer
exercises the zero-contract rejection path at the real production budget.
