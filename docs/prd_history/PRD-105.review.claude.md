## Minor Problems

**Problem 1:** R7 references `latest_hourly_run.json` but the only required field call-out
is for `latest_hourly_contract.json` (stay_flat_reason). The run summary artifact has
its own decision-quality content (run_id, generation_id, run_at_utc, posture, outcome
mirrors) but R7 does not enumerate any required fields for it.

- **Fix:** Either add a minimal required-field list for `latest_hourly_run.json` to R7
  (e.g., run_id, run_at_utc, posture, outcome), or split R7 into R7a (run.json fields)
  and R7b (contract.json fields including the stay_flat_reason divergence). Splitting
  is cleaner.

**Problem 2:** SCOPE's per-field metadata item 6 ("supports calibration review") allows
"yes / no / partial — with one-line reason". R3 does not require the one-line reason
for non-partial cases. An implementer could write `supports_calibration: yes` with no
justification, which weakens the calibration-readiness signal the map is supposed to
provide.

- **Fix:** Require the one-line reason on every value (yes / no / partial), not only
  on "partial". Update R3 and the FAIL condition accordingly.

**Problem 3:** R10's FAIL condition ("git status --short after implementation shows any
modified path other than docs/decision_quality_map.md and the registry/state
bookkeeping") is ambiguous about timing. If the registry/state bookkeeping is handled
by a separate process per CLAUDE.md, then during *this PRD's* implementation only
docs/decision_quality_map.md should be modified, with no bookkeeping touched yet.

- **Fix:** Tighten R10 FAIL to: "git status --short after implementation of this PRD
  shows any modified path other than docs/decision_quality_map.md. Registry and
  PROJECT_STATE bookkeeping is handled by a separate transition step and is out of
  scope for this PRD's diff." This matches the PROPOSED status and the user's
  instruction not to update PRD_REGISTRY.md or PROJECT_STATE.md yet.

**Problem 4:** R8 lists market_map.json fields as "grade tiers, watch zones, fib levels,
current_price" but does not require the implementer to enumerate every actual field
present in the artifact. By contrast, R4–R7 enumerate explicit field lists. This
asymmetry could allow an incomplete market_map section to pass.

- **Fix:** Either commit to a verified field list for market_map.json by reading
  market_map.py (slightly out of scope for the PRD review itself), or rephrase R8 to
  require that "every key present in a sample logs/market_map.json record is documented"
  with a grep-checkable validation step. The grep-based phrasing is preferable since
  it does not require pre-listing fields the implementer can verify.

## Risks

- **Scope creep risk:** low. R11's citation requirement and OUT OF SCOPE's exhaustive
  list together prevent the map from drifting into prescriptive territory. The Known
  Gaps section is bounded to the upstream-identified gaps.
- **Stale-on-arrival risk:** moderate. If audit.py, evaluation.py, or contract.py
  schemas change between PRD draft and implementation, the required-field lists will
  drift. Mitigation: implementation should re-verify each required field against
  current source before writing the map. No PRD change needed.
- **Implementer ambiguity risk:** low after the four Minor Problems above are
  addressed.
