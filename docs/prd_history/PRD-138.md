WHY NOW
PRD-136 added XAU/XAG to the dashboard with hardcoded neutral styling and
no notification counterpart; investigation after PRD-137 confirmed the
underlying payload already supplies `change_pct` for gold/silver and that
ordering tuples are already duplicated and divergent between
`dashboard_renderer.py` and `notifications/__init__.py`. PRD-138 fixes the
visual defect and resolves the ordering drift in one pass, before further
themed groupings (semis, AI infra, uranium, etc.) compound the duplication.

MAX EXPECTED DELTA
≤ 180 production LOC across 3 production files (new shared layout module,
renderer, notifications). Test additions excluded from ceiling. Exceeding
this halts implementation and triggers a PRD amendment.

GOAL
Establish a single shared macro-tape layout definition consumed by both
the dashboard renderer and the notification builder, and route XAU/XAG
through the same directional arrow/color path as BTC so spot-metals
render green/red from existing `change_pct` values.

SCOPE
- New module `cuttingboard/delivery/macro_tape_layout.py` defining three
  static, frozen row constants (`MACRO_ROW_1`, `MACRO_ROW_2`,
  `TRADABLES_ROW`) built from a minimal `TapeRow` dataclass.
- Renderer (`cuttingboard/delivery/dashboard_renderer.py`) imports and
  iterates the shared rows; removes local `_TAPE_DRIVER_DEFS`,
  `_TAPE_SPOT_METAL_DEFS`, `_TAPE_MM_SYMBOLS`.
- Renderer routes XAU/XAG through `_pct_arrow` and `_ARROW_CSS` exactly
  as BTC does today; hardcoded `tape-slot na` styling for spot metals is
  removed.
- Renderer emits rows in this visual order: MACRO_ROW_1 (XAU, XAG, BTC),
  MACRO_ROW_2 (VIX, DXY, 10Y, OIL), separator, TRADABLES_ROW
  (SPY, QQQ, GLD, GDX, SLV, XLE).
- Notifications (`cuttingboard/notifications/__init__.py`) consume the
  same `MACRO_ROW_1`/`MACRO_ROW_2`/`TRADABLES_ROW` and emit macro lines
  in the same order, adding OIL/XAU/XAG to the alert body.
- Tests covering layout parity, color routing, and ordering invariants.

OUT OF SCOPE
- Any new themed rows (semis, AI infra, Korea, EM growth, uranium,
  defense, robotics). PRD-138 establishes the primitive; thematic
  expansion is future-PRD work.
- Runtime-configurable layouts, plugin systems, registry loaders,
  external config-driven rows, dynamic discovery.
- Changes to regime logic, qualification, sizing, execution policy,
  market_map semantics, contract decision logic.
- Changes to ingestion, payload schema, contract payload keys, or any
  data source.
- Any tradables ordering beyond the corrected
  `SPY, QQQ, GLD, GDX, SLV, XLE` sequence (GLD→GDX gold→miners
  progression, SLV as higher-beta precious-metals expression, XLE as
  macro-energy tail). This is the canonical tradables ordering after
  PRD-138; the legacy `SPY, QQQ, GLD, SLV, XLE, GDX` order is retired.
- Coupling notifications to renderer HTML/CSS internals. Notifications
  consume row *semantics* (labels, payload keys, ordering) only.
- Pipeline cycle changes; XAU/XAG color resolves from current-run
  `change_pct` and does NOT require prior-snapshot diffing.

- `cuttingboard/delivery/dashboard_renderer.py` — macro-tape block only
  (the `# --- macro-tape ---` section). No changes to system-state,
  trend-structure, macro-pressure, qualification, or any other block.
  No public function signature changes.
- `cuttingboard/notifications/__init__.py` — `_MACRO_TAPE_SYMBOLS`,
  `_TRADABLES_ORDER`, `_macro_tape_block`, `_tradables_block`, and the
  helpers they depend on. No changes to focus tokens, blockers,
  confidence/regime lines, action label, or message assembly outside
  the macro/tradables blocks.
- `cuttingboard/delivery/macro_tape_layout.py` — new file. Pure
  semantic-definition layer: frozen dataclasses, row constants,
  ordering, and the label↔payload-key and payload-key↔raw-quote-symbol

  transformation helpers, business logic, runtime behavior, conditional
  row generation, I/O, logging, env/config reads, and imports from
  renderer or notifications. The module is consumed independently by
  renderer and notifications; it contains no consumer-specific code.

REQUIREMENTS

R1 — Shared Layout Module Exists and Is Frozen
A new module `cuttingboard/delivery/macro_tape_layout.py` defines:
  - `@dataclass(frozen=True) class TapeSlot` with fields `label: str`
    and `payload_key: str` (payload-key for macro_drivers blocks) OR
    `quote_symbol: str` (raw normalized-quote symbol) — exact shape
    decided at implementation but MUST be immutable and typed.
  - `@dataclass(frozen=True) class TapeRow` with fields `name: str` and
    `slots: tuple[TapeSlot, ...]`.
  - Module-level constants `MACRO_ROW_1`, `MACRO_ROW_2`, `TRADABLES_ROW`
    of type `TapeRow`, with the orderings specified in SCOPE.
  - No mutable module state. No registry. No dynamic discovery.
  - Module is PURE DATA / SEMANTIC STRUCTURE ONLY. Allowed contents:
    frozen dataclasses, row definitions, ordering constants,
    label↔payload-key mappings, payload-key↔raw-quote-symbol mappings.
    Forbidden: HTML helpers, renderer formatting logic, notification
    formatting logic, rendering utilities, transformation helpers,
    business logic, runtime behavior, conditional row generation.

FAIL: Module is missing, exports a mutable list, exposes a registry or
loader function, reads from any external source (env, file, config),
defines any function that returns formatted HTML or formatted
notification text, imports from `dashboard_renderer` or `notifications`,
or contains conditional row-construction logic that varies output
across calls with identical inputs.

R2 — Ordering Source of Truth
After PRD-138, neither `dashboard_renderer.py` nor
`notifications/__init__.py` contains a literal ordered tuple/list of
macro tape labels or tradables symbols. Both modules import their
ordering exclusively from `macro_tape_layout`.

FAIL: grep of the two consumer files finds any local ordering tuple
containing two or more of {VIX, DXY, 10Y, BTC, OIL, XAU, XAG} or of
{SPY, QQQ, GLD, GDX, SLV, XLE} outside an `import` statement or test
fixture.

R3 — Renderer Layout
The macro-tape block in `dashboard_renderer.py` emits HTML in this
order: macro bias, MACRO_ROW_1 (XAU, XAG, BTC), MACRO_ROW_2
(VIX, DXY, 10Y, OIL), separator, TRADABLES_ROW
(SPY, QQQ, GLD, GDX, SLV, XLE). Each macro row is a single `<div>`
containing one `<span class="macro-tape-slot tape-slot {dir}">` per
slot.

FAIL: Rendered HTML for a representative payload (fixture-mode) does
not contain the three rows in the order above, OR any macro row spans
multiple `<div>`s, OR row content does not match the canonical slot
order.

R4 — XAU/XAG Color Parity
XAU and XAG render with directional CSS classes (`up`, `down`, `flat`,
`na`) computed from `_pct_arrow(block["change_pct"])` exactly as BTC
does today. The hardcoded `tape-slot na` styling on the spot-metals row
is removed.

FAIL: For a payload where `macro_drivers["gold"]["change_pct"] > 0`,
the rendered XAU slot does NOT carry the `up` CSS class; OR for
`change_pct < 0`, it does not carry `down`; OR for missing/non-finite
`change_pct`, it does not carry `na`.

R5 — Notification Parity
`_macro_tape_block` emits rows in the order XAU, XAG, BTC, VIX, DXY,
10Y, OIL (rows present only when their underlying quote is available
and has both price and `pct_change_decimal`). `_tradables_block` emits
SPY, QQQ, GLD, GDX, SLV, XLE in that order. Both blocks derive their
ordering from `MACRO_ROW_1`, `MACRO_ROW_2`, and `TRADABLES_ROW`.

FAIL: For a fixture payload where all underlying quotes are present,
the assembled notification body does not contain the macro lines in
exactly the order XAU, XAG, BTC, VIX, DXY, 10Y, OIL, OR the tradables
lines in exactly the order SPY, QQQ, GLD, GDX, SLV, XLE.

R6 — Payload-Key vs Quote-Symbol Mapping
The shared layout module provides a deterministic mapping from each
macro slot to BOTH (a) its `macro_drivers` payload key (used by the
renderer) and (b) its raw normalized-quote symbol (used by
notifications). The mapping is static and exhaustive for the slots
listed in MACRO_ROW_1 and MACRO_ROW_2.

FAIL: Either consumer needs to translate slot identity locally (e.g.
hardcodes a label→payload-key or label→quote-symbol lookup), OR the
mapping is incomplete for any slot in MACRO_ROW_1/MACRO_ROW_2.

R7 — Observational-Only Boundary
PRD-138 makes no change to regime computation, qualification gates,
sizing, execution policy, market_map writes, contract payload schema,
or any decision-affecting code path.

FAIL: Any file outside the FILES list is modified; OR any function
flagged as part of the execution pipeline (per gitnexus_impact upstream
analysis) is touched.

R8 — Color Resolution Without Pipeline Cycle
XAU/XAG color resolves correctly on the first run after merge, using
the current run's `change_pct` from `macro_drivers["gold"]` /
`macro_drivers["silver"]`. No prior-snapshot artifact, no run-history
join, no additional cycle required.

FAIL: A fresh fixture-mode run produces XAU/XAG slots with the `na`
CSS class when the corresponding `change_pct` is finite and non-zero.

DATA FLOW
macro_tape_layout (static rows) → dashboard_renderer (HTML emission)
macro_tape_layout (static rows) → notifications._macro_tape_block (text emission)
contract.macro_drivers payload  → renderer._build_tape_slots → _pct_arrow → _ARROW_CSS → CSS class per slot
normalized_quotes               → notifications._macro_row → text line per slot

FAIL CONDITIONS
- macro_tape_layout module missing, mutable, or exposes runtime config (R1).
- Either consumer retains a local ordering tuple covering ≥2 of the
  macro/tradables symbol sets (R2).
- Rendered HTML row order or grouping deviates from spec (R3).
- XAU or XAG renders with wrong directional CSS for a payload with
  finite, non-zero `change_pct` (R4, R8).
- Notification body order deviates from XAU, XAG, BTC, VIX, DXY, 10Y,
  OIL (macro) or SPY, QQQ, GLD, GDX, SLV, XLE (tradables) when all
  underlying quotes are present (R5).
- macro_tape_layout contains any formatting/HTML/transformation
  function or imports from renderer/notifications (R1, purity rule).
- Either consumer hardcodes a slot→payload-key or slot→quote-symbol
  mapping locally instead of consuming it from the shared module (R6).
- Any file outside FILES is modified (R7).

REVIEW
Codex cross-review required per PRD_PROCESS § Cross-review gate: this
PRD changes runtime renderer + notification behavior, introduces a new
shared module, alters visible output strings, and resolves existing
architecture drift. Codex review must specifically address:
- abstraction sufficiency (TapeRow/TapeSlot shape correct for the
  immediate use, no premature generality);
- anti-bloat discipline (no registry, no dynamic discovery, no plugin
  surface);
- renderer/notification parity guarantees (single source of truth
  actually enforced by the implementation, not by convention);
- testability of shared row definitions (frozen, importable, no
  side effects);
- visible-string regression coverage (pre-edit grep of `tests/` for
  notification body assertions and dashboard HTML assertions completed
  before first implementation edit; surfaced test files match FILES).
