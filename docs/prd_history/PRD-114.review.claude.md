# PRD-114 Review — Claude (Adversarial)

Scope: independent challenge of PRD-114 (Watchlist Snapshot Sidecar,
PROPOSED). Findings are tagged with the Review Failure Taxonomy
(`docs/PRD_PROCESS.md`).

---

## Top-line position

**Conditional accept.** PRD-114 as drafted is the smallest defensible
v1 of a watchlist sidecar — but the sidecar's *existence* is itself
the largest unresolved question. If approved, freeze exactly as
drafted; resist any field re-introduction during implementation.

---

## F1 — Should this sidecar exist at all? [scope creep]

The strongest objection is existential. Every meaningful field the
user originally proposed (vwap relation, sma relations, trend
alignment, relative volume) is **already produced** by
`trend_structure_snapshot` for symbols in the tradable universe.
Watchlist ∩ tradable universe is large (SPY, QQQ, GDX, GLD, SLV, XLE,
NVDA, TSLA, META, AMZN, AAPL — all 11 are in the documented
universe).

If a future consumer wants "watchlist + dynamic context," that
consumer can read both `trend_structure_snapshot.json` and a static
curated-list constant in one render pass. No sidecar required. The
v1 artifact PRD-114 produces is essentially a frozen constant
serialized to disk on every run — minimal information value.

**Strongest counter-argument:** persistent on-disk artifact decouples
discretionary visibility from code edits and gives downstream PRDs
(PRD-115+) a stable contract to read against. Acceptable, but only if
the v1 schema is explicitly minimal (which the PRD now enforces) and
PRD-115 is gated on a real consumer need, not anticipatory build.

**Recommendation:** Accept v1, but require the next PRD that touches
this artifact to declare a concrete consumer in its FILES.

## F2 — Watchlist as overlay, not sidecar [hidden coupling risk]

A defensible alternative: make the watchlist a **static constant**
in a new `cuttingboard/watchlist_universe.py` (no I/O, no artifact)
and have any future consumer (a CONSUMER-class PRD) read it directly.
This eliminates the artifact path, the runtime hook, the
`artifact_flow_map.md` row, and the persistence step entirely.

Argument against: violates PRD-111 sidecar-first philosophy
(visibility features default to sidecar artifacts). The artifact
boundary makes the contract auditable; a constant module makes it
implicit.

**Verdict:** sidecar form is correct under doctrine; recording this
alternative as a documented rejection.

## F3 — `watch_reason` and `event_risk` are renderer-bait [derived semantics]

The original draft included `event_risk` and `watch_reason` as
free-text fields. PRD-114 retains `watch_reason` as **static** text
sourced from the frozen constant, and **drops `event_risk` entirely**
to PRD-115+. This is correct.

Risk to monitor in implementation: if `watch_reason` ever becomes a
function of dynamic state (e.g. "earnings within 5 days"), it crosses
into `threshold→label synthesis`. R1's "frozen constant of
`tuple[str, str, str]`" wording is the binding fence. Reviewers must
reject any implementation that turns `watch_reason` into a computed
value, even if computed deterministically.

## F4 — Curated-universe duplication with config.py [factual drift]

The watchlist symbols all already exist in CLAUDE.md's documented
universe (Indices / Commodities / High beta sections). Defining a
second constant in `watchlist_sidecar.py` creates two sources of
truth for "is this symbol tracked." Drift risk: someone removes NVDA
from `config.ALL_SYMBOLS` and the watchlist sidecar happily emits
`current_price = null` for it forever.

**Mitigation considered:** require watchlist symbols ⊆
`config.ALL_SYMBOLS` at builder construction. Rejected — this would
create hidden coupling from the sidecar back into config governance,
and PRD-114 explicitly defers to discretionary curation. Accept the
drift risk; document it; revisit if it bites.

## F5 — Initial universe deviation from request [scope creep]

Original draft listed `AMD`. AMD is not in CLAUDE.md's instrument
universe (High beta lists NVDA, TSLA, AAPL, META, AMZN, COIN, MSTR).
Including AMD in the watchlist constant would implicitly expand the
tracked universe via a SIDECAR-class PRD — exactly the kind of
boundary erosion PRD-113's CLASS matrix is designed to prevent.

**Action taken:** AMD removed from R1's frozen list. If AMD belongs
in the universe, it requires a CONTRACT or EXECUTION-class PRD that
adds it to `config.ALL_SYMBOLS`, not a sidecar.

`COIN` and `MSTR` from CLAUDE.md's High beta list were also omitted
from the user's draft; they remain absent here for the same reason
(no expansion via SIDECAR-class PRD).

## F6 — Pass-through of `current_price` is the only justifiable dynamic field [non-determinism risk minimized]

Including `current_price` in v1 is borderline — it's the one field
that makes the artifact non-redundant with a pure constant. It is
deterministic given the input quote map, comes from existing
upstream state (no new fetch), and matches the
`trend_structure_snapshot` pattern. Accept with R2's strict
pass-through-or-null fence.

Stricter alternative considered: drop `current_price` too, making
v1 purely static metadata. Rejected — the resulting artifact would
be indistinguishable from a constant module (see F2) and undermine
the sidecar form.

## F7 — Hidden coupling vector: dashboard auto-discovery [hidden coupling]

PRD-112 validated read-only dashboard consumption of
`trend_structure_snapshot`. The dashboard renderer may at some point
auto-scan `logs/*.json`. R7's leakage grep is the binding guard;
reviewers must keep this grep in CI scope (current PRD does not add
CI enforcement — flagging as a known gap, not a blocker for v1).

## F8 — Stale-data ambiguity [stale-data ambiguity]

The artifact has no freshness guarantee declared (no max-age, no
"stale" status). For an observe-only sidecar with no consumer in v1,
this is acceptable. When a consumer is introduced (PRD-115+), that
PRD must declare a freshness contract or explicitly mark the
artifact "always fresh as of last run."

## F9 — Missing fail condition: empty `normalized_quotes` [missing fail condition risk]

If `normalized_quotes` is empty (e.g. ingestion HALT upstream), the
builder will emit a record with all `current_price = null`. PRD-114
does not currently fail this case. Argument for accepting: HALT
already short-circuits the runtime path before
`_write_trend_structure_snapshot` and would short-circuit
`_write_watchlist_snapshot` too. Verify in implementation; if not
short-circuited, add an R11 requiring the watchlist write to be
skipped on HALT (matching trend_structure ordering).

**Action:** flagged for Codex to confirm runtime ordering relative
to HALT short-circuit.

## F10 — MAX EXPECTED DELTA realism [factual drift]

150 LOC for: a frozen 11-tuple constant, a ~30-line builder, a
~10-line runtime helper, and a ~5-line call-site insertion is
generous. Test file will be ~80–120 lines. If implementation comes
in materially under 150 production LOC (likely), no amendment
needed; if it exceeds, PRD-113 R8 binds (stop, amend, re-review).

## PRD-113 governance discipline check

- **CLASS inference:** SIDECAR is correct. Default tier T1 → CHANGE
  SURFACE mandatory. Confirmed present.
- **Forbidden-pattern inheritance:** R9 cites the catalog by
  reference rather than re-listing. Correct per PRD-113's governance
  compression rule (inference-first, not cargo-cult).
- **CHANGE SURFACE discipline:** Each constrained file has an
  explicit "how deeply it may change" sentence. Tightest fence is
  on `runtime.py` (append-only, no signature changes).
- **MAX EXPECTED DELTA enforcement:** Numeric ceiling stated; binding
  per PRD-113 R8.
- **Review taxonomy usefulness:** Nine of nine labels exercised
  across F1–F10 except `consumer contamination`, `ownership
  violation`, `silent fallback`, `partial malformed recovery`,
  `payload mutation`. Useful as a coverage probe — taxonomy is
  sufficient but not all labels apply per PRD; that is expected.

PRD-113 governance functions as designed for this PRD. No
adjudication artifact required unless Codex disagrees on F1, F5, or
F9 (the load-bearing findings).
