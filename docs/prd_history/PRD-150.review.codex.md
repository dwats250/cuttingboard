## VERDICT

REJECT — return to author for required edits before implementation.

The PRD is unusually well-structured (lane declaration, LOC ceilings,
FAIL line per requirement, observable validation steps) and its
qualification.py / regime.py / output.py / sidecar doctrine citations

## REQUIRED EDITS

### RE-1 — R6 must be rewritten against the actual contract surface.
**Severity: blocking.**

The PRD repeatedly says R6 restricts "the TRADES list" / "TRADES
contract decision" / "TRADES list of any contract artifact
(`logs/latest_hourly_contract.json` or its daily equivalent)". Verified
against the code:

- `cuttingboard/contract.py:62 build_pipeline_output_contract` emits
  these top-level fields (lines 99–127): `schema_version`,
  `generation_id`, `generated_at`, `session_date`, `mode`, `status`,
  `timezone`, `system_state`, `market_context`, `trade_candidates`,
  `rejections`, `audit_summary`, `artifacts`, `correlation`, `regime`,
  `macro_drivers`. There is no `trades` field and no `TRADES` list.
- Per-candidate trade gating lives on `trade_candidates[i].decision_status`
  (`contract.py:328`), with values from
  `VALID_DECISION_STATUSES` (`ALLOW_TRADE` / `BLOCK_TRADE` / etc.,
  `contract.py:618–642`). The "TRADES vs NO TRADE vs HALT" output
  contract referenced in CLAUDE.md is a *runtime* outcome string
  (`derive_run_status`, `contract.py:182–188`), not a list, and is
  exposed via the `status` field plus per-candidate `decision_status`.
- The contract artifact path is `logs/latest_contract.json` (top of
  `contract.py:42 LATEST_CONTRACT_PATH`) and the hourly path is
  `logs/latest_hourly_contract.json` (defined in
  `cuttingboard/runtime.py:169`). The PRD names one of them but
  conflates "TRADES list of any contract artifact" with the
  per-candidate decision_status semantics in `trade_candidates[]`.

Required rewrites:

1. State which contract surface is being filtered, in the exact
   vocabulary of the code. Options:
   - (a) The `trade_candidates` list emitted by `_build_trade_candidates`
     (`contract.py:285`) is filtered to symbols with `tier == PRIME`.
   - (b) Per-candidate `decision_status` is forced to `BLOCK_TRADE`
     (with a new `block_reason` value) for non-PRIME tiers, while the
     list itself remains the union of PRIME + QUALIFIED + … so that
     audit / dashboard consumers still see them.
   - (c) Top-level run `status` is forced to `NO TRADE` when no PRIME
     symbol exists.

   Pick ONE of (a/b/c) explicitly. Each has very different downstream
   blast radius and the PRD currently reads as if all three happen at
   once.

2. Replace the FAIL line *"A symbol with tier != PRIME appears in the
   TRADES list of any contract artifact"* with a FAIL line that names
   a real key path (e.g. `contract["trade_candidates"][*]["symbol"]`
   with `decision_status == ALLOW_TRADE`).

3. The FAIL line *"A run with zero PRIME symbols emits a contract
   decision of TRADES rather than NO TRADE"* needs to map onto
   either `status` or `outcome`. Currently neither field is named
   "TRADES" — `derive_run_status` returns strings the PRD must
   enumerate.

4. The "contract schema gains, loses, or reorders any field"
   FAIL line in R6 should explicitly enumerate the 15 top-level
   keys above so the assertion is mechanical, not subjective.

This is the single biggest blocking gap. HIGH-RISK lane review cannot
sign off on a contract change whose target field doesn't exist by the
name used.

  the artifact has a single producer and a non-overlapping path
  (`reports/daily_classification.md`), and the artifact_flow_map.md
  row is required in the same PRD. This part is well-formed.

### Where the PRD breaks

The dominant failure mode is **vocabulary drift between the PRD and
the code**:

- CLAUDE.md's "TRADES | NO TRADE | HALT" output contract is a
  conceptual triplet, not three field values in `contract.py`. The
  actual surface is `status` (top-level) plus `decision_status` per
  candidate, with a separate `outcome` upstream feeding
  `derive_run_status`. R6 treats "TRADES list" as if it's a real
  contract key — it isn't. Implementation guesswork would either
  (a) silently filter `trade_candidates` and miss the "zero PRIME →
  NO TRADE" FAIL, or (b) silently rewrite `derive_run_status` and
  miss the "contract schema gains/loses/reorders any field" FAIL.
- R1's "field preserved for downstream readers" claim is technically
  true at the dataclass level but operationally false: five branches
  go permanently dead. HIGH-RISK lane review can't accept a "no
  behavior change" framing on this.
- R3's "pure function" claim is undermined by silence on the flow
  gate and continuation paths, both of which produce
  QualificationResults that R3 must classify.

These are not deal-breakers in spirit — the design is reasonable —
but the PRD as written would generate at least one scope-lock or
contract-shape surprise mid-implementation. Per CLAUDE.md's
Visible-String Pre-Edit Audit and Strict Scope Locking rules, the
right move is to tighten the language now.

### REQUIRED EDITS (second pass)

#### RE2-1 — `derive_run_status` does NOT have `"NO_TRADE"` in its return alphabet. R6 is factually wrong.
**Severity: blocking.**

The PRD asserts (CHANGE SURFACE, R6, FAIL CONDITIONS):
> `derive_run_status` is extended to return `NO_TRADE` (string already in its return alphabet) when zero PRIME symbols exist...

Verified at `cuttingboard/contract.py:182-190`: `derive_run_status` returns only `STATUS_OK = "OK"` or `STATUS_STAY_FLAT = "STAY_FLAT"`. The module-level `STATUS_ERROR = "ERROR"` exists but is not returned by `derive_run_status`. There is no `"NO_TRADE"` constant in `contract.py` and no return path emits that literal.

This means:
- R6's "zero-PRIME → `status = NO_TRADE`" introduces a brand-new status literal, not "already in the alphabet."
- It expands `_VALID_STATUSES` (`contract.py:40`) from 3 to 4 members — a contract surface change the PRD claims it is NOT making.
- R6's own FAIL "the 15 top-level contract keys are preserved with no addition, removal, or reorder" is preserved, but the *value space* of the `status` key changes. The PRD treats key-set invariance as schema invariance; the schema also encompasses the enumerated value space.

Required:
1. Either (a) define and add `STATUS_NO_TRADE = "NO_TRADE"` to `contract.py`, list `contract.py:36-40` (the `_VALID_STATUSES` block) as a change site, and acknowledge in CHANGE SURFACE that the status value space expands from 3 to 4; OR (b) reuse an existing status (most naturally `STATUS_OK` with zero PRIME — which contradicts the spirit of the PRD); OR (c) emit `STATUS_STAY_FLAT` for zero-PRIME, which collapses the visibility distinction PRD-150 is trying to create.
2. Add `cuttingboard/contract.py` `_VALID_STATUSES` to the CHANGE SURFACE allowlist explicitly.
3. Correct the false claim "string already in its return alphabet."

#### RE2-2 — `block_reason` must equal `decision_trace["reason"]`; PRD violates this invariant.
**Severity: blocking.**

Verified at `cuttingboard/contract.py:691-692`:
```
assert block_reason == decision_trace["reason"], (
    f"trade_candidates[{index}].block_reason must match decision_trace.reason for BLOCK_TRADE"
)
```

The PRD says non-PRIME symbols are written with `decision_status = BLOCK_TRADE` and `block_reason = "NON_PRIME_TIER"`. For this to pass contract validation, `decision_trace["reason"]` must ALSO equal `"NON_PRIME_TIER"` — but `decision_trace` is built by the trade-decision pipeline (`cuttingboard/trade_decision.py` / `trade_policy.py` / `execution_policy.py`) before classification runs in the proposed data flow. Non-PRIME symbols won't have flowed through `create_trade_decision` at all (they're not in `qualified_trades`).

Two distinct problems:
1. `_build_trade_candidates` at `contract.py:285-343` iterates `trade_decisions`, and non-PRIME symbols (which include WATCHLIST/DEVELOPING/REJECT) will not appear in `trade_decisions` because they don't pass through `build_option_setups` / `create_trade_decision`. The PRD's claim that "every evaluated symbol remains an entry in `trade_candidates[]`" requires a new code path that synthesizes trade_candidates entries for non-qualified symbols — which is not in CHANGE SURFACE and not in the 250-LOC budget.
2. Even if synthesized, `block_reason == decision_trace["reason"]` invariant means PRD-150 must also set `decision_trace = {"reason": "NON_PRIME_TIER", ...}` for synthesized entries, which is a new decision_trace literal and a new shape the contract validator may reject (decision_trace likely has additional required keys).

Required:
1. Either (a) drop the "every evaluated symbol remains in trade_candidates[]" claim and accept that non-PRIME symbols stay in `rejections[]` / sidecar only; OR (b) add CHANGE SURFACE entry for the synthesis path, name the function, budget the LOC, and resolve the `decision_trace` shape requirement (read `_assert_trade_candidates_valid` at `contract.py:617-720` to enumerate every field that must be present on a BLOCK_TRADE candidate).
2. Address the `block_reason == decision_trace["reason"]` invariant explicitly. Either change the invariant (which touches the contract validator and is a separate scope-lock concern) or set `decision_trace.reason = "NON_PRIME_TIER"`.
3. Add a FAIL line covering this invariant: a non-PRIME candidate whose `decision_trace["reason"]` does not equal `"NON_PRIME_TIER"` is a contract validator failure.

#### RE2-3 — Runtime cite `runtime.py:828-850` is wrong; there is no per-PRIME alert loop there.
**Severity: blocking.**

Verified at `runtime.py:820-860`: lines 828-850 contain the `if qualification_summary.qualified_trades:` block that calls `build_option_setups`, then `validate_option_chains`, then constructs `trade_decisions` via a list comprehension. This is the trade-decision construction site, not a notification dispatch loop.

The actual Telegram dispatch is a single `send_notification(alert_title, alert_body)` call at `runtime.py:575` (LIVE mode, aggregate hourly notification) — NOT a per-symbol loop. There is no existing "per-PRIME alert loop" to gate.

This is the same mis-citation pattern flagged in the first-pass review (RE-7), only narrowed from a range to a specific wrong range. The revision moved the cite but did not verify it.

Two consequences:
1. R7's gating instruction ("the per-PRIME alert loop sits at `runtime.py:828–850` and is gated to tier == PRIME inline") instructs the implementer to insert PRIME gating into the trade-decision construction block — which would silently filter `trade_decisions` and break downstream contract assembly.
2. The "end-of-session QUALIFIED summary after the per-PRIME alert loop's last iteration returns" call-order assertion (R7 FAIL line, validation step) becomes unobservable because there is no loop.

Required:
1. Resolve whether PRD-150 is (a) introducing a new per-symbol Telegram loop (significant new code, not in budget), or (b) keeping the single aggregate `send_notification` at `runtime.py:575` and gating its *content* / suppression on whether any PRIME exists.
2. If (a), add the new loop's call site to CHANGE SURFACE with LOC budget and a FAIL line covering iteration order.
3. If (b), rewrite R7 to gate the aggregate notification and re-specify the QUALIFIED summary as either a second `send_notification` call or an appended body section to the existing call. Cite the correct line (`runtime.py:575`).
4. Either way, drop the "828–850" cite or replace it with the verified line of the structure being gated.
