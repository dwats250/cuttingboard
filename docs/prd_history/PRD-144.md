# PRD-144 — Redundant cron entries for 6:00 AM PT hourly alert resilience

STATUS
IN PROGRESS

LANE
MICRO

GOAL
On 2026-05-19 the 6:00 AM PT hourly Telegram report never sent because GitHub Actions silently dropped both scheduled firings that map to that PT slot — the `50 12 * * 1-5` warm-up (12:50 UTC) and the `0 13 * * 1-5` premarket cron (13:00 UTC). No workflow run exists between `2026-05-18T21:48Z` and the time of this PRD (~13:05 UTC 2026-05-19); the runner never executed, so the existing slot canonicalization and notify-path code paths were uninvolved. The same upstream-scheduler unreliability previously produced the off-cadence "1:27 PM / 2:48 PM PT" firings (PRD-141 motivation). Add belt-and-braces backup cron entries inside the premarket exemption window of the 13:00 UTC slot so that a single dropped firing no longer silently kills the 6 AM PT report. No runtime, notification, contract, or dashboard logic changes — the existing PRD-141 slot dedup behaviour absorbs any duplicate-on-success.

SCOPE
- Add `cron: '55 12 * * 1-5'` (12:55 UTC) as backup for the existing `50 12 * * 1-5` warm-up firing.
- Add `cron: '5 13 * * 1-5'` (13:05 UTC) as backup for the existing `0 13 * * 1-5` 6 AM PT firing.
- Both new entries fall inside the ±5-minute premarket tolerance window in `cuttingboard.notifications.hourly_slot.is_premarket_slot`, so they bypass the slot-dedup gate exactly like the entries they back up — no code change required for the new firings to behave correctly.
- Track this PRD in registry/state per existing convention.

NOTES
- The 13:50 UTC / "30 min before ORB" cron is NOT backed up here. It already shares the 6 AM PT canonical slot with `0 13`, so a dropped 13:50 firing is partially compensated by surviving 13:00/13:05 sends; a dedicated backup for the 13:50 ORB alert is deferred. If a dropped 13:50 firing is later observed in isolation, a follow-up micro-PRD adds `cron: '55 13 * * 1-5'`.
- The `*/30 14-21 * * 1-5` intraday cron is not backed up — it already provides natural redundancy (16 firings/day; dedup collapses to one send per PT hour).
- This PRD relies on PRD-141 slot canonicalization and PRD-142 cross-run slot persistence to absorb duplicate-on-success cases. If PRD-142 has not landed at implementation time, duplicate sends inside a PT hour are possible but bounded (at most one extra per backup-cron firing); this is the same risk surface the premarket window already accepts by design (13:00 + 13:50 both send).
