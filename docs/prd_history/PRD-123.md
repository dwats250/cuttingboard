# PRD-123 — Trend Structure Refresh Decoupling and Truthful Source Status

## STATUS
IN PROGRESS

## CLASS
VISIBILITY

## TIER
T1

## LANE
STANDARD

## GOAL
Decouple the trend_structure_snapshot writer from hourly-notify gating so any
`MODE_LIVE` pipeline run with valid quote/bar inputs refreshes the sidecar.
Correct the renderer's "usable" count so it reflects per-symbol data
availability rather than field-shape presence, and introduce explicit
`MARKET_CLOSED` / `AWAITING_DATA` source-health states so the panel stops
conflating "old file" (STALE) with "no live data" (no usable rows).
Notification policy is unchanged. `PipelineResult` shape is unchanged.

---

# SCOPE

## In Scope
- Introduce a narrow internal helper `_refresh_trend_structure_sidecar` in
  `cuttingboard/runtime.py` that takes `(mode, normalized_quotes,
  history_by_symbol, generated_at)`, gates on `mode == MODE_LIVE`, and on
  match delegates to the existing `_write_trend_structure_snapshot(...)`.
  The helper emits no notifications. Write failure remains best-effort,
  swallowed and logged by the underlying writer as today.
- Call this helper from within `_run_pipeline` at the point where the
  local `ohlcv` dict is fully populated and `normalized_quotes` /
  `run_at_utc` are in scope. The exact insertion point is implementation-
  determined but must be after the existing `ohlcv` population block in the
  `MODE_LIVE` branch and before `_run_pipeline` returns its `PipelineResult`.
- `PipelineResult` is **not** modified. No `history_by_symbol` field added,
  no new field of any kind. DataFrames stay local to `_run_pipeline`.
- The existing call in `_execute_notify_run` at `runtime.py:621` is unchanged
  — the hourly cadence continues to refresh the snapshot exactly as before
  via its own local `ohlcv`. There is no double-write risk: `_run_pipeline`
  is only invoked from `execute_run`; `_execute_notify_run` runs its own
  pipeline independently.
- Tighten `_trend_symbols_usable` in `dashboard_renderer.py`: a record only
  counts as usable when (a) every `_TREND_STRUCTURE_REQUIRED_FIELDS` key is
  present AND (b) `rec.get("data_status") != "MISSING"`.
- Extend `_trend_structure_source_health` precedence with two new states:
  - `MARKET_CLOSED` when snapshot is FRESH, `usable_count == 0`, and
    `inactive_session is True`.
  - `AWAITING_DATA` when snapshot is FRESH, `usable_count == 0`, and
    `inactive_session is False`.
  Neither new state may fire when lineage is `MIXED` / `STALE` / `MISSING`,
  or when `ts_generated_at_raw` parses as `PARSE_ERROR` (`INVALID`). The
  previous `FALLBACK` return-value for `usable_count == 0` is removed and
  replaced by the two new states. `FALLBACK`'s use in
  `_trend_structure_records` (all-or-nothing row rendering) is unchanged
  because that is a separate code path.
- Render the new states in the trend-structure panel header with the label
  `MARKET CLOSED — AWAITING INTRADAY DATA` and a `Last snapshot: <iso>` line
  below, sourced from `snapshot.get("generated_at")`. `STALE` retains its
  existing header rendering — the two are visually and semantically distinct.
- Tests covering: runtime non-hourly refresh; fixture/sunday do NOT write;
  no notification fires from the refresh path; renderer truthful usable
  count; renderer `STALE` vs `MARKET_CLOSED` vs `AWAITING_DATA`
  distinctness; existing `OK`/`MIXED`/`INVALID`/`INACTIVE_SESSION` paths
  unaffected.

## Out of Scope
- No `PipelineResult` shape changes.
- No changes to trade decisions, qualification, regime, scoring, options
  candidate generation, payload contract, market_map shape, or macro logic.
- No notification policy changes. `send_notification` continues to fire only
  from the existing `_execute_notify_run` path under `mode == MODE_LIVE`.
- No changes to `_execute_notify_run` itself.
- No changes to `build_trend_structure_snapshot` semantics; only writer
  call sites are involved.
- No new symbols added to `TREND_STRUCTURE_SYMBOLS`.
- No `INACTIVE_SESSION` semantic redesign. It remains the umbrella
  inactive-session label for system-state and market-map. The narrower
  `MARKET_CLOSED` state introduced here is specific to trend structure
  because its render fidelity depends on intraday availability.

# REQUIREMENTS

- **R1.** `cuttingboard/runtime.py` defines a new private helper:

  ```python
  def _refresh_trend_structure_sidecar(
      *,
      mode: str,
      normalized_quotes: dict[str, NormalizedQuote],
      history_by_symbol: dict[str, pd.DataFrame],
      generated_at: datetime,
  ) -> None:
      """PRD-123: refresh logs/trend_structure_snapshot.json from any
      MODE_LIVE pipeline run. Fixture/Sunday modes are excluded. Best-effort
      — write failures are swallowed and logged by the underlying writer."""
      if mode != MODE_LIVE:
          return
      _write_trend_structure_snapshot(
          normalized_quotes=normalized_quotes,
          history_by_symbol=history_by_symbol,
          generated_at=generated_at,
      )
  ```

  No notification calls, no I/O outside the underlying writer, no exposure of
  DataFrames beyond the function's own argument frame.
  - **FAIL:** Helper missing; OR signature differs; OR `mode != MODE_LIVE`
    is not the first branch; OR helper calls `send_notification` /
    `notification_*` / any other I/O.

- **R2.** `_run_pipeline` invokes `_refresh_trend_structure_sidecar(...)`
  **inside the `MODE_LIVE` code branch**, after the local `ohlcv` dict is
  populated (currently `runtime.py:794`) and after
  `validation_summary.valid_quotes` is established, before
  `PipelineResult` is returned. The placement is defense-in-depth: the
  helper checks `mode == MODE_LIVE` internally, AND the call site is
  inside the `MODE_LIVE` branch so fixture/sunday paths never reach the
  helper. The call passes `mode=mode`,
  `normalized_quotes=validation_summary.valid_quotes`,
  `history_by_symbol=<local ohlcv>`, `generated_at=run_at_utc`.
  - **FAIL:** No new call in `_run_pipeline`; OR call placed at a
    branch level reachable under `MODE_FIXTURE` or `MODE_SUNDAY` (must
    be inside the MODE_LIVE branch, not merely guarded by the helper);
    OR call placed before `ohlcv` is populated; OR call placed after
    `_run_pipeline` returns; OR the call adds a try/except that
    re-raises (write failure must remain best-effort).

- **R3.** `execute_run` is **not** modified by this PRD. The new write
  happens inside `_run_pipeline`. Notification policy is unchanged at
  every level of the call stack.
  - **FAIL:** `execute_run` body diff is non-empty for this PRD; OR any
    reachable path from `execute_run`/`_run_pipeline` newly invokes
    `send_notification`.

- **R4.** `_trend_symbols_usable(snapshot)` returns the count of
  `TREND_STRUCTURE_SYMBOLS` whose record (a) contains every key in
  `_TREND_STRUCTURE_REQUIRED_FIELDS` AND (b) has
  `data_status != "MISSING"`. Shape-present-but-data-MISSING rows do not
  count.
  - **FAIL:** A snapshot whose six curated symbols all have
    `data_status == "MISSING"` returns `usable_count > 0`; OR a snapshot
    with full data for all six returns `usable_count < 6`.

- **R5.** `_trend_structure_source_health` precedence is reordered so that
  `MARKET_CLOSED` is reachable under `inactive_session` while preserving
  PRD-117's coherence rule that an inactive-session panel does not show
  "SOURCE: MISSING" contradicting its "SESSION INACTIVE" body label.

  Explicit precedence:

  1. lineage `MIXED` → `MIXED`
  2. lineage `STALE` → `STALE`
  3. lineage `MISSING` → `MISSING`
  4. snapshot is not a dict:
     - `INACTIVE_SESSION` if `inactive_session` (PRD-117 coherence)
     - else `MISSING`
  5. freshness `PARSE_ERROR` → `INVALID`
  6. freshness `STALE` → `STALE`
  7. `usable_count == 0` (snapshot exists and is fresh):
     - `MARKET_CLOSED` if `inactive_session` **(NEW)**
     - else `AWAITING_DATA` **(NEW)**
  8. `inactive_session` → `INACTIVE_SESSION`  (rare: inactive with
     non-zero usable rows, e.g. cached intraday after session end)
  9. otherwise → `OK`

  The previously-trailing `usable_count == 0 → FALLBACK` return is
  **removed** from this function entirely. The two new states require
  the snapshot to be a dict and its `generated_at` to be FRESH (rows
  4–6 above already filter those out).
  - **FAIL:** Either new state fires when lineage is `MIXED`/`STALE`/
    `MISSING`, or when freshness is `PARSE_ERROR`/`STALE`; OR
    `MARKET_CLOSED` fires when `inactive_session is False`; OR
    `AWAITING_DATA` fires when `inactive_session is True`; OR a fresh
    snapshot with full data returns anything other than `OK`; OR
    `INACTIVE_SESSION` fires when `usable_count == 0` AND the snapshot
    is a fresh dict (must be `MARKET_CLOSED` instead); OR `FALLBACK`
    appears in this function's return surface; OR the `snapshot is not
    a dict` path returns `MISSING` under `inactive_session` (must be
    `INACTIVE_SESSION` to preserve PRD-117 coherence).

- **R6.** Trend-structure panel header renders:
  - `STALE` in its existing form when source-health is `STALE`.
  - `MARKET CLOSED — AWAITING INTRADAY DATA` when source-health is
    `MARKET_CLOSED` or `AWAITING_DATA`.
  - `Last snapshot: <ISO timestamp>` directly under the header in both new
    states, sourced from `snapshot.get("generated_at")`. Omitted in
    `STALE` / `MISSING` / `OK` / `MIXED` / `INVALID` to keep the existing
    layouts byte-identical.
  - `TREND SYMBOLS: <usable>/<total>` always reflects the corrected R4
    count.
  - **FAIL:** `STALE` conflated with `MARKET_CLOSED`/`AWAITING_DATA`
    in the header text; OR usable count contradicts the visible row
    statuses; OR `Last snapshot:` line emitted in non-degraded states;
    OR `Last snapshot:` line omitted in the two new degraded states.

- **R7.** `tests/test_runtime_trend_structure_refresh.py` covers:
  - A `MODE_LIVE` run through `_run_pipeline` invokes the helper and writes
    `logs/trend_structure_snapshot.json`. (Tests may mock the underlying
    `_write_trend_structure_snapshot` and assert one invocation with the
    expected kwargs to avoid filesystem dependency.)
  - The same test asserts `send_notification` is not invoked
    (monkeypatch-spy or mock).
  - A `MODE_FIXTURE` run does NOT invoke the helper's underlying write.
  - A `MODE_SUNDAY` run does NOT invoke the helper's underlying write.
  - A simulated underlying-write exception does not abort
    `_run_pipeline` and does not raise out of `_refresh_trend_structure_sidecar`.
  - **FAIL:** Any of the five cases unasserted.

- **R8.** `tests/test_dashboard_renderer.py` is **updated** (not merely
  augmented). The existing PRD-120 R14-7 test
  `test_prd120_trend_structure_source_fallback_zero_usable` at
  `tests/test_dashboard_renderer.py:2243–2268` asserts
  `SOURCE: FALLBACK` for a zero-usable trend snapshot. Under PRD-123 R5
  the `FALLBACK` return is removed from `_trend_structure_source_health`,
  so this test must be rewritten to assert `SOURCE: AWAITING_DATA` (or
  `SOURCE: MARKET_CLOSED` if the test forces inactive session) rather
  than `SOURCE: FALLBACK`. The `TREND SYMBOLS: 0/6` half of the existing
  assertion is preserved. In addition, the suite gains:
  - A fresh snapshot with all six curated symbols at

    (active session). Two parametrized cases.
  - A fresh snapshot with all six symbols carrying full data and finite
    numerics renders `TREND SYMBOLS: 6/6` and source-health `OK`.
  - An `mtime > 300s` snapshot renders source-health `STALE` regardless
    of inner row content (STALE wins via lineage precedence).
  - The string `MARKET CLOSED — AWAITING INTRADAY DATA` appears in the

  - **FAIL:** PRD-120 R14-7 test still asserts `SOURCE: FALLBACK` after

# FAIL CONDITIONS

- `_refresh_trend_structure_sidecar` invoked with `mode != MODE_LIVE` does
  anything other than return.
- `_run_pipeline` calls the helper from a code path reachable under
  `MODE_FIXTURE` or `MODE_SUNDAY` without re-checking mode.
- `PipelineResult` gains, removes, or renames any field as part of
  PRD-123. `dataclasses.fields(PipelineResult)` field-name set must be
  byte-identical before and after this PRD.
- `send_notification` reachable from any code path newly introduced.
- `usable_count` value disagrees with the number of rows the renderer
  actually displays as non-`MISSING`.
- `MARKET_CLOSED` or `AWAITING_DATA` returned when `usable_count > 0`,
  or when lineage is `MIXED`/`STALE`/`MISSING`, or when freshness is
  `STALE`/`PARSE_ERROR`.
- `STALE` label appearing on a fresh snapshot whose only fault is
  all-`MISSING` rows.
- `Last snapshot:` line appears in `OK` / `STALE` / `MISSING` / `MIXED`
  / `INVALID` renders.
