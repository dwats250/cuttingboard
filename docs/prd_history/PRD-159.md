# PRD-159: scripts/prd_open.sh — Stage 0 PRD scaffolder

Lane: MICRO / GOVERNANCE
Status: IN PROGRESS
Estimated size: ~80 LOC + tests
Author: Dustin
Filed: 2026-05-29 (reserved PROPOSED 2026-05-29; promoted to Stage 0 same day)

## Problem

CLAUDE.md § Operational rules requires "PRD file lands at Stage 0": the
first commit for any PRD is the `PRD-NNN.md` scaffold plus the IN PROGRESS
registry row plus the `prd_index.json` entry — *before* any implementation
commit. There is no tool for this. `scripts/prd_close.sh` mechanizes the
*closeout* edits, but the symmetric *opening* edits are done by hand.

The cost of the missing tool is concrete and recent: PRD-158 produced 9+
sequencing-gate hook fires because the PRD was authored in chat and the
registry was not updated until closeout. PRD-160 was built ahead of
PRD-159, and closing it forced a hand-patched index-contiguity fix plus a
PROPOSED placeholder row for 159 (this PRD). A single command that scaffolds
the three Stage-0 artifacts atomically removes the loophole.

## Goal

`scripts/prd_open.sh` — a mirror of `scripts/prd_close.sh` for Stage 0.
One command:

1. Scaffolds `docs/prd_history/PRD-NNN.md` with a `STATUS: IN PROGRESS`
   line plus Lane/Class header and skeleton sections (Problem, FILES,
   Acceptance criteria, Non-goals).
2. Inserts an `IN PROGRESS` row into `docs/PRD_REGISTRY.md`.
3. Inserts a sorted `prd_index.json` entry (status `IN PROGRESS`,
   commit `null`).
4. Optionally (`--commit`) commits the three edits as `PRD-NNN: stage 0`.

It must NOT touch `latest_complete` / `next_prd` — those track completion,
not opening, and are owned by `prd_close.sh` / the validator's invariant.

## Design (mirror of prd_close.sh)

- Bash wrapper + inline `python3` heredoc, same shape as `prd_close.sh`.
- Args: `--prd <NNN>`, `--title "<title>"`, `--lane <LANE>`,
  `--class <CLASS>`, optional `--summary "<one-line problem>"`,
  optional `--commit`. Missing required args → exit 2 with usage.
- `--title`/`--summary` flow through `re.sub` callable replacements (never
  template strings) to avoid the `re.PatternError: bad escape \d` class of
  bug that PRD-147 fixed in `prd_close.sh`.
- Refuse (exit 1) if `docs/prd_history/PRD-NNN.md` already exists — never
  overwrite an existing PRD.
- Registry row inserted after the last existing `| PRD-NNN |` row
  (mirrors prd_close; the normal case opens the highest-numbered PRD).
  Skip if a row for the PRD already exists.
- Index entry inserted in **sorted ascending** position (the validator
  enforces ascending order). Skip if an entry already exists.
- Default (no `--commit`): write edits, print a diff preview, instruct the
  user to review + commit — identical UX to `prd_close.sh`.

## FILES

- `scripts/prd_open.sh` — new Stage-0 scaffolder.
- `tests/test_prd_open.py` — new; drives the script in a temp dir against
  fixture `docs/` scaffolding.
- `docs/prd_history/PRD-159.md` — this file.

Bookkeeping (Stage 0 + closeout): `docs/PRD_REGISTRY.md`,
`docs/PROJECT_STATE.md`, `docs/prd_index.json`.

## Acceptance criteria

- Running `prd_open.sh --prd 200 --title "T" --lane MICRO --class GOVERNANCE`
  in a clean fixture tree creates `docs/prd_history/PRD-200.md` containing a
  `STATUS: IN PROGRESS` line, `Lane: MICRO / GOVERNANCE`, and a `# PRD-200`
  title header.
- The registry gains exactly one `| PRD-200 | — | T | IN PROGRESS | [PRD-200](prd_history/PRD-200.md) |` row.
- `prd_index.json` gains an entry `{number:200,title:"T",status:"IN PROGRESS",commit:null}` in sorted position; `latest_complete` / `next_prd` are unchanged.
- A title containing a regex metasequence (e.g. `\d`) does not crash the script.
- Re-running for an existing PRD number refuses (PRD-NNN.md present → exit 1) or skips the already-present registry/index rows, leaving them unchanged.
- Missing a required arg exits 2 with usage text.
- `tools/validate_prd_registry.py` exits 0 against the resulting fixture tree.
- Full suite passes; baseline maintained or increased.

## Non-goals

- Not changing `prd_close.sh`.
- Not auto-committing by default (symmetry with `prd_close.sh`; `--commit` opt-in).
- Not advancing `latest_complete` / `next_prd`.
- Not enforcing review-gate or FILES-completeness (that is authoring/review skill territory, not a scaffolder).
