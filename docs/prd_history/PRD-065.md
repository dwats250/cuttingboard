# PRD-065 — Signal Forge Interactive Dashboard Controls

**Status:** IN PROGRESS
**Commit:** —
**Depends on:** PRD-064 (visibility fields must be present in payload before this ships)

---

## GOAL

Enable client-side filtering and row expansion on the Signal Forge dashboard using the existing payload. No backend changes.

---

## SCOPE

- Add grade, visibility, and direction filter controls
- Add expandable row panel showing candidate detail
- All logic client-side; reads from `latest_payload.json` only

---

## OUT OF SCOPE

- No backend logic changes
- No new data fields (all consumed fields must exist in payload after PRD-064)
- No API changes
- No server-side rendering

---

## FILES

- `ui/index.html`
- `ui/app.js`
- `ui/styles.css`
- `tests/test_dashboard_renderer.py`

---

## REQUIREMENTS

### R1 — Filter controls

Three filter controls rendered above the candidate table:

| Control | Options |
|---|---|
| Grade | `ALL` (default), `A+`, `A`, `B` |
| Visibility | `DEFAULT` (initial), `ALL`, `ACTIVE`, `NEAR_MISS`, `BLOCKED` |
| Direction | `ALL` (default), `LONG`, `SHORT` |

Visibility semantics:

- `DEFAULT`: shows `ACTIVE` and `NEAR_MISS` only
- `ALL`: shows `ACTIVE`, `NEAR_MISS`, and `BLOCKED`
- `ACTIVE`: shows `ACTIVE` only
- `NEAR_MISS`: shows `NEAR_MISS` only
- `BLOCKED`: shows `BLOCKED` only

Filters are independent. A candidate is shown only if it passes all active filters simultaneously.

FAIL: any filter affects candidates it should not.
FAIL: filters do not compose (e.g., grade + direction applied independently).

### R2 — Default view

On load, show all `ACTIVE` and `NEAR_MISS` candidates. `BLOCKED` candidates are hidden unless explicitly selected via the Visibility filter.

FAIL: BLOCKED candidates appear in the default view.
FAIL: changing to Visibility=ALL does not reveal BLOCKED candidates.

### R3 — Row expansion

Each candidate row is clickable. On click, an inline detail panel expands below the row showing:

- `grade` (from market_map surface)
- `visibility_status`
- `visibility_reason`
- `enable_conditions` (as a list)
- `macro_alignment` (if present — added by PRD-066; render as absent if null)

Clicking the same row again collapses the panel.

FAIL: detail panel renders on a row that has no trade candidate data.
FAIL: panel does not collapse on second click.

### R4 — Missing field tolerance

Payload fields `visibility_status`, `visibility_reason`, `enable_conditions` may be null/absent in older payloads (pre-PRD-064). The dashboard must render without crash and display `—` in place of absent values.

FAIL: dashboard throws a JS error when visibility fields are absent.

### R5 — Determinism

Given identical payload, the rendered DOM must be identical across reloads. No timestamps, counters, or random keys in rendered output.

FAIL: repeated render of same payload produces different DOM structure.

## DATA FLOW

```
latest_payload.json
  → trade_candidates[].visibility_status
  → trade_candidates[].visibility_reason
  → trade_candidates[].enable_conditions
  → trade_candidates[].grade (from market_map surface if present)
app.js
  → filter() — pure function, returns filtered array
  → renderRow(candidate) — deterministic HTML
  → expandRow(symbol) — toggle detail panel
```

---

## FAIL CONDITIONS

- JS error thrown when visibility fields absent
- BLOCKED candidates visible in default view
- Filters not composable (each applied independently instead of AND)
- Row expansion does not collapse on second click
- Render output differs across identical-payload reloads

- Row expansion shows required fields and collapses on re-click
