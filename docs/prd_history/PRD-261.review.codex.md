FINDINGS (verbatim from captured stdout):

1. `git -c` is bypassable through shell expansion:
   `git -c $'core.fsmonitor\x3d/path/to/hook' status --short`
   Bash turns `\x3d` into `=`, so Git receives a real `-c name=value`,
   but the submitted command contains no literal `=`. `Bash(git *-c
   *=*)` does not match. The same construction bypasses the new Ruff and
   worktree rules, e.g. `ruff check $'--confi\x67' 'fix = true' .` and
   `git worktree $'a\x64d' /tmp/wt`. Claude Code warns that Bash
   argument-pattern rules are fragile; matching occurs before shell
   expansion.

2. The `git -c` rule has a real false positive:
   `git log -c --format=%H -1`
   This is a legitimate read-only combined-diff log command, but it
   matches `Bash(git *-c *=*)` because the later `--format=%H` supplies
   the `=`. It blocks the existing `Bash(git log *)` allow.

3. Ruff can enable fixes without `--config`:
   With `fix = true` in `ruff.toml` or `[tool.ruff]` in `pyproject.toml`,
   plain `ruff check .` matches the allow rule and none of the denies,
   yet applies fixes. The current resolved config is `fix=false`, but
   the policy does not protect against project or user configuration
   changing that default.

4. Ruff's output-file deny is bypassable:
   `ruff check -o/path/to/report .`
   Ruff accepts the attached short form, but it does not match
   `Bash(*ruff check* -o *)` or the long-form rule. It can overwrite the
   report path. An inherited `RUFF_OUTPUT_FILE` similarly makes a plain
   allowed `ruff check .` write output.

5. Pytest has an unguarded destructive flag:
   `python3 -m pytest --basetemp=ui tests`
   This matches `Bash(python3 -m pytest *)`. Pytest clears an existing
   `--basetemp` directory before the run, so this can erase and recreate
   a tracked child directory.

6. Force-push is still bypassable by option ordering:
   `git push --porcelain --force origin HEAD:refs/heads/main`
   It matches `Bash(git push:*)`, while each force deny requires
   `--force`/`-f` immediately after `git push`. `git push -qf …` has the
   same problem.

7. Destructive push modes have no deny:
   `git push -d origin refs/heads/main`, `git push --mirror origin`, and
   `git push --prune origin` all match the broad push allow. They delete
   refs; `--mirror` also force-updates refs.

8. The receive-pack execution backstop is bypassable by Git's accepted
   abbreviation:
   `git push --rece=/path/to/program .`
   Installed Git recognizes `--rece` as `--receive-pack`, but
   `Bash(git * --receive-pack*)` only matches the full spelling. This
   reopens executable selection through an allowed `git push`.

9. Other explicitly allowed Git commands still mutate state:
   `git fetch --prune` is directly allowed and deletes stale local
   remote-tracking refs; `git commit --amend --no-edit` is directly
   allowed and replaces the current branch tip. Neither has a deny
   backstop.

10. The active ignored local policy defeats the intended boundary.
    `settings.local.json` contains broad `Bash(python3:*)`,
    `Bash(python *)`, `Bash(git *)`, `Bash(bash *)`, and `Bash(gh pr *)`
    rules. Thus, for example, `git clean -fdx`, `python3 -c
    "…unlink()"`, and `gh pr close 147` are silently allowed unless a
    specific project deny happens to match. It is ignored rather than
    committed, but it is active runtime configuration and is
    specifically relevant because PRD-261 relies on project denies to
    constrain its blanket Git allow.
