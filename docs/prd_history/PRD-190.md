WHY NOW
TREND STRUCTURE's "SMA Composite" reads "SMA history insufficient" for all six
universe symbols every run while "vs SMA50" resolves ABOVE/BELOW. Same yfinance
daily series, different lookback: OHLCV_FETCH_MONTHS = 6 (config.py:97) → 6×31 = 186
calendar days (ingestion.py:319) ≈ ~126 trading bars. `_sma(closes, window)` returns
None when len < window (trend_structure.py:34), so sma_50 resolves (126 ≥ 50) but
sma_200 is null (126 < 200) → price_vs_sma_200 = INSUFFICIENT_HISTORY → the composite
degrades to "SMA history insufficient" (dashboard_renderer.py:109-119). Structural,
not staleness; reproduces independent of PRD-189.

MAX EXPECTED DELTA
One constant in config.py (OHLCV_FETCH_MONTHS 6 → 12) + one ~8-LOC regression test.
No ingestion.py change. ~2× daily bars/symbol per yfinance pull.

GOAL
sma_200 resolves for full-history universe symbols so the SMA Composite renders a real
ABOVE/BELOW composite instead of "SMA history insufficient", with the wider fetch frame
confirmed not to move any decision input.

SCOPE
- Bump OHLCV_FETCH_MONTHS 6 → 12 to guarantee ≥200 TRADING bars with holiday headroom
  (≈252 trading days; clears 200 even in holiday-dense quarters).
- The populated path is ALREADY built and tested: _resolve_sma_field returns
  ABOVE/BELOW the moment sma_200 is non-null (trend_structure.py:165), the composite
  lookup already carries the populated-pair entries (dashboard_renderer.py), and
  test_prd130_sufficient_history_populates_values exercises it. sma_200 simply flips
  null→populated — NO renderer, record-schema, or _data_status change.
- Add one R1 regression guard so a silent revert to a sub-200-bar window fails red.

OUT OF SCOPE
- No change to _sma / composite / classification logic (trend_structure.py and
  dashboard_renderer.py untouched); no record-schema or _data_status change.
- No data-source change (stays yfinance); no intraday / 1-minute window change.
- Length-aware (shape-aware) cache key is NOT built here. The OHLCV cache key is
  symbol-only (ingestion.py:369) and _is_fresh_ohlcv_cache checks only last-bar age
  vs OHLCV_STALE_HOURS (ingestion.py:146), never frame length. This is a LATENT
  symbol-only-key defect, deferred to a future follow-up PRD. It is NOT load-bearing
  for this PRD: correctness is guaranteed by the code-enforced OHLCV_STALE_HOURS (12h)
  TTL — any pre-bump ~126-bar / 6-month cache entry expires within ≤1 session, after
  which the next refetch uses the 12-month window (the "or expire" path in
  _is_fresh_ohlcv_cache, ingestion.py:146). The one-time `rm data/cache/*_ohlcv.parquet`
  is OPTIONAL acceleration only (collapses the ≤1-session wait to the first run); it is
  explicitly not required for correctness.
- Relaxing the composite's 200-line requirement is a separate CONSUMER PRD, not here.

(runtime/__init__.py) inherits whatever derived computes. _sma uses .tail(window) →
size-invariant by value (only resolves vs not).
