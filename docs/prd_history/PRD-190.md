WHY NOW
The TREND STRUCTURE "SMA Composite" column reads "SMA history insufficient" for
all six symbols, while "vs SMA50" resolves ABOVE/BELOW. Same source (yfinance
daily OHLCV, ingestion.py:118/137), same series — different lookback:
OHLCV_FETCH_MONTHS = 6 (config.py:97) → fetch window = 6 × 31 = 186 calendar
days (ingestion.py:319) ≈ ~126 trading bars. `_sma(closes, window)` returns None
when len < window (trend_structure.py:33–36), so sma_50 resolves (126 ≥ 50) but
sma_200 is null (126 < 200) → price_vs_sma_200 = INSUFFICIENT_HISTORY. The
composite degrades to "SMA history insufficient" if EITHER field is insufficient
(dashboard_renderer.py:104–120), so it reads insufficient on every run. This is
structural, not staleness — it reproduces independent of PRD-189.

MAX EXPECTED DELTA
One constant changed in config.py (OHLCV_FETCH_MONTHS 6 → ~12), PLUS a
length-aware cache-invalidation change in ingestion.py so a stale-by-shape
(~126-bar / 6-month) parquet entry is not served after the bump, plus tests. ~2×
daily bars fetched per symbol × 6 symbols per yfinance OHLCV pull.

GOAL
sma_200 resolves for full-history universe symbols so the SMA Composite renders a
real ABOVE/BELOW composite instead of "SMA history insufficient", with the
wider fetch frame confirmed not to move any decision input.

SCOPE
- Bump OHLCV_FETCH_MONTHS to guarantee ≥200 TRADING bars with holiday/halt
  headroom — size for ~12 months (≈252 trading days), NOT "a bit more than 6".
  ~12mo clears 200 bars even across holiday-dense quarters.
- 200DMA-in-composite call (RECOMMENDED, default KEEP): the SMA Composite
  requires both the 50 and 200 lines. Default recommendation: KEEP the 200 — the
  200DMA above/below is a real regime line and the column is named "Composite."
  If instead the call is to relax the 200 requirement, that is a renderer change
  (different FILES, CONSUMER lane) and must be its own PRD — not folded here.
- Length-aware cache invalidation: fetch_ohlcv() (ingestion.py:127–134) serves a
  cached parquet whenever _is_fresh_ohlcv_cache() passes, and that check tests
  ONLY the last bar's age vs OHLCV_STALE_HOURS (ingestion.py:146–158) — never the
  frame length. A fresh existing 6-month cache (~126 bars) is therefore served
  after the bump until its last bar ages past the TTL, so the first post-merge
  run can still render "SMA history insufficient." Make the cache shape-aware:
  the cache key incorporates the window length OR _is_fresh_ohlcv_cache() treats
  a frame shorter than the configured window as stale and forces a live refresh,
  so a config bump invalidates short entries on the next run.

OHLCV CONSUMERS (blast-radius enumeration — OHLCV_FETCH_MONTHS is global)
The constant is read once (ingestion.py:319) and sizes the frame returned by
ingestion.fetch_ohlcv(). Every fetch_ohlcv() consumer therefore receives a
larger frame after the bump:
- cuttingboard/derived.py:68 (compute_derived) →
    * volume_ratio (RVOL): derived.py:178–179 `avg_20d_vol = vol.iloc[-21:-1].mean()`
      — tail-windowed, so size-INVARIANT (see RVOL note).
    * EMAs ema9/ema21/ema50: derived.py:100–102 `close.ewm(span=…, adjust=False).mean().iloc[-1]`
      — recursive EMA with adjust=False is SEEDED FROM THE FIRST BAR, so a longer
      frame shifts the final value (effect decays with span; ema50 is the most
      exposed). THIS, not RVOL, is the most likely length-sensitive surface.
    * ATR: derived.py:159 `tr.ewm(alpha=1/ATR_PERIOD, adjust=False).mean().iloc[-1]`
      — same adjust=False length sensitivity as the EMAs.
    * momentum/levels: derived.py:167/170 use fixed iloc offsets — size-invariant.
- cuttingboard/runtime/__init__.py:412, 435, 744, 1893 (per-symbol fetch into the
  regime/structure path) — inherit whatever derived/structure compute from the frame.
- cuttingboard/watch.py:120/143 (daily_fetcher=fetch_ohlcv) → watch.py:161–162 use
  iloc[-1] (PDH/PDL) — size-invariant.
- cuttingboard/trend_structure.py (snapshot path): _sma uses .tail(window)
  (size-invariant by value, only resolves/doesn't); _relative_volume uses
  iloc[-21:-1] (trend_structure.py:81) — size-INVARIANT.

EMA/ATR SEED SENSITIVITY (the headline R2 check): ema9/ema21/ema50
(derived.py:100–102) and ATR (derived.py:159) are computed with
`ewm(..., adjust=False).mean().iloc[-1]`. With adjust=False the EWMA is a
recursive series SEEDED FROM THE FIRST BAR; that first-bar seed contamination
decays as the series lengthens, so adding ~126 older bars in front shifts the
RECENT EMA/ATR values (the .iloc[-1] we read), not just the deep history. This
is the actually-exposed surface: the bump that ENABLES sma_200 also silently
CHANGES today's ema9/21/50 and ATR on every symbol. ema50 and ATR are the most
exposed (longest effective memory).

RVOL (confirm-only): both RVOL computations (derived.py:179, trend_structure.py:81)
read a TAIL window (`iloc[-21:-1]`) and are therefore size-invariant; R2 confirms
this with one line and otherwise focuses on EMA/ATR.

CHANGE SURFACE
- config.py: the OHLCV_FETCH_MONTHS literal only.
- ingestion.py: the cache-freshness/key path (_is_fresh_ohlcv_cache and/or
  _ohlcv_cache_path / fetch_ohlcv) made length-aware so a short (pre-bump) cache
  entry is not served; no change to the yfinance fetch itself beyond the window.

REQUIREMENTS

R1 — Fetch window yields ≥200 trading bars with headroom
OHLCV_FETCH_MONTHS is sized so a normal-history symbol returns ≥200 daily bars
including holiday-dense windows.
FAIL: the chosen value can return <200 trading bars for a full-history symbol in
a holiday-dense 12-month span.

R2 — EMA/ATR adjust=False seed sensitivity is characterized
The downstream-consumer audit's headline check is the ewm(adjust=False) EMA/ATR
seed sensitivity (derived.py:100–102, 159): the audit confirms that lengthening
the fetch shifts the recent ema9/21/50 and ATR values (first-bar seed
contamination decaying further out), quantifies the shift per symbol, and
confirms RVOL is unchanged (tail-windowed, iloc[-21:-1]).
FAIL: the EMA/ATR value shift under the larger frame is not characterized per
symbol, or RVOL's size-invariance is not confirmed.

R3 — SMA Composite resolves
After the bump, full-history universe symbols render ABOVE/BELOW (not "SMA
history insufficient") in the SMA Composite column.
FAIL: sma_200 remains null for a full-history universe symbol.

R4 — Pre/post diff is the STANDARD-vs-escalate gate
The bump does not only ENABLE sma_200 — it silently CHANGES existing EMA/ATR
values on every symbol. Diff EMA/ATR/SMA outputs pre vs post across all six
universe symbols and confirm NO decision-surface field flips from warm-up alone:
price_vs_sma_50, price_vs_sma_200, the trend-structure label, and the regime
call. This diff is the GATE: a null diff (no field flips) keeps the PRD STANDARD;
any material decision-surface field that moves fires the R2 escalation to
HIGH-RISK and requires the moved field to be enumerated + accepted before merge.
FAIL: the six-symbol pre/post diff is not run, OR a decision-surface field
(price_vs_sma_50/200, trend label, regime call) flips and the PRD is not
escalated to HIGH-RISK.

R5 — The bump takes effect on the first post-bump run (cache cannot serve a short entry)
After the OHLCV_FETCH_MONTHS bump, the first run must actually fetch the longer
window: a pre-existing fresh ~126-bar / 6-month cache entry must NOT be served.
The cache key includes the window length, or _is_fresh_ohlcv_cache() treats a
frame shorter than the configured window as stale and forces a live refresh.
FAIL: with a fresh pre-bump 6-month cache present, the first post-bump run serves
~126 bars and price_vs_sma_200 stays INSUFFICIENT_HISTORY (R3 and the R4
EMA/ATR/SMA diff are then unsatisfiable on first run).
IMPLEMENTATION NOTE (2026-06-16): chose the length-aware-freshness mechanism (not
the cache-key-rename — that would break regime_history.py:74's hardcoded
SPY_ohlcv.parquet path, outside FILES). The freshness floor is pinned to the
LONGEST CONSUMER WINDOW (SMA-200 = 200 bars), config-independent — NOT the fetch
target (~252). Tying it to the fetch target would mark a holiday-shortened
~248-bar full-window frame as permanently stale and spin a re-fetch loop; the
200-bar floor clears every consumer (sma_200 is the deepest) with ~50 bars of
holiday headroom under the 12-month window. See cuttingboard/ingestion.py
_OHLCV_MIN_FRESH_BARS and tests/test_ingestion.py
(test_holiday_shortened_full_window_frame_is_fresh_no_refetch_loop).

DATA FLOW
