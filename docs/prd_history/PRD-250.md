## 1. Problem

A stale board presents as **fresh** with every workflow green. During the
2026-07-07 incident the published board read `UPDATED 2026-07-06 09:14:23 PT`
for ~23h while `hourly_alert.yml` runs all reported success — because the publish
tail was *skipped* (slot suppressed, `fresh=false`), not failed. Nothing on the
board, and nothing in CI, said "this read is old."

This is the worst failure class for a discretionary-sizing tool: **a frozen
board can feed a real trade.** The trader opens the page pre-market, reads a
regime/verdict that looks current, and sizes against it — unaware the data is a
day old. The danger is not that the board goes stale (bounded cron latency makes
occasional staleness inevitable); the danger is that **stale is indistinguishable
from fresh at read time.** The goal is to make staleness impossible to miss the
moment the trader looks.

## 2. Load-bearing decision — the freshness verdict is computed CLIENT-SIDE

This is the hinge the whole proposal turns on, and the reason a banner beats
every server-side option:

> The banner's "fresh vs stale" judgment MUST be computed **in the browser at
> view time** — JS embedded in the board reads the embedded `UPDATED` timestamp,
> compares it to the *viewer's* current clock, and renders `STALE · Xh old` past
> threshold. It MUST NOT be baked at publish/render time.

**Why client-side is non-negotiable.** A staleness label computed at render time
is written *into* the artifact — so a board that stops updating **freezes its own
freshness label along with everything else.** A frozen board would proudly
display "fresh" forever, which is exactly the failure we are trying to surface. A
publish-time check cannot detect the pathology (publish not happening) because it
only runs when publish *does* happen. Only a judgment made by the viewer's own
clock, against a timestamp frozen in the page, correctly reports "this page is N
hours old" no matter why the page stopped moving. The staleness verdict has to
live in the one component that keeps running when the pipeline stops: the
reader's browser.

**Accepted, recorded tradeoff — the verdict is viewer-clock-relative.** Because
the judgment compares the embedded timestamp to `Date.now()` on the reader's
device, a device with a materially wrong clock yields a wrong verdict — a
skewed-fast clock over-reports staleness, a skewed-slow one under-reports it.
This is accepted, not a latent surprise: the trader's own device is the read
surface, its clock is the same one they trade by, and the failure is
self-evident (a wildly wrong banner rides along with a wildly wrong device
clock). The alternative — a server-authoritative "now" — is rejected because it
reintroduces exactly the kind of component that stops when the pipeline stops,
defeating §2's whole point. Recorded here so the spec inherits this decision
rather than discovering it.

**Render-path dependency the SPEC must approve — not absorbed into banner work.**
The current `UPDATED` value is rendered as localized text
(`2026-07-06 09:14:23 PT`); parsing a localized "PT" string in-browser is fragile
(DST, locale). The robust form is for the renderer to *additionally* emit a
machine-readable ISO-8601 UTC value (e.g. a `data-updated-utc="…Z"` attribute on
the `#system-state` block or a `<meta>` tag) that the client JS reads. This is a
**distinct render-path change, named here and not made here** — it is a separate
dependency the spec must explicitly approve and FILES-scope, and the banner
cannot be implemented until that emit is agreed. It is not folded into "banner
work." To be confirmed against the render path Dustin will paste before spec.

## 3. Explicitly rejected alternative — widening `max_lag_minutes`

The freeze's proximate trigger was `routine_pt_slot(max_lag_minutes=25)`
suppressing scheduled runs that GitHub delivered 8–46 min late. The tempting
"fix" is to widen the window (25 → 60 → …). **Rejected, on principle:**

- **No fixed window beats unbounded latency.** GitHub scheduled-cron delay is
  best-effort and unbounded — it can and does exceed any constant you pick.
  Widening `max_lag` only chooses a larger number to lose against on the next bad
  day; it converts "often suppressed" into "less often suppressed," never
  "never." The failure recurs at a lower rate, which is *worse* — rare enough to
  stop watching for, common enough to eventually feed a trade.
- **Wrong axis entirely.** Widening treats the *symptom* (the board sometimes
  goes stale). The banner treats the *danger* (stale reads as fresh). Staleness
  is tolerable if it is visible; invisibility is the actual defect. If a wider
  window feels like the answer, that is the signal you are optimizing the wrong
  variable — recorded here so the instinct is not silently re-litigated.

The banner makes the freshness of the read a first-class, always-honest fact,
which is robust to *any* upstream cause (cron latency, suppressed slot, failed
publish, Pages outage) — not just the one that happened this time.

## 4. Threshold policy (to specify)

**(a) Derive the number from the decision, not from false-positive avoidance.**
The question is not "how long until a staleness alarm gets annoying" — it is
"what is the worst staleness a pre-market sizing decision can rest on before the
read no longer describes the environment being traded." A discretionary options
size rests on the current regime/macro read; during active pre-market hours that
read is refreshed each slot (~hourly) *precisely because* structure can move
within the hour. So the tolerance is one refresh cycle: a read that has missed a
full expected slot may already describe a superseded environment, and a sizing
decision should not rest on it. That yields **~90 min** = one slot cadence
(~60 min) + the routine lag margin (~25 min) + render/settle — i.e. "a full
refresh has come due and not arrived." 90 min stands **not** because it avoids
false positives but because it is the point past which the read may no longer be
about *today's* tape. If active-hours cadence ever tightens, this number tracks
the cadence, not an annoyance floor.

**(b) Key off "a refresh was due and missed," not raw age.** Raw wall-clock age
cannot separate "old because FROZEN" from "old because the market is closed and
no refresh was due" — off-hours the board is *supposed* to be stale, and an
age-only rule cries wolf every night. The banner must instead compute, from the
embedded slot schedule, **the most recent slot that should have published as of
the viewer's now**, and compare *that* to the embedded `UPDATED`:

- If the last *expected* slot is at or before `UPDATED` — nothing new was due —
  the board is current-for-the-schedule → **no alarm, regardless of age.** A
  Saturday board 40h old is correct, not stale.
- If the last *expected* slot falls meaningfully after `UPDATED` (a refresh came
  due and did not land, past the lag margin) — the board **missed a publish it
  owed** → `STALE · Xh old` alarm.

This makes off-hours quiet *by construction* (no slot is due, so none is missed)
and an active-hours freeze loud (a due slot is missing). The ~90 min of (a) is
the lag margin *inside* this rule — how long after a due slot before "late"
becomes "missed" — not a standalone age gate. It replaces the earlier flat
threshold, which papered over exactly this distinction.

**Open question (promoted, not papered).** The "expected slot as of now" rule
needs the session calendar client-side. The fixed 06:00–13:00 PT / Mon–Fri slot
set is trivial to embed, but **US market holidays and half-days are the residual**:
on a holiday no slot is due, yet a naive weekday rule would expect one and
false-alarm. Options for the spec to pick among: (i) embed a small holiday list
(accurate, needs yearly maintenance); (ii) accept holiday false-alarms as
rare-and-safe (a STALE banner on a day you weren't trading is harmless); or
(iii) have the renderer emit the *next-expected-slot* alongside the timestamp so
the client needs no calendar at all — which shifts a sliver onto the render path
and must be weighed against §2's ISO-8601 emit as its own spec-approved
dependency. **Not resolved here** — Dustin decides at spec.

## 5. Where the banner lives (presentation layer)

- Rendered board: `cuttingboard/delivery/dashboard_renderer.py` emits
  `ui/dashboard.html`, copied byte-identical to `ui/index.html` (the Pages-served
  artifact). The banner is a small **inline `<script>` / static JS asset** in the
  renderer's template, plus a banner container element at the top of the board.
  It runs on load, reads the embedded machine-readable timestamp (§2), computes
  age against `Date.now()`, applies the §4 threshold policy, and reveals the
  banner. **No data-path, contract, payload, or aggregation code is touched** —
  the banner reads only what is already in the page and writes nothing back.
- The banner element sits above `#system-state` so it is the first thing read.
- Exact template location and the machine-timestamp emission point: to confirm
  against the render path before spec (charge: do not touch it now).

## 6. NON-GOALS reconciliation (against `VISION.md`)

Presentation-only overlay; the claim is it trips no locked non-goal. Checked
against each:

| VISION non-goal / principle | Banner disposition |
|---|---|
| Not an automated execution engine; human at every seam | Read-only display; informs the human, acts on nothing. ✓ |
| Not backtesting / ML / multi-agent / HFT / general FinOS | Unrelated; a client-side freshness label. ✓ |
| **Description, not prediction** | Describes the *present* state of the read (how old it is); forecasts nothing. ✓ |
| **Read-only sidecars by default** | Pure presentation overlay; does not mutate the decision contract or any artifact — reads the embedded timestamp, writes nothing. ✓ |
| **The system serves the trader** (if it doesn't change a decision, it shouldn't exist) | Bears on the decision without making it: it states, at read time, whether the board describes the current session or a superseded one — a fact directly relevant to whether the read is fit to act on. It earns its keep by the four-questions standard (a stale read cannot honestly answer any of the four). It surfaces the condition; the trader draws the conclusion. ✓ |
| **Cuts before additions** | An addition; justified as a guard on the integrity of the single most-used artifact, not feature sprawl. Net: small, and it earns its keep by making an unfit-to-act read visible *as* unfit. ✓ |
| **Docs match code** | N/A for a proposal; the eventual PRD ships its own doc/test. |

**Informs, never instructs (load-bearing boundary).** The banner states the
staleness *condition* and stops — `STALE · Xh old`, a fact about the artifact. It
must never render an *instruction* or implied call to action ("stale — don't
trade", "hold off", "wait for the refresh"): the moment it tells the trader what
to *do*, it crosses from presentation into making the call — the exact trap the
system refuses ("does this change what I will actually do, or does it just help
me feel more informed"). The trader reads the condition and decides; the banner
does not decide for them. Any spec or copy wording that drifts toward directing
action is cut. This boundary is why every row above is phrased as *surfacing a
fact*, not *prompting a response*.

No conflict found. To be re-confirmed when Dustin pastes VISION's non-goals + the
render path (per his gate before spec).

## 7. OPTIONAL SECONDARY layer — CI / Telegram staleness ping (flagged)

**Client-side banner leads; this is secondary and separable — a real PRD could
ship the banner alone.** A secondary layer could have a scheduled check (or the
existing Telegram path) emit a "board is Xh stale" ping so a freeze surfaces even
when no one is looking at the page.

**Why it is the piece to scrutinize — a ping needs a dispatching actor.** The
distinction is not the soft "monitoring feels like automation." It is concrete:
the moment a ping exists, *something must decide to send it on a schedule* — a
dispatching actor that runs and acts (emits a message) with no human at that
seam. A scheduled actor that acts unprompted is precisely what the no-automation
non-goal guards against ("no execution automation… a human stays at every seam").
The client-side banner has **no such actor**: the viewer's browser computes a
verdict and paints it *when the human is already looking*; nothing is dispatched,
nothing runs on a schedule, nothing acts unprompted. That is the recorded line
between the two layers — **not message-vs-display, but dispatches-on-a-schedule
vs paints-for-a-present-reader.** The ping does not execute or size trades, and
the repo already runs a Telegram hourly path (NOTIFY_HOURLY), so it may well be
admissible — but it must clear the dispatching-actor question on its own, not
inherit the banner's clean pass. **Recommendation: keep it OUT of the lead
proposal; spec it as its own follow-up so that question gets a clean, isolated
decision rather than riding in on the banner.**
