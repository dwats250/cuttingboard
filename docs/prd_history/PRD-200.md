WHY NOW
PR #39 manually reconciled the registry to git ground truth on 2026-06-20, yet
`python tools/validate_prd_registry.py` already exits 1 on main again (11 drifts
across PRD-189/190/194/197/199 plus a stale latest_complete pointer), because
nothing on the auto-merge path runs it. The validator has thorough hermetic unit
tests (tmp_path fixtures) but is never executed against the real repo registry
where merges are decided -- the exact PRD-198 invariant 5 gap ("verify where
truth is determined"). Install the gate now so the next drift fails CI instead of
accumulating into another reconciliation PR.

AMENDED (2026-06-20, after CI run 27863721436): the first attempt wired the FULL
validator into CI and went red -- the commit-resolvability check (git cat-file -e
on each COMPLETE hash) cannot pass in a clean CI checkout because 19 historical
rows record hashes not reachable from main (squash-merged/rebased away). That
check is only reliable with a full local object store. This is itself PRD-198
invariant 5: local + both reviews were green; only CI caught it. Amended scope:
CI enforces the CONSISTENCY checks (the PR #39 drift class) via a narrow,
opt-in --skip-commit-resolvability flag; the 19-hash data drift is tracked as a
follow-up (see DISCOVERED / FOLLOW-UP).

GOAL
Make registry/index/state CONSISTENCY a blocking CI check on every PR by running
tools/validate_prd_registry.py (consistency checks only) inside the required
`test` job, after reconciling current drift to green so the gate-introducing PR
proves itself.

REQUIREMENTS

R1 -- Reconcile to consistency-green
For every tracked PRD (number >= TRACKING_START), title, status, and commit agree
byte-for-byte between docs/PRD_REGISTRY.md and docs/prd_index.json, and the index
latest_complete / next_prd pointers match the registry's latest COMPLETE row.

FAIL: `python tools/validate_prd_registry.py --skip-commit-resolvability` exits
nonzero on the PRD-200 branch HEAD.

R2 -- Narrow --skip-commit-resolvability flag
The validator gains a --skip-commit-resolvability flag (and a keyword-only
validate_repository(..., skip_commit_resolvability=...) parameter) that skips ONLY
_validate_commit_resolvable. All other checks (index schema/rules, registry<->index
agreement, history-doc existence, doc-status agreement) still run. Default (no
flag) behavior is identical to before.

FAIL: with skip_commit_resolvability=True, any non-resolvability error is
suppressed; OR without the flag the validator behaves differently than on
origin/main.

R3 -- Validator is a blocking step in the required `test` job
ci.yml's existing `test` job runs `python tools/validate_prd_registry.py
--skip-commit-resolvability`; the invocation is not isolated in a separate,
non-required job.

FAIL: no `test`-job step invokes the validator, OR it runs in a job other than the
branch-protection-required `test` job.

R4 -- Fail-loud, no silent fallback (PRD-198 invariant 1)
The validator step propagates its exit code: no continue-on-error: true and no
`|| true` / `|| echo` masking.

FAIL: the validator step sets continue-on-error: true or pipes the command
through a success-forcing fallback.

R5 -- Self-proving gate
This PR's own CI `test` check is green on the reconciled branch, demonstrating the
gate passes in the CI environment.

FAIL: the PRD-200 PR's CI `test` check is red.

R6 -- Red test proves the flag is narrow (PRD-198 invariant 4)
tests/test_prd_registry.py proves: (a) in a git tree, an unresolvable commit is
flagged by the default run but NOT when skip_commit_resolvability=True; and (b)
with the flag set, a registry/index consistency mismatch is STILL flagged.

FAIL: no test exercises the flag, OR a test asserts the flag suppresses a
consistency error.
