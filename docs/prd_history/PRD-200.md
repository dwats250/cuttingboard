CLASS
GOVERNANCE

WHY NOW
PR #39 manually reconciled the registry to git ground truth on 2026-06-20, yet
`python tools/validate_prd_registry.py` already exits 1 on main again (11 drifts
across PRD-189/190/194/197/199 plus a stale latest_complete pointer), because
nothing on the auto-merge path runs it. The validator has thorough hermetic unit
tests (tmp_path fixtures) but is never executed against the real repo registry
where merges are decided -- the exact PRD-198 invariant 5 gap ("verify where
truth is determined"). Install the gate now so the next drift fails CI instead of
accumulating into another reconciliation PR.

MAX EXPECTED DELTA
0 production-code LOC. <= 5 changed lines in .github/workflows/ci.yml. Remainder
is data-only reconciliation of docs/PRD_REGISTRY.md and docs/prd_index.json, plus
Stage-0/closeout pointers in docs/PROJECT_STATE.md. No validator-logic change.
Exceeding this stops implementation and re-triggers review.

GOAL
Make registry/index/state consistency a blocking CI check on every PR by running
the existing tools/validate_prd_registry.py inside the required `test` job, after
reconciling current drift to green so the gate-introducing PR proves itself.

SCOPE
- Reconcile docs/PRD_REGISTRY.md and docs/prd_index.json so the validator exits 0
  on the branch. The bare-hash + concise-title form is the canonical convention
  used by the index (all ~140 entries) and by the rest of the registry
  (PRD-195/196/198); the five recent rows (189/190/194/197/199) drifted by
  stuffing review-verdict prose and "(PR #NN)" into the title/commit fields. Trim
  those five registry rows to the canonical form (the verbose review/verification
  notes for 189/190/194/199 already live in docs/PROJECT_STATE.md; PRD-197 only
  drops a "(PR #29)" commit annotation). Bare hashes are additionally forced: the
  index commit field is COMMIT_RE-validated and rejects "(PR #NN)".
- Set docs/prd_index.json latest_complete to 199 (the actual max COMPLETE).
- Add fetch-depth: 0 to the `test` job checkout so the validator's commit-
  resolvability check can resolve historical COMPLETE commits.
- Add one step to the existing `test` job that runs the validator and fails the
  job (and thus the required `test` status check) on nonzero exit.

OUT OF SCOPE
- Any change to tools/validate_prd_registry.py logic or its unit tests.
- Any new CI job, new required status check, or branch-protection settings change
  (the gate rides the existing required `test` check).
- Reconciling rows below the validator's TRACKING_START (PRD < 56).
- The cuttingboard.yml / publish pipeline and all secret-bearing workflows.

CHANGE SURFACE
- .github/workflows/ci.yml: append-only within the existing `test` job. Add a
  `with: { fetch-depth: 0 }` to the actions/checkout step and one
  `- run: python tools/validate_prd_registry.py` step. No new job, no triggers
  change, no secrets; existing ruff and pytest steps unchanged.
- docs/prd_index.json: data-only (latest_complete integer). No key/schema change.
- docs/PRD_REGISTRY.md: data-only; trim only the five drifted rows to the
  canonical bare-hash + concise-title form. No other rows touched.
- docs/PROJECT_STATE.md: bookkeeping pointers only (active PRD; CI baseline).

REQUIREMENTS

R1 -- Reconcile to validator-green
For every tracked PRD (number >= TRACKING_START), the title, status, and commit
fields agree byte-for-byte between docs/PRD_REGISTRY.md and docs/prd_index.json,
and the index latest_complete / next_prd pointers match the registry's actual
latest COMPLETE row. tools/validate_prd_registry.py is the arbiter.

FAIL: `python tools/validate_prd_registry.py` exits nonzero on the PRD-200 branch
HEAD.

R2 -- CI checkout fetches full history
The `test` job's actions/checkout step sets fetch-depth: 0, so the validator's
commit-resolvability check (git cat-file -e on each COMPLETE hash) resolves
historical commits instead of false-failing under the default shallow clone.

FAIL: the `test` job checkout in ci.yml lacks fetch-depth: 0.

R3 -- Validator is a blocking step in the required `test` job
ci.yml's existing `test` job contains a step that runs
`python tools/validate_prd_registry.py`; the invocation is not isolated in a
separate, non-required job.

FAIL: no `test`-job step invokes tools/validate_prd_registry.py, OR the
invocation lives in a job other than the branch-protection-required `test` job.

R4 -- Fail-loud, no silent fallback (PRD-198 invariant 1)
The validator step propagates its exit code: no continue-on-error: true and no
`|| true` / `|| echo` masking.

FAIL: the validator step sets continue-on-error: true or pipes the command
through a success-forcing fallback.

R5 -- Self-proving gate
The PRD-200 PR runs the new gate: CI `test` is green on the reconciled branch,
demonstrating the validator passes in the CI environment (full history) on
reconciled data.

FAIL: the PRD-200 PR's CI `test` check is red.

DATA FLOW
docs/PRD_REGISTRY.md + docs/prd_index.json -> validate_prd_registry.py -> exit 0/1
validator exit code -> `test` job step result -> required `test` check -> merge gate

FAIL CONDITIONS
- `python tools/validate_prd_registry.py` exits nonzero on the branch HEAD.
- ci.yml `test` job checkout lacks fetch-depth: 0.
- No `test`-job step runs tools/validate_prd_registry.py, or it runs only in a
  non-required job.
- The validator step uses continue-on-error or a `|| true`-style fallback.
- A deliberate registry/index mismatch on a scratch branch does NOT turn CI
  `test` red (the guard is inert).
- The PRD-200 PR's CI `test` check is red.

VALIDATION
Manual:
- Before: run `python tools/validate_prd_registry.py`; expect exit 1 with the
  current drift errors. After reconciliation: re-run; expect exit 0 and no
  output.
- Inspect ci.yml: the `test` job checkout has fetch-depth: 0; a step runs
  `python tools/validate_prd_registry.py`; no continue-on-error / `|| true`.
- Negative test (PRD-198 invariant 4 -- every guard ships a red test): on a
  scratch branch off the PRD-200 branch, change one docs/prd_index.json entry
  title to mismatch the registry, push, and confirm the PR's CI `test` check goes
  red on the validator step; discard the scratch branch.
- Confirm the PRD-200 PR's CI `test` check is green on the reconciled branch.
- Full suite green in CI (test_prd_registry.py unchanged and passing).
