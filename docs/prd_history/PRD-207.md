PROVENANCE (the hollow-gate incident, this session)
Binding Codex cross-review of PRD-208 @ 7cbe0616 — run 28264070271, job
83746430244, step 4 ("Resolve + verify the served model … fail-closed"). The
step-4 env dump shows `CODEX_EVENTS: Model: gpt-4.1\n\n**Verdict** …` — i.e.
`CODEX_EVENTS` (wired to `steps.codex.outputs.final-message`, codex-review.yml:122)
is the assistant's PROSE final message, NOT the `--json` response-metadata JSONL.
`extract_model` JSON-greps `"model"[[:space:]]*:[[:space:]]*"…"` (:133-135) → the
prose has no such token → returns empty → the rollout fallback (:137-138) extracts
`"model"` from the codex session rollout, which records the REQUESTED/configured
model (`gpt-5-codex`) — laundering the request into `resolved-model`. The artifact
then reads `resolved-model=gpt-5-codex … allowlist-verified`, while the served
model's only signal (`Model: gpt-4.1`, off-allowlist) was never parsed. PRD-198 #2
(assert the resolved, not the requested) + #3 (authoritative source, not a prose
proxy). The existing contract test asserts these strings are PRESENT in the YAML
but never that the parser WORKS against real output (PRD-198 #4) — so the gate has
been green-on-the-letter while broken-on-the-meaning.

WHY NOW
Every HIGH-RISK PRD's binding Codex cross-review depends on this gate telling the
truth about which model authored the review. Right now it cannot: a routed/cheaper
substitute (here gpt-4.1) served the review and the gate reported gpt-5-codex. Until
fixed, no Codex APPROVE can be trusted as verified-real-Codex.

GOAL
`codex-review.yml` records `resolved-model` from the SERVED model in the runtime
`--json` response-metadata stream, fails closed (non-zero, no artifact) when the
served model cannot be read or is off-allowlist, and ships a test that fails when
the parser is fed a requested≠served / unreadable fixture.

SCOPE (read .github/workflows/codex-review.yml + tests/test_codex_review.py first)

REQUIREMENTS

R1 — resolved-model derives from SERVED response-metadata, never requested/config
`resolved-model` is parsed from the `--json` response-metadata JSONL event stream
(the served model). The requested input, the session rollout, and the prose
final-message are NEVER resolution sources.
FAIL: a run whose served metadata says model X but requested=Y records
resolved-model=Y (the requested), as run 28264070271 did (recorded gpt-5-codex
while the served signal was gpt-4.1).

R2 — fail-closed on unreadable served model (no requested fallback)
If the served model cannot be read from the response-metadata stream, the step
EXITS NON-ZERO and emits NO artifact — no fallback to rollout/requested/config
(PRD-198 #1 fail-loud + #3 authoritative-source).
FAIL: served-model unreadable yet an artifact is emitted with any resolved-model
value.

R3 — RED test: gate fails when served is off-allowlist / unreadable [GENUINE RED]
tests/test_codex_review.py exercises the resolution+allowlist logic against a REAL
captured-output fixture (not string-presence): a requested≠served fixture where the
served model is off-allowlist → the gate is INVALID (no artifact); an
unreadable-metadata fixture → fail-closed. The test FAILS against the current
prose/rollout-fallback workflow and passes only after the fix (PRD-198 #4: every
guard ships a red test that fails when violated).
FAIL: the test passes against the pre-fix workflow (i.e. it only checks that YAML
strings exist, not that the parser rejects an off-allowlist served model).

R4 — body model identity cannot contradict machine resolved-model
The artifact body no longer carries an independent prose `Model:` self-report that
can disagree with the machine `resolved-model` provenance line; the model identity
is recorded once, from the machine resolution (PRD-198 #6: pin identities; one
source of truth).
FAIL: the artifact shows a body `Model:` value different from the provenance
`resolved-model` (as PRD-208/210 artifacts did: body gpt-4.1/gpt-5.5 vs
resolved gpt-5-codex).

R5 — verification runs in CI where the decision is made
The resolution + fail-closed + allowlist verification execute inside
codex-review.yml on GitHub Actions (the gate's decision point), and the R3 test
runs in the required CI `test` job (PRD-198 #5 environment parity).
FAIL: the verification or its test runs only locally / is skipped in CI.
