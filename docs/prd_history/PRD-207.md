AMENDMENT (2026-06-27) — evidence-forced R-set revision; verdict CONFIRMED-PROSE-ONLY
The original R1/R2 assumed openai/codex-action's `--json` output carries a
structured SERVED-model field that the gate could parse. A read-only realizability
recon against the REAL incident stream (run 28264070271, job 83746430244) and the
pinned/latest action manifests proved that premise FALSE on this toolchain:
  * The pinned action (openai/codex-action@e0fdf01, v1.8) AND latest @main both
    declare exactly ONE output — `final-message` ("Raw output emitted by codex
    exec"), which in practice is the agent's PROSE last message
    (`agent_message.text` = "Model: gpt-4.1 …"). No output carries raw JSONL /
    response metadata / a served-model id.
  * codex CLI 0.142.2 emits NO served-model field in its `--json` stream: the
    quoted JSON key `"model"` appears 0 times across the full 1.75M-char captured
    stream. Event inventory: thread.started (thread_id only), turn.started,
    turn.completed (token usage only), item.* (command_execution / todo_list /
    error / agent_message). The served model's ONLY signals are (a) a STRUCTURED
    item.error fallback event when requested≠served (names no fallback model), and
    (b) the agent's PROSE self-report (banned by PRD-198 #3).
Conclusion: a positive served-model id is UNREALIZABLE on this toolchain. The
honest gate therefore certifies "the REQUESTED model was HONORED" (no silent
fallback), not "the served model is X". R1/R2 are amended below to that
fallback-error gate; an explicit CAPTURE MECHANISM requirement is added; RISKS /
NON-GOALS record the scope reduction. Recon detail: this session's chat verdict +
the captured stream at job 83746430244 (the artifact path is this PRD's PROVENANCE
+ AMENDMENT blocks; no separate recon file was persisted).

WHY NOW
Every HIGH-RISK PRD's binding Codex cross-review depends on this gate telling the
truth about whether the requested model actually authored the review. Right now it
cannot: a routed/cheaper substitute (here gpt-4.1) served the review and the gate
reported gpt-5-codex. Until fixed, no Codex APPROVE can be trusted as
verified-real-Codex.

GOAL (amended)
`codex-review.yml` certifies that the REQUESTED model was HONORED by detecting
codex's STRUCTURED item.error fallback event in the captured `--json` JSONL: if the
fallback event is present (requested≠served) it FAILS CLOSED (non-zero, no
artifact); if absent, the requested model was served, so `resolved-model =
requested` and is allowlist-checked. The gate NEVER resolves from the prose
final-message or the session rollout. It ships a test that fails when fed a
requested≠served (fallback-present) fixture or an unreadable-stream fixture.

CAPTURE MECHANISM (new requirement — name the source, PRD-198 #3)
The item.error fallback signal is emitted to the codex action step's STDOUT as
JSONL (because `codex-args: --json`). A `uses:` action step's stdout cannot be teed
to a file by the workflow, and the action surfaces only the prose `final-message`.
Therefore the gate resolves by READING THE ACTIONS JOB-LOG POST-HOC THROUGH THE
GITHUB API (e.g. `GET /repos/{owner}/{repo}/actions/jobs/{job_id}/logs`, or the run
logs, using GITHUB_TOKEN with `actions: read`) and parsing the structured item.error
line from that log. It MUST NOT run codex in a `run:` step (that consumes
OPENAI_API_KEY directly and breaks the secret-proxy model — NON-GOAL). Because a
job's own log is not reliably flushed/available via the API until the job
completes, the resolution + fail-closed + allowlist check + artifact assembly run
AFTER the codex review step's job has finished (a follow-on step/job that reads the
completed review job's log), not inside the same still-running step. The added
`actions: read` permission is scoped to the resolving job only; the key-holding
review job keeps `contents: read` and never gains write/actions scope.

RISKS
- Job-log availability/timing: the GitHub API may not expose a running job's log
  until completion; the resolver must read a COMPLETED job's log (topology above) or
  it will spuriously fail-closed. Validate against the real API behavior.
- Log-format drift: the resolver parses a structured JSONL line embedded in the
  Actions log (timestamp-prefixed). The matcher keys on the JSON object
  (`.item.type == "error"` AND `.item.message` matching the fallback text), not a
  brittle whole-line prose grep; pin against the REAL captured shape (R1).
- codex CLI version drift: `codex-version` is unpinned (the action installs a
  current CLI). A future CLI that DOES emit a structured served-model field, or that
  changes the fallback item.error wording/shape, would change this contract — the R3
  fixture is the canary, and the matcher must fail-closed on an unrecognized shape
  rather than silently pass (PRD-198 #1, #6).
- Fail-closed must not regress on a genuine honored run (no false INVALID when no
  fallback event is present).

REQUIREMENTS

R1 (amended) — honor verified via codex's STRUCTURED item.error fallback event
The gate stops reading a positive served-model id (none exists on this toolchain —
prose-only, PRD-198 #3 banned). Instead it parses the captured `--json` JSONL for
codex's structured fallback event and keys on its JSON shape, NOT a prose grep:
  a top-level event `{"type":"item.completed", "item":{ … }}` whose
  `item.type == "error"` AND whose `item.message` matches
  `^Model metadata for \`<requested>\` not found\. Defaulting to fallback metadata`
  (the exact item_0 message captured at job 83746430244).
- PRESENT (fallback fired) → the requested model was NOT honored → FAIL CLOSED
  (exit non-zero, NO artifact).
- ABSENT (no fallback) → the requested model WAS served → `resolved-model =
  requested`, then allowlist-checked against ALLOWED_CODEX_MODELS (fail-closed if
  off-allowlist).
The requested input, the session rollout, and the prose final-message are NEVER
resolution sources.
FAIL: a run whose stream contains the item.error fallback for the requested model
still emits an artifact recording resolved-model=requested as allowlist-verified
(as run 28264070271 did — fallback to gpt-4.1 fired, yet gpt-5-codex was recorded).

R2 (amended) — no proxy fallback; never resolve from prose; capture-source named
The rollout/session-config fallback is DELETED as a resolution source (it records
the requested model — a proxy, PRD-198 #3). The gate ALSO NEVER resolves from
`steps.codex.outputs.final-message` (the prose self-report, the incident's root,
PRD-198 #3). If the structured stream cannot be read at all (job-log unreadable /
no recognizable codex JSONL), the step EXITS NON-ZERO and emits NO artifact — no
fallback to rollout / requested / config / prose (PRD-198 #1 fail-loud + #3
authoritative-source). Resolution reads only the captured JSONL per CAPTURE
MECHANISM.
FAIL: the stream/job-log is unreadable yet an artifact is emitted with any
resolved-model value; OR any resolution path reads final-message/rollout/requested.

R3 (confirm — holds, retargeted to the fallback-error shape) — GENUINE RED test
tests/test_codex_review.py exercises the resolution logic against the REAL captured
stream (not string-presence):
  * requested≠served fixture (the captured stream WITH the item.error fallback) →
    the gate is INVALID / FAIL-CLOSED (no artifact);
  * honored fixture (a stream with NO item.error fallback) → PASS (resolved =
    requested, allowlist-checked);
  * unreadable-stream fixture → fail-closed.
The test FAILS against the current prose/rollout-fallback workflow and against any
prose-string-presence implementation, and passes only after the fix (PRD-198 #4:
every guard ships a red test that fails when violated).
FAIL: the test passes against the pre-fix workflow, or passes against an
implementation that merely greps prose for a model string (i.e. it doesn't prove
the fallback item.error drives fail-closed).

R4 (unchanged) — body model identity cannot contradict machine provenance
The artifact body no longer carries an independent prose `Model:` self-report that
can disagree with the machine provenance line; the model identity is recorded once,
from the machine resolution (PRD-198 #6: pin identities; one source of truth).
FAIL: the artifact shows a body `Model:` value different from the provenance
resolved-model line (as PRD-208/210 artifacts did: body gpt-4.1/gpt-5.5 vs
resolved gpt-5-codex).

R5 (unchanged) — verification runs in CI where the decision is made
The resolution + fail-closed + allowlist verification execute inside
codex-review.yml on GitHub Actions (the gate's decision point), and the R3 test
runs in the required CI `test` job (PRD-198 #5 environment parity).
FAIL: the verification or its test runs only locally / is skipped in CI.

DATA FLOW (amended)
codex (`--json`) → JSONL event stream on the action step's STDOUT (job log) →
[CAPTURE: read completed review job's log via GitHub API] → [R1 match item.error
fallback event] → PRESENT ⇒ fail-closed (R2) · ABSENT ⇒ resolved = requested →
allowlist check (fail-closed, R2) → artifact provenance (single identity source,
R4). The prose final-message, requested input, and session rollout are NEVER
resolution sources.

FAIL CONDITIONS
- artifact emitted when the item.error fallback fired (request not honored) (R1).
- resolution reads final-message/rollout/requested, or emits an artifact when the
  stream/job-log is unreadable (R2).
- contract test green against the pre-fix workflow or a prose-grep implementation
  (R3).
- body model identity diverges from the machine provenance line (R4).
- verification/test not exercised in CI (R5).

VALIDATION
- R1/R2: feed the resolver the captured stream WITH the item.error fallback →
  INVALID/non-zero, no artifact; a stream WITHOUT the fallback → artifact with
  resolved-model=requested (allowlist-checked); an unreadable stream → non-zero.
- R3 (RED): the fallback-present + unreadable fixtures fail on current code, pass
  after the fix; a prose-grep implementation still fails the fallback-present case.
- R4: rendered artifact has exactly one model-identity field; no contradictory
  prose line.
- R5: the test is collected by the CI `test` job; full suite green (CI is truth).
