REQUIREMENTS

R1 — Gate 8 failure message + pseudocode reflect equity-driven sizing
`docs/trade_qualification.md:192` currently reads `… — exceeds $200 maximum`
(a leftover of the retired `MAX_DOLLAR_RISK=200`); the code
(`qualification.py:432-437`) emits `… — exceeds budget ($<effective_target>
after <regime> multiplier)`. Update the failure-message example to the code's
actual format, and change the `:174` pseudocode `floor(150 / spread_cost)` to
divide by the regime-scaled `effective_target` it actually uses
(`qualification.py:421-430`).
FAIL: the doc still shows `exceeds $200 maximum`, or the Gate 8 pseudocode
still hardcodes `150` as the budget without the regime multiplier.

R2 — Regime enumeration includes EXPANSION
`docs/system_logic_map.md:26` lists the regime outputs as
`RISK_ON/RISK_OFF/NEUTRAL/CHAOTIC`, omitting EXPANSION — a first-class regime
(`config.py:124`, `regime.py`, branched on at `qualification.py:231,301,596`;
the entire continuation path is EXPANSION-only). Add EXPANSION to the list.
FAIL: the regime enumeration at `system_logic_map.md:26` still omits EXPANSION.

R3 — Continuation extension check documents its EMA21-None fail-open
The CONTINUATION table (`docs/trade_qualification.md:324`) presents check 6
(`EXTENDED_FROM_MEAN`) unconditionally, but the code
(`qualification.py:676-680`) guards it with `if dm.ema21 is not None:` and
passes the check when EMA21 is absent — a fail-open, matching the DIRECT-path
Gate 10 whose fail-open IS documented (`:277`). Add the fail-open note.
FAIL: the continuation check-6 row still implies the extension check always runs.

R4 — Correct the "halves" verb on the NEUTRAL multiplier
`docs/trade_qualification.md:250` says `NEUTRAL=0.6 halves it`; 0.6 is a 40%
reduction, not a halving (`config.py:122`). Correct the description; the
quoted value 0.6 is right and stays.
FAIL: the doc still says the 0.6 NEUTRAL multiplier "halves" the budget.

R5 — Reconcile the FVGZone shape vs PRD-007 R1
`PRD-007.md:33` specifies `FVGZone MUST contain: high, low, midpoint,
direction`; the shipped dataclass (`qualification.py:112-115`) has exactly
`upper_bound, lower_bound` (midpoint computed on demand at `:768`, direction
passed to `detect_fvg`). PRD-007 is COMPLETE, so this is historical spec
superseded at implementation — record it in `docs/DECISIONS.md` and add a
one-line clarification where `trade_qualification.md` describes the zone
bounds (PULLBACK section) so the closed PRD's field list is not mistaken for
current.
FAIL: neither DECISIONS nor `trade_qualification.md` records that the live
FVGZone is `upper_bound/lower_bound` (PRD-007's field list superseded).
