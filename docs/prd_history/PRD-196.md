# PRD-196: Robust PROJECT_STATE bullet matching in prd_close.sh

Lane: STANDARD / INFRA
Status: IN PROGRESS
Author: Claude Code (scoped per 2026-06-17 morning charge; held for Dustin)
Filed: 2026-06-17

## Problem

`scripts/prd_close.sh` updates a fixed set of `PROJECT_STATE.md` bullets at
closeout (the `- **Active PRD:**` pointer and the `- **Test baseline:**`
bullet). Its bullet matching is brittle: when a bullet's surrounding text
drifts from the exact pattern the script expects, the script emits a soft
`WARN: ... not found` and **silently skips that bullet**, then exits 0. The
closeout still "succeeds," so the stale bullet survives unless an operator
notices the warning and hand-patches the value.

This has now fired twice, each time on a *governance number*:

1. The `- **Active PRD:**` anchor (surfaced and realigned under PRD-183).
2. The `- **Test baseline:**` bullet (PRD-179 closeout, 2026-06-17): the
   script warned `test baseline bullet not found`, left the baseline at the
   prior `2736`, and the count had to be hand-patched. The hand-patched number
   was itself wrong against CI truth (sandbox `2768` vs CI `2773`), which is
   exactly how a silent skip poisons a downstream drift check.

A closeout tool that can silently leave a governance number stale is the wrong
failure mode. The fix is twofold: make the matching robust to benign format
variation, and make a genuine no-match **fail loud** (non-zero exit) instead of
WARN-and-continue.

## Scope / approach (to be finalized at implementation)

- Replace each brittle single-pattern bullet match with tolerant matching
  (anchor on the bold bullet label, e.g. `**Test baseline:**` / `**Active
  PRD:**`, independent of trailing prose).
- Convert "bullet not found" from a soft WARN to a hard error: if any bullet
  the closeout is contracted to update cannot be located, the script exits
  non-zero and writes nothing, so the operator fixes the anchor rather than
  shipping a stale governance number.
- Keep the registry-row / PRD-doc / prd_index edits unchanged (those already
  work); this PRD is strictly about the PROJECT_STATE bullet edits.

## FILES

- `scripts/prd_close.sh` — robust bullet matching + fail-loud on no-match.
- `tests/test_prd_close.py` — add a case asserting the script exits non-zero
  (and leaves PROJECT_STATE unmodified) when a contracted bullet is absent, and
  a case asserting the Test-baseline + Active-PRD bullets are updated across a
  benign format variation.

## Acceptance criteria

- Closing a PRD updates **every** contracted `PROJECT_STATE.md` bullet (Active
  PRD pointer, Test baseline) deterministically; no soft-skip path remains.
- A missing/unmatchable contracted bullet causes a non-zero exit with an
  explicit error naming the bullet; PROJECT_STATE is left unmodified in that case.
- `prd-closeout-verified` no longer needs a hand-patch step for the baseline.

## Non-goals

- Not changing the `PROJECT_STATE.md` format/schema itself.
- Not changing the `prd-closeout-verified` skill or any governance guardrail in
  `CLAUDE.md`.
- Not touching the registry-row, PRD-doc status, or `prd_index.json` edit paths.

## FAIL-line observability

- FAIL: `tests/test_prd_close.py` test asserting non-zero exit + unmodified
  PROJECT_STATE when a contracted bullet is absent (today the script would
  WARN, skip, and exit 0 — the test would fail against current `prd_close.sh`).
