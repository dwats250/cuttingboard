# PRD-196: prd_close.sh baseline hygiene (robust bullet matching + CI-sourced baseline)

Lane: STANDARD / INFRA
Status: IN PROGRESS
Author: Claude Code (scoped per 2026-06-17 morning charge; held for Dustin)
Filed: 2026-06-17

## Problem

`scripts/prd_close.sh` writes governance numbers into `docs/PROJECT_STATE.md`
during closeout. Two distinct defects have each produced a wrong or stale
governance number, both surfaced during the PRD-179 closeout (2026-06-17):

**(a) Silent bullet-match self-skip.** The script updates a fixed set of
PROJECT_STATE bullets (the `- **Active PRD:**` pointer and the
`- **Test baseline:**` bullet). Its matching is brittle: when a bullet's
surrounding text drifts from the exact expected pattern, the script emits a soft
`WARN: ... not found`, **silently skips that bullet, and still exits 0**. The
closeout "succeeds" while leaving the bullet stale. This has now fired twice on
governance bullets — the Active PRD anchor (realigned under PRD-183) and the
Test baseline bullet (PRD-179) — each forcing a manual hand-patch.

**(b) Baseline-source bug.** The recorded test baseline is whatever pass count
is handed to `--tests`, which in practice is the **local/sandbox** count. In
this environment 5 git-commit-signing tests fail env-only and pass in CI, so the
sandbox count is understated by 5 (PRD-179: sandbox 2768 vs CI truth 2773). A
sandbox number then becomes the recorded anchor — and the baseline is exactly
what future drift checks compare against, so an understated anchor silently
poisons every later "did the suite change?" read.

Together: a closeout tool can both silently skip a governance bullet AND, when
it does write one, record a sandbox-local number instead of CI truth. The fix is
to make bullet matching robust + fail-loud, and to source the baseline from CI
on merged `main` rather than from a hand-supplied/sandbox count.

## Scope / approach (to be finalized at implementation)

1. **Robust, fail-loud bullet matching.** Anchor each contracted bullet on its
   bold label (e.g. `**Test baseline:**`, `**Active PRD:**`) independent of
   trailing prose. If any contracted bullet cannot be located, the script exits
   non-zero and writes nothing — no soft WARN-and-continue path remains.
2. **CI-sourced baseline.** Derive the recorded baseline from the CI `test` job
   of the merge commit on `main` (the authoritative pass/xfail count), rather
   than trusting a hand-supplied `--tests` that may carry a sandbox count.
   Concretely (to finalize at build): resolve the merge SHA's CI run, read the
   pytest summary line from the `test` job, and record that count; fail loud if
   the CI count cannot be resolved rather than silently falling back to a local
   number.
3. Leave the registry-row / PRD-doc-status / `prd_index.json` edit paths
   unchanged — they already work; this PRD is strictly PROJECT_STATE hygiene.

## FILES

- `scripts/prd_close.sh` — robust + fail-loud bullet matching; CI-sourced
  baseline resolution (fail-loud if unresolved).
- `tests/test_prd_close.py` — (b1) assert non-zero exit + unmodified
  PROJECT_STATE when a contracted bullet is absent; (b2) assert the Active-PRD
  and Test-baseline bullets update across benign format variation; (b3) assert
  the baseline is taken from the CI-reported count, not a passed-in sandbox
  number (with the CI lookup stubbed/injected so the test is hermetic).

## Acceptance criteria

- Closing a PRD updates **every** contracted PROJECT_STATE bullet
  deterministically; no soft-skip path survives.
- A missing/unmatchable contracted bullet → non-zero exit naming the bullet;
  PROJECT_STATE left unmodified.
- The recorded Test baseline equals the CI `test`-job count for the merge commit;
  a sandbox count (understated by env-only signing failures) can never become
  the recorded anchor. If the CI count cannot be resolved, the script fails loud
  rather than recording a local number.
- `prd-closeout-verified` no longer needs a hand-patch step for the baseline.

## Non-goals

- Not changing the `PROJECT_STATE.md` format/schema itself.
- Not changing the `prd-closeout-verified` skill or any governance guardrail in
  `CLAUDE.md`.
- Not changing the registry-row, PRD-doc status, or `prd_index.json` edit paths.
- Not fixing the underlying sandbox signing-test failures (separate concern; the
  baseline simply sources CI truth instead).

## FAIL-line observability

- FAIL: `tests/test_prd_close.py` — (b1) non-zero exit + unmodified
  PROJECT_STATE when a contracted bullet is absent (today the script WARNs,
  skips, exits 0 → fails against current `prd_close.sh`); (b3) baseline equals
  the CI-reported count, not the passed-in sandbox number (today the script
  records `--tests` verbatim → fails against current `prd_close.sh`).
