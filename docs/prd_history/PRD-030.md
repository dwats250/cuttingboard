PRD-030 — SCENARIO ENGINE HARDENING (REGIME + LEVEL DRIVEN)

STATUS
COMPLETE
Commit: 83bdd3b

GOAL
Replace static premarket scenario templates with a deterministic,
rule-based scenario generator driven by:
- market_regime (RISK_ON / RISK_OFF / NEUTRAL / CHAOTIC / EXPANSION)
- gap_direction (derived from PRD-029 levels)
- derived levels: prior_high, prior_low, range_mid

Scenarios MUST be generated programmatically and cover all five
canonical regime states including EXPANSION, which the static
templates do not currently handle (falls through to DEFAULT).

---

SCOPE

- Modify cuttingboard/reports/premarket.py:
  - Remove static dicts: _SCENARIOS, _DEFAULT_SCENARIOS,
    _INVALIDATION, _DEFAULT_INVALIDATION
  - Add: _generate_scenarios(regime, gap_direction, levels) → list[dict]
  - Add: _generate_invalidation(regime, gap_direction, scenarios) → list[str]
  - Update build_premarket_report to call new generators

- Add tests/test_scenario_engine.py

---

OUT OF SCOPE

- No changes to postmarket logic
- No intraday data
- No VWAP or ORB logic
- No contract schema modification
- No execution pipeline changes
- No probabilistic scoring
- No changes to execution_posture logic

REQUIREMENTS

R1 — Scenario Generation Function

System MUST implement:
  _generate_scenarios(regime, gap_direction, levels) → list[dict]

Each scenario MUST have exactly these keys:
{
  "id":                 str,
  "condition":          str,
  "expected_behavior":  str,
  "preferred_direction": "LONG" | "SHORT" | "NEUTRAL"
}

No additional keys. No missing keys.

FAIL:
- Any scenario missing required keys
- Any scenario with extra keys
- preferred_direction outside {"LONG", "SHORT", "NEUTRAL"}

---

R2 — Deterministic Output

Given identical (regime, gap_direction, levels), output MUST be identical.

FAIL:
Two calls with identical inputs return different scenario lists.

---

R3 — Scenario Count

System MUST return exactly 2 or 3 scenarios per call.

FAIL:
- Fewer than 2 scenarios returned
- More than 3 scenarios returned

---

R4 — Level Token Enforcement

Each scenario.condition MUST contain at least one exact token from:
  "prior_high" | "prior_low" | "range_mid" | "gap_direction"

FAIL:
Any scenario.condition missing all tokens from the allowed set.

---

R5 — Regime Alignment (Primary Direction Axis)

Scenario preferred_direction MUST be consistent with market_regime.
Regime governs direction; gap_direction may not override it.

RISK_ON or EXPANSION:
  MUST include at least one scenario with preferred_direction == "LONG"
  MUST NOT include any scenario with preferred_direction == "SHORT"

RISK_OFF:
  MUST include at least one scenario with preferred_direction == "SHORT"
  MUST NOT include any scenario with preferred_direction == "LONG"

NEUTRAL:
  MUST include at least one scenario with preferred_direction == "NEUTRAL"
  MAY include directional scenarios as secondary

CHAOTIC:
  ALL scenarios MUST have preferred_direction == "NEUTRAL"

Unknown / null regime:
  ALL scenarios MUST have preferred_direction == "NEUTRAL"

FAIL:
- RISK_ON or EXPANSION scenario has preferred_direction == "SHORT"
- RISK_OFF scenario has preferred_direction == "LONG"
- CHAOTIC scenario has preferred_direction != "NEUTRAL"

---

R6 — Gap-Direction Sub-Branching (Secondary Condition Axis)

Within a regime, gap_direction MUST influence at least one scenario condition.

RISK_ON / EXPANSION + gap_direction == "UP":
  At least one condition MUST reference "prior_high" as continuation target

RISK_ON / EXPANSION + gap_direction == "DOWN":
  At least one condition MUST reference "prior_low" or "range_mid"
  as stabilization zone before re-entry

RISK_OFF + gap_direction == "DOWN":
  At least one condition MUST reference "prior_low" as continuation target

RISK_OFF + gap_direction == "UP":
  At least one condition MUST reference "range_mid" or "prior_high"
  as fade/rejection level

NEUTRAL (any gap_direction):
  At least one condition MUST reference "range_mid"
  OR reference both "prior_high" and "prior_low"

CHAOTIC or gap_direction == null:
  At least one condition MUST contain the token "gap_direction"
  to convey that state is undefined

FAIL:
Required level reference missing for the given regime/gap_direction combination.

---

R7 — Invalidation Derivation

_generate_invalidation(regime, gap_direction, scenarios) → list[str]
MUST return at least 2 invalidation conditions.

At least one condition MUST reference the opposite-side level of the
primary directional scenario:
  LONG primary scenario → invalidation references prior_high or prior_close
    breakdown (loss of upside reference)
  SHORT primary scenario → invalidation references prior_low or prior_close
    reclaim (loss of downside reference)

Each condition MUST contain at least one exact token from:
  "prior_high" | "prior_low" | "range_mid" | "gap_direction" | "prior_close"

FAIL:
- Fewer than 2 invalidation conditions returned
- No condition referencing the opposite-side level of the primary scenario
- Any condition missing all required level tokens

---

R8 — Scenario Uniqueness

No two scenarios in the returned list MAY share both identical condition
AND identical preferred_direction.

FAIL:
Two scenarios have identical condition AND identical preferred_direction.

---

R9 — Backward Compatibility

build_premarket_report(contract, levels=None) MUST:
  - Use new scenario engine when levels is provided
  - When levels is None, generate scenarios using regime alone
    (gap_direction treated as null, levels treated as empty dict)
  - Return the same top-level output keys as before

FAIL:
- Function raises when called with levels=None
- Top-level output key set changes

---

R10 — EXPANSION Regime Coverage

When market_regime == "EXPANSION":
  MUST generate scenarios specific to expansion context (breadth
  continuation / breakout) distinct from generic unknown/default scenarios.
  At least one condition MUST reference "prior_high" as a breakout level.
  ALL scenarios MUST have preferred_direction == "LONG".

FAIL:
EXPANSION regime falls through to default or unknown scenarios.

---

R11 — No External Data Dependencies

_generate_scenarios and _generate_invalidation MUST NOT:
  - fetch data
  - read files
  - write files
  - call providers
  - depend on runtime state

They MAY use constants or helper functions defined in premarket.py.

FAIL:
Any data fetch, file access, provider call, or runtime-state dependency
occurs inside either generator.

---

DATA FLOW

contract.system_state.market_regime ─┐
levels.gap_direction ─────────────────┼→ _generate_scenarios → scenario list
levels (prior_high/low/range_mid) ───┘                              │
                                                                     ↓
                                              _generate_invalidation → invalidation list
                                                                     │
                                              build_premarket_report ← both

---

FAIL CONDITIONS

- Missing or malformed scenario objects
- Non-deterministic output
- Scenario count outside [2, 3]
- Missing level tokens in conditions or invalidation
- Regime direction contradiction (R5)
- Required gap-direction reference missing (R6)
- Invalidation count < 2 or missing opposite-side level reference (R7)
- Duplicate scenarios (same condition + same direction) (R8)
- levels=None raises or changes output key set (R9)
- EXPANSION falls through to default/unknown scenarios (R10)
- External data dependency introduced (R11)

---

VALIDATION

Run:
pytest tests/test_scenario_engine.py -q

Tests MUST cover:

- Determinism: same inputs → same output
- Regime alignment per R5: RISK_ON, RISK_OFF, NEUTRAL, CHAOTIC, EXPANSION, null
- Gap-direction sub-branches per R6: UP / DOWN / FLAT / null — per regime
- Scenario count: 2 and 3 scenario paths both exercised
- Level token presence: every scenario.condition in every regime
- Invalidation minimum count and level token presence
- Opposite-side level in invalidation for RISK_ON and RISK_OFF
- Scenario uniqueness: no identical (condition, direction) pair
- Backward compatibility: levels=None does not raise; output keys unchanged
- EXPANSION scenarios distinct from default/unknown path
- No external data access within generator functions
