PRD-260 — Continuation decision geometry: synthesize the promoted candidate

Status: IN PROGRESS

WHY NOW
PRD-259 makes continuation acceptance possible for the first time, but
an accepted continuation result reaches `create_trade_decision` wearing
the geometry of the DIRECT candidate that just FAILED qualification
(stop = quote - 1x ATR14), not the breakout-level stop the gate
validated — the rendered invalidation level lies at the one seam the
system exists to make trustworthy. Confirmed independently four times
(Stage-0 audit, both PRD-259 review legs, the connector bot's live
repro: breakout stop 101, rendered stop 102). Dustin's 2026-07-15
ruling (docs/DECISIONS.md): this PRD is a BLOCKING DEPENDENCY of
PR #148's merge — the gate goes live once, correctly, or stays dead;
the pair lands in one squash or not at all.

MAX EXPECTED DELTA
Production code: <= 60 LOC across `cuttingboard/qualification.py`,
`cuttingboard/runtime/__init__.py`, `cuttingboard/output.py`,
`ui/app.js`. Tests: <= 200 LOC across the four test files in FILES.
Docs: CONTINUATION geometry paragraph in `docs/trade_qualification.md`;
promotion/closure notes in the first-fire sidecar. Review artifacts

GOAL
An accepted continuation candidate's TradeDecision, contract candidate,
and rendered card carry exactly the geometry the gate qualified on —
entry = accepted close, stop = breakout level, target = the synthetic
3x ATR14 ceiling labeled as such — with the failed direct candidate
totally replaced and the qualified/excluded contradiction removed.

SCOPE
- R1 (Gate A Q1a): at continuation acceptance, `qualify_all` builds a
  `TradeCandidate` (entry = accepted close, stop = `breakout_level`,
  target per R2, direction LONG) and returns it on a new
  `QualificationSummary.continuation_candidates` dict (additive field,
  default empty); the runtime merges it into `candidates` before
  `_run_decision_gates` — total promotion, the failed direct candidate
  for that symbol is replaced.
- R2 (Q2a): target = entry + `CONTINUATION_REWARD_ATR_MULTIPLE` x
  ATR14, so the rendered R:R equals the figure the gate qualified on.
- R3 (Q2a, card label): every card surface rendering a continuation
  candidate's target/R:R tags it as a synthetic ceiling — the rendered
  report (`output.py`) and the dashboard cards (`ui/app.js`), branching
  on the EXISTING `entry_mode` contract field; no schema change.
- R4 (Q3a): on promotion, the symbol's `excluded` entry is rewritten to
  record both facts ("DIRECT rejected (<original reason>); promoted via
  CONTINUATION") — audit trail preserved, contradiction removed.
- R5 (Q4): fail-loud guard — `_run_decision_gates` raises a named error
  (not KeyError) when a setup's symbol has no candidate, with a red
  test.
- R6: sizing invariance — PRD-256/259's validated dollars are unchanged
  by the synthesized candidate's presence at the options layer.

OUT OF SCOPE
- Sol findings E (qualify-only notification READY paths) and F
  (postmarket accepted-count silence) — ruled non-blocking, stay
  ticketed in the first-fire sidecar.
- Any change to gate order, thresholds, CONTINUATION_* constants, or
  the PRD-259 window/HOLD logic.
- Contract schema (`entry_mode` already exists in the candidate dict).

REQUIREMENTS

R1 — Synthesize and totally promote (red first)
Accepted continuation results produce a `TradeCandidate` carried on
`QualificationSummary.continuation_candidates` and merged over
`candidates[symbol]` in the runtime before decision assembly. The
proving test runs a direct-rejected symbol through to a continuation
acceptance and asserts the resulting TradeDecision has stop == the
gate's breakout level and entry == the accepted close; it MUST fail
against pre-change code (which renders the direct candidate's ATR
stop).

FAIL: an accepted continuation result reaches `create_trade_decision`
with any geometry other than entry = accepted close, stop = breakout
level, target = R2's value; or the proving test passes pre-change.

R2 — Synthetic-ceiling target
`target_price = entry + CONTINUATION_REWARD_ATR_MULTIPLE * dm.atr14`.
The TradeDecision's r_r therefore equals the gate's qualifying ratio
(reward / (entry - breakout_level)) within float tolerance.

FAIL: a test comparing the rendered r_r against the gate's qualifying
ratio for the same accepted candidate finds a mismatch.

R3 — Ceiling label on the card
When `entry_mode == "CONTINUATION"`, the rendered report's setup line
(`output.py`) and both dashboard card sites (`ui/app.js` primary card
and secondary table) append a visible "3xATR ceiling" tag to the
target/R:R display, sourced from the existing `entry_mode` field.

FAIL: a rendered report or dashboard card for a continuation candidate
shows target/R:R with no ceiling tag (asserted by a render test on
output.py; app.js verified by grep for the conditional at both R:R
sites plus the dashboard_preview manual step).

R4 — Exclusion rewrite on promotion
When the continuation pass promotes a symbol present in `excluded`, the
entry becomes "DIRECT rejected (<original reason>); promoted via
CONTINUATION". Symbols never direct-rejected gain no excluded entry.

FAIL: after a promotion, `excluded[symbol]` still carries the bare
direct-rejection text, or the symbol is absent from `excluded` when a
direct rejection had been recorded.

R5 — Fail-loud candidate join (red first)
`_run_decision_gates` raises a named, message-bearing error when a
setup's symbol is missing from `candidates` (PRD-198 invariant 1),
instead of a bare KeyError. Red test: constructed setup with no
candidate raises the named error; pre-change it raises KeyError.

FAIL: the missing-candidate path raises KeyError, or no red test proves
the named error.

R6 — Sizing invariance at the options layer
With the synthesized candidate present, `build_option_setups` produces
the same `max_contracts`/`dollar_risk` for a continuation result as the
candidate=None shape PRD-256/259 validated (strategy-aware max loss is
candidate-independent).

FAIL: a paired test (same continuation result, candidate=None vs
synthesized candidate) shows different final contracts or dollar_risk.

R7 — Promote the continuation candidate on the hourly path too (red
first; added by amendment, 2026-07-17)
The live hourly path (`runtime/__init__.py`'s `_HOURLY_MODES` branch)
calls `qualify_all` and builds `candidate_lines` via
`_build_hourly_candidate_lines`, but — unlike the daily pipeline's R1
merge — passed the ORIGINAL, un-merged `candidates` dict straight
through. For a promoted continuation symbol, `_hourly_rr` computed R:R
from the failed direct candidate's stale ATR-based stop/target instead
of the synthesized breakout-level geometry, and that wrong R:R was
persisted into `candidate_lines` / `latest_hourly_run.json` via
`alert_runner.py` — not displayed in the Telegram body today, but
computed, passed through the messaging path, and written to disk. Found
by the commissioned gpt-5.6-sol consumer-sweep review (Q5, PRD-260's own
HIGH-RISK ceremony), which built an in-memory repro against the real
`qualify_all` (direct: entry 100.0/stop 99.2/target 101.6 -> hourly line
computed 2.0:1; promoted: entry 100.0/stop 98.9/target 102.4 -> correct
value 2.2:1). Fix is the identical merge R1 already applies at the
daily-pipeline call site, mirrored at this second one — same shape,
same code, different location; not a new mechanism.

FAIL: a promoted continuation candidate's hourly `candidate_lines` entry
computes R:R from any geometry other than
`qualification_summary.continuation_candidates[symbol]`; or the proving
test (`tests/test_hourly_alert.py::test_hourly_candidate_line_uses_promoted_geometry_prd260_r7`)
passes against pre-R7 code.
