## Findings

### P1 — Use the scheduled start time for intraday slots (`scripts/resolve_run_mode.py`)
> In `.github/workflows/cuttingboard.yml` the `Determine run mode` step runs only
> after checkout, setup, dependency install, lint, pytest, engine-doctor, and
> artifact upload, so this samples the resolver execution time rather than the
> cron start time. For the shared `*/30` intraday cron, any 14:30/16:30/19:00/
> 20:00 run whose queue delay plus those pre-steps exceeds 15 minutes falls
> outside `_nearest_intraday_slot` and resolves to `noop`, meaning the new
> tolerance can still silently skip the intraday scans on busy or slow runs.

**Accepted (P1, correct).** Resolution: the `Determine run mode` step was moved to
run immediately after `Set up Python 3.11`, BEFORE install/lint/test/engine-doctor
(cuttingboard.yml). The resolver is stdlib-only, so the sampled time is now the
run start plus queue delay and checkout/setup only (seconds), within the ±15
window. Dedicated crons are unaffected (schedule-string resolution is
time-independent).

### P2 — Prevent inactive intraday crons from firing early slots (`scripts/resolve_run_mode.py`)
> Because this uses absolute distance, an inactive `*/30` fire immediately before
> an active slot can claim that future slot: for example, the 14:00 invocation
> that starts at 14:15 maps to the 14:30 `post_orb` scan, and the 19:30
> invocation that starts at 19:45 maps to the 20:00 `power_hour` scan. Those runs
> execute before their intended market-data window and then the audit dedup can
> suppress the correctly timed invocation.

**Accepted (P2, correct).** Resolution: slot attribution is now LATE-ONLY —
`_intraday_slot_for_start` attributes a start to a slot only when
`0 <= start − nominal <= tolerance` (a run never starts before its scheduled
time, since queue delay is non-negative). The fired-today dedup attributes each
audit record through the same late-only rule, so an earlier inactive fire (or a
late different-slot run) can never suppress a slot. Tests added:
`test_early_fire_does_not_claim_future_slot`, `test_late_only_tolerance_boundary`,
`test_late_other_slot_record_does_not_suppress`.

### P1 (commit 847c53f) — Return only CLI-supported run modes (`scripts/resolve_run_mode.py` / workflow)
> When a queued ORB/intraday schedule is now successfully resolved to
> `orb_trajectory` (and `post_orb`/`midmorning`/`power_hour`), the workflow
> invokes `python -m cuttingboard --mode <that value>`, but `build_parser()` only
> accepts `live`, `fixture`, `sunday`, `verify`, `prefetch` (runtime:165-168).
> Those rescued slots exit in argparse before writing the audit/scoreboard
> records this PR is trying to restore.

**Accepted (P1, correct — verified: `--mode orb_trajectory` exits code 2).** This
is a long-latent bug in the per-mode dispatch invocations: the slot names are
`--notify-mode` values, not `--mode` values, and the steps never ran before
because the noop freeze never reached them. Resolution (Dustin-approved scope
amendment): the four intraday/orb dispatch steps now invoke
`--mode live --notify-mode <slot>` (mirroring `--mode live --notify-mode
premarket`); `live` is the correct base — the intraday slots are notify variants
of the same live scan, and `notify_mode` alone drives the qualify-only/hourly
behavior. The runtime/ parser is correct and untouched. Added a CI guard
(`test_every_workflow_invocation_is_parser_valid`,
`test_every_scheduled_slot_is_wired_to_a_valid_dispatch_step`) asserting every
resolver-emittable slot maps to a parser-accepted invocation.

### P1 (commit d3ce8ae) — Route slot scans through the notify runner (cuttingboard.yml)
> `cli_main()` passes `args.notify_mode` into `execute_run()`, but the full
> pipeline path does not branch on it; `_execute_notify_run()` is the code that
> formats the orb_trajectory/post_orb/midmorning/power_hour notifications. Every
> rescued slot using this pattern runs a full live premarket pipeline/notification
> instead of the intended slot scan.

**Accepted (correct — verified).** `_execute_notify_run` (runtime:355) is called by
`alert_runner.py` (the hourly_alert.yml path), not `cli_main`. `cli_main` →
`execute_run` → `_run_pipeline`, which writes the audit record (runtime:956) but
sends a generic live notification and does not branch on the slot notify_mode.

### P2 (commit d3ce8ae) — Dedup needs an actual slot/mode marker (resolve_run_mode.py)
> A delayed dedicated run can write an audit row in an intraday slot's window
> (e.g. a late 13:50 orb_trajectory record at 14:33), causing the real 14:30
> post_orb to resolve to noop. Dedup needs an actual slot/mode marker, not
> wall-clock attribution.

**Accepted (correct).** The audit record has no slot marker (`router_mode` is the
sector regime), so wall-clock attribution is inherently heuristic.

**Resolution of both (Dustin-approved descope 2026-06-16):** PRD-189 is reduced to
resolving the dedicated premarket crons (prefetch/live/sunday) by cron-string
lookup. The intraday/orb slots, their crons, and the four dispatch steps were
REMOVED from cuttingboard.yml — hourly_alert.yml already covers that window, and
their correct notify path (`_execute_notify_run`) plus a per-slot audit dedup
marker (runtime/audit.py) are protected-`runtime/` changes deferred to PRD-192.
With no intraday slots, both findings are moot for PRD-189 (the dedup and the
`--notify-mode <slot>` invocations no longer exist here).

## Verdict
Five actionable findings across the review cycle (P1 sampling, P2 early-fire, P1
invalid --mode, P1 notify-routing, P2 dedup-marker). The first three were fixed;
the last two are resolved by descoping the intraday slots out of PRD-189 and into
PRD-192. As shipped, PRD-189 changes no contract/payload shape and its decision
surface is limited to dedicated-cron run-mode selection + renderer freshness
tokens.
