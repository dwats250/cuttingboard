# PRD-052 — Runtime Artifact Self-Healing (Legacy Tolerance)

**Status:** COMPLETE
**Commit:** —
**Author:** Dustin Watson
**Date:** 2026-04-29

---

## GOAL

Ensure runtime artifact writers tolerate legacy latest-artifact files that are parseable JSON objects but missing the configured timestamp key, without crashing the pipeline.

## SCOPE

- `safe_write_latest()` behavior
- summary file writing
- timestamp enforcement logic
- runtime operationalization tests

## OUT OF SCOPE

- contract schema
- `audit.jsonl`
- evaluation layer
- execution adapter (PRD-053)
- invalid JSON recovery for existing files

## FILES

MODIFY:
- `cuttingboard/runtime.py`
- `tests/test_operationalization.py`

## REQUIREMENTS

### R1 — Legacy tolerance

If an existing latest artifact is a JSON object but missing the configured `ts_key`, `safe_write_latest()` must not raise and must overwrite the file with the new valid data.

FAIL: the pipeline crashes because an existing latest artifact lacks `ts_key`.

### R2 — Deterministic overwrite

Overwrite behavior must be full replacement based solely on current run data.

FAIL: legacy contents are merged into the replacement payload.

### R3 — Strict validation for new writes

New writes must still include `ts_key` and must parse as valid timestamp data before write.

FAIL: a new artifact without `ts_key` is written successfully.

### R4 — No silent corruption

If the existing file is a JSON object and contains `ts_key`, the existing monotonic timestamp comparison must remain in place.

FAIL: equal or older timestamps overwrite a newer artifact.

### R5 — Optional logging

If a legacy overwrite occurs, the runtime may log `LEGACY_ARTIFACT_OVERWRITE`.

## DATA FLOW

`summary / contract payload` -> `safe_write_latest()` -> `latest_*.json`

## ACCEPTANCE CRITERIA

AC1: Given an existing latest artifact without `ts_key`, the pipeline completes successfully.

AC2: The artifact is replaced with valid data that includes `ts_key`.

AC3: No `RuntimeError` is raised in that legacy-missing-key scenario.

AC4: If the existing artifact has `ts_key`, the monotonic timestamp check still applies.

AC5: No changes to contract schema, audit output, or evaluation output.

## VALIDATION

- Create a latest artifact JSON object without `run_at_utc` or `generated_at`, run the pipeline, and confirm it completes successfully.
- Verify the file is replaced with a valid structure containing the timestamp key.
- Run again with older and equal timestamps and confirm the existing monotonic guard still rejects overwrite.
