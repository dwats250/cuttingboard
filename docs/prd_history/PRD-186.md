PRD-186 — Drift-review gate: per-PRD drift check + post-merge audit teeth + governance auto-merge carve-out

Status: IN PROGRESS

LANE
HIGH-RISK

CLASS
GOVERNANCE

WHY NOW
PRD-184 made auto-merge the landing path, removing the pre-merge human
checkpoint. The PRD-184 audit confirmed two gaps: the per-PRD independent
review checks only correctness (no VISION/PROJECT_STATE drift), and no doc
states that drift-review is now a post-merge concern. Two more follow: the
existing Alignment cadence is a label without a defined action, and
auto-merge could land changes to its own guardrails (the review skill / the
auto-merge policy) with no human read.

MAX EXPECTED DELTA
Doc/skill edits only across CLAUDE.md + the prd-review-claude skill +
DECISIONS. No code, no tests, no workflows.

GOAL
Close the four drift-review gaps: (1) a per-PRD drift check recorded in the
review artifact, (2) docs that frame drift-review as a post-merge audit
under auto-merge, (3) a post-merge audit with a defined trigger / drift
definition / remediation, and (4) a manual-merge carve-out so auto-merge
never lands changes to its own guardrails without a human read.

SCOPE
- Add a lightweight DRIFT CHECK to the prd-review-claude skill (a recorded
  review section + a Phase-2 verification item) and to the CLAUDE.md
  HIGH-RISK review gate.
- Reframe the CLAUDE.md auto-merge bullet: under auto-merge there is no
  pre-merge human checkpoint, so drift-review is a post-merge audit.
- Give the Alignment cadence teeth: each run audits PRDs merged since the
  last audit against VISION/PROJECT_STATE; defines drift; remediation is a
  corrective PRD.
- Add a governance auto-merge carve-out: PRs that change the review-gate
  skill or the governance/auto-merge policy are manual-merge-only.

OUT OF SCOPE
- No change to ci.yml, branch protection, or any workflow.
- No new prediction/sidecar/engine logic.
- The mechanical enforcement of the carve-out (CODEOWNERS + branch
  protection) is RECOMMENDED here (see R4) but not landed in this PRD; it
  needs repo-admin and is a follow-up.

REQUIREMENTS

R1 — Per-PRD drift check in the review gate
The prd-review-claude skill records a DRIFT CHECK in every review artifact:
(i) does the change conflict with a VISION non-goal/principle? (ii) does it
leave any PROJECT_STATE claim stale? The CLAUDE.md HIGH-RISK review gate
states the review artifact must carry this drift check.

FAIL: the skill's Review structure / Phase-2 has no DRIFT CHECK item, OR the
CLAUDE.md review-gate bullet does not require a VISION/PROJECT_STATE drift
check in the review artifact.

R2 — Auto-merge framed as post-merge drift audit
The CLAUDE.md auto-merge bullet states that under auto-merge there is no
pre-merge human checkpoint, so VISION/PROJECT_STATE drift-review is a
post-merge audit (review artifact + Alignment cadence), not a pre-merge gate.

FAIL: the auto-merge bullet still describes only the mechanical flow with no
post-merge-audit framing.

R3 — Alignment cadence has teeth
The Alignment cadence defines: a trigger (each run audits every PRD merged
since the last audit against VISION + PROJECT_STATE), what counts as drift,
and a remediation path (open a corrective PRD when drift is found; record the
audit in DECISIONS). A control with a defined action, not a label.

FAIL: the cadence does not name the merged-PRD audit trigger, OR does not
define drift, OR does not state the corrective-PRD remediation.

R4 — Governance auto-merge carve-out
CLAUDE.md states that PRs changing the review-gate skill (prd-review-claude)
or the governance/auto-merge policy are manual-merge-only and excluded from
auto-merge (open and hold for a human merge; do not queue `gh pr merge
--auto`). The PRD specifies HOW this is enforced and recommends one option.

FAIL: no CLAUDE.md rule excludes guardrail-changing PRs from auto-merge, OR
the PRD does not recommend an enforcement option with rationale.

ENFORCEMENT RECOMMENDATION (R4)
Options considered:
- (a) Policy-only: a CLAUDE.md rule the agent honors (skip `--auto` on
  guardrail PRs). Zero friction, but no mechanical backstop - relies on the
  agent classifying the PR correctly.
- (b) Mechanical CODEOWNERS on the review-skill file: mechanically gates the
  skill file, but the governance/auto-merge POLICY lives in CLAUDE.md;
  CODEOWNERS-protecting all of CLAUDE.md would gate every routine CLAUDE.md
  edit - friction.
- (c) Relocate the governance/auto-merge policy bullets into a dedicated
  protectable file, then CODEOWNERS covers {that file + the review skill}
  WITHOUT covering routine CLAUDE.md/PROJECT_STATE content.

RECOMMENDED: (c). It is the only option that is both mechanical (a CODEOWNERS
+ branch-protection "require review from Code Owners" gate that auto-merge
cannot satisfy, forcing a human merge) AND friction-free for routine
CLAUDE.md/PROJECT_STATE edits (those files are not in a CODEOWNERS path).
(a) has no backstop; (b) cannot cover the CLAUDE.md policy without taxing
every CLAUDE.md edit. This PRD lands (a) as the immediate, agent-honored
carve-out (this PRD itself is manual-merge, demonstrating it); (c) is the
recommended mechanical hardening and needs repo-admin (enable "require Code
Owner review" on the protected paths) + the relocation, tracked as a
follow-up. Note: on this single-owner repo (no CODEOWNERS today) a "require
Code Owner review" gate can never be independently satisfied, so (c) does not
manufacture a second reviewer - it blocks auto-merge and forces a manual
admin-merge. That is the intended effect (stop auto-merge from landing a
guardrail change without a human read), not a defect.
