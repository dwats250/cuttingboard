## WHY NOW

The real finding is three distinct failure shapes across 25 docs and six
status-header conventions, discovered incrementally — not the single
instance this PRD was first framed around.

`tools/validate_prd_registry.py`'s `_validate_doc_status_agreement`
(PRD-164 R6b) only fires when a PRD doc's trailing status line matches the
exact convention `STATUS: COMPLETE @ <hash>`. When a PRD doc still reads
IN PROGRESS or PROPOSED while its registry row reads COMPLETE, the regex
never matches, the check silently `continue`s, and no error is raised
anywhere — a closeout that flips the registry but never flips the doc's own
status word produces zero signal. That is Shape 1.

This was found as a single instance (PRD-088) during a read-only
closeout-integrity audit. Simulating a proposed fix against every
registry-COMPLETE row before writing this PRD surfaced 13 more Shape-1
instances — PRD-107, 108, 109, 110, 111, 129, 135, 136, 137, 138, 152, 197
— spanning a historical closeout period where doc-status flipping wasn't
consistently enforced. Widening the parser to recognize the six header
conventions those docs actually use (REQUIREMENTS R1) then surfaced Shape
2: PRD-056 and PRD-057, whose docs carry no status line at all, in any
convention. Total at that point: 15 rows, and a first version of this PRD's
check shipped against them using the pass condition "COMPLETE appears
anywhere in the extracted word set."

That pass condition was itself the same class of defect it was written to
fix — a detector satisfied by a partial match, not genuine agreement. A
`chatgpt-codex-connector` review on this PRD's own PR caught it directly:
PRD-197's doc has *two* status declarations (a header, which the first
version's fix corrected, and a separate trailing `STATUS: IN PROGRESS`
line, which it didn't), and "COMPLETE somewhere in the doc" passed it
anyway because the header now said COMPLETE. Re-simulating the full
registry with a strict "the doc's status words must be exactly `{COMPLETE}`"
rule surfaced Shape 3 in ten more rows, in two mirror-image forms:

- **Group A (8 rows: PRD-140, 144, 145, 146, 147, 148, 154, 155).** The
  header — the field a human reads first — still says PROPOSED or IN
  PROGRESS, while a separate, correctly-formatted trailing
  `STATUS: COMPLETE @ <hash>` line already exists lower in the doc. The
  correct trailing line is exactly what let both the pre-existing hash
  check and this PRD's first-version check pass these silently.
- **Group B (2 rows: PRD-189, 194) plus PRD-197 (already inside the
  original 15, now fully closed rather than partially):** the inverse —
  header correctly says COMPLETE, but a leftover trailing
  `STATUS: IN PROGRESS` line was never updated at closeout.

Total: **25 registry-COMPLETE rows, three distinct failure shapes, six
status-header conventions, and a pass condition that would have shipped
with its own partial-match hole** had the review not caught it before
merge.

This does not qualify for the WONTFIX-HISTORICAL treatment PRD-243 gave the
phantom-SHA debt, for any of the 25. That waiver applies because the
correct commit values are genuinely unrecoverable — squash-merged away,
gone from retention. Here every correct value is already known: the
registry says COMPLETE and the work landed (spot-verified for PRD-088
against git/GitHub history; the other 24 carry the same registry-level
truth, and their commit cells are separately covered by the
already-adjudicated WONTFIX-HISTORICAL debt or, for PRD-189/194/197,
already resolvable and now written into the trailing line itself). A
waiver is for debt that cannot be paid, not debt that is tedious to pay. A
dated exemption floor (mirroring `TRACKING_START`/
`SECOND_MODEL_DISPOSITION_START`) was considered and rejected, twice: it
would ship the detector with a permanent hole over exactly the rows that
proved the gap exists — a documented exemption from day one, recreating
the blind spot this PRD closes. Splitting the 10 newly-found rows into a
follow-up PRD was considered and rejected for the same reason the original
15 weren't split: it would ship a check that goes red on ten known rows at
its first CI run.

A second review pass on the tightened check caught a third, distinct
partial-match hole before merge: `_match_status_candidate`'s prefix check
(`upper.startswith(candidate)`) accepted a mistyped `Status: COMPLETED` as
`COMPLETE`, since `"COMPLETED".startswith("COMPLETE")` — silently treating
a malformed declaration as valid agreement. This is the third instance of
the same underlying pattern in this one check: "COMPLETE appears anywhere"
(the pass-condition fix above), a hash-format-only regex (the original 13),
and now a prefix match with no token boundary. Fixed by requiring the
character immediately after a matched candidate to be absent or
non-alphanumeric — pinned with a regression test for `COMPLETED` (the real
near-miss) and `INCOMPLETE` (a lexically related word that was never
actually a prefix match, pinned anyway to document that status matching is
whole-token, not substring/fuzzy). No live doc in this repo is currently
mistyped this way — the same argument that covered the earlier two holes
until one didn't, which is exactly why this is fixed rather than deferred.

A fourth review pass (on the token-boundary commit) caught two more,
underscoring the same lesson yet again: (a) the boundary check accepted
any non-alphanumeric follower, so `COMPLETE-ish` still passed — every real
convention in this repo separates a status word from any trailer with
whitespace, so the boundary is now whitespace-or-end specifically, not
"non-alphanumeric"; (b) a doc with one valid `Status: COMPLETE` line and a
second, separate malformed line (`STATUS: COMPLETED @ abc`) still passed,
because an unmatched line was silently dropped rather than recorded —
`_scan_doc_status_lines` now tracks malformed status-shaped lines
explicitly, and `_validate_doc_status_word_agreement` fails a doc that
carries any, not just a doc missing a valid one.

Recording malformed lines instead of dropping them immediately surfaced a
fifth problem, self-caught by running the tightened check against the
full live registry before shipping it (not by review this time): the
label match was case-insensitive, so ordinary body prose beginning a line
with lowercase "status" — data-flow bullets, code-enum descriptions,
field docs, all unrelated to the PRD's own completion state — was
misread as a malformed declaration. 16 false positives resulted. Every
real convention in this corpus capitalizes the label ("Status" or
"STATUS"); the label match now requires exact capitalization. A
second, related false positive survived that fix: a capitalized "Status"
used as an ordinary noun ("Status column: PARTIAL · Tier header: ...",
describing a dashboard column) rather than a declaration. Every real
convention has the label immediately followed by nothing or ":" — never a
space then another word — closing that gap too. Both false-positive
shapes are now pinned as regression tests directly, and the full live
registry (not just the 25 corrected docs) was re-verified clean after
each tightening, not merely re-simulated.

## GOAL

Add a validator check that reads a PRD doc's own status word directly —
independent of hash format and independent of which of the six observed
header conventions the doc uses — and flags any registry-COMPLETE row
whose doc does not declare status words that are exactly `{COMPLETE}`:
no recognizable status line at all, a recognized-but-wrong word, or two
conflicting declarations (a stale header alongside a correct trailer, or
the reverse) are all failures, not passes. Correct the 25 known-stale docs
in the same PRD, since the check alone breaks CI and the corrections alone
leave the gap; they are one indivisible unit (same pairing discipline as
PRD-259/260).

## SCOPE

- New `_extract_doc_status_words(text)` helper in
  `tools/validate_prd_registry.py`: scans a PRD doc's text line by line,
  recognizing every status-header convention observed in this repo's
  history —
  1. `**Status:** COMPLETE` (markdown bold)
  2. `Status: COMPLETE` / `STATUS: COMPLETE` (plain, colon same line)
  3. bare `STATUS` on its own line, value on the next line
  4. `\*\*Status:\*\*` (literal escaped asterisks — not real markdown)
  5. `## STATUS` (H2 heading, value on the next line)
  6. `- **STATUS:**` (bullet-list item)
  — by stripping bullet markers and markdown/escape decoration
  (`-`, `•`, `\`, `*`, `#`, `>`) from each line before checking whether the
  cleaned line's label reads "status", and returning the SET of every
  recognized `ALLOWED_STATUSES` value found anywhere in the doc (not just
  the header, and not just the first match — a doc may legitimately
  declare status in more than one place, e.g. a header plus a trailing
  hash-pinned line, and both must agree).
- New `_validate_doc_status_word_agreement(root, registry_rows, errors)`:
  for every registry row with `status == COMPLETE` (number >=
  `TRACKING_START`, file cell not `—`/empty, doc exists — same skip
  convention as the existing hash-based check), call
  `_extract_doc_status_words` on the doc text. **Pass only if the returned
  set equals exactly `{"COMPLETE"}`.** Otherwise: if the set is non-empty,
  emit a "Doc status disagreement" error naming every word found (this
  covers both a single wrong word and a doc that declares COMPLETE
  alongside a conflicting word); if the set is empty (no recognized
  convention matched, or the doc has no status line at all), emit a "Doc
  status unreadable" error. An unrecognized, absent, or internally
  inconsistent status declaration is a FAILURE, not a skip.
- Wire the new function into `validate_repository`, alongside the existing
  `_validate_doc_status_agreement` call. The existing hash-agreement check
  is untouched — this is an additional check, not a replacement.
- Correct the status word to COMPLETE in the 21 docs that currently read a
  stale word somewhere: the original 13 (PRD-088, 107, 108, 109, 110, 111,
  129, 135, 136, 137, 138, 152, 197) plus Group A's 8 (PRD-140, 144, 145,
  146, 147, 148, 154, 155) — a one-word edit in each file's existing status
  line, no other content touched.
- Add a status line reading `STATUS` / `COMPLETE` to PRD-056.md and
  PRD-057.md, which currently have none — placed immediately after each
  doc's title line, matching the bare-STATUS convention already used
  elsewhere in this era of the registry (e.g. PRD-081, PRD-107).
- Correct the stale trailing `STATUS: IN PROGRESS` line to
  `STATUS: COMPLETE @ <hash>` (hash matching the doc's own registry commit
  cell) in PRD-189.md, PRD-194.md, and PRD-197.md — Group B plus the
  completion of PRD-197's already-in-scope fix. This also lets the
  pre-existing hash-based check (`_validate_doc_status_agreement`) start
  passing on these three for the first time, since they previously had no
  line in that convention at all.
- New `_match_status_candidate(value_upper, ranked)` helper, shared by both
  extraction call sites: matches a candidate only when it is the whole
  value or followed by whitespace, closing the prefix-match hole
  described in WHY NOW.
- New `_scan_doc_status_lines(text)`, replacing the body of
  `_extract_doc_status_words` (now a thin wrapper over it): tracks every
  status-shaped line that fails to parse into a recognized word as
  `malformed`, separate from the recognized `words` set, so a doc with
  one valid and one malformed declaration cannot pass by the malformed
  line contributing nothing. The label match requires exact
  capitalization (`Status`/`STATUS`, never lowercase `status`) and the
  label immediately followed by nothing or `:` (never a space then
  another word), excluding both body prose and noun usage from being
  treated as an attempted declaration at all.
- New tests in `tests/test_prd_registry.py` covering: each of the six
  recognized conventions individually (agreement); the unrecognized-format
  and no-status-line failure cases; single-word disagreement; the existing
  hash check's independence from the new check; pinning the actual defect
  a review caught — one test reproducing Group A's shape (stale header,
  correct trailer) and one reproducing Group B's shape (correct header,
  stale trailer), each asserting the check fails on that doc; and pinning
  the token-boundary fix — `COMPLETED` and `INCOMPLETE` each asserted as
  NOT extracted as `COMPLETE`, plus an integration test that a
  registry-COMPLETE row with a `COMPLETED`-mistyped doc is flagged "Doc
  status unreadable."

## REQUIREMENTS

R1 — Status-word extraction recognizes all six observed conventions
`_extract_doc_status_words` must return `{"COMPLETE"}` when applied to a
fixture doc using any of the six conventions listed in SCOPE with the value
COMPLETE, and the correct non-COMPLETE word for IN PROGRESS/PROPOSED
fixtures in each convention.

FAIL: any of the six convention fixtures returns the wrong word set or an
empty set when a recognizable status line is present.

R2 — Unrecognized or absent status line is a failure, not a skip
When a registry-COMPLETE row's doc contains no line `_extract_doc_status_words`
recognizes as a status declaration (including a doc missing any status
line at all, and a doc using a deliberately invented seventh format), the
returned set is empty and `_validate_doc_status_word_agreement` emits a
"Doc status unreadable" error for that row rather than skipping it.

FAIL: a doc with an unrecognized/absent status line and a registry-COMPLETE
row produces no error.

R3 — Doc/registry disagreement on a recognized word is flagged
When a registry-COMPLETE row's doc reads a recognized but non-COMPLETE
status word (IN PROGRESS, PROPOSED, PATCH, DEPRECATED), the check emits a
"Doc status disagreement" error naming the word found.

FAIL: a doc reading a non-COMPLETE status word and a registry-COMPLETE row
produces no error.

R4 — Existing hash-agreement check is untouched
`_validate_doc_status_agreement` (the `STATUS: COMPLETE @ <hash>` ->
registry commit-set membership check) keeps its exact current behavior;
this PRD adds `_validate_doc_status_word_agreement` as an additional call
in `validate_repository`, never replacing or gating the existing call.

FAIL: any existing test in `tests/test_prd_registry.py` for
`_validate_doc_status_agreement` (the `test_r6_doc_status_*` family)
changes behavior or is deleted.

R5 — All 25 known-stale docs read exactly {COMPLETE} after this PRD
After the doc corrections, running `_validate_doc_status_word_agreement`
against the current registry produces zero "Doc status disagreement" or
"Doc status unreadable" errors for PRD-056, 057, 088, 107, 108, 109, 110,
111, 129, 135, 136, 137, 138, 140, 144, 145, 146, 147, 148, 152, 154, 155,
189, 194, 197.

FAIL: `python3 tools/validate_prd_registry.py` (full mode, run from a
checkout with complete history) reports a doc-status error for any of the
25 named PRDs.

R6 — No new doc-status errors introduced for any other registry-COMPLETE row
Running the full validator against the corrected registry produces no
doc-status error for any registry-COMPLETE row other than the 25 named in
R5.

FAIL: the validator reports a "Doc status disagreement" or "Doc status
unreadable" error for a PRD number outside the 25 named in R5.

R7 — The pass condition rejects a partial match (Group A and Group B)
`_validate_doc_status_word_agreement` fails a doc whose extracted status-word
set contains `"COMPLETE"` alongside any other word — a stale header next to
a correct trailer (Group A shape), or a correct header next to a stale
trailer (Group B shape) — not only a doc where `"COMPLETE"` is absent
entirely. The pass condition is `words == {"COMPLETE"}`, never
`"COMPLETE" in words`.

FAIL: a fixture doc reproducing the Group A shape (stale header, correct
trailing `STATUS: COMPLETE @ <hash>` line) or the Group B shape (correct
header, stale trailing `STATUS: IN PROGRESS` line) produces no error.

R8 — A matched candidate requires a whitespace-or-end token boundary
`_match_status_candidate` accepts a candidate match only when the matched
text is the entire value, or the character immediately following it is
whitespace. A value that extends a candidate with another letter, digit,
or directly-attached punctuation (e.g. `COMPLETED` or `COMPLETE-ish`
extending `COMPLETE`) is not treated as a match to that candidate.

FAIL: `_extract_doc_status_words` applied to a doc reading
`**Status:** COMPLETED` or `**Status:** COMPLETE-ish` (or any other
non-whitespace-separated extension of a valid status word) returns a
non-empty set.

R9 — A malformed status-shaped line is recorded, never silently dropped
`_scan_doc_status_lines` records the raw text of every line that matches
the label-and-boundary shape of a status declaration but whose value does
not match any `ALLOWED_STATUSES` candidate, in a `malformed` list separate
from the recognized `words` set. `_validate_doc_status_word_agreement`
fails a doc whose `malformed` list is non-empty, even when `words` is
exactly `{"COMPLETE"}`.

FAIL: a doc with one valid `Status: COMPLETE` declaration and one
separate malformed status-shaped line (e.g. `STATUS: COMPLETED @ abc`)
produces no error against a registry-COMPLETE row.

R10 — The label match excludes prose and noun usage
A line counts as a status-declaration label only when, after decoration
stripping, it begins with the exactly-capitalized token `Status` or
`STATUS`, AND that token is immediately followed by nothing (bare label)
or `:` — never by a space and further text. Plain lowercase `status`
(body prose) and a capitalized `Status` used as a noun followed by other
words before any colon (e.g. `Status column: ...`) are excluded entirely
— not recognized, and not recorded as malformed.

FAIL: `_scan_doc_status_lines` applied to a doc containing
`- status: one of VALID, INCOMPLETE, CONFLICTED, UNKNOWN` (lowercase
prose) or `Status column: PARTIAL · Tier header: ...` (noun usage)
returns a non-empty `malformed` list for either line, or the full live
registry (not just the 25 corrected docs) produces any `Doc status`
error when run through `tools/validate_prd_registry.py`.

## DATA FLOW

## TOOLING NOTE — GitHub CI verification (durable, unrelated to the check itself)

Found while verifying this PRD's own CI state: this repo's CI runs entirely
as GitHub Actions check-runs, and never populates GitHub's legacy combined
commit-status endpoint. A `get_status`-style call (the combined-status API)
returns `pending` / zero checks for this repo's commits regardless of
whether CI actually passed, failed, or never ran — the same empty result
in every case, not a real "pending" read. Checking that endpoint and
reporting its result as CI state is silently wrong, not merely incomplete.
Use the check-runs API (`get_check_runs` / equivalent) instead, and verify
against actual job log content (not just the conclusion field) when the
result matters for a merge decision. Worth writing down once rather than
re-discovering per session.
