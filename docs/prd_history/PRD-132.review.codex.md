## Questions and Answers

1. **Literal RVOL threshold vocabulary** - Acceptable. The 11-string closed
   vocabulary is mechanically structural: the nine R1 rows are literal
   combinations of `price_vs_vwap` and RVOL band strings (PRD-132 lines
   172-182), and the two R2 strings are VWAP data/computation states
   (lines 223-233). The three RVOL display fragments - `"RVOL >= 1.5x"`,
   `"RVOL < 1.5x"`, and `"RVOL unavailable"` - are numerical predicates or a
   data-state sentinel, not quality or directional language. R1 explicitly
   forbids magnitude adjectives and reuses the PRD-131 deny-list (lines
   190-199), and the OUT OF SCOPE section repeats the same boundary (lines
   113-126). The lock-step coupling is also acknowledged: the renderer-local
   threshold and displayed `"1.5x"` substrings must agree (lines 61-67), and
   any future threshold tune must update every R1 display string atomically
   (lines 201-204).

2. **Symmetric AT_LEVEL handling** - Correct call. The token emitter confirms
   `AT_LEVEL` is just a current-sample equality state from `_cmp()` (ABOVE,
   BELOW, AT_LEVEL only; `trend_structure.py` lines 106-117), while VWAP can
   also produce `DATA_UNAVAILABLE` or `NOT_COMPUTED` through the VWAP resolver
   (lines 152-162). Given that input domain, preserving RVOL banding for
   `AT_LEVEL` is operationally meaningful as a current structural fact:
   price is at VWAP and participation is at/above threshold, below threshold,
   or unavailable. The revised 3 x 3 table (PRD-132 lines 172-188) removes
   the earlier asymmetry and the defense burden that would come from hiding
   participation only in the AT_LEVEL branch. `"At VWAP, RVOL >= 1.5x"` does
   not itself imply a directional setup; it says nothing about long/short,
   support/resistance, follow-through, confirmation, or quality. The phrase is
   acceptable only because R1/R4/R5 keep it display-only and forbid decision
   feedback (lines 190-204, 251-258, 296-315).

3. **PRD-131 isolation** - Partially specified, not fully protected. The
   intended isolation is clear: PRD-132 says there must be no change to
   PRD-131's SMA composite vocabulary, mapping, or render position (lines
   104-109), the new column must be appended after `"SMA Composite"` (lines
   80-85, 152-155, 360-366), and R6 says PRD-131's mapping, helper, and
   vocabulary remain byte-for-byte unchanged (lines 317-325). The current
   source anchor matches that intended insertion point: PRD-131's mapping and
   helper live in `dashboard_renderer.py` lines 77-104, the header literal
   `"SMA Composite"` is the last existing trend-structure header (lines
   1803-1808), and the per-row `_cells` tuple currently appends
   `_trend_structure_composite_display(_rec)` as the final existing cell
   (lines 1820-1830). However, R6's enforcement is too narrow. Its FAIL clause
   protects diffs to named non-delivery modules and modifications to
   `_TREND_STRUCTURE_COMPOSITE_DISPLAY` / `_trend_structure_composite_display`
   (PRD-132 lines 327-335), but it does not explicitly protect the header
   literal `"SMA Composite"`, the 12-string SMA vocabulary literals, or the
   render-order invariant that "Intraday Context" must be strictly additive
   after the existing SMA Composite cell.

4. **Renderer-only scope** - Mostly compliant. The FILES list contains only
   one production module under `cuttingboard/`, and it is
   `cuttingboard/delivery/dashboard_renderer.py` (PRD-132 lines 139-145).
   OUT OF SCOPE forbids changes to `trend_structure.py`, runtime, regime,
   qualification, output, market_map, payload schema, contract schema,
   notifier, scoring, sizing, trade policy, and `config.py` (lines 93-112).
   `_INTRADAY_RVOL_THRESHOLD` is intentionally renderer-local rather than
   `config.py` because it is display classification only (lines 61-67,
   110-112), so qualification, regime, policy, and sizing modules should have
   no legitimate import path to it. R4(a)/(b) would catch helper, threshold,
   or literal vocabulary appearances under non-delivery `cuttingboard/`
   modules (lines 262-277), while R4(c) correctly inherits the PRD-131
   Round-3 machine-readable artifact scope and excludes `*.html` destinations
   (lines 279-294). The indirect leak channel R4 does not fully prove is
   dynamic reconstruction of the phrase in a non-delivery module without the
   exact literal; the FILES allowlist and R6 diff check should catch that if
   they are enforced, and emitted logs/payloads/market_map artifacts would be
   caught by the machine-readable grep if the exact phrase appears there.

## Blocking Findings

B1. R6 does not fully protect PRD-131 isolation. PRD-132 requires no diff to
PRD-131's mapping, helper, render position, header literal `"SMA Composite"`,
or 12-string SMA composite vocabulary, and requires the new `"Intraday
Context"` column to be strictly additive after `"SMA Composite"`. But the R6
FAIL/test language only names non-delivery module diffs and the two PRD-131
symbols (lines 327-335). Before implementation, R6 should add an explicit
assertion or diff guard for the trend-structure header order, the existing
per-row `_cells` order, the `"SMA Composite"` literal, and the 12 PRD-131
display strings.

### Blocking Findings (Round 3 — Post-Impl)

six PRD-131 isolation invariants. The implemented tests start at R6(b)
(`tests/test_dashboard_renderer.py:3423`) and continue through R6(f) at

B2. R7's FAIL language requires each R1/R2 display string to have a targeted
test asserting the rendered fragment contains that exact string for the
corresponding input record. The implementation tests all 9 R1 strings through
the pure helper at `tests/test_dashboard_renderer.py:3221-3231` and tests the
two R2 strings through helper precedence at lines 3248-3258, but only one
intraday phrase is asserted in rendered HTML at lines 3298-3317. Runtime output
is correct in the source, but the rendered-fragment coverage promised by R7 is
not present for the other 10 vocabulary strings.
