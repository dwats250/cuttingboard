WHY NOW
A prior Codex read-only review found that
`.github/workflows/hourly_alert.yml` runs `python3
scripts/check_readiness.py` (line 53) BEFORE the "Publish hourly
artifacts" step (line 55) that performs the dashboard render
(`python3 -m cuttingboard.delivery.dashboard_renderer ... --output
ui/dashboard.html`, line 63-66) and the index copy (`cp
ui/dashboard.html ui/index.html`, line 77). `scripts/check_readiness.py`
validates `ui/dashboard.html` and `ui/index.html` against
`REQUIRED_HTML_MARKERS` (system-state, macro-tape, candidate-board,
RUN SNAPSHOT) and forbidden patterns. Because readiness runs before
render/copy, it validates the previously committed (stale) HTML
artifacts rather than the freshly rendered artifacts the workflow is
about to publish. A regression that breaks fresh render or copy would
pass readiness, since readiness never sees the fresh output.

This PRD follows PRD-125 (OHLCV Cache Freshness), PRD-126 (Fixture
Mode No-Live-OHLCV Boundary), and PRD-127 (Hourly Alert Action
Language Alignment) â€” none of which are modified here.
