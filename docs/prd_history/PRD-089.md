# PRD-089 - Dashboard Artifact Coherence Guard

**Status:** COMPLETE
**Owner:** Codex

---

## GOAL

Prevent the public dashboard from displaying incoherent state assembled from mismatched artifact families.

The dashboard can currently combine active trade/setup state from one artifact with stale candidate-board or contract data from another artifact. This can produce misleading output such as TRADE SETUP ACTIVE, PREMARKET_CONTEXT, STALE, and NO ACTIONABLE SETUPS on the same page.

This PRD adds a renderer-only coherence guard so stale or mixed artifacts are clearly identified, active trade/setup labels are suppressed when payload/run artifacts are incoherent, and stale Candidate Board data is not displayed as current.

---

## SCOPE

1. Update cuttingboard/delivery/dashboard_renderer.py.
2. Update tests/test_dashboard_renderer.py.
3. Add renderer-only artifact timestamp extraction for:
   - selected payload artifact
   - selected run artifact
   - logs/market_map.json
   - logs/latest_hourly_contract.json when contract entries are used
4. Use existing timestamp fields where available:
   - payload: meta.timestamp, timestamp, generated_at
   - run: run_at_utc, timestamp, generated_at
   - market_map: generated_at
   - contract: generated_at
5. Use DASHBOARD_STALE_AFTER_SECONDS as the deterministic coherence threshold if it already exists. If it does not exist, add a renderer-local constant with that name.
6. Treat selected payload/run as the dominant artifact family.
7. If selected payload and selected run differ beyond the threshold:
   - render prominent MIXED_ARTIFACTS warning
   - do not render TRADE SETUP ACTIVE
   - do not render action wording that implies a current actionable trade setup
8. If market_map.generated_at is stale relative to the coherent selected payload/run baseline:
   - Candidate Board must show a stale/suppressed state
   - Candidate Board must not render old candidate cards as if they belong to the selected run
9. If latest_hourly_contract.generated_at is stale relative to the coherent selected payload/run baseline:
   - contract-derived entry anchors must not be treated as current
   - PRD-088 current_price fallback may still be used only when market_map is coherent and meaningful level context exists
10. Add compact artifact diagnostics showing source labels and timestamps for:
    - payload
    - run
    - market_map
    - contract
11. Preserve normal dashboard behavior when artifacts are coherent.
12. Preserve existing static dashboard deployment path.

## ACCEPTANCE CRITERIA

1. Renderer computes artifact timestamps for selected payload, selected run, market_map, and latest_hourly_contract when available.
2. Renderer includes compact artifact diagnostics with both source path labels and timestamps.
3. Payload/run timestamp comparison uses DASHBOARD_STALE_AFTER_SECONDS.
4. If payload/run are mixed beyond the threshold, dashboard renders MIXED_ARTIFACTS.
5. If MIXED_ARTIFACTS is active, dashboard does not render TRADE SETUP ACTIVE.
6. If MIXED_ARTIFACTS is active, dashboard does not render active/actionable trade wording from stale or mismatched artifacts.
7. If payload/run are coherent but market_map.generated_at is stale beyond the threshold, Candidate Board renders a stale/suppressed message and no candidate cards.
8. If payload/run are coherent but latest_hourly_contract.generated_at is stale beyond the threshold, contract entries are ignored for level anchors.
9. If contract entries are ignored because the contract is stale, PRD-088 current_price fallback still works when market_map is coherent and level context is meaningful.
10. Coherent artifacts preserve existing System State, Macro Tape, Macro Pressure, Candidate Board, and PRD-088 level diagram behavior.
11. PRD-088 behavior remains intact:
    - valid level context renders level diagram
    - valid price without level context renders Level context unavailable
    - invalid or missing anchor renders Chart unavailable - no price data
12. No runtime, notification, market_map schema, data-fetching, grading, lifecycle, workflow, generated dashboard artifact, or trade-decision files are changed.
13. Dashboard renderer tests pass.

## IMPLEMENTATION NOTES

This PRD is a renderer safety layer only.

Before:
- dashboard could combine latest_run.json, latest_payload.json, market_map.json, and latest_hourly_contract.json from different timestamps or modes
- active trade/setup labels could appear beside stale Candidate Board output
- contract entry anchors could come from a different artifact family than the selected payload/run
- public dashboard could imply current trade permission when artifacts were mixed

After:
- mixed payload/run artifacts are explicitly flagged
- active trade/setup labels are suppressed when selected payload/run are incoherent
- stale market_map Candidate Board output is suppressed instead of displayed as current
- stale contract entries are ignored as level anchors
- PRD-088 current_price fallback remains usable when market_map is coherent
- artifact source/timestamp diagnostics are visible
- trade logic and artifact generation remain unchanged
