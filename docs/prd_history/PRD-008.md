PRD-008 — Expansion Regime Detection and Continuation Entry Mode

STATUS
COMPLETE
Commit: 30b583a0

GOAL
Add EXPANSION as a third regime state and a CONTINUATION entry pathway so the system participates in broad momentum-driven advances instead of outputting NO TRADE.

SCOPE
- Add EXPANSION constants and detect_expansion_regime() to regime.py
- Add EXPANSION to REGIME_RISK_MULTIPLIER in config.py
- Add ENTRY_MODE_CONTINUATION, detect_continuation_breakout(), and _qualify_continuation_candidate() to qualification.py
- Update output.py with EXPANSION banner and alternate NO TRADE message
- Add EXPANSION to runtime VALID_REGIMES and VALID_POSTURES

OUT OF SCOPE
- No changes to the 8-vote macro regime model
- No changes to validation or ingestion layers

FILES
M cuttingboard/config.py
M cuttingboard/regime.py
M cuttingboard/qualification.py
M cuttingboard/output.py
M cuttingboard/runtime.py

REQUIREMENTS

R1 — EXPANSION Detection
detect_expansion_regime() MUST evaluate breadth and leadership confirmation independently of the vote model.
EXPANSION MUST NOT be derived from vote net score.

FAIL: EXPANSION state is triggered by vote net score alone.

R2 — CONTINUATION Entry Mode
qualify_all() MUST activate CONTINUATION path only when regime == EXPANSION.
CONTINUATION candidates MUST use EXPANSION_RR_RATIO for gate 7.

FAIL: CONTINUATION path activates under non-EXPANSION regime.

R3 — Runtime Acceptance
EXPANSION MUST appear in VALID_REGIMES and VALID_POSTURES.
EXPANSION_LONG MUST appear in VALID_POSTURES.

FAIL: runtime.py raises ValueError on EXPANSION regime output.

DATA FLOW
breadth + leadership signals → detect_expansion_regime() → EXPANSION regime
EXPANSION regime → qualify_all() CONTINUATION path → TradeCandidate with CONTINUATION entry_mode

FAIL CONDITIONS
- CONTINUATION activates without EXPANSION
- EXPANSION not accepted by runtime validator

VALIDATION
Run: pytest -q
Manual: fixture with EXPANSION conditions — confirm CONTINUATION candidates produced, EXPANSION banner in output.
