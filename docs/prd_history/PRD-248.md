WHY NOW
The PRD-244 governance refactor introduced a new prd_history artifact class,
`PRD-NNN.proposal.md` (merged via PR #119). `prd_eval.sh`'s registry-gap
allowlist covers `.review.`/`.adjudication.md`/`.codex_prompt.md`/`.impl_notes.`
but not `.proposal.md`, so `docs/prd_history/PRD-244.proposal.md` trips the
UserPromptSubmit registry-gap warning on every prompt. Surfaced repeatedly
across the PRD-245/246/247 arc; PRD-247's DECISIONS entry named this as its
own INFRA PRD.

MAX EXPECTED DELTA
<= 5 production lines (one allowlist clause + comment) in prd_eval.sh; test
edits confined to FILES.

GOAL
Treat `PRD-NNN.proposal.md` as the non-PRD sidecar it is — allowlisted from
the registry-gap check exactly like review/adjudication/codex-prompt/impl-notes
artifacts — so a proposal artifact no longer demands a registry row, while a
genuine unregistered `PRD-NNN.md` is still flagged.

SCOPE
- R1–R3 below; nothing else. No change to the gap-detection logic, prompt
  parsing, or emit path.

OUT OF SCOPE
- The retired keyword detectors (already gone, PRD-243).
- Any other hook or the protect_files hard-block set.
- Retroactively registering `PRD-244.proposal.md` (the point is that a
  proposal artifact needs no row).

CHANGE SURFACE
`prd_eval.sh`: the sidecar-exclusion tuple (currently `.review.` /
`.adjudication.md` / `.codex_prompt.md` / `.impl_notes.`) gains one
`.proposal.md` clause, and the header comment's exclusion list is updated to
match. No change to prompt parsing, the registry-row scan, or the JSON emit.
`tests/test_prd_eval_hook.py`: additive tests only.

REQUIREMENTS

R1 — `.proposal.md` is allowlisted
The registry-gap check skips `PRD-*.proposal.md` files, the same class as the
existing review/adjudication/codex-prompt/impl-notes sidecars.
FAIL: a `PRD-NNN.proposal.md` with no registry row produces REGISTRY GAP
output naming it.

R2 — the guard still fires (mutation-verified red test)
The allowlist change must not suppress genuine gaps: a real `PRD-NNN.md` with
no registry row still produces the REGISTRY GAP output. A negative test proves
R1's exclusion fails-closed — reverting the `.proposal.md` clause turns the R1
test red, and the R2 test stays green (proving the guard itself is intact).
FAIL: the allowlist change causes a genuine unregistered `PRD-NNN.md` to be
skipped, or R1 merges without a test that goes red when the clause is reverted.

R3 — comment truth
The hook's header comment (`prd_eval.sh:3-5`) enumerates the excluded sidecar
classes; it must include `.proposal.md`.
FAIL: the comment still lists the exclusions without proposal.
