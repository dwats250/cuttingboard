# PRD-161: Add tradable qualified fixture for PRD-157 sizing gate

Lane: MICRO / FIXTURE
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-04

## Problem

The PRD-157 fixture-verification gate (the binding pause before B1/B2/W4) is
BLOCKED. See `audits/recon-2026-05-24/prd-157-fixture-gate-report.md`.

The only fixture in the repo (`tests/fixtures/2026-04-12.json`) yields a single
sized candidate that is **non-tradable** (`^VIX`, a member of
`config.NON_TRADABLE_SYMBOLS`), `decision_status=BLOCK_TRADE`, `size_multiplier=0.0`,
under `outcome=NO_TRADE`. The PRD-157 sizing fields (`position_size`,
`dollar_risk`, `estimated_debit`) are arithmetically correct and coherent on that
record, but they have never been gut-checked against a **realistic tradable
instrument** ÔÇö which is exactly what the gate exists to confirm.

This PRD adds fixture coverage that produces at least one qualified, tradable
candidate so the three sizing fields can be verified against a symbol that would
actually be traded.

## Realizability constraint (binding)

Fixture mode forces every candidate to `BLOCK_TRADE`:

- `cuttingboard/trade_decision.py:101` ÔÇö `status = ALLOW_TRADE if chain.classification == VALIDATED else BLOCK_TRADE`.
- `cuttingboard/runtime.py:1558-1564` (`_fixture_chain_results`) ÔÇö fixture mode
  unconditionally stamps every setup `classification=MANUAL_CHECK`.

Therefore `decision_status=ALLOW_TRADE`, `policy_allowed=true`, and
`size_multiplier>0` are **UNREALIZABLE in fixture mode** without changing
`_fixture_chain_results` (production logic), which is out of scope. The sizing
fields are populated regardless of block status, so this PRD verifies them on a
tradable `MANUAL_CHECK` (blocked-but-sized) candidate.

## Implementation approach (decided 2026-06-04)

A genuine end-to-end fixture where a *tradable* symbol qualifies would require
committed OHLCV data crafted to pass every qualification gate ÔÇö but
`data/cache/*_ohlcv.parquet` is git-ignored (only `.gitkeep` is tracked), so it
cannot be relied on in CI, and the repo's sanctioned end-to-end pipeline tests
(`tests/test_runtime_decision.py`) do not craft real OHLCV ÔÇö they **inject** a
qualified candidate at the `qualify_all` / `build_option_setups` stages and let
the real contract assembly compute sizing.

This PRD mirrors that sanctioned pattern: it drives `runtime._run_pipeline` in
fixture mode with a qualified TRADABLE candidate (`SPY`) injected at the
qualification/option stages, leaving the chain + decision + contract assembly
REAL, then asserts the payload `top_trades` carries coherent sizing. No committed
OHLCV/JSON data fixture is required, and no `cuttingboard/` code changes. This
verifies the PRD-157 sizing passthrough on a tradable candidate (the gate's
intent); it does not prove a tradable genuinely qualifies from market data (no
sanctioned test does ÔÇö that path is mocked everywhere in the suite).

## Acceptance criteria

A fixture-mode `_run_pipeline` run with an injected qualified `SPY` emits, in the
built payload's `sections.top_trades`, at least one candidate (the "gate
candidate") for which:

- AC1 ÔÇö the candidate `symbol` is **not** in `config.NON_TRADABLE_SYMBOLS`.
- AC2 ÔÇö `position_size` is numeric and `> 0`.
- AC3 ÔÇö `dollar_risk` is numeric and `> 0`.
- AC4 ÔÇö `estimated_debit` is numeric and `> 0`.
- AC5 ÔÇö `position_size * estimated_debit <= dollar_risk` (sizing coherence /
  risk-cap), asserted against **deterministic expected values** pinned in the test.
- AC6 ÔÇö the fixture is deterministic: two consecutive runs produce byte-identical
  `top_trades` sizing for the gate candidate.
- AC7 ÔÇö no production decision logic is modified (grep guard: the diff touches
  only `tests/fixtures/`, `tests/`, and `docs/`; `cuttingboard/` is untouched).

### Dropped (unrealizable in fixture mode ÔÇö see Realizability constraint)

- ~~`decision_status == "ALLOW_TRADE"`~~
- ~~`policy_allowed == true`~~
- ~~`size_multiplier > 0`~~

These three cannot be produced by any fixture under current routing
(`_fixture_chain_results` Ôćĺ `MANUAL_CHECK` Ôćĺ `BLOCK_TRADE`). They are documented
here as out-of-reach-by-design, not omitted by oversight. The live ALLOW_TRADE
sizing path remains unverified by fixtures and must be confirmed against a real
pipeline run (or a captured real-run snapshot) before W4 ships.
