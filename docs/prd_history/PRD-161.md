# PRD-161: Add tradable qualified fixture for PRD-157 sizing gate

Lane: MICRO / FIXTURE
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-04

## Problem

The PRD-157 fixture-verification gate (the binding pause before B1/B2/W4) is
BLOCKED. See `audits/recon-2026-05-24/prd-157-fixture-gate-report.md`.

The only fixture in the repo (`tests/fixtures/2026-04-12.json`) yields a single
sized candidate that is **non-tradable** (`^VIX`, a member of
`config.NON_TRADABLE_SYMBOLS`), `decision_status=BLOCK_TRADE`, `size_multiplier=0.0`,
under `outcome=NO_TRADE`. The PRD-157 sizing fields (`position_size`,
`dollar_risk`, `estimated_debit`) are arithmetically correct and coherent on that
record, but they have never been gut-checked against a **realistic tradable
instrument** — which is exactly what the gate exists to confirm.

This PRD adds fixture coverage that produces at least one qualified, tradable
candidate so the three sizing fields can be verified against a symbol that would
actually be traded.

## Realizability constraint (binding)

Fixture mode forces every candidate to `BLOCK_TRADE`:

- `cuttingboard/trade_decision.py:101` — `status = ALLOW_TRADE if chain.classification == VALIDATED else BLOCK_TRADE`.
- `cuttingboard/runtime.py:1558-1564` (`_fixture_chain_results`) — fixture mode
  unconditionally stamps every setup `classification=MANUAL_CHECK`.

Therefore `decision_status=ALLOW_TRADE`, `policy_allowed=true`, and
`size_multiplier>0` are **UNREALIZABLE in fixture mode** without changing
`_fixture_chain_results` (production logic), which is out of scope. The sizing
fields are populated regardless of block status, so this PRD verifies them on a
tradable `MANUAL_CHECK` (blocked-but-sized) candidate.

## Acceptance criteria

A fixture pipeline run (`--mode fixture --fixture-file tests/fixtures/sizing_gate_tradable.json`)
emits at least one `sections.top_trades[*]` candidate (the "gate candidate") for
which:

- AC1 — the candidate `symbol` is **not** in `config.NON_TRADABLE_SYMBOLS`.
- AC2 — `position_size` is numeric and `> 0`.
- AC3 — `dollar_risk` is numeric and `> 0`.
- AC4 — `estimated_debit` is numeric and `> 0`.
- AC5 — `position_size * estimated_debit <= dollar_risk` (sizing coherence /
  risk-cap), asserted against **deterministic expected values** pinned in the test.
- AC6 — the fixture is deterministic: two consecutive runs produce byte-identical
  `top_trades` sizing for the gate candidate.
- AC7 — no production decision logic is modified (grep guard: the diff touches
  only `tests/fixtures/`, `tests/`, and `docs/`; `cuttingboard/` is untouched).

### Dropped (unrealizable in fixture mode — see Realizability constraint)

- ~~`decision_status == "ALLOW_TRADE"`~~
- ~~`policy_allowed == true`~~
- ~~`size_multiplier > 0`~~

These three cannot be produced by any fixture under current routing
(`_fixture_chain_results` → `MANUAL_CHECK` → `BLOCK_TRADE`). They are documented
here as out-of-reach-by-design, not omitted by oversight. The live ALLOW_TRADE
sizing path remains unverified by fixtures and must be confirmed against a real
pipeline run (or a captured real-run snapshot) before W4 ships.
