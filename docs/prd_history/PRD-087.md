# PRD-087 — Pipeline Command Timeout Hardening

**Status:** COMPLETE
**Owner:** Codex

---

## GOAL

Prevent scheduled Cutting Board pipeline runs from silently hanging by adding
bounded command execution and clearer mode-level logging around every
`python -m cuttingboard` workflow invocation.

The current failure pattern shows scheduled workflow runs reaching approximately
15 minutes and ending as cancelled/failure without a useful traceback or
failed-step root cause. This PRD adds explicit command-level and job-level
timeout boundaries so future failures identify the exact pipeline mode that hung.

---

## SCOPE

1. Update `.github/workflows/cuttingboard.yml` only.
2. Add job-level `timeout-minutes: 20` to `jobs.pipeline`.
3. Wrap every `python -m cuttingboard` execution in GNU timeout:
   ```
   timeout 8m python -m cuttingboard ...
   ```
4. Preserve every existing mode and notify-mode argument:
   - `prefetch` / `premarket`
   - `live` / `premarket`
   - `sunday` / `premarket`
   - `verify` / `premarket`
   - `orb_trajectory` / `orb_trajectory`
   - `post_orb` / `post_orb`
   - `midmorning` / `midmorning`
   - `power_hour` / `power_hour`
5. Before each timeout command, print a UTC timestamp using:
   ```
   date -u +"%Y-%m-%dT%H:%M:%SZ"
   ```
6. After each timeout command, print a UTC timestamp using the same format.
7. Preserve existing START and END log banners.
8. If timeout exits nonzero, allow the workflow step to fail normally.
9. Preserve all existing schedules, workflow_dispatch inputs, comments,
   concurrency behavior, permissions, lint, tests, engine doctor, artifact
   upload, commit message generation, artifact commit logic, and dashboard
   rendering behavior.

## REQUIREMENTS

**R1.** `jobs.pipeline` contains exactly one job-level timeout:
```yaml
timeout-minutes: 20
```
FAIL: `timeout-minutes: 20` not found under `jobs.pipeline` in the workflow file.

**R2.** Every step that executes `python -m cuttingboard` wraps it with GNU timeout:
```
timeout 8m python -m cuttingboard
```
FAIL: Any `python -m cuttingboard` line exists without the `timeout 8m` prefix.

**R3.** The workflow contains at least 8 timeout-wrapped Cutting Board commands (one per mode).
FAIL: `s.count("timeout 8m python -m cuttingboard") < 8`.

**R4.** No unwrapped `python -m cuttingboard` command remains in the workflow file.
FAIL: Any line matches `"python -m cuttingboard" in line and "timeout 8m python -m cuttingboard" not in line`.

**R5.** A UTC timestamp is printed before each timeout-wrapped mode execution.
FAIL: Timestamp count `< 16` (8 modes × 2 timestamps each).

**R6.** A UTC timestamp is printed after each timeout-wrapped mode execution.
FAIL: See R5 — same assertion covers both before and after.

**R7.** Existing START and END banners remain present for all mode steps.
FAIL: Any `=== START <mode> ===` or `=== END <mode> ===` banner is absent.

**R8.** Existing mode routing is unchanged:
- `workflow_dispatch` mode controls manual runs.
- Scheduled UTC time routing maps to the same `job_mode` values.
- Sunday routing is unchanged.
- `noop` behavior is unchanged.

FAIL: Any `job_mode` assignment, time comparison, or dispatch routing is altered.

**R9.** All `notify-mode` arguments remain unchanged.
FAIL: Any `--notify-mode` value differs from the original.

**R10.** The workflow fails normally if a timeout occurs (exit code 124 propagates through `set -eo pipefail`).
FAIL: Any timeout failure is suppressed (`|| true` or `continue-on-error` added to a mode step).

**R11.** Lint, test, engine doctor, artifact upload, commit message generation, dashboard render, and artifact commit logic are functionally unchanged.
FAIL: Any non-mode step is modified.

## DATA FLOW

No runtime data flow change. This PRD operates entirely in the CI layer.

```
GitHub Actions trigger
  → jobs.pipeline (timeout-minutes: 20)
    → Determine run mode
    → [mode step] date timestamp (before)
                  timeout 8m python -m cuttingboard --mode X --notify-mode Y
                  date timestamp (after)
    → Commit artifacts / Push (unchanged)
```

## VALIDATION

Run locally after patch:

from pathlib import Path

## IMPLEMENTATION NOTES

This PRD does not fix the underlying hang directly. It changes the workflow
failure contract so the next scheduled failure exposes the exact mode that hangs
instead of producing a vague cancelled/failure result after approximately 15 minutes.

Expected improvement is diagnostic clarity:
- **Before:** scheduled run ends cancelled/failure with little or no useful traceback.
- **After:** specific mode step fails after 8 minutes, preserving START logs and timestamp context.

If the next run fails after this PRD, investigate the specific timed-out mode for
network calls, quote fetches, notification sends, verify behavior, dashboard
rendering, or artifact commit behavior.
