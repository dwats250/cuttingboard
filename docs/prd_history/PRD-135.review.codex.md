## Summary

PRD-135 is correctly scoped as read-only, but the milestone document is not yet accurate enough as a durable architecture checkpoint. The main problems are overstated contract/downstream claims, incomplete artifact coverage, and incorrect notifier/dashboard lineage descriptions.

## Findings

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:55-61` says the three sidecars sit downstream of the contract and are re-read by downstream consumers. Current code does not support that broadly: `market_map` is built before contract assembly and is used immediately by `build_visibility_map()` / `apply_overnight_policy()` in `cuttingboard/runtime.py:888-970`; `watchlist_snapshot.json` has no v1 consumer per `docs/artifact_flow_map.md:118-124`.

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:35-47` calls the listed outputs a fixed hourly artifact set but omits `logs/audit.jsonl`, `logs/evaluation.jsonl`, `logs/performance_summary.json`, and the non-hourly/default `logs/latest_run.json` / `logs/latest_payload.json` / `logs/latest_contract.json` family. Runtime appends audit/evaluation/performance outputs at `cuttingboard/runtime.py:1007-1034`; artifact-family distinctions are documented in `docs/artifact_flow_map.md:55-97`.

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:70-72` underspecifies the dashboard publish path. The published hourly Pages path uses CLI overrides `--payload logs/latest_hourly_payload.json`, `--run logs/latest_hourly_run.json`, and `--output ui/dashboard.html`, then shell-copies `ui/dashboard.html` to `ui/index.html` in `.github/workflows/hourly_alert.yml:57-62`. `reports/output/dashboard.html` is local/default output, not the CI published artifact, per `docs/artifact_flow_map.md:171-193`.

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:86` incorrectly says the contract participates in dashboard coherence enforcement. `validate_coherent_publish()` checks only `payload.meta.generation_id`, `run.generation_id`, and `market_map.generation_id` in `cuttingboard/delivery/dashboard_renderer.py:328-424`. The contract carries `generation_id` (`cuttingboard/contract.py:90-97`) but is not part of the fail-closed publish equality gate.

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:94-98` misstates the hourly notifier inputs and write boundary. `_execute_notify_run()` formats the hourly notification before building the hourly contract/payload (`cuttingboard/runtime.py:540-605`), and `format_hourly_notification()` accepts regime, validation, qualification, candidate lines, market_map, and normalized_quotes, not a canonical contract/payload pair (`cuttingboard/notifications/__init__.py:511-588`). Also, notification transport writes audit records to `logs/audit.jsonl` via `send_notification()` / `send_telegram()` / `write_notification_audit()` (`cuttingboard/output.py:700-733`, `cuttingboard/audit.py:242-307`), so "does not write logs JSON artifacts" is too broad.

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:100` understates evaluation. Evaluation is downstream-only, but it is an active runtime call, not just a doctrine reference: `_run_pipeline()` calls `run_post_trade_evaluation()` and then `run_performance_engine()` at `cuttingboard/runtime.py:1030-1034`. `cuttingboard/evaluation.py:37-71` appends evaluation records from prior audit decisions.

- `docs/milestones/ENGINE_MILESTONE_2026-05-12.md:106-109` overstates "every consumer" and "no hidden coupling discovered." The dashboard reads sidecar artifacts directly outside the contract, market_map feeds visibility/overnight presentation before contract return, and watchlist has no consumer. This should be narrowed to avoid claiming a single contract-only fan-out.

## Required Edits

- Replace the "Three deterministic, read-only sidecars sit downstream of the contract" sentence with wording that separates sidecar producers from consumers: market_map is built during runtime and consumed by renderer plus next-run lifecycle injection; trend_structure is renderer-consumed; watchlist_snapshot is observe-only with no v1 consumer.

- Expand "Major Artifacts" to include audit/evaluation/performance artifacts and explicitly distinguish hourly artifacts from default/non-hourly latest artifacts.

- Rewrite "Dashboard Publish Path" to state that local defaults read `logs/latest_payload.json` / `logs/latest_run.json`, while the hourly CI publish path overrides to `logs/latest_hourly_payload.json` / `logs/latest_hourly_run.json`, writes `ui/dashboard.html`, and shell-copies `ui/index.html`.

- Change the lineage model sentence to: payload, run, and market_map participate in `validate_coherent_publish()`; contract carries `generation_id` but is not checked by that gate.

- Rewrite the notifier section to say hourly notifications are formatted from regime/validation/qualification/candidate lines/normalized_quotes plus optional market_map, not from contract/payload, and note that transport writes notification audit events to `logs/audit.jsonl`.

- Add runtime evaluation/performance to the engine flow and boundary sections: `run_post_trade_evaluation()` reads prior audit records and appends `logs/evaluation.jsonl`; `run_performance_engine()` reads evaluation output and writes `logs/performance_summary.json`; neither feeds back into decisions.

- Narrow the "Single canonical contract" and "Determinism at every layer" strengths so they do not imply sidecars/evaluation/notifier all consume only the contract or that watchlist has a downstream consumer.
