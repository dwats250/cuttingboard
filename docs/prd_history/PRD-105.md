PRD-105 — Decision Quality Evidence Map

STATUS
PROPOSED

GOAL
Create a deterministic documentation audit of every decision-quality evidence field
currently produced by Cutting Board so future PRDs (Sunday Report sidecar, Threshold
Review framework, Trend Structure sidecar) can be scoped against a known field
inventory rather than guessed.

SCOPE
This PRD adds documentation only.

Create:
- docs/decision_quality_map.md

The map must anchor on (and not duplicate):
- docs/system_logic_map.md
- docs/artifact_flow_map.md

The map must enumerate every decision-quality field across the following artifacts:
- logs/audit.jsonl
- logs/evaluation.jsonl
- logs/performance_summary.json
- logs/latest_hourly_run.json
- logs/latest_hourly_contract.json
- logs/market_map.json

For each artifact and each field, the map must document:
1. Field name (exact key as it appears on disk).
2. Writer module and function (where known).
3. Current readers (where known); if no reader exists, state "no reader".
4. Whether the field is persisted to disk in the artifact above.
5. Whether the field is aggregated anywhere in the repo today; if yes, by which
   module/function; if no, state "not aggregated".
6. Whether the field can support calibration review. Value must be exactly one of
   "yes", "no", or "partial", and must be followed by a one-line reason on every
   value (not only "partial").
7. Field category, exactly one of:
   - decision-affecting
   - decision-adjacent
   - audit-only
   - evaluation-only
   - display-only

The map must explicitly flag the following known gaps as a dedicated section
titled "Known Gaps":
- no structured block_reason rollup across runs
- no threshold-friction summary (no aggregator of decision_trace, regime_failure_reason,
  or excluded_symbols keys)
- no weekly decision-quality summary artifact
- no documented evidence-to-PRD calibration convention
- no human-disagreement surface inside the repo (intentional; tracked in Obsidian only)
- evaluation.jsonl only evaluates ALLOW_TRADE candidates; rejected candidates have no
  counterfactual outcome record
- stay_flat_reason is set on contract.system_state but is not persisted directly in
  audit._build_record (audit.py)

Each gap entry must include:
- one-line description
- the artifact/field/path that exhibits the gap
- whether closing the gap would require a new PRD (yes / no), and the gap's
  candidate downstream PRD name (Sunday Report sidecar, Threshold Review framework,
  audit schema patch, etc.)

OUT OF SCOPE
- no aggregation code
- no sidecar code
- no threshold changes
- no audit schema changes
- no evaluation schema changes
- no contract changes
- no payload changes
- no dashboard changes
- no notification changes
- no source code changes
- no test changes
- no Obsidian integration
- no log artifact changes
- no report artifact changes
- no ui artifact changes
- no PRD_REGISTRY.md changes (this PRD is PROPOSED; registry row is added when
  status moves to IN PROGRESS per CLAUDE.md PRD lifecycle)
- no PROJECT_STATE.md changes until implementation begins

FILES
A docs/decision_quality_map.md

REQUIREMENTS

R1 — Anchor section present
docs/decision_quality_map.md begins with an "Anchors" section that explicitly references
docs/system_logic_map.md and docs/artifact_flow_map.md and states that this map is a
field-level extension of those documents. The map must not redefine artifact writers,
artifact readers, or sidecar boundary rules already documented in those anchor files;
it must reference them.

FAIL: docs/decision_quality_map.md does not contain a section literally titled
"Anchors" that names both anchor files by path.

R2 — Per-artifact field inventory
docs/decision_quality_map.md contains exactly one heading section per artifact in the
artifact list above. Each section is titled with the artifact's logs/ path. Each
section enumerates every decision-quality field present in that artifact today.

FAIL: any of the six artifacts is missing a heading section, or any artifact's section
contains zero field entries.

R3 — Per-field metadata
For every field listed under any artifact section, all seven metadata items from the
SCOPE list (field name, writer, readers, persisted, aggregated, supports calibration,
category) are documented. Category must be exactly one of the five enumerated values.
The "supports calibration" item must be one of "yes", "no", or "partial", and every
value (yes, no, or partial) must be accompanied by a one-line reason.

FAIL: any field entry omits one or more of the seven metadata items, or uses a category
value not in the enumerated set, or uses a "supports calibration" value outside
{yes, no, partial}, or has a "supports calibration" value without an accompanying
one-line reason.

R4 — audit.jsonl coverage
The audit.jsonl section enumerates at minimum these top-level fields, with metadata for
each: outcome, regime, posture, confidence, net_score, vix_level, router_mode,
energy_score, index_score, symbols_validated, symbols_qualified, symbols_near_a_plus,
symbols_watchlist, symbols_excluded, regime_short_circuited, regime_failure_reason,
qualified_trades, trade_decisions, watchlist, near_a_plus, excluded_symbols,
suppressed_candidates, halt_reason, alert_sent, report_path. Per-candidate sub-fields
within qualified_trades and trade_decisions must also be enumerated, including at
minimum: symbol, direction, strategy, structure, dte, contracts, dollar_risk, entry,
stop, target, risk_reward, decision_status, block_reason, decision_trace, policy_allowed,
policy_reason, size_multiplier, downside_permission, intraday_state.

FAIL: any field in this list is absent from the audit.jsonl section of the map.

R5 — evaluation.jsonl coverage
The evaluation.jsonl section enumerates at minimum: evaluated_at_utc, decision_run_at_utc,
symbol, direction, entry, stop, target, evaluation.result, evaluation.R_multiple,
evaluation.time_to_resolution.

FAIL: any field in this list is absent from the evaluation.jsonl section of the map.

R6 — performance_summary.json coverage
The performance_summary.json section enumerates at minimum: buckets (per-symbol),
buckets[symbol].total_trades, wins, losses, flats, win_rate, avg_r_win, avg_r_loss,
expectancy, insufficient_data, generated_at. The section also documents the
performance_engine.py _MIN_SAMPLE = 5 floor and the consequence (no expectancy stats
emitted below the floor).

FAIL: any field in this list is absent, or the _MIN_SAMPLE floor is not documented.

R7a — latest_hourly_run.json coverage
The latest_hourly_run.json section enumerates the decision-quality fields surfaced in
this artifact as it exists today, citing runtime.py as the writer. At minimum, the
section documents these fields with full per-field metadata: run_id, run_at_utc,
posture, outcome. Any additional decision-quality fields written into this artifact
by runtime.py must also be enumerated.

FAIL: latest_hourly_run.json section is missing, or any of run_id, run_at_utc,
posture, or outcome is absent from that section.

R7b — latest_hourly_contract.json coverage
The latest_hourly_contract.json section enumerates the decision-quality fields surfaced
in this artifact as it exists today, citing contract.py as the writer. At minimum, the
section documents that contract.system_state.stay_flat_reason is present in this
artifact and explicitly documents the divergence: stay_flat_reason is set on
contract.system_state but is not persisted by audit._build_record (audit.py); a
calibration consumer wanting STAY_FLAT reasons across runs must read the contract
artifact, not audit.jsonl.

FAIL: latest_hourly_contract.json section does not document stay_flat_reason, or does
not explicitly document the audit-vs-contract divergence for this field.

R8 — market_map.json coverage
The market_map.json section must document every key present in a representative
per-symbol record from a current logs/market_map.json file (a "representative record"
is any one symbol entry from the artifact at implementation time; the implementer
selects it). For each key in that representative record, the seven-item metadata from
R3 must be provided. Per system_logic_map.md, market_map grades are display-only and
must be categorized as such in this map.

The implementer must record (inline in the section) which symbol record was used as
the representative sample so the coverage check is reproducible.

FAIL: market_map.json section is missing; or any key present in the chosen
representative per-symbol record is undocumented; or the representative symbol is
not named inline; or any market_map grade is categorized as decision-affecting (would
contradict system_logic_map.md and require a new PRD).

R9 — Known Gaps section
docs/decision_quality_map.md contains a section literally titled "Known Gaps". The
section enumerates every gap listed in SCOPE. Each gap entry includes the one-line
description, the artifact/field/path that exhibits the gap, whether closing the gap
requires a new PRD, and a candidate downstream PRD name.

FAIL: "Known Gaps" section is missing, or any of the listed gaps is absent, or any gap
entry omits one of the four required items.

R10 — No source/test/log/report/ui modifications
This PRD is documentation-only. The only file produced by the implementation of this
PRD is docs/decision_quality_map.md. Registry and PROJECT_STATE bookkeeping is a
separate transition step handled outside this PRD's diff and is out of scope for this
PRD's implementation purity check.

FAIL: git status --short, evaluated against the implementation diff of this PRD only,
shows any modified path other than docs/decision_quality_map.md.

R11 — No new architecture claims
The map must not introduce new architecture claims, sidecar rules, or mutation
boundaries beyond those already documented in system_logic_map.md and
artifact_flow_map.md. Where the map needs to assert behavior, it must cite the
specific anchor section and quote the relevant rule.

FAIL: any normative statement in the map ("must", "must not", "may not", "is forbidden")
appears without an inline citation to system_logic_map.md or artifact_flow_map.md.
