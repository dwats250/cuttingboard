CLASS

WHY NOW
PRD-130 replaced the ambiguous `"UNKNOWN"` trend-structure tokens
with five deterministic states (`AT_LEVEL`, `INSUFFICIENT_HISTORY`,
`DATA_UNAVAILABLE`, `NOT_COMPUTED`, plus the renderer-only
`SESSION_UNAVAILABLE` branch) alongside the existing `ABOVE` /
`BELOW` comparison tokens. The dashboard now exposes those raw
tokens verbatim in adjacent cells. An operator scanning the panel
must still mentally combine `price_vs_sma_50` and `price_vs_sma_200`
on every row to recover the composite SMA position. This PRD adds a
single deterministic composite-display cell that flattens the two
SMA comparison tokens into one short structural phrase, reducing
decoding effort without adding any inference, history, trajectory,
or forecast vocabulary. An earlier draft of this PRD (rejected on
cross-review) attempted operator-narrative phrasing
("recovery attempt", "pullback within uptrend", "inflection")
and smuggled in directional history not present in the input
record; this revision removes that vocabulary entirely.

<= 60 production-code lines in
`cuttingboard/delivery/dashboard_renderer.py` (one literal mapping
table + one pure helper + one render-site insertion), plus matching
targeted tests. No new modules. No changes to
`cuttingboard/trend_structure.py` or any other production module.
Exceeding this ceiling MUST stop implementation, amend the PRD with

GOAL
Add a deterministic, read-only **composite display** cell to the
dashboard trend-structure panel that translates the two SMA
comparison tokens (`price_vs_sma_50`, `price_vs_sma_200`) into a
single short structural phrase using strictly position-based,
non-inferential vocabulary — leaving every artifact, contract,
runtime, qualification, sizing, payload, notifier, and decision
surface unchanged.

SCOPE

  helper `_trend_structure_composite_display(record: dict) -> str`
  whose input is a single trend-structure record dict already

  `(price_vs_sma_50, price_vs_sma_200)` maps to exactly one phrase.

  session-inactive short-circuits (R3), and the vocabulary-leak

REQUIREMENTS

R1 — Deterministic SMA composite display mapping
The helper `_trend_structure_composite_display(record)` MUST be a
pure, total function over the enumerated token domain. The mapping
table for the two SMA comparison tokens is closed and binding:

| price_vs_sma_50 | price_vs_sma_200 | Composite display              |
|-----------------|------------------|--------------------------------|
| ABOVE           | ABOVE            | Above SMA50 and SMA200         |
| ABOVE           | BELOW            | Above SMA50, below SMA200      |
| BELOW           | ABOVE            | Below SMA50, above SMA200      |
| BELOW           | BELOW            | Below SMA50 and SMA200         |
| AT_LEVEL        | ABOVE            | At SMA50, above SMA200         |
| AT_LEVEL        | BELOW            | At SMA50, below SMA200         |
| ABOVE           | AT_LEVEL         | Above SMA50, at SMA200         |
| BELOW           | AT_LEVEL         | Below SMA50, at SMA200         |
| AT_LEVEL        | AT_LEVEL         | At SMA50 and SMA200            |

The vocabulary above is the complete operational vocabulary for
the comparison-token branch of this helper — no other strings may
be emitted by this branch. Each phrase is a strict positional
restatement of the two input tokens. No scoring, weighting,
aggregation, inference, history, trajectory, momentum, or
direction is permitted in the phrasing.

FAIL: any input combination of comparison tokens
`(ABOVE | BELOW | AT_LEVEL) × (ABOVE | BELOW | AT_LEVEL)` maps to a
string not in the table above, OR any composite display string
contains a substring matching the forbidden vocabulary list
(case-insensitive) enumerated in OUT OF SCOPE: `recovery`,
`pullback`, `inflection`, `established`, `weakness`, `weak`,
`strong`, `firm`, `soft`, `confirmation`, `breakout`, `breakdown`,
`rebound`, `reversal`, `momentum`, `trending`, `likely`,
`probable`, `expected`, `imminent`, `confidence`,
`high-probability`, `uptrend`, `downtrend`, `bullish`, `bearish`.
Detected by a targeted test that enumerates the 9 comparison-token
cells and runs both a literal-equality assertion and a regex
assertion against the forbidden vocabulary list.

R2 — Explicit precedence slot table for unknown-state branches
The helper MUST resolve unknown-state tokens by traversing the
following precedence slots **in order**, and the first matching
slot wins. The slot list is closed:

1. **Session-inactive (renderer branch)** — handled by the
   existing INACTIVE_SESSION_LABEL / market-closed short-circuit
   above the per-row render site. The composite display helper is
   never called on this branch (see R3). This slot is listed first
   for completeness; it is satisfied by inheritance, not by new
   code inside the helper.
2. **`DATA_UNAVAILABLE` on any of (`price_vs_sma_50`,
   `price_vs_sma_200`)** → emit `"Structure unavailable"`.
3. **`INSUFFICIENT_HISTORY` on any of (`price_vs_sma_50`,
   `price_vs_sma_200`)** → emit `"SMA history insufficient"`.
4. **`NOT_COMPUTED` on any of (`price_vs_sma_50`,
   `price_vs_sma_200`)** — not currently emitted by
   `cuttingboard/trend_structure.py` for SMA fields, but reserved
   for future emitter expansion. If reached, emit
   `"Structure not computed"`. Implementations MUST include this
   slot even though no production code path emits it today, so
   that a future emitter change cannot fall through to an "else"
   branch outside the closed vocabulary.
5. **Comparison-token combination** — both SMA tokens are in
   `{ABOVE, BELOW, AT_LEVEL}`; resolve via the R1 table.

The precedence order is binding and deterministic. The closed
unknown-state vocabulary is exactly:
`{"Structure unavailable", "SMA history insufficient",
"Structure not computed"}`. No other unknown-state string may be
emitted.

FAIL: any record whose tokens trigger more than one of slots
2–4 resolves to a string not produced by the highest-precedence
matching slot; OR slot 4 falls through to slot 5 (or an "else")
when a future emitter adds `NOT_COMPUTED` to an SMA field; OR any
unknown-state branch emits a string outside the closed set above.
Detected by per-slot targeted tests, including a synthetic-record
test exercising slot 4 to prove the helper is total.

R3 — Inactive-session and snapshot-absent short-circuit
The trend-structure panel's existing INACTIVE_SESSION_LABEL branch
and snapshot-absent branch (PRD-117 / PRD-120 / PRD-130) MUST
short-circuit *before* per-row composite display rendering. No
composite display cell is emitted on those branches.

FAIL: the panel renders a composite display cell on an
inactive-session or missing-snapshot run. Detected by a targeted
test asserting the rendered fragment for those branches contains
no string from the closed R1/R2 vocabulary.

R4 — Pure / read-only / no decision feedback / vocabulary leak gate
`_trend_structure_composite_display` MUST be a pure function: no
global mutation, no I/O, no time-of-day reads, no environment
reads. Its output MUST NOT be read by any module under
`cuttingboard/` outside `cuttingboard/delivery/`. No qualification
gate, scoring module, sizing module, regime module, output
payload, notifier, market_map, or contract schema gains a
dependency on composite display output.

The grep gate is strengthened from helper-name-only to a
**vocabulary leak grep**:

a) `grep -RIn "_trend_structure_composite_display" cuttingboard/`
   MUST return matches only inside
   `cuttingboard/delivery/dashboard_renderer.py`.

b) For each composite display string defined in R1 and R2 (the
   12-string closed vocabulary:
   `"Above SMA50 and SMA200"`, `"Above SMA50, below SMA200"`,
   `"Below SMA50, above SMA200"`, `"Below SMA50 and SMA200"`,
   `"At SMA50, above SMA200"`, `"At SMA50, below SMA200"`,
   `"Above SMA50, at SMA200"`, `"Below SMA50, at SMA200"`,
   `"At SMA50 and SMA200"`, `"Structure unavailable"`,
   `"SMA history insufficient"`, `"Structure not computed"`),
   `grep -RIFn "<string>" cuttingboard/` MUST return matches only
   inside `cuttingboard/delivery/dashboard_renderer.py`. This
   catches copied literals, renamed aliases, and parallel mapping
   tables in notifier / payload / market_map / qualification
   code that the helper-name grep would miss.

c) For each composite display string, `grep -RIFn "<string>"`
   restricted to **machine-readable artifact paths** MUST return
   zero matches. Specifically, the search set is:

   - `logs/**` (all files under `logs/`)
   - `reports/**/*.json` (all JSON files anywhere under `reports/`)
   - `reports/**/*.txt`, `reports/**/*.md`, `reports/**/*.csv`,
     `reports/**/*.jsonl` (other machine-readable formats under
     `reports/`)

   Rendered HTML files are explicitly **excluded** from sub-check
   (c): `*.html` under `ui/`, `reports/`, or anywhere else is the
   intended destination of these vocabulary strings and MUST NOT
   be flagged by the leak gate. The intent of (c) is "vocabulary
   strings MUST NOT enter sidecar artifacts, log payloads, or
   any other machine-readable surface that a downstream pipeline
   module could consume" — not "vocabulary strings must never
   appear anywhere on disk". Rendered dashboard HTML is the
   intended destination and is by definition not a decision input.

FAIL: any of (a), (b), or (c) returns a match outside its allowed
location. Detected by a dedicated targeted test that builds the
12-string vocabulary set and runs the three greps via
`subprocess.run(["grep", ...], check=False)` and asserts on exit
codes / match counts. The sub-check (c) test MUST explicitly
exclude `*.html` paths from its search set so that a legitimate
local render to `reports/output/dashboard.html` (or any other
HTML destination) does not fail the gate.
