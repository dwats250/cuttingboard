## SUMMARY

PRD-149 cleanly diagnoses the 21:30 UTC + cron drift root cause behind
the observed 3:00 PM PT STAY FLAT alerts and lands on a sound design:
a PT-anchored allowed-slot resolver (`routine_pt_slot`) gates routine
sends and supplies the canonical `slot_utc` already consumed by
`cuttingboard/runtime.py:545–550` for hourly-mode title rendering. The
slot set, the 25-min lag, the 06:00-only redundancy, the force-slot
bypass, and the dedup-file invariant are all correct. The required
edits below are localized to (a) the R7 cron block — internally
redundant and inconsistent with R7's own "list each unique UTC minute
once" rule, and (b) two small consistency items in R3 and R6 that
would otherwise create scope-lock ambiguity at implementation time.

## REQUIRED EDITS

**E1. Collapse the R7 cron block to its unique-UTC-minute form.**

The block at `PRD-149.md:175–192` lists nine cron lines, of which
`0 14 * * 1-5` is declared twice (lines 179 and 187), and the
`0 15-20` / `0 16-21` ranges fully subsume the standalone `0 15` on
line 188 (and the `0 14` line on 179 overlaps with `0 16-21`'s lower
edge in PST mapping context). R7's own prose at lines 194–198 says
"list each unique UTC minute once" — but the canonical YAML it gives
violates that rule. Two readers will produce two different
`hourly_alert.yml` files from this PRD.

Replace the YAML block with the deduplicated authoritative form:

```yaml
schedule:
  # 06:00 PT  (PDT: 13:00 UTC; PST: 14:00 UTC, also = 07:00 PDT)
  - cron: '0 13 * * 1-5'
  - cron: '0 14 * * 1-5'
  # 06:05 PT redundancy backup (06:00 slot only)
  - cron: '5 13 * * 1-5'
  - cron: '5 14 * * 1-5'
  # 06:30 PT  (PDT: 13:30; PST: 14:30 — also fires 07:30 PDT,
  # which the runner gate suppresses as outside the allowed set)
  - cron: '30 13 * * 1-5'
  - cron: '30 14 * * 1-5'
  # 07:00 PT – 13:00 PT top-of-hour
  # PDT mapping → 14:00–20:00 UTC (14:00 already declared above)
  # PST mapping → 15:00–21:00 UTC
  - cron: '0 15-21 * * 1-5'
```

FAIL: the implemented `.github/workflows/hourly_alert.yml`
`schedule:` block contains any UTC cron line not in the seven unique
lines above (`0 13`, `0 14`, `5 13`, `5 14`, `30 13`, `30 14`,
`0 15-21`), or contains any duplicate of those lines.

**E2. Replace R3's `state_key=now_utc.isoformat()` with a stable
slot-class key.**

The existing PRD-141 same-slot suppression uses
`state_key=slot_utc.isoformat()` (per the audit call at
`cuttingboard/alert_runner.py:64–72`). PRD-149 R3 deviates by using
`state_key=now_utc.isoformat()` for outside-window suppression, which
yields one unique state_key per second of wall-clock time and
prevents downstream auditors from grouping suppressed-after-window
events. Use a stable, slot-class key:

```python
state_key=f"outside:{now_utc.astimezone(_PT_TZ).strftime('%Y-%m-%dT%H')}"
```

(or equivalent — the binding requirement is that two outside-window
suppressions in the same PT hour share a state_key). The audit
schema does not require uniqueness; consistency with PRD-141 wins.

FAIL: two outside-window suppressions in the same PT hour produce
distinct `state_key` values in the notification audit log.

**E3. R6 promises `is_premarket_slot` "remains exported" — clarify the
test contract so the helper isn't accidentally inlined or
re-namespaced.**

R6 says the helper "remains in the module for compatibility" but
gives no observable FAIL anchored to the public surface. Tighten:

```
FAIL: `from cuttingboard.notifications.hourly_slot import
is_premarket_slot` raises ImportError; `is_premarket_slot.__module__`
is not `cuttingboard.notifications.hourly_slot`.
```

This makes the "still exported" claim binary and CI-checkable.

**A2. Note the operational consequence of R7's "wasted fires."**

In PDT, `0 21 * * 1-5` fires at 14:00 PT (outside window, gate
suppresses). In PST, `0 13 * * 1-5` fires at 05:00 PT (also
suppressed). These cost a handful of GitHub Actions minutes per
weekday with no decision value. Acceptable trade for DST-resilient
declaration, but worth a sentence under SCOPE so a future maintainer
doesn't "tidy up" the cron and break PST coverage.

PROJECT_STATE.md but doesn't enumerate this. The closeout skill
(`prd-closeout-verified`) will enforce it; no PRD edit needed if you
rely on the skill.

**A4. Registry-row hook friction is benign here.**
