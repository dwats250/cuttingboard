PRD-175 — Historical regime scoreboard aggregation sidecar

Status: IN PROGRESS

LANE
STANDARD

CLASS
SIDECAR

WHY NOW
The dashboard emits regime/posture conclusions with no track record. logs/audit.jsonl
already records one append-only row per run with the regime call; nothing aggregates
it into an inspectable daily history. This is description, not prediction: it records
what the system said and what the market did next.

MAX EXPECTED DELTA
120 production LOC across 1 new module + 1 workflow step. Exceeding stops
implementation and re-triggers SIDECAR review.

GOAL
A daily, append-only regime-history artifact (logs/regime_history.jsonl) aggregated
from logs/audit.jsonl, with the prior day's record carrying SPY's next-session
percent change, produced by a post-run workflow step. Render is OUT of scope
(absorbed by PRD-177).

SCOPE
- New pure aggregation module cuttingboard/delivery/regime_history.py: read
  logs/audit.jsonl, fold each date's pipeline records into one summary record, and
  append/replace that date's row in logs/regime_history.jsonl (idempotent per date).
- Attach SPY next-session percent change to the prior date's record using the cached
  SPY daily-close series (data/cache/SPY_ohlcv.parquet, "Close" column).
- Invoke the aggregation as a post-run step in .github/workflows/hourly_alert.yml,
  after audit.jsonl is written and before the artifact commit, so the new artifact
  is committed and pushed.

OUT OF SCOPE
- Any dashboard render of the scoreboard (PRD-177 absorbs the render block).
- Any hit-rate, accuracy, win-score, or grading metric. v1 shows call + reference
  move side by side; grading embeds a prediction claim and needs its own PRD.
- Per-symbol outcome tracking (SPY is the single reference asset).
- Any change to cuttingboard/audit.py or the audit record schema.
- Backfill beyond what logs/audit.jsonl already contains.

FILES
A cuttingboard/delivery/regime_history.py
M .github/workflows/hourly_alert.yml
A tests/test_regime_history.py

CHANGE SURFACE
- cuttingboard/delivery/regime_history.py is a new read-only sidecar: it reads
  audit.jsonl and the SPY OHLCV cache and writes only logs/regime_history.jsonl.
  It MUST NOT import or mutate runtime decision state, payloads, or the contract.
- .github/workflows/hourly_alert.yml: one added run step only; no change to the
  alert run command or the existing commit/push contract.

REQUIREMENTS

R1 — Daily aggregation from available fields
The aggregator folds all pipeline records for a date into one summary record built
only from fields that exist in audit.jsonl: regime, posture, confidence, net_score,
vix_level, plus date, first/last run_at_utc, and run count.
FAIL: the summary record references any audit field not actually written by
cuttingboard.audit (e.g. risk_on_votes/total_votes), or omits date/regime/posture.

R2 — Correct run-record filtering
Pipeline records are selected by the presence of "outcome"+"run_at_utc" and the
ABSENCE of an "event" key; notification records (event=="notification") are excluded.
FAIL: a notification record is counted in any date's aggregation.

R3 — Idempotent per date
Re-running aggregation for an already-aggregated date replaces that date's single
record; it never appends a duplicate.
FAIL: two records share the same date in logs/regime_history.jsonl.

R4 — Prior-day reference move
When a new date's record is written, the immediately prior date's record is updated
in place with spy_close_change_pct derived from the SPY daily-close series.
FAIL: after two consecutive trading days of runs, the older record lacks
spy_close_change_pct.

R5 — Sidecar isolation
The module writes only logs/regime_history.jsonl and never writes back to audit.jsonl,
payloads, or pipeline state.
FAIL: the module opens any path other than logs/audit.jsonl (read),
data/cache/*_ohlcv.parquet (read), logs/regime_history.jsonl (write).

DATA FLOW
logs/audit.jsonl (read) + data/cache/SPY_ohlcv.parquet (read) ->
regime_history.aggregate(date) -> logs/regime_history.jsonl (append/replace).
Workflow: run alert -> write audit.jsonl -> run aggregation -> commit -> push.

FAIL CONDITIONS
- Summary references a non-existent audit field or omits date/regime/posture.
- Notification record counted in aggregation.
- Duplicate date rows in regime_history.jsonl.
- Older record missing spy_close_change_pct after a second trading day.
- Module writes any path outside its declared read/write set.

VALIDATION
Manual:
- Feed a synthetic audit.jsonl spanning two dates (with one notification record);
  run aggregation; confirm one record per date, no notification counted, and the
  older record carries spy_close_change_pct.
- Re-run aggregation for the same date; confirm no duplicate row.
- python3 -m pytest tests/test_regime_history.py -q passes; full suite green; ruff clean.
