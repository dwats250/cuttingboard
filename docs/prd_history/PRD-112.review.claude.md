## 1. PRD-112 vs. a thinner "dashboard ingestion adapter" PRD

**Decision: PRD-112 as drafted.** The renderer already has a sidecar-read
idiom; we are not introducing a new ingestion concept. A separate adapter
PRD would invent abstraction without removing duplication and would violate
the "no speculative logic" rule. PRD-112 reuses `_load_json` and the
constant-at-module-top pattern.

**Binding:** PRD-112 forbids introducing a generic "sidecar adapter" class.
The renderer continues to read each sidecar by direct path, as today.

## 2. Payload mutation: required or avoidable?

**Avoidable. Forbidden by sidecar doctrine.**

`docs/sidecar_doctrine.md` lines 71–84 ("Contract isolation") and lines
86–101 ("Dashboard as consumer"): the dashboard reads sidecar artifacts
directly from canonical paths. The contract is not the transport for
sidecar data. The renderer reading
`logs/trend_structure_snapshot.json` directly is the doctrine-mandated
path; merging fields into payload would be a doctrine violation.

**Binding:** R2 forbids any payload, contract, run-snapshot, or market_map
mutation. Any future PRD that needs trend-structure inside the contract is
a separate, decision-affecting PRD.

## 3. Doctrine compliance (PRD-111 / sidecar doctrine)

**Compliant** with three reinforcements added to PRD-112:

- Renderer reads sidecar directly (doctrine §"Dashboard as consumer").
- `artifact_flow_map.md` updated to declare renderer as consumer (doctrine
  §"Producer / consumer separation" and §"No-hidden-coupling doctrine").
- No write-back from renderer to sidecar path; producer remains
  `runtime._write_trend_structure_snapshot` only (R2).

## 4. Exact FILES list

**Sufficient and tight.** Three files only:

```
M cuttingboard/delivery/dashboard_renderer.py
M tests/test_dashboard_renderer.py
M docs/artifact_flow_map.md
```

No CSS files because the existing dashboard CSS is inlined in the renderer
template (per `tests/test_dashboard_renderer.py` fixture style — plain
dicts, no external CSS). No JS because the existing dashboard does not use
JS-loaded sections.

**Risk:** If the renderer's CSS scope blocks a clean table layout, the
implementer may be tempted to add a stylesheet. Reject. Use existing
table/badge classes only; if layout is suboptimal, accept it for the
read-only panel and revisit in a separate UI PRD.

R1 (constant), R3 (curated 6, fixed order, exact 6 rows), R4 (sidecar fields
only, no computation), R6 (artifact map row update), R7 (leakage grep), R8

R5 (graceful degradation) is binary across four enumerated states. Good.

**No subjective acceptance criteria remain.**

## 6. Failure conditions

Each requirement has an inline `FAIL:` line. The aggregated `FAIL
CONDITIONS` block is non-redundant and binary. The `git diff --name-only`
guard in VALIDATION is the single strongest scope-lock check.

## 7. Stale / missing sidecar behavior

R5 enumerates four states: missing, malformed, stale, unparseable
`generated_at`. Each maps to a deterministic UI outcome. Aligns with the
`DASHBOARD_STALE_AFTER_SECONDS` constant already defined at
`dashboard_renderer.py:43`. **Accept.**

**Reinforcement:** stale-state behavior MUST mirror the macro snapshot's
existing degradation pattern. If implementation discovers the macro path
treats stale differently, follow the macro path; do not invent a new
pattern.

## 8. UI drift risk

Concerns and mitigations:

- **Mobile readability:** R3 fixes 6 rows; even on narrow viewports, 6 rows
  with 9 columns is borderline. PRD-112 instructs reusing the existing
  responsive scale. **Binding addition:** column priority on narrow
  viewports collapses to `symbol`, `data_status`, `entry_context`,
  `trend_alignment`. Hide other columns via existing CSS breakpoints.
  Implementer may NOT introduce new CSS; if no breakpoint pattern exists,
  fall back to a plain horizontally-scrollable table.

- **Section placement drift:** PRD-112 says "after macro blocks, before
  candidate board." Implementer must read existing renderer order and
  insert at exactly that point; do not reorder unrelated sections.

- **Color/badge invention:** R4 caps formatting to existing badge CSS for
  enum values that already have classes (e.g., `_LIFECYCLE_BADGE_CSS`). If
  no class exists for a `trend_alignment` value, render plain text — do
  NOT invent a new badge class.

## 9. Read-only table vs. compact card vs. collapsed panel

**Read-only table.** Justification:

- 6 rows × 9 columns is too wide for a card grid and too dense for a
  collapsed panel header.
- A table aligns with the candidate-board pattern already used elsewhere
  in the dashboard.
- A collapsed panel adds JS state (`<details>` would be acceptable but
  introduces visibility friction for a visibility-first system).

**Binding:** flat read-only table; no `<details>` toggle; no card grid.

## 10. UI publish workflow

R10 makes regenerated `ui/dashboard.html` and `ui/index.html` mandatory in
the implementing commit, matching the user's standing dashboard publish
rule (Memory: "Dashboard publish rule"). **Accept.**

## Independent risks Claude flags

**a) Race with sidecar producer.**
The producer writes via `.tmp → replace`. The renderer reads via
`_load_json`. Atomic replace plus a single-shot read makes a torn read
impossible on POSIX. No additional locking required.

**b) Symbols absent from sidecar but in `TREND_STRUCTURE_SYMBOLS`.**
Possible if the producer writes a partial snapshot. R3 specifies
placeholder rows with `data_status = MISSING`. Good.

**c) Sidecar contains a banned symbol.**
PRD-110 R2 forbids this on the producer side, but the renderer must NOT
trust the producer to enforce. R3's "symbols present in the sidecar but
NOT in `TREND_STRUCTURE_SYMBOLS` are NOT rendered" is the defensive lock.
Test (e) covers it. Good.

**d) Test fixture coupling.**
`tests/test_dashboard_renderer.py` uses plain dict fixtures and
`monkeypatch.setattr(... _MACRO_SNAPSHOT_PATH)`. PRD-112 R9 follows the
same pattern for `_TREND_STRUCTURE_PATH`. No new fixture infrastructure
needed.

**e) Cross-review gate.**
This PRD changes dashboard behavior → Codex review IS required per
CLAUDE.md "Cross-review gate" → "Invoke Codex when… dashboard behavior
changes." Codex must be invoked before implementation.

**f) Contract integrity.**
The PRD does not touch `cuttingboard/contract.py`,
`cuttingboard/output.py`, qualification, regime, decision, or notification
paths. Output contract (`TRADES | NO TRADE | HALT`) is preserved.

**g) Strategy mutation.**
Rejected by R2, R7, and the SCOPE / OUT OF SCOPE bounds. No strategy
mutation possible within the FILES list.
