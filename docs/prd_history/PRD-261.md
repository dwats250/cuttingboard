R5 — Fix R1's `git log -c` false positive
`"Bash(git *-c *=*)"` denies `git log -c --format=%H -1` — a legitimate,
read-only combined-diff log invocation — because the wildcard between
`-c` and `=` doesn't require adjacency, so `--format=%H` later in the
same command supplies the `=` the pattern is looking for. Fix by
anchoring `-c` to the position immediately after `git` (a global option
can only mean config-injection there; a subcommand's own `-c` flag,
like `git log -c`, cannot appear in that position).

FAIL: `git log -c --format=%H -1` is denied, or the original
`git -c core.fsmonitor=/path/to/hook status --short` attack is no longer
denied after the fix.

Accepted trade-off, not silently dropped: anchoring to `git -c` no
longer catches `-c name=value` preceded by another global option (e.g.
`git -C . -c core.fsmonitor=x status`). Wildcard-only patterns can't
express "either immediately after git, or after exactly one other global
option, but never after a subcommand name" without reintroducing the
`git log -c` false positive for the `-C`-first case too. Given the
standing accident-prevention-not-security-boundary doctrine and finding
#1 below (shell-expansion evasion already defeats this whole class
regardless), closing the common case cleanly beats a broader pattern
that's both incomplete and re-breaks `git log -c`.

R6 — Fix pre-existing `ruff check -o` space-adjacency bug
`"Bash(*ruff check* -o *)"` requires a space on both sides of `-o`,
missing the attached form `-o<path>` (verified live: `ruff check
-o/path/to/report .` writes the file, exit 0). Fix by dropping the
trailing-space requirement only: `"Bash(*ruff check* -o*)"`. The leading
space must stay — dropping it too would make the pattern match inside
`--output-format` (a real, distinct, harmless ruff flag: `--output` and
`-o` differ by exactly one dash, so " -o" — space then single dash —
does not appear inside "--output-format", which has two dashes before
the first o).

FAIL: `ruff check -o/path/to/report .` executes without a confirmation
prompt, or `ruff check --output-format=json .` (unrelated, legitimate
flag) is newly denied.

R7 — Close the `git push` destructive/reorder cluster
Three related gaps in one requirement, same command surface:
1. Force-push reordering: the existing denies
   (`"Bash(git push --force:*)"` etc.) are anchored to the start of the
   command, so `--force`/`-f` positioned anywhere other than
   immediately after `git push` bypasses them (verified:
   `git push --porcelain --force origin HEAD:refs/heads/main` matches
   none of the three existing force denies).
2. No deny at all for other destructive push modes: `-d`/`--delete`
   (deletes remote refs), `--mirror` (force-overwrites all refs),
   `--prune` (deletes remote refs not present locally) — all reachable
   through the broad `"Bash(git push:*)"` allow with zero backstop.
3. `--receive-pack` abbreviation bypass: `"Bash(git * --receive-pack*)"`
   only matches the full spelling; git accepts unambiguous abbreviations,
   and `--rece` is already unambiguous against every other `git push`
   flag (verified via `git push --help`: the only other flag sharing the
   `--rec` prefix is `--recurse-submodules`, which diverges at the 5th
   character — `e` vs `u` — so `--rece` cannot mean anything else).
   `--exec` (a documented alias for the same option) is already covered
   by the existing `"Bash(git * --exec*)"` deny.

FAIL: any of `git push --porcelain --force origin HEAD:refs/heads/main`,
`git push -d origin refs/heads/main`, `git push --mirror origin`,
`git push --prune origin`, or `git push --rece=/path/to/program origin
main` executes without a confirmation prompt; or a legitimate push to a
branch whose name happens to contain "-f"/"-d" as a substring (e.g.
`add-feature`, `some-dev-branch`) is newly denied.

Accepted trade-off, not silently dropped: the new `-f`/`-d` short-form
denies require the flag to appear as its own space-bounded token
(` -f`/` -d`, boundary satisfied by a following space or end-of-string)
specifically to avoid denying ordinary branch names containing those
letter pairs. This means a combined short-option form
(`git push -qf origin main`, quiet+force combined per POSIX getopt
convention) is NOT caught — documented as the same class of residual
gap as finding #1 (pattern-matching cannot fully parse combined/expanded
shell forms without a real argument parser).

BOT THREAD DISPOSITION (R4)
R4 asked to verify, not assume, whether GitHub's reply API accepts a
reply on a thread belonging to an already-merged PR before falling back
to a top-level comment. Verified: attempting an in-thread reply
(`gh api repos/.../pulls/147/comments/3582912995/replies -f
body="..."`) was explicitly DENIED at the permission layer — not an
API-side rejection (a 4xx from GitHub saying the thread is locked or
read-only), a tool-call denial in this session. That is a different
failure mode than R4 anticipated (an API limitation tied to the PR being
merged).

The concrete, verifiable reason, not an analogy: posting either a reply
or a fresh top-level comment requires a request body, which `gh api`
only accepts via `-f`/`--raw-field` (or `-F`/`--field`) — and
`"Bash(gh api *-f*)"` (added in PRD-258 R4) denies any `gh api` call
carrying that flag, unconditionally, regardless of which endpoint or
whether the target PR is open or merged. Confirmed live in the
fresh-context Claude review of this PRD: a bare `gh api` GET (no
`-f`/`-X`) against the same PR is NOT denied; the block is on the
write-shaped flag, not on the PR's merged state. The planned
top-level-comment fallback needs the identical `-f body=...` flag, so it
would hit the identical deny — attempting it would not be gathering new
information, it would be re-running a call already known to fail for a
reason unrelated to which endpoint it targets.

Honest caveat, not papered over: this same `-f body=...` shape was used
successfully, many times, earlier in PRD-258's own bot-disposition rounds
(all 22 in-thread replies on PR #147 itself), after the `"Bash(gh api
*-f*)"` deny already existed. That is a genuine inconsistency with the
explanation above, not resolved here — plausibly a difference between
this session's own live permission handling and the fresh-context
reviewer's separate session, or a live human decision independent of
`.claude/settings.json` entirely, rather than the deny rule mechanically
firing every time regardless of context. Recorded as an open question
about this session's own permission behavior, not asserted as settled.

What was found, verified live by the reviewer against the real
permission engine (not a simulation): R7's two short-form deny patterns,
`"Bash(git *push* -f:*)"` and `"Bash(git *push* -d:*)"`, matched
*nothing at all* — not even the exact literal-order case the
pre-existing (pre-PRD-261) pattern used to catch. `git push -f
<remote> <ref>` and `git push -d <remote> <ref>` both executed without
any confirmation prompt. This is a regression, not merely an incomplete
fix.

Root cause (confirmed by both the reviewer and independently
re-confirmed live by this session afterward): the trailing `:*` suffix
is only a safe shorthand for "prefix, then a space-or-end-of-string
boundary" when everything before it is a plain literal with no embedded
`*`. Embedding a glob wildcard inside that same "prefix" — as both
broken patterns did, via `*push*` earlier in the pattern — makes the
whole thing require the literal text `git *push* -f` (asterisks and
all) to open the command, which no real invocation ever contains, so
the pattern can never fire. This directly contradicts this session's
own earlier-established understanding of `:*` semantics (recorded via a
`claude-code-guide` subagent citation and used throughout PRD-258/259):
that understanding was incomplete — it held for the simple case
(pure-literal-prefix + `:*`, e.g. the surviving `"Bash(gh pr merge:*)"`,
re-verified still working) but not for this compound case. Grepped the
*entire* deny list (not just R7) for any other pattern combining an
earlier `*` with a trailing `:*`: none found — this was a two-pattern,
self-contained regression from this amendment's own first draft, not a
pre-existing or wider problem.

Real side effect, recorded transparently rather than smoothed over: the
reviewer's live verification of `-d` (after first safely testing with a
fake remote name, which proved the bug without touching anything real)

only, no repeat of the real-origin exposure): both patterns changed from
`...-f:*)"` / `...-d:*)"` to a bare trailing `*` with no colon at all —

already-fixed `"Bash(*ruff check* -o*)"` style. A bare trailing `*`
(unlike `:*`) correctly handles both the interior case (`-f` followed by
more tokens) and the end-of-string case (`-f` as the very last token, no
trailing space) since a plain `*` matches zero-or-more characters
