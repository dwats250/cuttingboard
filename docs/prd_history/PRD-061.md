PRD-061 — PRD Registry Numbering Guard

STATUS
COMPLETE

GOAL
Add deterministic repo-level PRD tracking so future agents cannot invent PRD numbers from chat history.

SCOPE
- Add `docs/prd_index.json` as the machine-readable PRD tracking source.
- Add a deterministic validator for tracked PRD registry/index consistency.
- Add isolated tests for valid state, validator failures, parser ignores, and non-mutation behavior.
- Add the PRD-061 registry/history records.

OUT OF SCOPE
- CI enforcement
- GitHub Actions changes
- pre-commit hook changes
- automatic registry rewriting
- automatic history doc generation
- branch inspection
- git log parsing
- trading system changes
- dashboard changes
- runtime changes
- payload changes
- contract changes
- notification changes
- macro pressure integration
- event-risk feature work
- future PRD drafting

FILES
A docs/prd_index.json
A tools/validate_prd_registry.py
A tests/test_prd_registry.py
A docs/prd_history/PRD-061.md
M docs/PRD_REGISTRY.md

REQUIREMENTS

R1 — Machine Index
`docs/prd_index.json` MUST contain `tracking_start`, `latest_complete`, `next_prd`, and sorted PRD entries from PRD-056 through the current tracked PRD state.

FAIL: `docs/prd_index.json` is missing, malformed, unsorted, duplicated, or omits a required tracked PRD number.

R2 — Exact Registry Titles
Tracked entries in `docs/prd_index.json` MUST use the current `docs/PRD_REGISTRY.md` titles exactly. Existing titles MUST NOT be normalized, shortened, rewritten, or simplified.

FAIL: A tracked PRD title differs between `docs/prd_index.json` and `docs/PRD_REGISTRY.md`.

R3 — Deterministic Validator
`tools/validate_prd_registry.py` MUST validate schema, statuses, duplicate numbers, missing tracked numbers, commit hashes, `latest_complete`, `next_prd`, matching history docs, and registry/index agreement.

FAIL: The validator exits 0 for an invalid tracked registry/index state.

R4 — Registry Row Coverage
Any normal integer `PRD-NNN` row in `docs/PRD_REGISTRY.md` with `NNN >= tracking_start` MUST exist in `docs/prd_index.json`.

FAIL: A normal integer tracked registry row such as `PRD-062` exists but `docs/prd_index.json` lacks number `62`.

R5 — Parser Boundaries
The validator MUST parse only the main PRD registry table and ignore Init rows, audit/report tables, decimal PRDs, letter suffix PRDs, PATCH suffix rows, and non-table prose.

FAIL: An ignored historical or special PRD row causes validation failure.

R6 — Non-Mutation
The validator MUST NOT modify `docs/prd_index.json`, `docs/PRD_REGISTRY.md`, or tracked PRD history files.

FAIL: File contents change after running the validator.

DATA FLOW
`docs/prd_index.json` -> validator schema checks -> deterministic pass/fail
`docs/PRD_REGISTRY.md` -> main table parser -> tracked entry comparison
`docs/prd_history/PRD-###.md` -> existence checks for tracked entries

FAIL CONDITIONS
- `docs/prd_index.json` is missing, malformed, unsorted, duplicated, or omits a required tracked PRD number.
- A tracked PRD title differs between `docs/prd_index.json` and `docs/PRD_REGISTRY.md`.
- The validator exits 0 for an invalid tracked registry/index state.
- A normal integer tracked registry row such as `PRD-062` exists but `docs/prd_index.json` lacks number `62`.
- An ignored historical or special PRD row causes validation failure.
- File contents change after running the validator.

VALIDATION

- `python3 -m pytest tests/test_prd_registry.py -q`
- `python3 tools/validate_prd_registry.py`
- `python3 -m pytest tests/ -q`
