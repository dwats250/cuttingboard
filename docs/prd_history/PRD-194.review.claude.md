## HEADLINE FINDING (mechanism pivot — discovered during implementation)

The PRD as authored specified publishing by **rebasing the main-based artifact commit
onto `origin/publish`** (ci_push_artifacts.sh: "four `main` refs become
`$PUBLISH_BRANCH`"). This is DEFECTIVE: the artifact commit's parent is `main`'s tip,
whose `logs/audit.jsonl` is the frozen copy, while `publish`'s `audit.jsonl` is the
accumulated copy. Rebasing 3-way-merges these divergent append-only histories — both
sides add lines at EOF — which git resolves as a CONFLICT, not a superset. Publishing
would fail intermittently, defeating the PRD's entire purpose.

Surfaced to Dustin mid-implementation. Decision: **worktree overlay** with a hard
requirement — **delta-append only THIS run's new `audit.jsonl` rows onto the publish
tip's CURRENT file (never wholesale-copy/clobber)**; `ui/` + `regime_history` +
`latest_*` are full-regen overwrites. SUPERSEDED in the PRD's SCOPE, CHANGE SURFACE,
and R5. Implemented in `tools/ci_push_artifacts.sh`:
- `git worktree add` on the publish tip; commit's parent = publish tip => clean
  fast-forward (cb-publish serialization keeps the tip from moving).
- `audit.jsonl`: `tail -n +(base_rows+1)` of POST_SHA's file appended onto the
  worktree copy, where base = `CB_PUBLISH_BASE_SHA` (the tip we restored from). This
  preserves any rows another writer landed since restore — the never-clobber guarantee.

## BINDING AMENDMENTS (applied)

- **A1 — R3/R4 restore gate.** Replaced the blanket `|| true` with
  `tools/ci_restore_publish_state.sh`: explicit `git ls-remote --exit-code --heads`
  gate; branch present -> fetch + checkout, a real checkout error FAILS the job (no
  swallow); branch absent -> bootstrap from main's checked-in copy. A legitimately
  absent path on an existing branch (macro state.json pre-first-publish) is logged +
  skipped — not a swallowed error, so R3's fail-live clause holds for `audit.jsonl`
  (always present post-seed). Helper records `CB_PUBLISH_BASE_SHA`.
- **A2 — R2 deploy trigger.** Confirmed all three writers push with the default
  `GITHUB_TOKEN` (no PAT in any workflow env). A GITHUB_TOKEN push to `publish` does
  not fire `on: push`, so `pages.yml` deploys via `workflow_run` on all three writer
  names; `push:[main]` dropped; checkout `ref: publish`.

## RESIDUAL (accepted, recorded)

`cb-publish` serialization eliminates concurrent-append corruption, but GitHub retains
only the most-recent PENDING run per concurrency group, so bunched triggers can cancel
a pending writer — one dropped append, recovered by the next run's R3 read-back. The
delta-append (never-clobber) mechanism means a dropped run loses only ITS rows, never
prior history. Accepted; revisit with a push-retry loop only if it proves material in
practice.

## RECOMMENDED (post-merge, non-blocking)

- The worktree publish + delta-append paths are NOT exercisable in CI unit tests (they
  need a live multi-branch remote). The R7 guards pin CONTENT, not runtime behavior.
  The mandatory validation is the **manual rollout**: seed `publish` from `main`, leave
  it unprotected, dispatch all three writers, and confirm (a) each pushes to `publish`
  not `main`, (b) `audit.jsonl` accumulates without loss across two runs, (c) Pages
  redeploys `publish`'s `ui/` from each writer, (d) `main` is byte-unchanged.
- If the cancel-residual drops appends often, add a single fetch+rebuild+push retry in
  the publish helper (delta-append is idempotent against the current tip).
