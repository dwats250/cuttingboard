SCOPE
- Deny `git -c <name>=<value> ...` (any subcommand): a global option that
  sets config inline for one invocation, bypassing the `git *config*`
  deny (which only matches the literal `config` *subcommand*) and
  re-arming the same fsmonitor/textconv/ext-diff/gpg.program helper paths
  PRD-258 already closed one way.
- Deny `ruff check --config ...`: `--config` can set arbitrary
  configuration keys, including enabling fixes (`--config 'fix = true'`)
  without any `--fix`-shaped token ever appearing, bypassing all four
  existing ruff write-flag denies.
- Extend the `git worktree` deny from `remove` only to the full mutating
  subcommand set (`add`, `move`, `prune`, `lock`, `unlock`, `repair`) —
  PRD-258 only asked for `worktree list` to stay read-only-allowed and
  only denied `remove`, leaving the other five mutating subcommands
  reachable via local settings' blanket `git *` with no project deny.

REQUIREMENTS

R1 — Deny `git -c` config overrides
Add a deny pattern catching `git -c <name>=<value>` in any position
(global options can precede the subcommand, same lesson as PRD-258's
`checkout`/`reset`/`rebase` middle-wildcard fix) across any git
invocation this PRD's allow-list permits.

FAIL: `git -c core.fsmonitor=/path/to/hook status --short` (or any other
allowed git subcommand preceded by `-c <name>=<value>`) executes without
a confirmation prompt.

R2 — Deny `ruff check --config`
Add a deny pattern catching `--config` on any of the three existing
`ruff check` allow forms (bare binary, `python -m`, `python3 -m`),
positioned so it matches regardless of whether `--config` is the first
token after `check` or preceded by other arguments (same
space-adjacency lesson PRD-258's round 3 already applied to `--fix`).

FAIL: `ruff check --config 'fix = true' <path>` (or the equivalent
`python -m ruff check` / `python3 -m ruff check` forms) executes without
a confirmation prompt.

R3 — Extend `git worktree` denies beyond `remove`
Add deny patterns for `git worktree add`, `move`, `prune`, `lock`,
`unlock`, `repair` — the full mutating subcommand set per `git worktree
-h` — using the same middle-wildcard construction as `remove` (and as
R1/R2 above) so a preceding token or global option doesn't bypass them.

FAIL: any of `git worktree add`/`move`/`prune`/`lock`/`unlock`/`repair`
executes without a confirmation prompt.

R4 — Disposition the three originating bot threads
The three findings arrived on PR #147, which merged before they were
seen. Before assuming they cannot be dispositioned in-thread, verify
directly whether GitHub's review-comment reply API accepts a reply on a
thread belonging to a merged PR (it may — review threads are not
locked by merge, only by an explicit lock action). If it does, reply
in-thread citing this PRD and the fixing commit, matching PRD-228's
standard disposition, same as every other round in PRD-258. If it
genuinely does not, disposition via a single top-level PR comment on
#147 naming all three thread IDs and the fixing commit, and record that
fallback (with the verification result) in this PRD's VALIDATION section
— do not assume the reply API is unavailable without trying it first.

FAIL: none of the three threads has a disposition record (in-thread
reply, or the documented top-level-comment fallback) citing this PRD.
