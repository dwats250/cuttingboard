Status: IN PROGRESS

WHY NOW
`detect_continuation_breakout`'s lookback window (`df.iloc[-(n+1):-1]`)
includes the hold candle the HOLD gate re-examines, so `hold_close <=
breakout_level` is a mathematical certainty for any valid OHLC bar
(High >= Close) — `NO_HOLD_CONFIRMATION` fires on 100% of breakouts,
verified against 3,984 real symbol-days with zero exceptions
(`docs/prd_history/PRD-256.hold-confirmation-gate.proposal.md`).
PRD-256 fixed what continuation trades are CHARGED; this gate decides
whether they are ACCEPTED, and it has never once fired correctly.
Promoted from the sidecar on Dustin's 2026-07-14 Gate A ruling
(direction: strictly-prior window; recorded in docs/DECISIONS.md).
This change is strictly risk-increasing — it turns a dead path live on
real money — and carries an explicit stop-and-report clause (R4).

MAX EXPECTED DELTA
Amended 2026-07-14 (R7 amendment, below): production code <= 15 LOC in
`cuttingboard/qualification.py`, all inside
`detect_continuation_breakout` (slice + docstring/comment) and
`_qualify_continuation_candidate`'s HOLD check (multi-hold-candle
completeness, R7). Tests: <= 160 LOC across the two continuation test
files. Docs: gate 3/4 rows + prose in `docs/trade_qualification.md`,
the sidecar's PROMOTED note + R3 characterization appendix, one new
first-fire-consumers proposal sidecar, DECISIONS/PROJECT_STATE entries.
Review artifacts additional per the standing REVIEW-stage carve-out.

GOAL
Make the continuation path's HOLD-confirmation gate satisfiable by
computing `breakout_level` from bars strictly prior to the bar under
test, then characterize — on real cached OHLCV, before any merge — what
the corrected gate actually accepts, so Dustin rules on observed
acceptance behavior rather than on the arithmetic fix alone.

REQUIREMENTS

R1 — Strictly-prior lookback window (red first)
`breakout_level` = max High over the `CONTINUATION_BREAKOUT_BARS` bars
ending strictly before the hold candle(s):
`df.iloc[-(n + 1 + hold_candles):-(1 + hold_candles)]`. The existing
length guard (`len(df) < n + 1 + hold_candles`) already reserves exactly
these bars. The proving test constructs an all-valid-OHLC sequence
(every bar High >= max(Open, Close), Low <= min(Open, Close)) that
clears every continuation gate; it MUST fail with
`NO_HOLD_CONFIRMATION` against pre-change code and pass post-change.

FAIL: the proving test passes against pre-change code, or any valid-OHLC
input reaches `qualified=True` through a window that includes the hold
candle.

R2 — Valid-OHLC fixtures only
Every fixture in the two test files that passes the HOLD gate uses
valid OHLC bars. The invalid-bar fixtures (`_breakout_df` bar 8 High=98
< Close=100; `_qualified_df` bar 8 High=101 < Close=102) are rebuilt.

FAIL: `grep`-verifiable audit of the two test files finds any
HOLD-passing fixture bar with High < Close.

R3 — Real-data characterization of the corrected gate
Replay the corrected, unmodified `_qualify_continuation_candidate`
against the real cached OHLCV (`data/cache/*_ohlcv.parquet`, all 16
tradable symbols, full cached history, real Wilder ATR14, real
day-over-day VIX) and append to the HOLD-gate sidecar: acceptance count
and rate; per-symbol and temporal concentration; per-candidate anatomy
(entry, stop = breakout level, risk, RR, binding gates); accepted
candidates' real max-loss dollars through the real, unmodified
`build_option_setups` under both CREDIT and DEBIT IV resolutions at the
re-converged `MAX_RISK_PCT_PER_TRADE` budget; and a
pre-fix control run over the same harness confirming 0 acceptances.

FAIL: characterization is missing, covers fewer than all 16 tradable
symbols, omits real dollar figures at the re-converged budget, omits
the pre-fix zero-acceptance control, or is derived from synthetic
rather than cached data.

R4 — Stop-and-report clause
If R3 shows acceptance that looks wrong — too many, too loose,
concentrated somewhere suspicious — the fix is NOT submitted for merge;
the findings are reported for Dustin's ruling instead. Either way the
REVIEW report presents R3's findings and the risk-direction statement
(dead path goes live) before any merge.

FAIL: the PR is marked ready-for-review without R3's characterization
present in-tree.

R5 — Docs match code
`docs/trade_qualification.md` CONTINUATION gates 3-4 describe the
strictly-prior window (gate 3: close clears the 5-bar high ending
before the hold candle; gate 4: the hold close also held above that
same prior level).

FAIL: the doc still describes, or is silent about, a lookback that
includes the hold candle.

R7 — Multi-hold-candle completeness (amendment, 2026-07-14)
Added on the commissioned second-model review's diff verdict
(`docs/prd_history/PRD-259.review.codex.md`): the HOLD gate validates
EVERY completed hold candle's close against `breakout_level`, not only
the oldest (`df.iloc[-(hold_candles+1):-1]["Close"]`, reject when any
is `<= breakout_level`). Identical behavior at the current
`CONTINUATION_HOLD_CANDLES = 1`; proven by a red test at h=2 (an
intervening dip below the level must reject, and did not pre-change).

FAIL: with `CONTINUATION_HOLD_CANDLES` monkeypatched to 2, a sequence
whose intermediate hold candle closes below `breakout_level` reaches
`qualified=True`, or the h=2 red test passes against pre-R7 code.
