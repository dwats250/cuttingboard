REQUIREMENTS

R1 — Deny `git -c` config overrides
Add a deny pattern catching `git -c <name>=<value>` in any position
(global options can precede the subcommand, same lesson as PRD-258's
`checkout`/`reset`/`rebase` middle-wildcard fix) across any git
invocation this PRD's allow-list permits.

FAIL: `git -c core.fsmonitor=/path/to/hook status --short` (or any other
allowed git subcommand preceded by `-c <name>=<value>`) executes without
a confirmation prompt.

R2 — Deny `ruff check --config`
Add a deny pattern catching `--config` on any of the three existing
`ruff check` allow forms (bare binary, `python -m`, `python3 -m`),
positioned so it matches regardless of whether `--config` is the first
token after `check` or preceded by other arguments (same
space-adjacency lesson PRD-258's round 3 already applied to `--fix`).

FAIL: `ruff check --config 'fix = true' <path>` (or the equivalent
`python -m ruff check` / `python3 -m ruff check` forms) executes without
a confirmation prompt.

R3 — Extend `git worktree` denies beyond `remove`

`unlock`, `repair` — the full mutating subcommand set per `git worktree
-h` — using the same middle-wildcard construction as `remove` (and as
R1/R2 above) so a preceding token or global option doesn't bypass them.

FAIL: any of `git worktree add`/`move`/`prune`/`lock`/`unlock`/`repair`
executes without a confirmation prompt.

IMPLEMENTATION NOTES

R1 implemented as `"Bash(git *-c *=*)"`, not a bare `-c` substring match:
git's global config-override always takes `-c <name>=<value>` form, so
requiring the `=` excludes `git log -c` (a real, unrelated, harmless
per-subcommand flag meaning "show combined diff for merges" — verified
via `git log -h`, no `=` argument, would have been a false-positive
casualty of a looser pattern). Confirmed by the matching simulation
(`git log -c` and `git log --oneline -c -p main` both correctly NOT
denied; every `git -c <name>=<value> ...` form, in any position, is).

R3 implemented as five new entries, not six: `"Bash(git *worktree
*lock*)"` covers both `lock` and `unlock`, since `unlock` contains
`lock` as a literal substring — a separate `*unlock*` entry would be
redundant. Confirmed by the matching simulation (`git worktree unlock`
denied via the `lock` pattern).

Found and cleaned up while implementing (not part of R1-R3, a direct
side-effect of touching this file): the seven original space-anchored
`gh api * -X*`-style deny entries from PRD-258 R4 were still present
alongside their PRD-258-round-3 corrected (`gh api *-X*`-style, no
required space) replacements — fully superseded, dead weight, not a
correctness bug (an extra strict-subset rule never independently
matters) but clutter. Removed the seven dead originals.
