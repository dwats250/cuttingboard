R1 implemented as `"Bash(git *-c *=*)"`, not a bare `-c` substring match:
git's global config-override always takes `-c <name>=<value>` form, so
requiring the `=` excludes `git log -c` (a real, unrelated, harmless
per-subcommand flag meaning "show combined diff for merges" — verified
via `git log -h`, no `=` argument, would have been a false-positive
casualty of a looser pattern). Confirmed by the matching simulation
(`git log -c` and `git log --oneline -c -p main` both correctly NOT
denied; every `git -c <name>=<value> ...` form, in any position, is).

*lock*)"` covers both `lock` and `unlock`, since `unlock` contains
`lock` as a literal substring — a separate `*unlock*` entry would be
redundant. Confirmed by the matching simulation (`git worktree unlock`
denied via the `lock` pattern).

Found and cleaned up while implementing (not part of R1-R3, a direct

BOT THREAD DISPOSITION (R4)
R4 asked to verify, not assume, whether GitHub's reply API accepts a
reply on a thread belonging to an already-merged PR before falling back
to a top-level comment. Verified: attempting an in-thread reply
(`gh api repos/.../pulls/147/comments/3582912995/replies -f
body="..."`) was explicitly DENIED at the permission layer — not an
API-side rejection (a 4xx from GitHub saying the thread is locked or
read-only), a tool-call denial in this session. That is a different
failure mode than R4 anticipated (an API limitation tied to the PR being
merged).

The concrete, verifiable reason, not an analogy: posting either a reply
or a fresh top-level comment requires a request body, which `gh api`
only accepts via `-f`/`--raw-field` (or `-F`/`--field`) — and
`"Bash(gh api *-f*)"` (added in PRD-258 R4) denies any `gh api` call
carrying that flag, unconditionally, regardless of which endpoint or
whether the target PR is open or merged. Confirmed live in the
fresh-context Claude review of this PRD: a bare `gh api` GET (no
`-f`/`-X`) against the same PR is NOT denied; the block is on the
write-shaped flag, not on the PR's merged state. The planned
top-level-comment fallback needs the identical `-f body=...` flag, so it
would hit the identical deny — attempting it would not be gathering new
information, it would be re-running a call already known to fail for a
reason unrelated to which endpoint it targets.

Honest caveat, not papered over: this same `-f body=...` shape was used
successfully, many times, earlier in PRD-258's own bot-disposition rounds
(all 22 in-thread replies on PR #147 itself), after the `"Bash(gh api
*-f*)"` deny already existed. That is a genuine inconsistency with the
explanation above, not resolved here — plausibly a difference between
this session's own live permission handling and the fresh-context
reviewer's separate session, or a live human decision independent of
`.claude/settings.json` entirely, rather than the deny rule mechanically
firing every time regardless of context. Recorded as an open question
about this session's own permission behavior, not asserted as settled.

Disposition recorded here instead, in-tree, citing the exact thread IDs
and the fixing commit — durable and equally satisfies PRD-228's "no
substantive thread left dangling" intent, even though it is not a GitHub
reply:
- Thread `3582912995` ("Deny git -c config overrides") — ACTIONED,
