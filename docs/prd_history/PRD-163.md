# PRD-163: Fix regime permission wording for EXPANSION posture

Lane: MICRO / SURFACE
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-05

## GOAL

The dashboard System State block presents contradictory user-facing
instructions when the regime is `EXPANSION`. On the live run
`hourly-20260604T171635Z` (recorded in `docs/PROJECT_STATE.md`, 2026-06-04
real-run inspection) the **Regime** field rendered `Stand down` while the
**Permission** field in the same block read
`EXPANSION — momentum allowed. Continuation entries.` — one field tells the
trader to stand down, the other tells them momentum entries are allowed.

Root cause: `_regime_to_permission_verb(regime)` in
`cuttingboard/delivery/dashboard_renderer.py` (line 1027) maps only
`RISK_ON`→`"Longs allowed"` and `RISK_OFF`→`"Shorts allowed"`, and falls
**every other regime** — including `EXPANSION` — through to a catch-all
`return "Stand down"`. `EXPANSION` is a first-class regime literal
(`cuttingboard/regime.py:34`, `EXPANSION = "EXPANSION"`) carried into
`payload["summary"]["market_regime"]` via `contract.py:237`
(`_safe_str(regime.regime)`), so it reaches the verb translator as the
literal string `"EXPANSION"` and silently degrades to `"Stand down"`.
EXPANSION is a breadth-and-leadership-confirmed risk-on advance whose paired
posture is `EXPANSION_LONG` (`regime.py:41`); the Permission line for that
posture (`runtime.py:_PERMISSION_LINES["EXPANSION_LONG"]`) correctly reads
"momentum allowed. Continuation entries." The Regime field must agree with
that directionally instead of contradicting it.

## SCOPE

- Add an `EXPANSION` branch to `_regime_to_permission_verb` in
  `cuttingboard/delivery/dashboard_renderer.py` (line 1027) mapping it to a
  long-directional permission verb that agrees with the EXPANSION_LONG
  Permission line. **Recommended verb: `"Momentum longs allowed"`** — it
  does not say "Stand down", agrees directionally with the Permission
  field's "momentum allowed", and stays distinct from the plain-`RISK_ON`
  `"Longs allowed"` so the two regimes are not conflated. (Minimal
  alternative for review: reuse `"Longs allowed"`. See "Design decision for
  review" below.)
- Leave the catch-all `return "Stand down"` in place for all other regimes
  (`NEUTRAL`, `CHAOTIC`, `None`, and any future literal) — these are
  directionless or blocked states for which "Stand down" remains correct.
- Add a focused renderer test asserting the EXPANSION wording and the
  absence of "Stand down" for an EXPANSION payload.
- Track this PRD in registry/state per existing convention.

## DESIGN DECISION FOR REVIEW

Two verb options satisfy every acceptance criterion. Flagging for the
review gate to confirm before implementation:

- **(Recommended) `"Momentum longs allowed"`** — distinct from `RISK_ON`'s
  `"Longs allowed"`, mirrors the Permission line's "momentum allowed", and
  preserves that EXPANSION is a stronger, breadth-confirmed regime than a
  plain risk-on tape. Cost: one new visible string.
- **(Minimal) reuse `"Longs allowed"`** — zero new strings, directionally
  coherent, but collapses the EXPANSION/RISK_ON distinction in the Regime
  field (the badge CSS class still differs, `class="badge EXPANSION"`).

Per VISION "cuts before additions" the minimal option is defensible; the
recommendation favors coherence with the existing Permission language. The
reviewer picks one; R1/R3 are written against the recommended verb and will
be repointed if the minimal option is chosen.
