PRD-035 — Signal Forge dashboard — contract extension + UI dashboard

STATUS
COMPLETE

GOAL
Extend contract.json with a regime block carrying all macro signal data, then render a
Signal Forge dashboard above the existing contract view in the UI.

SCOPE
- Add a `regime` block to contract.json output in output.py
- Add Signal Forge section to ui/index.html (above contract-view)
- Render Signal Forge from contract data in ui/app.js
- Style Signal Forge in ui/styles.css

OUT OF SCOPE
- Changes to regime classification logic (regime.py is read-only)
- Adding raw per-symbol price data beyond what RegimeState already holds
- Any new ingestion or derived metric
- Backtesting or historical views
- Any UI section other than the Signal Forge strip

FILES
M cuttingboard/output.py
M ui/contract.json
M ui/index.html
M ui/app.js
M ui/styles.css

REQUIREMENTS

R1 — Contract regime block
output.py must write a top-level `regime` key in the contract JSON when a RegimeState is
available. If regime is None, `regime` must be null. The block must contain exactly these
fields, no more:

  {
    "classification": "<RISK_ON|RISK_OFF|NEUTRAL|CHAOTIC>",
    "posture":        "<posture string>",
    "confidence":     <float 0.0–1.0>,
    "net_score":      <int>,
    "risk_on_votes":  <int>,
    "risk_off_votes": <int>,
    "neutral_votes":  <int>,
    "total_votes":    <int>,
    "vote_breakdown": { "<key>": "<RISK_ON|RISK_OFF|NEUTRAL>", ... },
    "vix_level":      <float|null>,
    "vix_pct_change": <float|null>
  }

The 8 vote_breakdown keys must match the canonical order from regime.py:
  SPY pct_change, QQQ pct_change, IWM pct_change, VIX level,
  VIX pct_change, DXY pct_change, TNX pct_change, BTC pct_change

FAIL: contract.json parsed in Python has no `regime` key, or `regime` is missing any
      listed field, or contains a field not listed above.

R2 — ui/contract.json sample updated
The sample contract.json must include the `regime` block with representative values so
UI development can proceed without a live run.

FAIL: ui/contract.json does not contain a `regime` key after this PRD is merged.

R3 — Signal Forge section in index.html
index.html must contain a `<div id="signal-forge">` placed immediately before
`<div id="contract-view">`. It must contain five named cells:
  #sf-regime, #sf-confidence, #sf-votes, #sf-vix, #sf-corr

FAIL: `<div id="signal-forge">` is absent, or any of the five cell IDs is missing.

R4 — Signal Forge rendered from contract data in app.js
When a contract is loaded, app.js must populate Signal Forge using `contract.regime`.
If `contract.regime` is null, the entire `#signal-forge` block must be hidden.

Cell rendering rules:
  #sf-regime     — classification tag (colored by RISK_ON/RISK_OFF/NEUTRAL/CHAOTIC)
                   + posture label beneath
  #sf-confidence — numeric value (2 decimal places) + filled bar reflecting the value
  #sf-votes      — 8 pips in canonical key order; green=RISK_ON, red=RISK_OFF, grey=NEUTRAL
  #sf-vix        — vix_level (1 decimal) + vix_pct_change as signed % (1 decimal);
                   null values render as "—"
  #sf-corr       — correlation.risk_modifier from contract.correlation (already present);
                   label reads "Corr ×<value>" (e.g. "Corr ×0.70")

FAIL: any cell not populated after contract load, or Signal Forge shown when regime is null.

R5 — Signal Forge styles in styles.css
styles.css must define layout and color rules for #signal-forge and its cells.
Signal Forge must render as a horizontal strip. It must not affect existing contract-view
layout or styles.

FAIL: Signal Forge cells overlap, or existing contract-view blocks shift position.

DATA FLOW
regime.py:RegimeState → output.py:_build_contract_dict → contract.json["regime"]
contract.json["regime"] → app.js:renderSignalForge() → #signal-forge DOM cells

FAIL CONDITIONS
- contract.json missing `regime` key or any sub-field listed in R1
- vote_breakdown has fewer or more than 8 keys, or keys differ from canonical order
- #signal-forge absent from index.html
- any of the 5 cell IDs absent from index.html
- Signal Forge renders with null regime (must be hidden)
- confidence bar does not reflect numeric value
- vote pips count != total_votes value from contract
- #sf-corr reads from wrong field (must use contract.correlation.risk_modifier)
- existing contract-view blocks displaced by Signal Forge styles

VALIDATION
Manual:
1. Run the pipeline and confirm contract.json contains a `regime` block with all 11 fields.
2. Load the updated sample ui/contract.json in the UI — Signal Forge strip renders above contract view.
3. Verify #sf-regime shows classification tag + posture.
4. Verify #sf-confidence shows 0.75 and bar is ~75% filled.
5. Verify #sf-votes shows 8 pips with correct colors matching vote_breakdown values.
6. Verify #sf-vix shows level and signed pct, or "—" for null.
7. Verify #sf-corr shows "Corr ×0.70" (or whatever the sample value is).
8. Set `regime: null` in sample JSON — confirm Signal Forge is hidden, contract view unaffected.
9. Confirm existing contract blocks (signal bar, primary trade, rejections, etc.) are unchanged.
