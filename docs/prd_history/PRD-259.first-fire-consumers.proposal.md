Fast-follow proposal — first-fire consumers of an accepted continuation
candidate

Status: PROPOSAL — not started, no Gate A yet. Not a registered PRD (a
`.proposal.md` sidecar, allowlisted from the registry-gap check per
PRD-248; no registry row, no PRD number until Dustin rules).

ORIGIN
Surfaced during PRD-259's Stage-0 downstream-consumer audit
(2026-07-14). PRD-259 makes the continuation HOLD gate satisfiable for
the first time; every consumer below was built during the era in which
`_qualify_continuation_candidate` could not accept, so none has ever
executed with an accepted continuation result. Both findings verified
by direct read of current `main`-equivalent code, not inferred.

FINDING A — runtime decision loop hard-index KeyError (LATENT, not
currently reachable — corrected 2026-07-14 by the commissioned
second-model disposition, `docs/prd_history/PRD-259.review.codex.md`,
and re-verified by direct read)
`cuttingboard/runtime/__init__.py` (`_run_decision_gates`) builds
`TradeDecision`s via `create_trade_decision(candidates[symbol], ...)` —
a hard `[]` index. `candidates` comes from
`options.py::generate_candidates`, which SKIPS any symbol whose quote
is missing (`quote is None: continue`). The continuation pass in
`qualification.py::qualify_all` needs only `df`/`dm` — no quote. An
earlier draft of this finding claimed a same-day quote failure makes
the crash reachable; that overstated it: `_run_pipeline` constructs
`ohlcv` as a dict comprehension OVER `candidates`
(`runtime/__init__.py`, `ohlcv = {symbol: df for symbol in candidates
...}`), so a candidate-less symbol has `df=None` and rejects
`DATA_INCOMPLETE` before it can ever be accepted. The join invariant
(every accepted result has a candidate) is real but currently protected
ONLY by that candidate-scoped OHLCV acquisition — widening OHLCV
acquisition later re-arms the KeyError silently.

FINDING B — TradeDecision geometry substitution (silent, worse)
`cuttingboard/trade_decision.py::create_trade_decision` sources
`entry`/`stop`/`target`/`r_r` exclusively from the DIRECT
`TradeCandidate` (`stop = entry - 1x ATR14`, `target = entry + 2x
ATR14` from `options.py::_build_candidate`). The continuation gate
qualified the candidate on a DIFFERENT geometry: `stop_price =
breakout_level`, `reward = 3x ATR14` (`qualification.py`,
`_qualify_continuation_candidate` gates 7-8). That geometry is
discarded; the rendered invalidation level — one of VISION's four
questions — is a number the continuation logic never validated. Every
accepted continuation candidate WITH a quote takes this path silently.

FINDING B SHARPENED + FURTHER FINDINGS (2026-07-14, commissioned
second-model disposition, `docs/prd_history/PRD-259.review.codex.md`;
each independently formed from a spoiler-scrubbed snapshot)
- B sharpened: because production `ohlcv` is candidate-scoped (Finding
  A correction), every real continuation acceptance HAS a candidate —
  and specifically one that already FAILED direct qualification (the
  continuation loop only runs for symbols not qualified/watchlisted via
  DIRECT). So the first accepted continuation trade will render the
  entry/stop/target of a candidate the system just rejected.
- C: contradictory emission — direct hard failure writes
  `excluded[symbol]`; a later continuation acceptance appends to
  `qualified` without removing the exclusion. Contract and audit then
  emit the SAME symbol as both a trade candidate and a qualification
  rejection.
- D: one-contract floor vs adjusted budget — `options.py`'s
  `max(1, ...)` sizes one contract even when that contract's
  strategy max loss exceeds `ACCOUNT_EQUITY x MAX_RISK_PCT_PER_TRADE x
  risk_modifier`. Pre-existing on all paths; first-fires here for
  credit-resolving continuation accepts under correlation penalties.
- E: qualification-only notification paths (`notifications/formatter.py`
  READY block, hourly focus) announce from `qualified_trades` without
  proving a decision/contract exists, and `QualificationResult` carries
  no geometry for them to render.
- F: postmarket "qualified for execution" counts qualification-level
  acceptance even when downstream gates block execution; it also has no
  branch surfacing `continuation_accepted_count` at all.

OPEN DESIGN QUESTIONS (need their own Gate A before any build)
1. Should the decision loop skip, synthesize, or fail-loud on a
   continuation-accepted symbol with no candidate (Finding A)? Fail-loud
   is the PRD-198 default; silent skip hides an accepted trade.
2. What geometry should a continuation TradeDecision carry (Finding B)?
   The gate's own stop (= breakout level) is the natural candidate but
   threading it requires either extending QualificationResult or
   special-casing decision assembly — a modeling decision, not a patch.
