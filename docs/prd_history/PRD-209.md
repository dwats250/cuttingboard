PROVENANCE
diag-2026-06-24/qqq-sma-history.md (recon F08). The diagnosis verified two
count-blind gates in cuttingboard/ingestion.py:
  1. `_fetch_ohlcv_from_yfinance` (:325-361) accept gate is EMPTY-ONLY
     (`if df.empty: raise`, :339) — a short-but-nonempty provider response is
     accepted and cached.
  2. `_is_fresh_ohlcv_cache` (:147-167) validates only the LAST BAR'S DATE
     (`return last_bar.date() >= most_recent_completed_session_date(...)`, :167) —
     a short-but-recent cache is served as "fresh" and never refetched within a
     trading day.
PRD-190 raised OHLCV_FETCH_MONTHS 6→12 so a FULL fetch yields ≥200 bars; it added
NO floor against a SHORT frame. That is the gap this PRD closes.

WHY NOW
The trend-structure SMA columns (sma_50 needs ≥50 closes, sma_200 needs ≥200)
silently go null/unavailable when a symbol's daily frame is short, and — because
the cache is count-blind — the short frame persists across same-day slots until a
session rollover forces a refetch. A bar-count floor on both gates makes a
truncated frame heal-on-refetch or fail loudly (PRD-198 #1), never serve-as-fresh.

GOAL
A daily OHLCV frame with fewer closes than the largest consumer needs (sma_200 ⇒
200) is NEVER served as fresh and NEVER cached as complete: a short cache triggers
a heal-refetch; a short fresh fetch is rejected and either heals or fails loudly;
a genuinely short-history symbol degrades to an honest INSUFFICIENT_HISTORY without
refetch-looping.

SCOPE (read ingestion.py first; ground every edit in the actual functions)
- Add a parameterized floor constant to config.py (e.g. OHLCV_FULL_MIN_BARS = 200),
  defined as the largest SMA-consumer window (sma_200). Both gates reference it; no
  literal int in ingestion.py, and the two gates use the SAME constant.
- `_is_fresh_ohlcv_cache` (ingestion.py:147-167): in addition to the existing
  date check, require `len(df) >= OHLCV_FULL_MIN_BARS` for a frame to count as
  fresh — EXCEPT a frame already marked "short-by-nature" (see edge below). A
  short, date-current cache is therefore NOT fresh → fetch_ohlcv falls through to a
  live refetch (heal immediately).
- `_fetch_ohlcv_from_yfinance` (:325-361): after a successful download, if
  `len(df) < OHLCV_FULL_MIN_BARS`, do NOT accept it as complete — retry within the
  existing FETCH_RETRIES loop; if every retry is still short, return None AND log
  loudly (PRD-198 #1 fail-loud), and NEVER write the short frame to the cache via
  `_write_ohlcv_cache` (so a short frame cannot poison the cache).
- EDGE — genuinely short-history symbol must not infinite-refetch (charge): the
  12-month window yields ≥200 bars for any symbol listed ≥~200 trading days; a
  symbol whose provider MAX-available history is < floor (newly listed) legitimately
  cannot reach 200. For such a symbol the system must render INSUFFICIENT_HISTORY
  (honest) and NOT refetch on every consecutive same-day slot. Specify the
  distinguisher the implementer must use (recommended: persist a "short-by-nature"
  marker once a refetch returns a frame no longer than the cached one — i.e. the
  provider's ceiling — and treat that marker as cache-fresh by date; do not loop).
  The floor applies to the ≥200 consumers; it must not convert a genuinely-short
  symbol into a per-slot refetch storm.
- F15 (Juneteenth SPY-cache mislabel) is NOT covered by this primitive and is NOT
  folded: regime_history._load_spy_close_series (delivery/regime_history.py:81-100)
  reads the SPY parquet DIRECTLY via pd.read_parquet (isolated reader, PRD-175 R5),
  not ingestion.fetch_ohlcv/_is_fresh_ohlcv_cache. F15 is a warning-classification
  bug in regime_history.aggregate (holiday-gap vs truncated-cache), a different
  module and mechanism → stays on the PRD-198 #3 (authoritative-source) track.
  Folding would break single-concern.

DESIGN NOTES
- The floor is a CACHE/FETCH-completeness gate, not a data-quality judgment: it
  asks "does this frame have enough bars for the ≥200 consumer", nothing about
  values. Pure count.
- Heal-on-refetch is the primary remedy (a short cache refetches to full); loud-
  raise is the fallback when the provider itself can't supply a full frame.
- Reuse the existing FETCH_RETRIES/backoff loop; no new fetch machinery.

NON-GOALS
- No change to the quote path (_try_yfinance_quote), intraday bars
  (fetch_intraday_bars), or OHLCV_MIN_BARS (the derived-layer ewm floor stays 21).
- No change to trend_structure.py, derived.py, regime, qualification, payload, or
  any renderer (PRD-208 owns the SMA presentation).
- No change to OHLCV_FETCH_MONTHS or the fetch window.
- No F15 remediation here.
- No new data source/provider.

RISKS
- Over-eager floor could refetch-storm genuinely-short symbols — the EDGE
  requirement + its FAIL line guard this.
- The `_cache_only` test mocks may assume the old freshness logic; the new floor
  must not break them (verify in FILES finalization).
- If the OPEN DIAGNOSTIC QUESTION resolves to "not a short-cache incident," the
  motivation narrows to latent-bug hardening (still valid) — recorded, not hidden.

- ingestion.py: `_is_fresh_ohlcv_cache` (add count check + short-by-nature

REQUIREMENTS

R1 — Short cache is not served as fresh; triggers a heal-refetch  [GENUINE RED TEST]
A cache whose last bar is the most-recent completed session but with
`len(df) < OHLCV_FULL_MIN_BARS` (and not short-by-nature) is NOT fresh:
`_is_fresh_ohlcv_cache` returns False, so fetch_ohlcv performs a live refetch.
(Fails on current code, which returns True for a short date-current cache.)
FAIL: a 30-bar, date-current cache is treated as fresh and returned by fetch_ohlcv
without a refetch.

R2 — Short fresh fetch is rejected, never cached, and fails loudly
`_fetch_ohlcv_from_yfinance`: a downloaded frame with `len < OHLCV_FULL_MIN_BARS`
(and not the provider ceiling) is not accepted; after retries exhaust it returns
None with a loud ERROR log, and `_write_ohlcv_cache` is NEVER called with a
sub-floor frame.
FAIL: a sub-floor downloaded frame is written to the OHLCV cache, OR a sub-floor
fetch returns/over-writes silently (no loud log).

R3 — Floor is parameterized at the largest consumer need, shared by both gates
The floor is config.OHLCV_FULL_MIN_BARS (= the sma_200 window, 200), referenced by
BOTH _is_fresh_ohlcv_cache and _fetch_ohlcv_from_yfinance.
FAIL: a literal bar-count int appears in ingestion.py, OR the two gates use
different floors.

R4 — Genuinely short-history symbol degrades honestly without refetch-looping  [edge]
A symbol whose provider max-available history is < floor renders
INSUFFICIENT_HISTORY (via the unchanged builder) and does NOT trigger a live
refetch on every consecutive same-day run (short-by-nature marker honored).
FAIL: a symbol with genuinely < floor available history triggers a live OHLCV
refetch on every same-day slot.

R5 — No regression for full-history symbols
A symbol with ≥ floor bars caches and serves exactly as before; sma_50/sma_200
resolve unchanged.
FAIL: a ≥ floor symbol's cache-hit/refetch behavior or sma resolution differs from
the pre-PRD baseline for identical input.

DATA FLOW
yfinance daily download → _fetch_ohlcv_from_yfinance (R2: reject/loud-raise
sub-floor, never cache short) → _write_ohlcv_cache (full frames only) →
_is_fresh_ohlcv_cache (R1: count + date, short-by-nature exception) → fetch_ohlcv →
derived / trend_structure / qualification (UNCHANGED consumers). The floor sits
entirely inside ingestion.py + the config constant.

FAIL CONDITIONS
- Short date-current cache served as fresh without refetch (R1).
- Sub-floor frame cached, or sub-floor fetch fails silently (R2).
- Magic-number floor, or divergent floors between gates (R3).
- Genuinely-short symbol refetch-storms every slot (R4).
- Full-history symbol behavior changes (R5).
