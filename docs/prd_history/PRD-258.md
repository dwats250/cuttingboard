WHY NOW
Too many permission prompts fire on boring, reversible commands (reads,
greps, tests, lint, git inspection, gh reads), diluting the prompts that
matter; the merge gate (Dustin's, always) already holds regardless of what
the allow-list permits, so widening it for reversible operations only moves
where friction lands, not what's ultimately controlled. `.claude/settings.json`
is a GOVERNANCE HIGH-RISK FILES entry in the INFRA row of the CLASS Matrix
(`docs/PRD_PROCESS.md`), so this rides full HIGH-RISK ceremony rather than
the MICRO lane a pure widening-of-ALLOW might otherwise seem to deserve — a
lane-relief carve-out for "safe-direction" permission changes was considered
and explicitly declined here, because legislating that relief from inside
the PR that would benefit from it is exactly the self-relief pattern
`docs/DECISIONS.md`'s PRD-255 entry forbids. If narrower ceremony for this
class of change is warranted, that is a separate ruling at a future gate,
not something this PRD grants itself.

GOAL
Widen `.claude/settings.json`'s `permissions.allow` to cover commands that
are reversible and boring (git inspection, `git commit`, `gh` reads,
test/lint invocations, read-only shell utilities, and a curated safe subset
of the repo's own `scripts/`/`tools/` invocations), while every irreversible
or destructive operation — force-push, checkout/reset/rebase, branch/worktree
deletion, `gh pr create`/`ready`/`merge`, mutating `gh api` calls, `rm`/`mv`/
`sed -i`, and any write to `.claude/hooks/`, `.claude/skills/`, or
`.claude/settings.json` itself — stays gated exactly as before.

R2 — Git inspection: add safe subcommands, hold `branch`/`fetch`/`ls-remote`
to exact-match forms where a wildcard reaches unsafe territory
Add to `allow`: `"Bash(git status *)"`, `"Bash(git diff *)"`,
`"Bash(git log *)"`, `"Bash(git show *)"`, `"Bash(git blame *)"`,
`"Bash(git merge-base *)"`, `"Bash(git cat-file *)"`,
`"Bash(git rev-parse *)"`, `"Bash(git worktree list *)"`. Each was checked
(`git <subcommand> --help`, grepped for `output=`/`upload-pack`/`--exec`)
and carries no arbitrary-file-write or command-exec flag.

`git branch` is the one inspection subcommand with a destructive flag
family (`-d`/`-D`/`--delete`, also `-m`/`-M` rename) reachable via the SAME
subcommand word, and git's option parser accepts these flags interspersed
with other arguments in more than one position — a single trailing-wildcard
rule (`"Bash(git branch *)"` + a deny carve-out for `-d`/`-D` prefixes, the
same shape used for `git push`/`git push --force`) cannot be proven to
catch every interspersed ordering the way the `git push --force` carve-out
can (force-push's flag is always the second token; branch's delete flag is
not guaranteed to be). Add only exact-match, argument-frozen forms instead:
`"Bash(git branch)"`, `"Bash(git branch -a)"`, `"Bash(git branch -r)"`,
`"Bash(git branch -v)"`, `"Bash(git branch --show-current)"`. Each is a
complete, closed string with nothing after it can match — there is no
wildcard for a destructive flag to hide inside. A `git branch` invocation
not matching one of these five exact strings (e.g. `--list <pattern>`, or
any `-d`/`-D`/`-m`/`-M`/`--delete` form) falls through to the default
(prompt) — intended, not a gap.

`git fetch` and `git ls-remote` are likewise held to exact-match forms,
NOT a trailing wildcard, after the commissioned second-model review
(`docs/prd_history/PRD-258.review.codex.md`) found `--upload-pack=<exec>`
— a real, documented flag on both subcommands (verified via
`git fetch --help`/`git ls-remote --help`) that git executes as a shell
command when fetching from a local path, turning an inspection-looking
command into arbitrary code execution — and, on `git fetch` specifically,
that an explicit destination refspec combined with `--force`/
`--update-head-ok` can overwrite a local branch, which a trailing wildcard
cannot be bounded against for the same interspersion reason as `git
branch`. Add only: `"Bash(git fetch)"`, `"Bash(git fetch origin)"`,
`"Bash(git fetch --all)"`, `"Bash(git fetch --prune)"`,
`"Bash(git ls-remote)"`, `"Bash(git ls-remote origin)"`. A blanket
`"Bash(git * --upload-pack*)"` deny is added as a second, independent
layer (see R4) in case a future PRD re-widens either rule to a wildcard
without re-deriving this reasoning.

`git diff`/`git log`/`git show` keep their trailing wildcard (inspection
needs free-form revision ranges/paths/flags that an exact-match list
cannot practically enumerate), but all three inherit `--output=<file>`
from git's shared diff machinery (verified via `--help`; no short-flag
synonym exists) — an inspection command silently gains a write-to-any-path
capability via a documented flag. Closed with a blanket
`"Bash(git * --output*)"` deny (R4) rather than an exact-match rewrite of
three high-traffic, argument-varying commands. `git blame`/`cat-file`/
`rev-parse`/`merge-base`/`commit`/`status` were each checked and carry
neither `--output=` nor `--upload-pack`/`--exec`.

FAIL: `git checkout`, `git reset`, `git rebase`, `git branch -d`/`-D`/
`--delete`/`-m`/`-M`, or `git worktree remove` is reachable without a
prompt (via a new allow entry OR because a new entry's wildcard is broad
enough to also match one of these); OR any of the nine listed safe
subcommands still prompts; OR `git fetch`/`git ls-remote` is allowed via a
trailing-wildcard form rather than the six exact-match strings above.

R4 — `gh` reads: PR list/diff, and a defended `gh api` read path
Add to `allow`: `"Bash(gh pr list *)"`, `"Bash(gh pr diff *)"`,
`"Bash(gh api repos/dwats250/cuttingboard/pulls/*)"`,
`"Bash(gh api repos/dwats250/cuttingboard/issues/*)"` (the second rule's
trailing wildcard already covers `.../issues/*/reactions` — no separate
reactions-specific entry is needed or added). The owner/repo segment is
hardcoded to this repo rather than wildcarded (`repos/*/...`): the
authenticated `gh` token's access is not scoped to this repo alone, and an
open owner/repo wildcard would let a `gh api` read reach any other repo
the token can see — no destructive risk, but wider than this PRD's actual
purpose (bot-channel checks on this repo's PRs), so it is narrowed rather
than left open on a "read-only, so it doesn't matter" assumption.

Both `gh api` rules use a trailing wildcard, which — per Claude Code's
documented matching (a wildcard matches any sequence of characters
including spaces, and can span multiple arguments) — would ALSO match a
mutating call whose flags happen to land after the endpoint path in the
same command string (e.g. `gh api repos/x/y/issues/146/reactions -f
content=+1`), not just the safe bare-GET form. Add a `deny` backstop that
catches the mutating flag regardless of where in the command it appears
(before or after the endpoint, since deny is evaluated before allow and
always wins regardless of rule specificity): `"Bash(gh api * -X*)"`,
`"Bash(gh api * --method*)"`, `"Bash(gh api * -f*)"`,
`"Bash(gh api * -F*)"`, `"Bash(gh api * --raw-field*)"`,
`"Bash(gh api * --field*)"`, `"Bash(gh api * --input*)"`. This is the same
broad-allow-plus-narrow-deny shape already used for `git push`/`git push
--force`, applied here because (unlike `git branch`, R2) the dangerous
token (`-X`, `-f`, etc.) is a flag with its own leading `-`/`--` marker
that cannot be confused with a positional path segment, so a
wildcard-bounded deny reliably catches it in any position.

Two corrections from the commissioned second-model review
(`docs/prd_history/PRD-258.review.codex.md`), both fixed here rather than
carried as a known gap: (1) the original five deny patterns each required
a literal trailing space (e.g. `"gh api * -X *)"`, matching `-X POST` but
NOT `-XPOST` or `--method=POST` — `gh`'s flag parser accepts both spaced
and attached/equals forms (verified via `gh api --help`), so the space
requirement was a real bypass; every deny pattern now attaches the
wildcard directly (`"* -X*)"`, no space) to catch both forms. (2) `gh api`
has TWO distinct, case-sensitive field flags — `-f`/`--raw-field` (string
parameter) and `-F`/`--field` (typed parameter, confirmed by `gh api
--help`) — and the original denies covered only the lowercase form;
`-F`/`--field` is now denied too.

FAIL: `gh pr list`/`gh pr diff` or a bare-GET `gh api` call on either path
family still prompts; OR any `gh api` call carrying `-X`/`--method`/`-f`/
`-F`/`--raw-field`/`--field`/`--input` anywhere in the command, in either
spaced or attached/equals form, does NOT prompt (deny backstop failed).

BOT REVIEW ROUND 3 — the bot re-scanned after each fixing commit (5
inline threads became 27) and found a pattern-construction bug affecting
this PRD's own deny rules, plus several genuinely new gaps

**1. A `deny`-pattern construction bug, found once and then verified
against every rule in the file.** Several rules added earlier this PRD
(`*ruff check * --fix*`, `rg * --pre*`, `git branch -d*`, `gh api * -X*`,
etc.) hard-coded a literal space between the wildcard and the dangerous
flag/subcommand. That space assumes another token always sits between the
fixed prefix and the flag — true when there's a variable subcommand in
between (`git * --output*` matches `git diff --output=x` fine, since
`diff` supplies the space), false when the flag can immediately follow a
multi-word fixed literal with nothing between (`ruff check --fix .`, `gh
api -X POST ...`, `git branch foo -d`). Removed the hard-coded space from
every affected pattern (`*ruff check*--fix*`, `rg *--pre*`, `git *branch
*-d*`, `gh api *-X*`, etc. — full list in `.claude/settings.json`),
letting the wildcard itself absorb an optional preceding token. Verified
with a 66-case simulation (every fixed pattern against both its direct
form and its "flag-first" form, plus ~30 legitimate-usage false-positive
checks) before committing — see VALIDATION.

**2. A middle-wildcard fix for `checkout`/`reset`/`rebase`, closing the
global-options-before-subcommand bypass.** Git accepts global options
before the subcommand (`git -C <path> reset --hard`, `git -c x=y
checkout other`), so the original `"Bash(git checkout*)"`-style denies
(anchored to the start of the string) never matched when anything
preceded the subcommand name. Fixed to `"Bash(git *checkout*)"` /
`*reset*` / `*rebase*` (middle wildcard, no anchor), which catches the
evasion form without breaking the direct form — the first attempt at
this fix accidentally left a **mandatory** space in the pattern
(`"Bash(git * checkout*)"`), which broke the direct case entirely
(`git checkout foo` no longer matched); caught by the same 66-case
simulation before it shipped, not left as a regression.

**3. New setup-primitive denies: `git config`, `git remote add`/
`set-url`.** Several bot findings (git blame/diff/log/show running a
configured `textconv`/`ext-diff` helper with no flag at all, `git status`
running a configured `core.fsmonitor` hook, `git log`/`show
--show-signature` running a configured `gpg.program`, `git fetch`
following a remote configured with a custom transport helper like
`foo::bar`) all depend on `.git/config` (or `.gitattributes`, for
`textconv`/`ext-diff`) already naming a helper program — none of them
need a flag on the allowed command itself. Two of these configs
(`git config`, `git remote add`/`set-url`) were themselves reachable
within this session's own granted surface: `.gitattributes` is writable
via the already-allowed `Write`/`Edit` tools (not blocked by
`protect_files.sh`, which only covers `.env`/`.git/*`/`*.lock`/workflow
files), and local settings already grants bare `"Bash(git remote:*)"`.
Denying `git config`/`git remote add`/`git remote set-url` closes the
two-step *local* escalation path (write the helper's config, then run
the now-armed "safe" command) while leaving the residual — a repo that
arrives with a malicious `.gitattributes`/`.git/config`/`gpg.program`
already configured — as a supply-chain/clone-integrity question, the
same class already dismissed for the `git commit` hooks finding and
recorded in `docs/DECISIONS.md`'s accident-prevention-layer entry.

**4. Direct flag additions:** `ruff check --add-noqa`/`-o`/
`--output-file` (rewrites files, same class as `--fix`), `gh api
--hostname` (redirects the read off `github.com`, bypassing the
hard-coded repo scope), `gh pr list`/`diff --web`/`-w` (spawns a
configured browser command), `gh pr diff <URL>` (reads a PR from
whatever repo the URL names, bypassing the `-R`/`--repo` deny via a
different argument shape entirely — a positional URL, not a flag), `git
branch -f`/`--force` (force-moves a branch ref, the same family as
`-d`/`-D` but not named in the original sweep).

**5. Dismissed as no NEW risk — read-outside-repo findings.** `git diff`
`--no-index`-style reads of arbitrary filesystem paths (e.g. `git diff
/etc/hostname /dev/null`), `git blame --contents <file>` (reads an
arbitrary file as blame's base image), and `tools/validate_prd_registry.py
<other-repo-path>` (reads another repo's registry files) are all real,
but none of them widen this repo's actual exposure: `cat`, `grep`,
`head`, `tail` are already unconditionally allowed with **no path
scoping at all**, predating this PRD entirely — an agent can already
`cat /etc/hostname` or `grep -r secret /home/dustin` today. These three
findings are additional *paths* to a read capability that was never
closed and was never in this PRD's scope to close (widening `git`/`gh`,
not retroactively sandboxing the pre-existing coreutils allows).

**6. Dismissed with a direct factual correction — the P1 shell-chaining
claim.** One finding (P1, the highest severity of this round) claimed
`git status ; git reset --hard` bypasses the new `reset` deny because
the compound string starts with the allowed `git status` prefix.
Independently verified against Claude Code's own documented behavior
(not taken on the bot's word, nor asserted from memory without
re-checking): Claude Code parses compound commands and evaluates each
subcommand independently on `&&`, `||`, `;`, `|`, `|&`, `&`, and
newlines — "a rule must match each subcommand independently." `git
reset --hard` as its own subcommand matches only the `deny` rule, never
an `allow`, so it is blocked regardless of what `git status` resolves to.
This is factually distinct from the `bash -c`-style bypass
`docs/DECISIONS.md` already records as real and unfixable at this layer
(that one hands the dangerous string to a *different top-level command*,
never matching the deny pattern at all; chaining via `;` keeps `git
reset --hard` as its own literal subcommand, which the deny pattern does
match).

**7. Dismissed — `git commit -e`/`--edit` (editor-launching).** Real
(`git commit --allow-empty -e -m msg` can run a configured `core.editor`
script), but a short flag like `-e` cannot be denied by substring match
without false-positiving on ordinary commit messages containing that
two-character sequence inside a word (e.g. "fix **pre-e**xisting bug"
contains the literal substring `-e`... no wait, verified: **word**
substrings like "pre-existing" do contain a literal `-e`, confirmed by
direct string inspection, not assumed). Not fixed: this repo's actual
commit usage always passes `-m` explicitly and has no interactive
terminal to hand an editor to in this session context; the practical
exposure is low and a broad-substring deny risks blocking legitimate
commit messages instead.
