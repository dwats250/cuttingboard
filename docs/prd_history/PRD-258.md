PRD-258 — Widen the Claude Code Bash permission allow-list for reversible operations

Status: IN PROGRESS

LANE
HIGH-RISK

CLASS
INFRA

WHY NOW
Too many permission prompts fire on boring, reversible commands (reads,
greps, tests, lint, git inspection, gh reads), diluting the prompts that
matter; the merge gate (Dustin's, always) already holds regardless of what
the allow-list permits, so widening it for reversible operations only moves
where friction lands, not what's ultimately controlled. `.claude/settings.json`
is a GOVERNANCE HIGH-RISK FILES entry in the INFRA row of the CLASS Matrix
(`docs/PRD_PROCESS.md`), so this rides full HIGH-RISK ceremony rather than
the MICRO lane a pure widening-of-ALLOW might otherwise seem to deserve — a
lane-relief carve-out for "safe-direction" permission changes was considered
and explicitly declined here, because legislating that relief from inside
the PR that would benefit from it is exactly the self-relief pattern
`docs/DECISIONS.md`'s PRD-255 entry forbids. If narrower ceremony for this
class of change is warranted, that is a separate ruling at a future gate,
not something this PRD grants itself.

MAX EXPECTED DELTA
1 production file (`.claude/settings.json`, permissions block only — no
hooks changes), plus this PRD doc, `docs/PRD_REGISTRY.md`, and
`docs/PROJECT_STATE.md`. No `cuttingboard/*` file touched.

GOAL
Widen `.claude/settings.json`'s `permissions.allow` to cover commands that
are reversible and boring (git inspection, `git commit`, `gh` reads,
test/lint invocations, read-only shell utilities, and a curated safe subset
of the repo's own `scripts/`/`tools/` invocations), while every irreversible
or destructive operation — force-push, checkout/reset/rebase, branch/worktree
deletion, `gh pr create`/`ready`/`merge`, mutating `gh api` calls, `rm`/`mv`/
`sed -i`, and any write to `.claude/hooks/`, `.claude/skills/`, or
`.claude/settings.json` itself — stays gated exactly as before.

SCOPE
- Add git inspection commands to `allow`: `status`, `diff`, `log`, `show`,
  `blame`, `fetch`, `merge-base`, `cat-file`, `ls-remote`, `rev-parse`,
  `worktree list`, and a narrowly-enumerated (non-wildcarded) safe subset of
  `branch` forms.
- Add `git commit` to `allow`, unconditionally (ruled: commit is not the
  gate; the merge is).
- Add `gh pr list`, `gh pr diff`, and `gh api` reads on this repo's
  `repos/dwats250/cuttingboard/pulls/*` and
  `repos/dwats250/cuttingboard/issues/*` to `allow`; add a `gh api`
  mutating-flag deny backstop (see R4) so the read path can't be walked
  into a write.
- Add `python -m ruff`/`python3 -m ruff` to `allow` (pytest via `-m` and bare
  `ruff check` are already present); widen `validate_prd_registry.py`'s
  existing exact-match entries to cover its one real flag
  (`--skip-commit-resolvability`).
- Add a curated, individually-justified subset of `scripts/`/`tools/`
  invocations to `allow` — NOT a blanket `scripts/*`/`tools/*` rule (see R6
  for the read-vs-destructive audit of every file in both directories).
- Add `wc` and `diff` to the existing read-only-shell-utility allow set
  (`rg`/`grep`/`cat`/`head`/`tail`/`ls`/`find`/`jq` already present).

OUT OF SCOPE
- Blanket `Bash(*)` allow. `protect_files.sh` (`PreToolUse` on `Write`/`Edit`
  only) is the mechanism that blocks writes to protected paths, and it was
  deliberately never extended to intercept `Bash` (PRD-254, decided, not
  deferred — `docs/CLAUDE_HOOKS.md` lines 43-53: "the hook catches accidents,
  not intent... extending the matcher to Bash would add a knob chasing a
  threat model this project doesn't have"). A blanket `Bash(*)` allow would
  let `sed -i` (or any other file-mutating shell command) reach
  `.claude/settings.json`, `.claude/hooks/*`, or any other protected path
  with zero guard — protect_files.sh only ever covered the Write/Edit tools,
  never Bash, so a blanket Bash allow removes the one layer that currently
  makes an accidental protected-path Bash-write require a prompt at all.
- Refactoring the LANE rule so safe-direction permission changes (widening
  DENY, narrowing ALLOW) ride MICRO. Considered, explicitly declined (see
  WHY NOW) — not ruled out forever, just not self-granted here.
- Any change to `gh pr merge`'s deny status (PRD-254 R1; stays denied) or to
  `gh pr create`/`gh pr ready` (never allowed; stay gated by omission, no
  deny entry needed since default is ask).
- `git checkout`, `git reset`, `git rebase`, `git branch -d`/`-D`/`--delete`,
  `git worktree remove`, `git push --force`/`--force-with-lease`/`-f` (already
  denied) — none added to `allow`; no change to their gated status.
- Extending `protect_files.sh`'s matcher to `Bash` — out of scope for the
  reason stated above (this is a permissions-file change, not a hook-logic
  change).
- A blanket `scripts/*`/`tools/*` allow. Both directories were read in full
  before drafting (R6); several files are individually destructive,
  environment-specific (CI-only), or make live network calls, and are

- `tools/engine_doctor.py`, `tools/macro_awareness_collector.py`,

  left ungated by default (no allow entry added) rather than assumed safe.

R2 — Git inspection: add safe subcommands, hold `branch` to exact-match only
Add to `allow`: `"Bash(git status *)"`, `"Bash(git diff *)"`,
`"Bash(git log *)"`, `"Bash(git show *)"`, `"Bash(git blame *)"`,
`"Bash(git fetch *)"`, `"Bash(git merge-base *)"`, `"Bash(git cat-file *)"`,
`"Bash(git ls-remote *)"`, `"Bash(git rev-parse *)"`,
`"Bash(git worktree list *)"`. None of these subcommands has a
destructive-flag form (fetch's `--prune` only removes stale
remote-tracking refs, never a local branch).

`git branch` is the one inspection subcommand with a destructive flag
family (`-d`/`-D`/`--delete`, also `-m`/`-M` rename) reachable via the SAME
subcommand word, and git's option parser accepts these flags interspersed
with other arguments in more than one position — a single trailing-wildcard
rule (`"Bash(git branch *)"` + a deny carve-out for `-d`/`-D` prefixes, the
same shape used for `git push`/`git push --force`) cannot be proven to
catch every interspersed ordering the way the `git push --force` carve-out
can (force-push's flag is always the second token; branch's delete flag is
not guaranteed to be). Add only exact-match, argument-frozen forms instead:
`"Bash(git branch)"`, `"Bash(git branch -a)"`, `"Bash(git branch -r)"`,
`"Bash(git branch -v)"`, `"Bash(git branch --show-current)"`. Each is a
complete, closed string with nothing after it can match — there is no
wildcard for a destructive flag to hide inside. A `git branch` invocation
not matching one of these five exact strings (e.g. `--list <pattern>`, or
any `-d`/`-D`/`-m`/`-M`/`--delete` form) falls through to the default
(prompt) — intended, not a gap.

FAIL: `git checkout`, `git reset`, `git rebase`, `git branch -d`/`-D`/
`--delete`/`-m`/`-M`, or `git worktree remove` is reachable without a
prompt (via a new allow entry OR because a new entry's wildcard is broad
enough to also match one of these); OR any of the eleven listed safe
subcommands still prompts.

R3 — `git commit`: unconditional allow
Add `"Bash(git commit *)"` to `allow`. Ruled (not this PRD's own
invention): commit is fully reversible on a branch nothing merges from
without Dustin's manual merge; the merge is the gate, not the commit.
`--amend` on an already-pushed commit only matters in combination with a
force-push, which stays denied independently — no interaction risk.

FAIL: `git commit` still prompts for a plain commit invocation.

R4 — `gh` reads: PR list/diff, and a defended `gh api` read path
Add to `allow`: `"Bash(gh pr list *)"`, `"Bash(gh pr diff *)"`,
`"Bash(gh api repos/dwats250/cuttingboard/pulls/*)"`,
`"Bash(gh api repos/dwats250/cuttingboard/issues/*)"` (the second rule's
trailing wildcard already covers `.../issues/*/reactions` — no separate
reactions-specific entry is needed or added). The owner/repo segment is
hardcoded to this repo rather than wildcarded (`repos/*/...`): the
authenticated `gh` token's access is not scoped to this repo alone, and an
open owner/repo wildcard would let a `gh api` read reach any other repo
the token can see — no destructive risk, but wider than this PRD's actual
purpose (bot-channel checks on this repo's PRs), so it is narrowed rather
than left open on a "read-only, so it doesn't matter" assumption.

Both `gh api` rules use a trailing wildcard, which — per Claude Code's
documented matching (a wildcard matches any sequence of characters
including spaces, and can span multiple arguments) — would ALSO match a
mutating call whose flags happen to land after the endpoint path in the
same command string (e.g. `gh api repos/x/y/issues/146/reactions -f
content=+1`), not just the safe bare-GET form. Add a `deny` backstop that
catches the mutating flag regardless of where in the command it appears
(before or after the endpoint, since deny is evaluated before allow and
always wins regardless of rule specificity): `"Bash(gh api * -X *)"`,
`"Bash(gh api * --method *)"`, `"Bash(gh api * -f *)"`,
`"Bash(gh api * --raw-field *)"`, `"Bash(gh api * --input *)"`. This is the
same broad-allow-plus-narrow-deny shape already used for `git push`/
`git push --force`, applied here because (unlike `git branch`, R2) the
dangerous token (`-X`, `-f`, etc.) is a flag with its own leading `-`/`--`
marker that cannot be confused with a positional path segment, so a
wildcard-bounded deny reliably catches it in any position.

FAIL: `gh pr list`/`gh pr diff` or a bare-GET `gh api` call on either path
family still prompts; OR any `gh api` call carrying `-X`/`--method`/`-f`/
`--raw-field`/`--input` anywhere in the command does NOT prompt (deny
backstop failed).

R5 — Test/lint: `python -m ruff`, widen `validate_prd_registry.py`
Add `"Bash(python -m ruff *)"` and `"Bash(python3 -m ruff *)"` (pre_push_check.sh
invokes ruff this way, not via the bare `ruff` binary already in `allow`).
Replace the two existing exact-match `validate_prd_registry.py` entries
with wildcarded forms: `"Bash(python3 tools/validate_prd_registry.py *)"`,
`"Bash(python tools/validate_prd_registry.py *)"` — verified against the
script's own `argparse` block (`tools/validate_prd_registry.py:404-420`):
an optional positional `root` path and one boolean flag
(`--skip-commit-resolvability`), no destructive behavior either way.

Do not add a bare `"Bash(python -m *)"`/`"Bash(python3 -m *)"` rule: `python
-m` is a module-execution mechanism, not a tool name, and would also cover
`python -m pip install`, `python -m venv`, `python -m http.server`, and
other module invocations with no relationship to test/lint — CLAUDE.md's
own approvals rule requires explicit approval for dependency changes, which
a blanket `python -m *` would silently bypass.

FAIL: `python -m ruff`/`python3 -m ruff` still prompts; OR
`validate_prd_registry.py --skip-commit-resolvability` still prompts; OR a
bare `"Bash(python -m *)"`/`"Bash(python3 -m *)"` entry exists in `allow`.

- `tools/ci_push_artifacts.sh`, `tools/ci_restore_publish_state.sh` — CI-only

- `tools/prune_run_history.sh` — `git rm -f`s historical `run_*.json` files

- `tools/engine_doctor.py`, `tools/macro_awareness_collector.py`,
