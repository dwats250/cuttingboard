- DEDUP: PRD-141 already keys fired-today dedup on the canonical PT-hour slot

  called only at cuttingboard/runtime/__init__.py:956 (the premarket _run_pipeline
  path). The hourly path writes only a NOTIFICATION audit record
  (write_notification_audit). So the realizable marker is a notify_mode tag on the
  notification audit record (observability), not a pipeline-record dedup field.

Cut from the prior draft: re-homing slots to cuttingboard.yml; the

GOAL
Every notification audit record emitted on the hourly/intraday and premarket
notify paths carries the notify_mode that produced it, so audit rows are
self-describing (which slot/mode fired) instead of a hardcoded alert_title
"hourly". INTERFACE_LOCK.md no longer claims dedicated cuttingboard.yml live crons
for the four intraday slots.

SCOPE
- cuttingboard/audit.py (write_notification_audit, :249): add an optional
  keyword-only `notify_mode: Optional[str] = None` parameter and include it as a
  `notify_mode` field in the emitted record dict. Default None -> existing callers
  unchanged (the field is present-but-null for untagged records).
- cuttingboard/output.py: thread the optional `notify_mode` through
  send_notification (:688) and send_telegram (:529) to the write_notification_audit
  calls inside send_telegram (:559, :636, :663). Default None; no behavior change
  for any existing caller.
- cuttingboard/runtime/__init__.py: pass notify_mode on the hourly/intraday notify
  path -- the success send (send_notification at :483, and the failure send at
  :569) and the premarket-suppression notification audit (:940) -- using the
  notify_mode already in scope (_execute_notify_run param :358; _run_pipeline param).
- cuttingboard/alert_runner.py: pass `notify_mode=NOTIFY_HOURLY` at the two
  suppressed-path write_notification_audit calls (:67 outside-window, :83
  same-slot). NOTIFY_HOURLY is already imported (:48).
- docs/prd_history/INTERFACE_LOCK.md (:76-79): rewrite the orb_trajectory /
  post_orb / midmorning / power_hour rows to record that intraday coverage is
  FOLDED INTO hourly_alert.yml's routine window (not dedicated cuttingboard.yml
  `--mode <slot>` live crons, which PRD-189 dropped to noop-on-schedule).
- tests/test_notification_audit.py: assert the notify_mode field is present and
  carries the threaded value on a tagged write, and is None on an untagged
  (default) write. Red against today's records (no such field) first.

OUT OF SCOPE
- No re-homing of the four slots to cuttingboard.yml; no new crons; the dropped-
  slots resolver guard (tests/test_resolve_run_mode.py) stays green unchanged.
- No new dedup mechanism (PRD-141 already dedups on the canonical slot).
- No change to write_audit_record (the pipeline record); no new pipeline-record
  field. The hourly path writes no pipeline record, so nothing to add there.
- No change to notification routing, retry, priority, or send behavior -- the new
  parameter is metadata-only and optional.

RETAINED UNTAGGED (dead-branch enumeration)
- The runner-level backstop send (cuttingboard/alert_runner.py, "HALT - SYSTEM
  ERROR", reason="runner_level_exception") is left UNTAGGED (notify_mode=None) by
  design. It is the catastrophic last-resort path, semantically distinct from a
  normal hourly notify run, and NOTIFY_HOURLY is a lazy import inside the runner's
  try block -- referencing it in the except path would risk a NameError if the
  failure was that import itself. The audit reason "runner_level_exception"
  already identifies the record; tagging it would add risk to the last-resort
  path for no observability gain.
