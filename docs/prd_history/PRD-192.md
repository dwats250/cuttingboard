PROBLEM
PRD-189 scoped the cuttingboard.yml run-mode resolver to the dedicated premarket
crons (prefetch / live / sunday) and removed the orb_trajectory / post_orb /
midmorning / power_hour slots, because Codex review surfaced two facts:

- Their real per-slot behavior lives in runtime `_execute_notify_run` (the
  `alert_runner` / `hourly_alert.yml` path), NOT `cli_main` / `_run_pipeline`.
  Running them via `python -m cuttingboard --mode live --notify-mode <slot>`
  executes a full live pipeline + a generic live notification at intraday times,
  not the intended slot scan; it also overlaps the intraday coverage
  `hourly_alert.yml` already provides.
- Robust per-slot fired-today dedup needs a slot/mode marker in the audit record.
  Today's pipeline records carry no slot identifier (`router_mode` is the sector
  regime), so cuttingboard-side dedup could only be heuristic wall-clock
  attribution of `run_at_utc`.

SCOPE (to be detailed at Stage 0)
- Decide the correct home + entrypoint for the four intraday/orb slots: route
  through `alert_runner` / `_execute_notify_run`, or fold into `hourly_alert.yml`
  (which already covers the window), and wire the workflow(s) accordingly.
- Add a per-slot/mode marker to the audit record (`runtime/audit.py`) so any
  fired-today dedup keys on the actual slot, not wall-clock attribution.
- If the intraday slots return to `cuttingboard.yml`, re-introduce
  queue-delay-tolerant resolution for the shared `*/30` cron (the late-only,
  start-sampled, attribution-dedup design explored in the PRD-189 review thread).
- Reconcile `docs/prd_history/INTERFACE_LOCK.md` — its orb_trajectory / post_orb /
  midmorning / power_hour rows currently list them as live scheduled invocations,
  stale since PRD-189 dropped those crons to noop-on-schedule. Update the rows to
  match whatever intraday routing this PRD lands.

NON-GOALS
- No change to the premarket cron-string resolver landed in PRD-189.
