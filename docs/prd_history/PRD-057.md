# PRD-057: Lifecycle Visibility on Signal Forge Dashboard

## GOAL

Surface PRD-056 lifecycle metadata on Signal Forge candidate cards. Candidates display a lifecycle badge derived from `lifecycle["grade_transition"]` (suppressed for UNCHANGED). High-grade cards (A+, A, B) render an additional transition detail row. Removed symbols render in a compact section after tier groups.

This PRD modifies the dashboard renderer only. No engine, notification, or lifecycle computation changes.

---

## SCOPE

- Add lifecycle badge to each candidate card (all grades; suppressed when transition is UNCHANGED)
- Add lifecycle transition detail row to A+, A, B cards only
- Add removed_symbols section inside candidate-board after tier groups

- Update `tests/test_dashboard_renderer.py` for new rendering behavior

## REQUIREMENTS

### R1 — Lifecycle badge

For each candidate card, if `symbol_entry["lifecycle"]` exists and `lifecycle["grade_transition"]` is not `"UNCHANGED"`, render one badge:

```html
<span class="lifecycle-badge lifecycle-{lowercase}">UPGRADED</span>
```

Allowed transitions and class mapping:
- `NEW` → `lifecycle-new`
- `UPGRADED` → `lifecycle-upgraded`
- `DOWNGRADED` → `lifecycle-downgraded`
- `UNKNOWN` → `lifecycle-unknown`

Suppression rule: do not render a badge when `grade_transition == "UNCHANGED"`.

Badge renders in the card header section after the GRADE value, before BIAS.

FAIL: A card with `lifecycle.grade_transition = "UPGRADED"` does not contain `class="lifecycle-badge lifecycle-upgraded"`.
FAIL: A card with `lifecycle.grade_transition = "DOWNGRADED"` does not contain `class="lifecycle-badge lifecycle-downgraded"`.
FAIL: A card with `lifecycle.grade_transition = "UNCHANGED"` contains any `lifecycle-badge` element.
FAIL: A card without a `lifecycle` key contains any `lifecycle-badge` element.

### R2 — Lifecycle CSS classes

`_CSS` must include definitions for these lifecycle classes:
- `.lifecycle-badge` — base badge style (inline-block, padding, border-radius, font-size)
- `.lifecycle-new` — distinct (cyan/teal family)
- `.lifecycle-upgraded` — positive (green family)
- `.lifecycle-downgraded` — negative (red family)
- `.lifecycle-unknown` — muted

FAIL: `grep "lifecycle-upgraded" cuttingboard/delivery/dashboard_renderer.py` returns no match.
FAIL: `grep "lifecycle-downgraded" cuttingboard/delivery/dashboard_renderer.py` returns no match.
FAIL: `grep "lifecycle-badge" cuttingboard/delivery/dashboard_renderer.py` returns no match.

### R3 — Card render order

Candidate card render sequence:

```
SYMBOL
GRADE (+ lifecycle badge if rendered)
BIAS
STRUCTURE
--- for A+, A, B cards only ---
lifecycle-detail row (if lifecycle exists)
setup_state row (existing behavior, unchanged)
trade_framing fields (existing behavior, unchanged)
invalidation (existing behavior, unchanged)
risk (existing behavior, unchanged)
reason_for_grade (existing behavior, unchanged)
```

lifecycle-detail format:
```html
<div class="lifecycle-detail">LIFECYCLE: {previous_grade} → {current_grade} | {previous_setup_state} → {current_setup_state}</div>
```

- Null previous values render as `—` (em dash)
- Row renders for A+, A, B cards only, when lifecycle exists
- Row does not render for C, D, F cards
- Row does not render when lifecycle is absent

FAIL: A B-grade card with lifecycle does not contain `class="lifecycle-detail"`.
FAIL: An F-grade card with lifecycle contains `class="lifecycle-detail"`.
FAIL: A card with no lifecycle key contains `class="lifecycle-detail"`.
FAIL: lifecycle-detail appears after setup_state in the HTML output.

### R4 — Removed symbols section

If `market_map["removed_symbols"]` exists and is non-empty, render inside `candidate-board` after all tier groups:

```html
<div class="removed-symbols">
  <div class="tier-header">REMOVED</div>
  <div class="removed-row">{symbol} — removed (prev: {previous_grade})</div>
  ...
</div>
```

Rules:
- Section absent when `removed_symbols` is empty or key is absent
- Removed entries do not appear in any tier group
- All values HTML-escaped via `_esc()`

FAIL: `removed_symbols` with one entry does not produce `class="removed-symbols"` in HTML.
FAIL: `removed_symbols = []` produces `class="removed-symbols"` in HTML.
FAIL: A symbol in `removed_symbols` also appears in a tier group.

### R5 — No badge when lifecycle absent

When a symbol entry has no `lifecycle` key, no lifecycle badge or detail row renders for that card.

FAIL: A symbol without a `lifecycle` key produces any output matching `lifecycle-badge` or `lifecycle-detail`.

### R6 — Existing block IDs preserved

Always present: `dashboard-header`, `macro-tape`, `system-state`, `candidate-board`, `run-health`.
Conditional: `run-delta` present when `previous_run` is provided; `run-history` present when `history_runs` is non-empty.

FAIL: Any always-present block ID is absent from HTML output.
FAIL: `run-delta` absent when `previous_run` is provided.
FAIL: `run-history` absent when `history_runs` is non-empty.

### R7 — No regression

All previously passing tests pass without loosening any existing assertion.

FAIL: Any previously passing test fails after this change.

## DATA FLOW

1. `_render_candidate_card(w, sym, entry)` reads `entry.get("lifecycle")`.
2. If lifecycle exists and `grade_transition != "UNCHANGED"`: emit lifecycle badge after GRADE value.
3. If grade in `_HIGH_GRADES` and lifecycle exists: emit lifecycle-detail row before setup_state.
4. After all tier groups in `candidate-board` (only when `market_map is not None`):
   - Read `removed_symbols = market_map.get("removed_symbols") or []`
   - If non-empty, emit `removed-symbols` section.

No file I/O, no engine calls, no lifecycle computation.
