WHY NOW
This session proved the hazard concretely: local `codex exec` failed from an
ephemeral container (`x-deny-reason: host_not_allowed` to api.openai.com; no
mounted `~/.codex` / `OPENAI_API_KEY`), while the PRD-197 GitHub Actions path
succeeded — and the only thing that picked the right route was a human
remembering the asymmetry. The CODEX PATH DIAGNOSIS (this session) confirmed every
prior local `codex exec` run was on Dustin's host (`~/.codex/config.toml` →
`[projects."/home/dustin/cuttingboard"]`), never an in-container run. The route
should be guaranteed by a tool, not by memory.

MAX EXPECTED DELTA
≤300 production LOC in `scripts/codex_review.sh`; ≤8 files in FILES. Exceeding
either stops implementation and re-triggers review.

GOAL
A single `codex-review <target>` command that runs a read-only Codex review from
ANY environment by auto-selecting the channel (host CLI vs PRD-197 GitHub
Actions), so "run a Codex review" means one thing whether on Dustin's host or in
an ephemeral container.

SCOPE
- New `scripts/codex_review.sh` entrypoint (mirrors `scripts/prd_open.sh` /
  `scripts/prd_close.sh` as a bash tool) implementing capability-based routing:
  - Fast local gate (filesystem/env, instant): `codex` on PATH AND
    (`~/.codex/auth.json` exists OR `OPENAI_API_KEY` is set).
  - Gate pass → attempt local `codex exec -s read-only`; on connectivity/egress
    failure → fall back to the CI route (asher-style probe-then-fall-back).
  - Gate fail → CI route directly.
  - CI route: dispatch `.github/workflows/codex-review.yml` against the target,
    poll for the resulting `codex-review/PRD-<n>-<sha>` artifact branch, and
    surface the verdict + provenance (resolved-model, allowlist-check) the way
    PRD-197 already records it.
- `tests/test_codex_review_router.py` — hermetic route-selection tests (fake
  `codex` on PATH, env toggles; no network, no real dispatch), mirroring the
  `tests/test_prd_open.py` subprocess-harness pattern. Named to avoid colliding
  with the existing PRD-197 `tests/test_codex_review.py` (the workflow contract).
- `CLAUDE.md`: point the Codex-invocation workflow pattern (currently
  ``codex exec -s read-only - < prompt``, ~:210-211) at `codex-review <target>`
  as the canonical review invocation.
- `docs/DECISIONS.md`: record that the host/CI route is now tool-guaranteed,
  retiring the host-dependence hazard this session surfaced (the hazard is NOT
  yet a DECISIONS entry — this PRD adds it; it does not cite an existing line).

OUT OF SCOPE
- Widening the PRD-197 workflow's dispatch inputs. The CI route uses
  `codex-review.yml` exactly as-is.
- Closing the host/CI review-surface asymmetry (CI can only review a PRD target).
  That is the separate "fire reviews without scaffolding a PRD" work (Path B).
  Path-B note: CI Codex (PRD-197) runs a FIXED generic consistency pass — it
  cannot carry a custom review focus. Path-B flexibility = arbitrary target AND
  arbitrary prompt, not target alone. Demonstrated in the PRD-205 review

- Provisioning OpenAI credentials or network egress into containers. The container
  route is the GHA dispatch precisely so no OpenAI key ever touches the container.
- Any dashboard / pipeline / runtime / contract change. None of those files are
  touched. This PRD must not bundle with PRD-204.

REQUIREMENTS

R1 — One environment-agnostic entrypoint
`codex-review <target>` produces a read-only Codex review from either environment
without the caller choosing a channel.
FAIL: invoking the command requires the caller to pass a host-vs-CI flag, OR it
refuses to run on an environment whose route's preconditions are satisfied.

R2 — Capability-based gate, never path/host-based
The local route is attempted iff `codex` is on PATH AND (`~/.codex/auth.json`
exists OR `OPENAI_API_KEY` is set). The route decision reads no hostname and no
repo path.
FAIL: the route decision branches on hostname or repo path, OR the local route is
attempted with `codex` absent or with neither credential present.

R3 — Fall back to CI on any local-environment failure
The local gate checks credential *existence*, not *validity*; the `codex exec`
attempt is the only true capability probe. When the local gate passes but the
attempt fails for any local-ENVIRONMENT reason the CI route could satisfy —
egress-block, auth/401, stale-or-revoked credential, missing/broken `codex`
binary — the command falls back to the CI route. IF exit codes make it cheap,
the router distinguishes environment failures (→ fall back) from THIS-REVIEW
failures (malformed/invalid target → surface and do NOT retry on CI, which would
fail identically and burn a dispatch); if not cheap, "any non-zero local exit →
fall back" is an acceptable approximation, and the burns-a-dispatch trade-off on
an invalid target is then a known, documented cost.
FAIL: a local-environment failure (egress / auth / stale credential / broken
binary) aborts the command instead of falling back to CI; OR (when exit-code
discrimination is implemented) an invalid-target failure silently retries on CI
and burns a dispatch with no diagnostic.

R4 — CI route dispatches PRD-197 and surfaces the verdict + provenance
The CI route dispatches `.github/workflows/codex-review.yml` against the target,
polls for the `codex-review/PRD-<n>-<sha>` artifact branch, and prints the verdict
plus the resolved-model and allowlist-verification provenance from the landed
artifact.
FAIL: the CI route does not dispatch `codex-review.yml`, OR it returns without
surfacing the landed artifact's resolved-model / allowlist provenance.

R5 — Read-only on every route
Both routes run Codex read-only — `-s read-only` on the host route; the workflow
is read-only by construction. The command never opts into workspace-write.
FAIL: the host route omits `-s read-only` or passes `-s workspace-write` anywhere.

R6 — Fail-loud when no route can run
When the local gate fails AND no `workflow_dispatch`-capable GitHub token is
available, the command exits non-zero with a diagnostic naming the missing
capability.
FAIL: with no viable route the command exits 0 or emits a success-shaped message.

R7 — Documented asymmetry
The command's `--help`/usage and the docs state that the CI route reviews only a
PRD target (PRD-197 as-is) while the host route reviews an arbitrary
branch/diff/file, and that closing the gap is out of scope (Path B).
FAIL: help or docs claim the CI route can review an arbitrary diff/file.

R8 — Retire the host-dependence hazard in docs
`CLAUDE.md`'s canonical Codex-invocation pattern points at `codex-review <target>`,
and a `docs/DECISIONS.md` entry records that the route is now tool-guaranteed.
FAIL: CLAUDE.md still names a bare `codex exec` as the canonical review invocation
with no reference to the router, OR no DECISIONS entry records the change.

DATA FLOW
target + environment → capability gate (codex on PATH ∧ credential present?) →
  pass → `codex exec -s read-only` → [any environment-class local fail:
    egress / auth / stale credential / broken binary] → CI route
  pass → `codex exec` → [invalid-target fail] → surface, no CI retry
  fail → CI route
CI route → dispatch codex-review.yml(prd=target, sha) → poll codex-review/PRD-*
  branch → read PRD-<n>.review.codex.md → print verdict + resolved-model/allowlist

FAIL CONDITIONS
- The command requires a manual host-vs-CI choice (R1).
- The route decision reads hostname/repo path, or attempts the local route without
  codex+credential (R2).
- A local-environment failure (egress / auth / stale credential / broken binary)
  aborts instead of falling back to CI; or an invalid-target failure silently
  burns a CI dispatch (R3).
- The CI route does not dispatch codex-review.yml or hides the provenance (R4).
- Any route runs Codex non-read-only (R5).
- No viable route yields exit 0 / success-shaped output (R6).
- Docs claim the CI route reviews arbitrary diffs (R7).
- CLAUDE.md still canonicalizes bare `codex exec`, or no DECISIONS entry (R8).

VALIDATION

  with a fake `codex` that emits a host-not-allowed/connection error, the host
  route falls back to CI (R3 — RED against a no-fallback implementation first).
- Unit: with the local gate failing and no dispatch-capable token, the command
  exits non-zero with the capability-naming diagnostic (R6 — RED first).
- Unit: assert the host invocation string contains `-s read-only` and never

- Manual (host, post-merge): `codex-review 205` on Dustin's host runs the local
