WHY NOW
Live-site report (Dustin, 2026-07-07 GDX screenshot): the per-symbol candidate
card still runs ~2.5 phone screens after the PRD-213..225 density passes. The
GDX card renders ~12 stacked lines before the level ladder for ~5 real decision
inputs:
- an 8-line identity block — SYMBOL/GRADE/BIAS/STRUCTURE each rendered as a
  stacked label-over-value pair;
- `LIFECYCLE: B → B | DEVELOPING → DEVELOPING` — a no-op transition (nothing
  changed) rendered as noise;
- `STATE: DEVELOPING` — a third surfacing of the same state already carried by
  GRADE ("Developing") and by the LIFECYCLE line;
- the `IF NOW` verdict — the single "is this tradable now" answer — buried at
  line 7;
- `RISK: …` — the `trade_framing.downgrade` string, which restates the
  INVALIDATION line directly above it (both reference the same reclaim level).

PRD-215 already moved REASON/PLAY/WATCH behind a default-collapsed DETAIL
disclosure and accented ENTRY/INVALIDATION; this PRD finishes the job on the
always-visible block above the ladder.

MAX EXPECTED DELTA
~40 production LOC, all in cuttingboard/delivery/dashboard_renderer.py
(_render_candidate_card body only — one header line, verdict reorder, one

GOAL
Reorganize and surface the card's EXISTING rendered fields so each high-grade
card reads header → verdict → in/out couplet → watch → ladder, targeting one
phone screen. Presentation-layer only: no field is computed differently and no
datum is lost (STATE folds into the header; RISK's content is the redundant
restatement of INVALIDATION).

SCOPE (in render order, high-grade cards)
- Header (one line) replaces the stacked SYMBOL/GRADE/BIAS/STRUCTURE block:
  `SYMBOL · GRADE · STATE · BIAS STRUCTURE` (e.g. `GDX · B · DEVELOPING ·
  BEARISH PULLBACK`). GRADE is the letter grade; STATE is `setup_state`
  (suppressed from the header when absent or `DATA_UNAVAILABLE`). This is where
  `setup_state` now lives — the standalone STATE line is removed, not the datum.
- Verdict at position 2: the `IF NOW` value renders immediately after the
  header, as the first field after identity.
- LIFECYCLE line: render only when the transition is real — i.e.
  `previous_grade != current_grade` OR `previous_setup_state !=
  current_setup_state`. A no-op transition (both equal, e.g.
  `B → B | DEVELOPING → DEVELOPING`) is suppressed. The lifecycle BADGE is
  unchanged (it already suppresses UNCHANGED via `_LIFECYCLE_BADGE_CSS`).
- Entry/invalidation couplet, adjacent, relabeled to the in/out frame and
  keeping the existing cyan `value-actionable` accent:
  - `IN  →` carries `trade_framing.entry`
  - `OUT →` carries `invalidation[0]`
- Drop the standalone RISK line (`trade_framing.downgrade`) — but FOLD its one
  non-redundant clause into OUT (review advisory, below). `downgrade` is
  `wait if price {loses|reclaims} {nearest} or structure turns choppy`
  (market_map.py:500/511): the price clause restates INVALIDATION, but the
  structural clause after " or " ("structure turns choppy") is a distinct
  structural-invalidation path the couplet does not express. OUT renders the
  invalidation joined with that structural clause, read verbatim from the
  field — presentation-only compose, derive nothing. The `downgrade` field is
  untouched upstream.
- Drop the standalone STATE line (`candidate-state`): `setup_state` now shows in
  the header.
- Single WATCH line: the `what_to_look_for` items (already one list, already
  behind DETAIL since PRD-215) render as one semicolon-joined value under one
  WATCH label instead of one label-per-item.
- One small CSS rule added: `.card-header{font-weight:bold;margin-bottom:6px}`
  for the new header line.
- Dead-branch enumeration: `_grade_to_action` (PRD-158 translation 11) was the
  GRADE-line's only consumer. The header shows the letter grade, so the helper
  and its local caller are now dead and are removed in this PRD. No other reader
  exists (grep: only the retired call site referenced it); no test imports it.

CHANGE SURFACE
cuttingboard/delivery/dashboard_renderer.py — edits confined to the high-grade
(`is_high`) branch of `_render_candidate_card` (lines ~1743–1801): replace the
stacked identity writes with one header line; move the IF NOW write above
ENTRY/INVALIDATION; add a real-transition guard around the LIFECYCLE detail
write; relabel ENTRY→"IN →" / INVALIDATION→"OUT →"; delete the STATE
(`candidate-state`) and RISK (`candidate-risk`) writes; join the WATCH loop into
one line. No new CSS class is required (header can reuse existing label/value
markup or a single line); no change to `_render_level_diagram`, the low-grade
branch, the tier loop, or any function signature.

REQUIREMENTS

R1 — Single-line header replaces the stacked identity block
A high-grade card renders SYMBOL, GRADE (letter), STATE (setup_state when
present), BIAS, and STRUCTURE on ONE line, not five stacked label/value pairs.
FAIL: a high-grade card renders a standalone `<div class="label">GRADE</div>`
or `BIAS` or `STRUCTURE` label/value pair (the old stacked block survives).

R2 — Verdict is first after the header
The `IF NOW` value renders before ENTRY/INVALIDATION and before any DETAIL
disclosure on a high-grade card that has a verdict.
FAIL: the `IF NOW` value's position in the card HTML is after the ENTRY/`IN →`
value.

R3 — No-op LIFECYCLE is suppressed
The LIFECYCLE detail line renders only when previous_grade != current_grade OR
previous_setup_state != current_setup_state; a both-equal transition is not
rendered.
FAIL: a card whose lifecycle has previous_grade == current_grade AND
previous_setup_state == current_setup_state renders `class="lifecycle-detail"`.

R4 — In/out couplet, no RISK, no standalone STATE
A high-grade card renders the entry under an `IN →` label and the invalidation
under an `OUT →` label, both keeping the `value-actionable` accent; it renders
NO `class="candidate-risk"` element and NO `class="candidate-state"` element.
FAIL: `class="candidate-risk"` or `class="candidate-state"` appears on a
high-grade card, or the entry/invalidation lose the `value-actionable` class.

R6 — OUT carries downgrade's structural clause (review advisory)
When `trade_framing.downgrade` contains a structural clause after " or " (e.g.
"structure turns choppy") that the invalidation does not already express, the
OUT line renders it, joined to the invalidation. The redundant price clause of
`downgrade` is NOT reintroduced, and the standalone RISK line stays gone.
FAIL: a card whose downgrade carries "… or structure turns choppy" renders an
OUT line without "structure turns choppy", or reintroduces a standalone RISK
line / the "wait if price …" price clause.

R5 — Single WATCH line
The `what_to_look_for` items render as one semicolon-joined value under a single
WATCH label inside DETAIL, not one WATCH label per item.
FAIL: a card with ≥2 `what_to_look_for` items renders ≥2 `<div class="label">
WATCH</div>` labels.

FAIL CONDITIONS
- Stacked GRADE/BIAS/STRUCTURE label/value pairs survive on a high-grade card (R1).
- IF NOW renders after IN → (R2).
- A no-op lifecycle transition renders lifecycle-detail (R3).
- candidate-risk or candidate-state appears on a high-grade card, or the
  actionable accent is dropped (R4).
- ≥2 WATCH labels render for ≥2 watch items (R5).

REVIEW ADVISORY CLOSED
The fresh-context Claude review returned six-PASS with one non-blocking
advisory: dropping the RISK line silently removed `downgrade`'s structural
clause ("… or structure turns choppy"), which is NOT redundant with the OUT
couplet (only the price clause is). Closed by FOLDING that structural clause
into OUT (renderer, presentation-only compose from the invalidation + downgrade
fields — verbatim), NOT by restoring the standalone RISK line. So "no data
removed" now holds strictly, not by dedup rationale. Covered by R6 and
tests/test_dash_candidates.py::test_out_line_folds_downgrade_structural_clause
(asserts on rendered output). Scope unchanged: dashboard_renderer.py + the two
test files only.
