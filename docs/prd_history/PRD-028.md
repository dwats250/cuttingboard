PRD-028 — PRD System Hardening

STATUS
COMPLETE

GOAL
Eliminate PRD drift and patch proliferation by enforcing file existence, a canonical template, per-requirement fail conditions, hard scope boundaries, and a formal lifecycle state model.

SCOPE
- Introduce docs/PRD_TEMPLATE.md as the canonical starting point for all PRDs
- Introduce docs/PRD_PROCESS.md defining lifecycle states and enforcement rules
- Update docs/PRD_REGISTRY.md: add File column linking each row to its prd_history file; add Status column using formal lifecycle states

OUT OF SCOPE
- No changes to any pipeline module
- No changes to runtime behavior
- No new runtime dependencies
- No changes to tests

FILES
A docs/PRD_TEMPLATE.md
A docs/PRD_PROCESS.md
M docs/PRD_REGISTRY.md

REQUIREMENTS

R1 — PRD File Existence
Every PRD MUST have a corresponding file at docs/prd_history/PRD-NNN.md.
A PRD is not active until this file exists.
File MUST be created with status IN PROGRESS before any implementation begins.

FAIL: A registry row exists with no corresponding file in prd_history/.

R2 — Canonical Template
docs/PRD_TEMPLATE.md MUST define the required sections in fixed order:
GOAL / SCOPE / OUT OF SCOPE / FILES / REQUIREMENTS / DATA FLOW / FAIL CONDITIONS / VALIDATION
All PRDs created after this PRD MUST follow this template exactly.
Section names and order MUST NOT deviate.

FAIL: A new PRD file is missing any required section or has sections in a different order.

R3 — Per-Requirement Fail Conditions
Every numbered requirement (R1, R2, ...) in a PRD MUST include an inline FAIL line.
The FAIL line MUST describe an observable, binary outcome.
No subjective language ("unclear", "poor", "insufficient") is permitted in FAIL lines.

FAIL: A requirement block contains no FAIL line, or the FAIL line is non-binary.

R4 — Scope Lock
The FILES section defines a hard boundary.
Any file modified during implementation that does not appear in FILES constitutes a scope violation.
PRD_PROCESS.md MUST state this rule explicitly.

FAIL: Implementation commits touch a file not listed in FILES with no corresponding PRD amendment.

R5 — Lifecycle States
PRD_PROCESS.md MUST define exactly five states:
PROPOSED — drafted, not yet approved for implementation
IN PROGRESS — file exists, implementation has begun
COMPLETE — implementation merged, commit hash recorded
PATCH — corrective PRD targeting a specific defect in a prior PRD
DEPRECATED — requirement superseded or withdrawn before completion

The PRD_REGISTRY.md Status column MUST use only these values.

FAIL: A registry row contains a status value not in the five defined states.

R6 — Registry File Links
PRD_REGISTRY.md MUST include a File column.
Each row with a corresponding prd_history file MUST link to it.
Rows without a file MUST show — in the File column.

FAIL: A row has a prd_history file that is not linked in the registry.

R7 — Patch PRD Root Cause
Any PRD with status PATCH MUST include a ROOT CAUSE section.
ROOT CAUSE MUST identify exactly one of:
- missing fail condition
- ambiguous requirement
- hidden dependency

FAIL: A PATCH PRD file has no ROOT CAUSE section, or ROOT CAUSE lists more than one cause without a primary.

DATA FLOW
PRD_TEMPLATE.md → copied per new PRD → prd_history/PRD-NNN.md
PRD_PROCESS.md → defines lifecycle rules referenced by registry and all PRD files
PRD_REGISTRY.md → File column links to prd_history/PRD-NNN.md; Status column enforces lifecycle state vocabulary

FAIL CONDITIONS
- Any prd_history/ file missing a required template section
- Any registry row with status not in {PROPOSED, IN PROGRESS, COMPLETE, PATCH, DEPRECATED}
- Any PATCH PRD without a ROOT CAUSE section
- Any requirement block without a binary FAIL line
- Implementation touching files outside FILES

VALIDATION

- Confirm docs/PRD_PROCESS.md exists and contains: lifecycle state definitions, scope lock rule, patch root cause rule
- Confirm PRD_REGISTRY.md has File and Status columns
- Confirm this PRD (PRD-028) row links to this file and status reads COMPLETE after merge
