THE GAP, IN FULL:

The full daily pipeline (`_run_pipeline`) merges the synthesized
continuation candidate into `candidates` at runtime/__init__.py:997-1004
before any downstream consumer runs — this is exactly what PRD-260
built and what Leg-1's review verified. But a SEPARATE, SHORTER "live
hourly" path (runtime/__init__.py:438 and :448) passes the ORIGINAL,
un-merged `candidates` dict straight to `_build_hourly_candidate_lines`,
which calls `_hourly_rr` (runtime/__init__.py:318) — reading the failed
DIRECT candidate's ATR-based stop and target, not the synthesized
continuation candidate's breakout-level geometry. This hourly candidate
line is then persisted into `candidate_lines` (runtime/__init__.py:1976)
and is live through `alert_runner.py:97`, ending up serialized into
`latest_hourly_run.json`.

Concrete in-memory repro against the real `qualify_all` (from the
model's own verification, not asserted):
- Direct geometry: entry 100.0, ATR stop 99.2, target 101.6 — rejected
  (stop-distance + extension failures).
- Continuation accepted, synthesized as: entry 100.0, breakout stop
  98.9, target 102.4.
- Current hourly line: `NVDA | LONG | TREND | 2.0:1` (computed from the
  rejected direct candidate's stale geometry).
- Line using the promoted candidate instead: `NVDA | LONG | TREND |
  2.2:1`.

Scope note carried over verbatim from the model: "The actual Telegram
body currently does not display that R:R, but the stale line is
constructed, parsed by the messaging path, and serialized into
`latest_hourly_run.json`." — so this is not purely inert; the wrong
number is computed, passed through the messaging path, and persisted to
a JSON artifact, even though it isn't rendered in the Telegram message
body today.
