STAGE-0 DECISION (locked 2026-06-22) — preserve-prior-with-staleness-marker
Mechanism 1 (preserve-prior), NOT mechanism 2 (fail-loud). Rationale, recorded:
aggregate()'s destructive rebuild corrupts already-committed, already-correct
returns when the SPY parquet is absent — the prior values are the trustworthy
actor and the rebuild is the unreliable one. No downstream action keys off the
scoreboard return (it is a published DISPLAY value; the only reader is
`dashboard_renderer.py:2439` via `.get`), so fail-loud would blank the surface on
every parquet-absent run (every Sunday, every premarket gap) and train the eye to
ignore the alarm. So: preserve last-known-good, and mark it stale so a genuinely
stuck value cannot masquerade as fresh. R3 ("loud, not silent") is satisfied by a
WARNING log + a stdout preserved-count in `main()`, not by exiting non-zero.

DRIFT FLAGGED (handoff vs repo)
- The handoff cited "the nullable-meta-field pattern from PRD-200
  (`meta.nyse_session_closed_reason`)" as the marker precedent. RECON: that field
  does not exist anywhere in the repo, and PRD-200 is the registry/index/state CI
  consistency PRD, not a payload-field PRD. The marker shape was therefore chosen
  to match THIS record's actual style instead: the `regime_history` summary is a
  flat dict whose fields are always present (set to null when absent), so the
  marker `spy_close_change_pct_stale` is an always-present boolean (True only when
  a value was carried forward), consistent with the record and readable via
  `.get(..., False)`.

CLASS
EXECUTION / INFRA (scoreboard sidecar; read-only observational artifact)

WHY NOW
The published dashboard scoreboard ("SPY next" column) loses all realized
forward-return values — every historical row, not just the latest — on the
weekend Sunday publish, then self-heals Monday. Root cause traced in recon:

- `regime_history.aggregate()` is a **full destructive rebuild**: it recomputes
  `spy_close_change_pct` for every date from `spy_closes` and overwrites
  `logs/regime_history.jsonl` via `_write_history(...)`
  (`cuttingboard/delivery/regime_history.py:138-144`).
- `spy_closes` comes from `_load_spy_close_series()`, which **silently returns
  `[]`** when `data/cache/SPY_ohlcv.parquet` is missing (`regime_history.py:74-76`).
  An empty series makes `_next_session_change()` return `None` for **every** date
  (`:91-104`).
- The SPY parquet is gitignored ("regenerated at runtime"), is not staged by the
  Commit step (`reports/ logs/ ui/` only), and is restored by no `actions/cache`.
  So on any fresh runner it exists only if that run wrote it.

The single code path that reliably writes the SPY parquet is
`_collect_trend_structure_history` (`runtime/__init__.py:1879-1896`), which fetches
every `TREND_STRUCTURE_SYMBOLS` (incl. SPY, `config.py:188`) regardless of posture
— and that lives ONLY in the hourly `_execute_notify_run` path (called at
`runtime/__init__.py:549`). The premarket pipeline `_run_pipeline`
(`:611-960`) does NOT call it and fetches OHLCV only for trade `candidates`, and
only when `mode != MODE_SUNDAY` (`:705`).

So the destructive wipe triggers on **any** `aggregate()` run that lands on a
runner with no SPY parquet — not Sunday alone:
- **Sunday premarket** (`cuttingboard.yml:263`, `_run_pipeline`, `mode=sunday`):
  OHLCV fetch fully gated off → parquet always absent → **persistent** wipe (no
  weekend hourly run heals it).
- **Weekday `live` premarket** (`cuttingboard.yml:263`, `mode=live`): wipes too
  whenever SPY is not a generated candidate, on a fresh runner — but the next
  hourly run repopulates it, masking the defect intra-day.

This is a PRD-198 invariant #1 violation (fail-loud, never silent-fallback): a
missing source silently degrades a published artifact. It also contradicts the
`aggregate()` docstring's stated contract (`:124-129`), which promises a
**gain-only** rebuild ("a prior day gains its reference move once the next
session's close exists") — when the parquet is absent the rebuild does the
opposite, a prior day *loses* its already-computed move.

GOAL
A rebuild never replaces a previously-computed `spy_close_change_pct` with `null`
because the SPY series was empty/absent. The fix is **mode-agnostic** — it must
hold for every caller (`cuttingboard.yml`, `hourly_alert.yml`), not gate on
Sunday/`session_type`. A test enforces the non-destruction.

SCOPE (to be detailed at Stage 0)
- `regime_history.py`: when `_load_spy_close_series()` yields an empty series (or
  a given date has no resolvable next-session close), do NOT overwrite an existing
  populated `spy_close_change_pct` in the prior `logs/regime_history.jsonl` with
  `None`. Two candidate mechanisms to weigh at Stage 0 (decision recorded in the
  PRD, not pre-judged here):
  1. **Preserve-prior (merge):** read the existing history, and for any date whose
     freshly-computed value is `None` but whose persisted value is non-null, keep
     the persisted value. Idempotent; keeps the docstring's gain-only contract
     literally true.
  2. **Fail-loud / skip-write:** when the SPY series is empty, exit non-zero (or
     skip the rebuild) so the workflow does not stage a wiped scoreboard, rather
     than writing all-`None`. Honors PRD-198 invariant #1 directly.
  The recon recommends (1) as the safe default (keeps the published artifact warm
  even on the Sunday run) with the empty-series case logged loudly; (2) is the
  stricter fail-closed alternative. Stage 0 picks one.
- Correct the `aggregate()` docstring to match whichever contract lands (the
  current "gain-only / idempotent" prose is the *intended* contract — fix the code
  to honor it, do not weaken the prose to legitimize destruction).

NON-GOALS
- No change to how the SPY parquet is produced or persisted — cache persistence is
  PRD-193's territory (defense-in-depth; this PRD fixes the published regression
  independently of whether the cache is ever warmed on Sunday).
- No Sunday/`session_type`-gated branch — the trigger is parquet-absence, not the
  calendar; a calendar gate would leave the weekday-premarket case unfixed.
- No renderer change — `_fmt_pct_signed` already renders `n/a` correctly for a
  null value (`dashboard_renderer.py:1015-1020`); the defect is the persisted null.
- No change to `_next_session_change` semantics (D scored by D+1 close).

REQUIREMENTS

R1 — A rebuild never wipes a populated return because the series was empty
Running `aggregate()` with an absent/empty SPY series preserves every
previously-persisted non-null `spy_close_change_pct` (mechanism 1), OR refuses to
write a degraded file (mechanism 2). Either way the published scoreboard never
regresses populated rows to `n/a` due to a missing parquet.
FAIL: after an `aggregate()` run with an empty SPY series, a date that had a
non-null `spy_close_change_pct` in the prior history is `null` in the new file.

R2 — Mode-agnostic
The guard holds for every caller, with no `session_type` / Sunday branch.
FAIL: the fix only protects the Sunday run, leaving the weekday-premarket
empty-parquet path able to wipe values.

R3 — Loud, not silent
The empty-series / unresolved-close case is logged (or exits non-zero per the
chosen mechanism), never a silent substitute-and-continue.
FAIL: an empty series degrades the artifact with no log/non-zero signal.

R4 — Test enforces non-destruction, and fails against today's code
A test seeds a history with populated returns, runs `aggregate()` with an empty
`spy_closes`, and asserts the populated values survive (R1). It must FAIL against
the current destructive rebuild before the fix.
FAIL: the test passes against today's all-`None` overwrite.

R5 — Preserved values carry an observable staleness marker
A value carried forward from the prior history (because the SPY series could not
recompute it this run) is flagged with `spy_close_change_pct_stale = True`; a
freshly-computed or genuinely-not-yet-computed value is `False`. The marker is the
hook the Lead-5 staleness-styling follow-up will read; this PRD only PRODUCES it
(no renderer change).
FAIL: a preserved value is indistinguishable from a freshly-computed one (no
marker), OR a freshly-computed value is marked stale.

R6 — Docstring restored to the safe (gain-only) contract
The `aggregate()` docstring describes the non-destructive, gain-only,
preserve-and-mark behavior the code now honors; it is NOT rewritten to bless
destruction. This closes the PRD-198 invariant #1 violation.
FAIL: the docstring still implies an unconditional recompute that can null a
populated value, or omits the preserve/marker behavior.

R7 — Production preserve-prior reads the PUBLISHED scoreboard, not main's fallback
Both publishing workflows restore `logs/regime_history.jsonl` from the `publish`
branch BEFORE the Aggregate step, so `aggregate()`'s preserve-prior reads the live
published values rather than main's frozen fallback. Guarded by env-parity tests in
`tests/test_ci_artifact_hygiene.py` that assert each workflow's restore arg list
includes `logs/regime_history.jsonl` and that the restore precedes the aggregate
step; the guards fail red if the arg is dropped (PRD-198 invariants #4, #5).
FAIL: a publishing workflow aggregates regime history without first restoring
`logs/regime_history.jsonl` from `publish`, so a publish-only date is wiped to null
in production despite the in-process preserve-prior logic.

DATA FLOW
absent/empty SPY parquet → `_load_spy_close_series()` → empty series →
[today] `_next_session_change` returns None for all dates → `_write_history`
overwrites populated values with null → published scoreboard shows all `n/a`.
[after] empty series → preserve prior populated values (or skip write) → published
scoreboard retains realized returns.
[production path] workflow restores `logs/regime_history.jsonl` from `publish` →
`aggregate()` reads the LIVE published prior (not main's frozen fallback) →
preserve-prior holds for publish-only dates too (Amendment 1 / R7).
