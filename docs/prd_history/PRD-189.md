WHY NOW
Every scheduled `cuttingboard.yml` run resolves to `noop` and returns SUCCESS.
The "Determine run mode" step (cuttingboard.yml:95–126) matches `date -u +%H%M`
EXACTLY against cron minutes (1250/1300/1350/1430/1630/1900/2000). GitHub
Actions queue delay means scheduled runs start off-the-minute, so the gate falls
through to `noop` and skips prefetch/live/sunday/verify/orb/post_orb/midmorning/
power_hour and the commit+push. Confirmed on run 27564681927 (started 17:37Z):
every pipeline step `skipped`, job conclusion `success`. The last pipeline-shaped
record in logs/audit.jsonl is 2026-05-13 — a manual workflow_dispatch (off-cron
11:25Z), i.e. the last time anyone forced the pipeline to run. A pipeline doing
zero work is indistinguishable from a healthy one; that invisibility is why the
freeze hid for ~33 days. The SCOREBOARD, the RUN SNAPSHOT, and the audit
pipeline-record feed have all been frozen the entire time, while the separate
hourly_alert.yml workflow kept the quote/trend surfaces current — so the
dashboard reads "fresh" over stale state.

MAX EXPECTED DELTA
A new unit-testable resolver entrypoint (~60–90 lines) + its tests; the
cuttingboard.yml "Determine run mode" step rewired to call it (dispatch `if:`
guards unchanged); the renderer freshness block extended from one age token to
three per-surface tokens (~30–50 lines) + tests. No change to any pipeline
mode's body, regime, qualification, sizing, payload schema, or notifications.

GOAL
(A) A scheduled run that starts late still resolves to the correct intended slot
and runs that slot's pipeline exactly once. (B) The dashboard surfaces freshness
PER DATA SURFACE so a frozen scoreboard cannot read "fresh" merely because quotes
are current — making A's success (or failure) directly observable on the
dashboard.

SCOPE
A — Mode resolution:
- Replace exact-minute matching with nearest-scheduled-slot-within-tolerance:
  map the run's actual UTC start to the closest scheduled slot whose nominal
  time is within a tolerance window (±N minutes), else `noop`.
- Extract the resolver out of inline workflow bash into a unit-testable Python
  entrypoint (scripts/resolve_run_mode.py) that the workflow calls; the workflow
  retains only the call and the per-mode dispatch.
- Dedup / single-fire: resolve to EXACTLY ONE slot, and guard the new failure
  this introduces — two `*/30` invocations (or one late run) matching the same
  or two adjacent windows must not double-fire ORB/power-hour.
- "Fired-today" source of truth: derive it from the EXISTING committed
  logs/audit.jsonl — "is there already a pipeline-shaped record for slot X on
  this UTC day?" — NOT a new marker file. Rationale: a marker file is a new
  state surface that can desync from the actual committed record of work, and
  audit.jsonl is the same record B's scoreboard-age freshness reads, so the
  dedup guard and the freshness instrument stay coherent by construction. (The
  resolver therefore reads audit.jsonl, which the checkout already restores.)
- Schedule shape — decision RECORDED in this PRD: keep `*/30 14-21` + the router
  vs. explicit per-slot crons. Neither option removes Actions queue delay, so
  the tolerant resolver is required regardless of which schedule shape is
  chosen; the choice only affects how many `noop` invocations are scheduled.

B — Per-surface freshness:
- Render freshness as distinct, separately-labeled ages, not one token:
  (1) quote age, (2) live-state age (latest_run.json run_at_utc),
  (3) scoreboard age (newest logs/regime_history.jsonl record date).
- Quote age MUST reuse the existing contract.stale_data_detected /
  fetched_at_utc signal (contract.py:269–274), which is currently computed and
  discarded — do not rebuild a quote-freshness computation.

OUT OF SCOPE
- No change to what any pipeline mode DOES once selected (prefetch/live/sunday/
  verify/orb/post_orb/midmorning/power_hour bodies unchanged).
- No change to regime, qualification, sizing, payload schema, or notifications.
- No auto-remediation, alerting, or escalation on detected staleness — surfacing
  only (description, not prediction).
- No new pipeline modes and no new schedules; if the recorded decision is to
  rewrite crons, that rewrite is in-scope but adds no mode/schedule semantics.

CHANGE SURFACE
- cuttingboard.yml: the "Determine run mode" step body only — it calls
  resolve_run_mode.py and consumes job_mode. Per-mode step `if:` guards
  unchanged.
- scripts/resolve_run_mode.py: new; pure slot/tolerance/dedup resolution reading
  run start time + logs/audit.jsonl.
- dashboard_renderer.py: the RUN SNAPSHOT / freshness render block only; adds
  scoreboard-age + live-state-age tokens beside the existing token.

REQUIREMENTS

R1 — Late start resolves to the intended slot
A run whose UTC start is within tolerance of a scheduled slot resolves to that
slot's job_mode (not `noop`).
FAIL: a start time inside a slot's tolerance window resolves to `noop`.

R2 — Single-fire, deduped from audit.jsonl
A single run resolves to at most one slot; a slot for which a pipeline-shaped
record already exists in logs/audit.jsonl for the current UTC day resolves to
`noop`. No marker file is introduced.
FAIL: one run resolves to two slots, OR a slot whose pipeline-shaped record
already exists for today fires again, OR dedup relies on any state surface other
than audit.jsonl.

R3 — Resolver is unit-testable outside the workflow
The slot/tolerance/dedup logic lives in an importable entrypoint with tests
covering on-time, late-within-tolerance, late-beyond-tolerance, and
already-fired-today cases.
FAIL: the resolution logic remains inline in workflow bash with no unit test.

R4 — Scoreboard/live-state age rendered distinctly from quote age
The dashboard renders three separately-labeled freshness tokens: quote age,
live-state age, scoreboard age.
FAIL: a stale scoreboard (newest record > 1 day old) renders a "fresh" / "<1 min"
freshness label anywhere on the dashboard.

R5 — Quote age reuses the existing stale signal
Quote-age freshness derives from contract.stale_data_detected / fetched_at_utc
(contract.py:269–274), not a newly-introduced quote-freshness computation.
FAIL: a second, independent quote-freshness calculation is added.

DATA FLOW
schedule fire → resolve_run_mode.py (run start time + audit.jsonl fired-today
check → job_mode) → workflow dispatches exactly one pipeline mode → mode writes
a pipeline-shaped record to logs/audit.jsonl → hourly regime_history aggregation
folds it → scoreboard appends a dated row → renderer diffs `now` against
{quote fetched_at, latest_run.run_at_utc, newest regime_history date} → three
per-surface age tokens.

FAIL CONDITIONS
- A late scheduled run resolves to `noop` within tolerance.
- A slot double-fires in one UTC day, or dedup uses a marker file.
- The dashboard shows "fresh" while the scoreboard's newest row is stale.

VALIDATION
- Unit: resolver tests for on-time / late-in-tolerance / late-out-of-tolerance /
  already-fired-today (audit.jsonl with a same-day pipeline record).
- Integration acceptance (SHARED A↔B): on the FIRST scheduled slot after this
  lands, a pipeline-shaped record appears in logs/audit.jsonl AND the scoreboard
  gains a new dated row; the dashboard's scoreboard-age token drops from ~30+
  days to current. B is the instrument that proves A worked.
- Manual: render the dashboard against a deliberately stale regime_history
  fixture; confirm scoreboard-age reads stale while quote-age reads fresh.
