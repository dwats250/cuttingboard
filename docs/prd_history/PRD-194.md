WHY NOW
The published dashboard froze. The originating cause (scheduled runs resolving to
noop on a wall-clock match) was fixed by PRD-189, but fixing it un-masked a second,
currently-active blocker: the artifact-publish path direct-pushes to `main`, and
`main` branch protection (added by PRD-182/184, merged 2026-06-14) now rejects every
such push. The first post-PRD-189 live slot (run 27637384167, 2026-06-16 17:56 UTC)
ran the full pipeline, committed the 2026-06-16 scoreboard row in the runner, then
failed at the push:

    remote: error: GH006: Protected branch update failed for refs/heads/main.
    remote: - Required status check "test" is expected.
    ! [remote rejected] HEAD -> main (protected branch hook declined)

`tools/ci_push_artifacts.sh` runs `git push origin HEAD:main` from three workflows
(cuttingboard.yml:241, hourly_alert.yml:147, macro_awareness.yml:72). Every artifact
push has failed since branch protection landed; the last successful artifact commit
on `main` is e3213b1 (CB hourly, 2026-06-12 17:21 UTC). The dashboard, the scoreboard
(regime_history), and the macro sidecar are all frozen behind this wall.

This finishes the decoupling PRD-178 began. PRD-178 decoupled PREVIEW from production
(an on-demand, never-deploy render loop); it explicitly held production publish out of
scope (PRD-178 OUT OF SCOPE; R3 forbade touching pages.yml / hourly_alert.yml /
cuttingboard.yml). Production publish was never decoupled — it still direct-pushes to
the protected `main`. PRD-194 decouples production publish onto a dedicated unprotected
branch, so `main` keeps its full review/merge gate and the bot never pushes to it.

DESIGN DECISION (resolved 2026-06-16; see docs/DECISIONS.md)
Option (b): publish artifacts to a dedicated UNPROTECTED branch that GitHub Pages
deploys from. Rejected alternatives: (a) bot bypass of `main` protection — erodes the
PRD-182/184/186 guardrail; (c) a PR + auto-merge per artifact — a full ~2,700-test
`test` run and a PR per hourly publish, with minutes of latency on a "fresh" surface.

State-log home: b1. The publish branch carries BOTH the rendered `ui/` AND the
read-back state (the accumulating `logs/audit.jsonl`, the scoreboard source). The
pipeline checks out `main` for CODE (always runs current reviewed code) and fetches
STATE from the publish branch at run start, so the scoreboard keeps accumulating
without `main` ever receiving a bot push.

PUBLISH BRANCH NAME: `publish` (long-lived, unprotected — no required `test` check).

MAX EXPECTED DELTA
No production Python LOC. Edits to 4 workflow YAML files + 1 shell script, plus the
two workflow-content guard tests, the CLAUDE.md governance wording, and bookkeeping.
A one-time rollout step seeds the `publish` branch from `main`. Exceeding this (e.g.
needing renderer or runtime changes) stops implementation and amends this PRD.

GOAL
Scheduled and hourly runs publish a fresh dashboard + an appended scoreboard to the
live site again, with `main` branch protection fully intact and no automated push to
`main`. After this lands, CLAUDE.md:52-54 ("no direct-push path") is TRUE.

SCOPE
- Create an unprotected `publish` branch. Seed it once from `main` at rollout
  (`git push origin main:publish`) so pages.yml and the state-restore step have a
  branch from the first run (deterministic bootstrap; see R4).
- `tools/ci_push_artifacts.sh`: push target `HEAD:main` -> `HEAD:$PUBLISH_BRANCH`
  (default `publish`); rebase base `origin/main` -> `origin/$PUBLISH_BRANCH`. The
  branch is a single variable so the test can pin it and a future rename is one edit.
- State read-back (R3): in each state-writing workflow, AFTER `actions/checkout@v6`
  of `main` and BEFORE the pipeline runs, restore the read-back state from the publish
  branch:
      git fetch origin "$PUBLISH_BRANCH" || true
      git checkout "origin/$PUBLISH_BRANCH" -- logs/audit.jsonl 2>/dev/null || true
  `logs/audit.jsonl` is the single canonical read-back artifact (the scoreboard source;
  `logs/regime_history.jsonl` is regenerated from it by the aggregator each run, so it
  need not be restored — it is rebuilt then published). Everything else under `logs/`
  is per-run output, not accumulating state.
- First-run / absent-branch bootstrap (R4): if `origin/$PUBLISH_BRANCH` does not exist
  (ls-remote miss) the restore is skipped and the run seeds from `main`'s checked-in
  `logs/audit.jsonl`; the push step creates the branch via `git push origin
  HEAD:$PUBLISH_BRANCH`. The one-time rollout seed above makes this path rare, but it
  must be safe (no hard-fail when the branch is absent).
- Shared publish concurrency (R5): give all THREE state-writers the SAME repo-wide
  concurrency group so publishes serialize across workflows:
      concurrency:
        group: cb-publish
        cancel-in-progress: false
  This eliminates the `logs/audit.jsonl` append race: with serialization, each publish
  job starts from the current tip of `publish` (via the R3 restore), appends, and
  fast-forward-pushes; no two appends ever rebase-merge.
- `pages.yml`: deploy from `publish` — `actions/checkout@v6` with `ref: publish`, and
  the push trigger `branches: [main]` -> `branches: [publish]`. Pages is UI-ONLY: it
  reads/deploys, never writes state.
- CLAUDE.md:52-54: correct the "no direct-push path" wording to describe the publish
  branch (artifact publishing targets the unprotected `publish` branch; `main` receives
  only PR-gated merges) — making the statement true rather than aspirational.

WRITE-STATE vs UI-ONLY (explicit, per the audit mandate)
- cuttingboard.yml  — STATE-WRITER (appends logs/audit.jsonl, regenerates
  logs/regime_history.jsonl, renders ui/). Shared group cb-publish.
- hourly_alert.yml  — STATE-WRITER (appends logs/audit.jsonl, regenerates
  regime_history, renders ui/). Shared group cb-publish.
- macro_awareness.yml — STATE-WRITER (writes the macro sidecar; pushes via
  ci_push_artifacts.sh). Shared group cb-publish.
- pages.yml         — UI-ONLY (deploys `publish`'s ui/; never writes state).

OUT OF SCOPE
- OHLCV cache (`data/cache`) persistence — that is PRD-193. The read-back state here is
  `logs/audit.jsonl` only; the cache is deliberately NOT moved to the publish branch.
- Any renderer / runtime / aggregation logic change. The render and aggregation
  commands are byte-unchanged; only WHERE the result is pushed and WHENCE state is
  restored changes.
- Re-homing the intraday/orb slots (PRD-192) and the prefetch slot (PRD-193).
- Changing the `test` required-status-check on `main` — `main` protection is unchanged;
  this PRD routes AROUND it, it does not weaken it.
- The PRD-189 hourly_alert.yml render-before-aggregate ordering nit (parked).

CHANGE SURFACE

  variable with a `publish` default. Push must FAIL CLOSED on a non-fast-forward
  (never auto-merge two divergent audit.jsonl appends — serialization makes this a
  no-op, but the guard stays).
- Each state-writer workflow: add the shared concurrency group + the R3 state-restore

REQUIREMENTS

R1 — No automated push to `main`
After this PRD, no workflow pushes to `main`. `tools/ci_push_artifacts.sh` pushes only
to `$PUBLISH_BRANCH`; the three state-writers carry no other push step.
FAIL: `HEAD:main` (or any `git push` targeting main) appears in ci_push_artifacts.sh or
any of the three state-writer workflows.

R2 — Pages deploys the publish branch
pages.yml checks out `ref: publish` and deploys its `ui/`; its push trigger is
`branches: [publish]`.
FAIL: pages.yml checks out the default branch (no ref) or triggers on push to main.

R3 — State read-back before the run
Each state-writer restores `logs/audit.jsonl` from `origin/$PUBLISH_BRANCH` after
checkout and before the pipeline step, so the run appends onto the live-accumulating
history rather than `main`'s frozen copy.
FAIL: a state-writer runs the pipeline against `main`'s checked-in audit.jsonl without
attempting the publish-branch restore.

R4 — Absent-branch bootstrap is safe
If `origin/$PUBLISH_BRANCH` is absent, the restore is skipped (no hard-fail) and the
push step creates the branch.
FAIL: a missing publish branch makes a state-writer job error out, or the push step
cannot create the branch.

R5 — Publishes serialize across workflows (append-race elimination)
cuttingboard.yml, hourly_alert.yml, and macro_awareness.yml share one repo-wide
concurrency group (`cb-publish`, cancel-in-progress: false), so no two publish jobs
append to logs/audit.jsonl concurrently, and ci_push_artifacts.sh's push is a
fast-forward (its rebase never silently merges divergent JSONL appends).
FAIL: the three workflows carry different concurrency group names, or the push path can
rebase-merge two divergent audit.jsonl appends instead of failing closed.

R6 — `main` protection intact
The `test` required-status-check on `main` is unchanged; no bypass/allowance is added
for the bot. (Verified out-of-tree against branch-protection settings at closeout.)
FAIL: `main` protection is weakened, or the bot is granted any push allowance on main.

R7 — Workflow-content guards updated
tests/test_dashboard_renderer.py:105 and tests/test_ci_artifact_hygiene.py are updated
so their anchors match the new push target/branch and still fail (not smoke-pass) when
violated; a new assertion pins ci_push_artifacts.sh's target to `$PUBLISH_BRANCH` and
pins the shared concurrency group on all three state-writers.
FAIL: a workflow could revert to pushing `main`, or drop the shared concurrency group,
while the hygiene suite stays green.

R8 — CLAUDE.md governance wording corrected
CLAUDE.md:52-54 describes the publish-branch reality (artifact publishing targets the
unprotected publish branch; main takes only PR-gated merges).
FAIL: CLAUDE.md still asserts "no direct-push path" in a way contradicted by the
workflows, or omits the publish-branch mechanism.

DATA FLOW
Run: checkout main (code) -> fetch+restore logs/audit.jsonl from origin/publish (R3) ->
pipeline appends audit.jsonl + renders ui/ + regenerates regime_history -> commit ->
ci_push_artifacts.sh fast-forward push to publish (R1/R5) -> pages.yml deploys publish's
ui/ (R2). main never receives a bot push (R1/R6).

FAIL CONDITIONS
- Any workflow still pushes to main, or main protection is weakened.
- Pages deploys from main instead of publish.
- A state-writer ignores the publish-branch restore and appends onto main's frozen
  audit.jsonl (re-freeze).
- The three publish workflows are not serialized and two appends rebase-merge.
- The hygiene suite stays green while a workflow reverts the push target or drops the
  shared concurrency group.
