currently-active blocker: the artifact-publish path direct-pushes to `main`, and

such push. The first post-PRD-189 live slot (run 27637384167, 2026-06-16 17:56 UTC)
ran the full pipeline, committed the 2026-06-16 scoreboard row in the runner, then
failed at the push:

    remote: error: GH006: Protected branch update failed for refs/heads/main.
    remote: - Required status check "test" is expected.
    ! [remote rejected] HEAD -> main (protected branch hook declined)

`tools/ci_push_artifacts.sh` runs `git push origin HEAD:main` from three workflows

on `main` is e3213b1 (CB hourly, 2026-06-12 17:21 UTC). The dashboard, the scoreboard
(regime_history), and the macro sidecar are all frozen behind this wall.

This finishes the decoupling PRD-178 began. PRD-178 decoupled PREVIEW from production
(an on-demand, never-deploy render loop); it explicitly held production publish out of
scope (PRD-178 OUT OF SCOPE; R3 forbade touching pages.yml / hourly_alert.yml /
cuttingboard.yml). Production publish was never decoupled — it still direct-pushes to
the protected `main`. PRD-194 decouples production publish onto a dedicated unprotected
branch, so `main` keeps its full review/merge gate and the bot never pushes to it.

DESIGN DECISION (resolved 2026-06-16; see docs/DECISIONS.md)
Option (b): publish artifacts to a dedicated UNPROTECTED branch that GitHub Pages
deploys from. Rejected alternatives: (a) bot bypass of `main` protection — erodes the
PRD-182/184/186 guardrail; (c) a PR + auto-merge per artifact — a full ~2,700-test
`test` run and a PR per hourly publish, with minutes of latency on a "fresh" surface.

State-log home: b1. The publish branch carries BOTH the rendered `ui/` AND the
read-back state (the accumulating `logs/audit.jsonl`, the scoreboard source). The
pipeline checks out `main` for CODE (always runs current reviewed code) and fetches
STATE from the publish branch at run start, so the scoreboard keeps accumulating
without `main` ever receiving a bot push.

PUBLISH BRANCH NAME: `publish` (long-lived, unprotected — no required `test` check).

TWO-TREE SYNC INVARIANT (must not be collapsed into one direction):
  - logs/ accumulated state flows publish -> run: RESTORE from publish, then publish
    only THIS run's delta/append onto the live tip. **publish is the authority.**
  - ui/ static assets flow main -> publish: FULL-SYNC from CURRENT `origin/main` (the
    latest REVIEWED main, NOT the run's POST_SHA — a run based on an older main must not
    roll publish back to stale JS/CSS; Codex P2) every publish; generated pages
    (dashboard/index/contract) are published only by a run that regenerated them.
    **main is the authority.**
  These run in OPPOSITE directions on purpose; a single shared sync direction would
  either freeze the scoreboard (if ui-style main->publish) or serve stale reviewed
  assets (if logs-style publish->run). Keep them separate.

FAIL: a state-writer runs against `main`'s frozen copy of any file in its read-back set
without attempting the publish-branch restore; OR a present-branch checkout error is
swallowed; OR the two sync directions are collapsed.

R4 — Absent-branch bootstrap is safe
If `origin/$PUBLISH_BRANCH` is absent, the restore is skipped (no hard-fail) and the
publish step creates the branch (`git push origin $POST_SHA:refs/heads/$PUBLISH_BRANCH`).
FAIL: a missing publish branch makes a state-writer job error out, or the push step
cannot create the branch.

R5 — Delta-append never clobbers + bounded retry handles races (append-integrity)
The publish step DELTA-APPENDS this run's new logs/audit.jsonl rows onto the publish
tip's CURRENT file (base = CB_PUBLISH_BASE_SHA) — it never wholesale-copies/clobbers —
so even if the tip carries rows from another writer, no accumulated history is lost.
ui/ and regime_history are full-regen overwrites. Cross-workflow publish races are NOT
serialized by a shared lock (Codex P2 — that over-serialized the hourly alert); instead
a non-fast-forward push RETRIES (bounded, with jitter): re-fetch the tip, re-apply this
run's delta onto the moved tip, re-push. Each writer keeps its own per-workflow group
for self-serialization (hourly-alert's is load-bearing for 06:00/06:05 backup dedup).
No rebase, no 3-way merge of the appends.
FAIL: the publish step wholesale-copies/overwrites logs/audit.jsonl instead of
delta-appending; OR it lacks a bounded non-fast-forward retry; OR a shared cb-publish
