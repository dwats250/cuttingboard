## 1. Ingestion path: A vs B vs C

**Verdict: A — direct read of `logs/trend_structure_snapshot.json`.**

Reasoning, governance-first:

- `docs/sidecar_doctrine.md` §"Contract isolation" and §"Dashboard as
  consumer" are categorical: the contract is not the transport for
  sidecar data; the renderer reads sidecar artifacts from canonical paths.
  Option B (payload adaptation) would invert this rule and is forbidden
  unless promoted by a decision-affecting PRD.
- Option C (thin adapter layer) introduces an abstraction without removing
  duplication. The renderer already has three direct sidecar reads
  (`_MACRO_SNAPSHOT_PATH`, `_HOURLY_CONTRACT_PATH`, `MARKET_MAP_PATH`).
  Adding a fourth on the same template is incremental, not architectural.
  An adapter layer would also expand FILES and create a hidden-coupling
  surface (the adapter would need to know all sidecars).
- Option A is the doctrine-mandated path and the lowest-risk diff.

**Binding:** Reject A's stronger forms (e.g., a `_load_sidecar()` helper
that takes a name). Use the same direct constant + `_load_json` pattern as
the macro snapshot. No new abstraction layer.

## 4. Deterministic acceptance criteria

PRD-112 R1–R10 are binary, observable, and grep-checkable. Specific
strengths:

- R3's "row count is not exactly 6" check is the strongest defensive
  assertion in the PRD.
- R7's leakage grep reuses PRD-107 R2 / PRD-110 R7 patterns, preserving
  continuity.
- R8's workflow-YAML diff guard is the cleanest scope lock.
- R10's `git status --short ui/` clean check is binary.

Specific weaknesses:

- **R5 (stale)** — the threshold is `DASHBOARD_STALE_AFTER_SECONDS`
  evaluated relative to "the run timestamp used by the rest of the
  dashboard." That phrase is implementer-interpretive. **Codex requires:**
  name the exact field used (likely `payload["generated_at"]` or
  `run["started_at"]`) and bind the renderer to use whichever the macro
  snapshot already uses. Do not invent a third reference timestamp.

- **R4** — "rounding, percent suffix on `relative_volume`" is the only
  arithmetic permitted. The PRD should explicitly forbid `if/else` that
  derives a new enum from numeric thresholds (e.g., classifying
  `relative_volume > 1.5` as "ELEVATED"). **Codex requires:** add an
  explicit FAIL line "Renderer source contains numeric comparisons against
  any sidecar value used to derive a displayed label."

- **R9** — six tests are enumerated but "non-deterministic" is the FAIL
  trigger. The PRD should also forbid wall-clock dependence in stale
  tests. **Codex requires:** stale test must use a frozen reference
  timestamp passed explicitly (e.g., `monkeypatch` of the renderer's clock
  source) — not `datetime.now()`.

## 5. Deterministic stale-data behavior

PRD-112 R5 enumerates four states. Adequate, with two reinforcements:

1. **Reference timestamp must be named** (see §4 above). Without it,
   "stale" is ambiguous and tests are flaky.
2. **No partial fall-back.** If the file is malformed, the renderer
   renders the empty 6-row state. It must NOT attempt to parse what it
   can and render the rest. **Codex binding:** add to R5 a FAIL line
   "Renderer attempts partial JSON recovery (e.g., per-symbol salvage
   when top-level parse succeeds but a record is malformed) — must
   degrade the entire section to MISSING state."

   Rationale: partial recovery is a hidden-coupling surface where the
   renderer silently invents a contract not enforced by the producer.

## 6. Safest UI form

**Verdict: compact read-only table.**

- Compact cards: 6 cards × 9 fields creates inconsistent vertical rhythm
  on desktop and forces a grid system the dashboard does not have today.
- Collapsible section (`<details>`): adds JS-free progressive disclosure
  but hides visibility-first information by default. Visibility-first is
  the system's stated direction (`docs/PROJECT_STATE.md`).
- Existing panel integration (e.g., merging into the candidate board):
  high coupling risk — candidate board rows are decision-relevant; trend
  structure rows are observe-only. Merging mixes two different contracts
  inside one DOM region. Doctrine violation.
- Compact table: aligns with the macro tape and candidate board patterns
  already in the renderer; renders deterministically; no JS state; mobile
  degrades to horizontal scroll.

**Codex binding:** flat read-only table; no `<details>`; no card grid; no
merge into candidate-board markup. Mobile column priority must be in the
PRD body (see §2 gap).

## 7. UI publish workflow

**Yes — required.** PRD-112 R10 already mandates:

```
python3 -m cuttingboard.delivery.dashboard_renderer --output ui/dashboard.html
cp ui/dashboard.html ui/index.html
```

This matches the standing dashboard publish rule (Memory: "Dashboard
publish rule"). The default renderer output path is
`reports/output/dashboard.html`; CI redirects to `ui/dashboard.html` and
copies to `ui/index.html`. The implementing commit must perform the same
two steps locally so the published Pages artifact reflects the new
section.

**Codex requires** (already in R10 — flagged for emphasis): post-commit
`git status --short ui/` must be clean.

## 9. Codex deliverables

### Architecture verdict
PRD-112 is correctly scoped as a renderer-only consumer change. The
direct-read pattern is doctrine-compliant and reuses an established
idiom. No adapter layer, no payload mutation, no contract change.

### Scope risks
1. Mobile column-priority rule lives only in Claude review — must be
   lifted into PRD body (R3 / R3a).
2. R5 stale-reference timestamp is implementer-interpretive — must name
   the exact field.
3. R5 must explicitly forbid partial JSON recovery.
4. R4 must explicitly forbid numeric-comparison-derived display labels.
5. R9 stale test must forbid wall-clock dependence.

### Revised PRD guidance
Apply the five revisions above to PRD-112 before freezing. The revisions
are mechanical tightenings (no scope expansion, no behavior change).
Per CLAUDE.md cross-review gate, mechanical incorporation of accepted
review findings is Claude-only — no second Codex pass required after
these revisions land.
