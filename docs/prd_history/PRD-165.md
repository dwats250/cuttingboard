## AMENDMENT (2026-06-05, pre-implementation)

Design decisions locked by Dustin after the Codex review:
- **D1 → bold-only.** ENTRY/INVALIDATION emphasis is a dedicated CSS class with
  `font-weight:bold` (no color/size/border change) — the minimal "cuts before
  additions" treatment.
- **D2 → conditional collapse** (already resolved at review).
- **D3 → split.** R3 (RUN SNAPSHOT freshness) is removed from this PRD and
  deferred to **PRD-167**. PRD-165 now implements **R1 + R2 only**.

## WHY NOW

The dashboard is the trader-facing decision surface, and two display weaknesses
dull it: (1) each candidate card renders ~10 label+value pairs at identical
visual weight, so ENTRY and INVALIDATION — the only falsifiable, actionable
fields — read no louder than SYMBOL/GRADE/STATE/REASON; (2) the trend-structure
table carries columns that are NOT COMPUTED / INSUFFICIENT HISTORY for 5–6 of 6
symbols, spending horizontal space on noise. PRD-162 and PRD-160 (the
state-correctness prerequisites) have landed, so hierarchy work is no longer
wasted on a card that shows contradictory state. (The RUN SNAPSHOT freshness
contract miss is real but split to PRD-167 per D3.)

## REQUIREMENTS

R1 — ENTRY and INVALIDATION are bold.
In a high-grade candidate card, the ENTRY and INVALIDATION value rows render
with a dedicated CSS class (not the generic `.value` shared by
REASON/PLAY/WATCH), and that class sets `font-weight:bold` in `_CSS`
(`dashboard_renderer.py:556-648`).

FAIL: in a rendered high-grade card, the ENTRY/INVALIDATION value elements carry
the same CSS class as the REASON value element, OR the dedicated class's `_CSS`
rule does not set a bold `font-weight` distinct from `.value`.

R2 — Uniformly-empty trend-structure columns conditionally collapse.
A column ("vs VWAP", "vs SMA200", "Alignment", "Entry Context") is hidden only
when its cells are NOT_COMPUTED / INSUFFICIENT_HISTORY / DATA_UNAVAILABLE across
*every* rendered symbol; a column with at least one healthy value still renders
in full, preserving the PRD-131/132 header set and column order
(`tests/test_dashboard_renderer.py:2577-2580`, `:2769-2778`, `:2903-2924`) and
the healthy-snapshot expectations (`:1138-1155`). The collapse decision runs
*after* the PRD-112 source-valid / all-or-nothing degrade gate and must not
salvage partial rows: a single missing required trend key degrades the whole
section, exactly as today (`tests/test_dashboard_renderer.py:1192-1208`).

FAIL: (a) a healthy snapshot no longer renders populated "Alignment" /
"Entry Context", or SMA Composite / Intraday Context order changes; OR (b) with
"vs VWAP"/"vs SMA200"/"Alignment"/"Entry Context" uniformly unavailable across
all symbols the table still emits a full populated column for each; OR (c)
deleting one required trend key renders a healthy value from another symbol
rather than degrading the whole section.

## DESIGN DECISIONS (resolved)

- **D1 — RESOLVED to bold-only.** ENTRY/INVALIDATION get a dedicated class with
  `font-weight:bold`; no color, size, or border change. Dustin's call,

- **D2 — RESOLVED to conditional collapse** per Codex review: an unconditional
  cut breaks healthy-snapshot and PRD-131/132 header/order tests.
- **D3 — RESOLVED to split.** R3 (RUN SNAPSHOT freshness) moves to PRD-167; this
  PRD ships R1 + R2.
