# PRD-165: Candidate-card visual hierarchy, dead-column pruning, RUN SNAPSHOT freshness

Lane: HIGH-RISK / CONSUMER
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-05

## WHY NOW

The dashboard is the trader-facing decision surface, and three display
weaknesses dull it: (1) each candidate card renders ~10 label+value pairs at
identical visual weight, so ENTRY and INVALIDATION — the only falsifiable,
actionable fields — read no louder than SYMBOL/GRADE/STATE/REASON; (2) the
trend-structure table carries columns that are NOT COMPUTED / INSUFFICIENT
HISTORY for 5–6 of 6 symbols, spending horizontal space on noise; (3) the RUN
SNAPSHOT timestamp renders via the generic `.value` class with no
freshness/staleness token — confirmed on the 2026-06-04 real-run inspection and
a direct PRD-158 §4.1 RUN-SNAPSHOT-freshness contract miss. PRD-162 and PRD-160
(the state-correctness prerequisites) have landed, so hierarchy work is no
longer wasted on a card that shows contradictory state.

## GOAL

Make the candidate card and trend-structure table read at a glance: ENTRY and
INVALIDATION stand out, uniformly-empty trend columns stop consuming space, and
the RUN SNAPSHOT shows whether the data is fresh or stale.

## SCOPE

- **R1 — ENTRY/INVALIDATION emphasis.** Give the ENTRY and INVALIDATION
  label+value rows a distinct visual treatment in high-grade candidate cards.
  Currently both render via the generic `.value` class
  (`dashboard_renderer.py:1482` ENTRY, `:1486` INVALIDATION), identical to
  REASON/PLAY/WATCH (`.value` defined at `:562`).
- **R2 — Prune dead trend-structure columns.** The columns "vs VWAP",
  "vs SMA200", "Alignment", "Entry Context" (headers rendered at
  `dashboard_renderer.py:2006-2010`, cells at `:2026-2030` via `_ts_display`,
  `:77`) are NOT COMPUTED / INSUFFICIENT HISTORY for most symbols. Cut or
  conditionally collapse them (see Design Decision D2).
- **R3 — RUN SNAPSHOT freshness.** Render a relative freshness / staleness
  token on the RUN SNAPSHOT block (`dashboard_renderer.py:1862-1863`) by wiring
  the already-present-but-unused `_compute_timestamp_freshness` (`:222`) and
  `DASHBOARD_STALE_AFTER_SECONDS` (`:167`). Satisfies PRD-158 §4.1.

## REQUIREMENTS

R1 — ENTRY and INVALIDATION are visually emphasized.
In a high-grade candidate card, the ENTRY and INVALIDATION value rows render
with a distinct emphasis treatment (a dedicated CSS class), not the generic
`.value` class shared by REASON/PLAY/WATCH.

FAIL: in a rendered high-grade card, the ENTRY value element and the
INVALIDATION value element carry the same CSS class as the REASON value
element.

R2 — Uniformly-empty trend-structure columns do not consume a full column.
A trend-structure column whose cells are NOT COMPUTED / INSUFFICIENT HISTORY /
DATA UNAVAILABLE across every rendered symbol is cut or collapsed per the
chosen Design Decision D2 mechanism.

FAIL: with a trend_structure snapshot in which "vs VWAP", "vs SMA200",
"Alignment", and "Entry Context" are uniformly NOT_COMPUTED /
INSUFFICIENT_HISTORY for all rendered symbols, the rendered table still emits a
full populated column (header + per-row cells) for each of those four.

R3 — RUN SNAPSHOT renders a freshness/staleness token.
The RUN SNAPSHOT block renders a relative freshness indicator derived from
`_compute_timestamp_freshness` / `DASHBOARD_STALE_AFTER_SECONDS`, in addition
to (or replacing) the absolute timestamp.

FAIL: rendering a payload whose `meta.timestamp` is older than
`DASHBOARD_STALE_AFTER_SECONDS` produces a RUN SNAPSHOT block containing only an
absolute timestamp with no staleness token.

## DESIGN DECISIONS FOR REVIEW

- **D1 (R1 mechanism — needs Dustin's taste).** How should ENTRY/INVALIDATION
  stand out — bold weight, accent color, a left border/box, or larger size?
  This is a visual-judgment call on the live decision surface; flagging for
  Dustin rather than picking unilaterally.
- **D2 (R2 cut vs collapse).** *Cut* the four columns entirely (simplest,
  "cuts before additions") versus *conditionally collapse* (hide a column only
  when uniformly unavailable, so it reappears once intraday data computes).
  Recommendation: conditional collapse, since these columns *do* populate
  intraday for liquid names — but confirm.
- **D3 (R3 split?).** The candidate noted R3 "may be split out as a smaller
  dedicated PRD-158 renderer follow-up instead of waiting on the full hierarchy
  pass." If R1/R2 need design iteration, R3 (a clean contract fix) could ship
  first as its own PRD. Reviewer to decide bundle vs split.

## DATA FLOW

`payload.meta.timestamp` → `_compute_timestamp_freshness` → RUN SNAPSHOT token (R3)
`trend_structure_snapshot.json` → per-column availability scan → cut/collapse (R2)
`trade_framing.entry` / `invalidation[0]` → emphasized card rows (R1)

## FAIL CONDITIONS

- ENTRY/INVALIDATION share the REASON value's CSS class (R1).
- A uniformly-empty trend column still renders a full column (R2).
- A stale RUN SNAPSHOT renders no staleness token (R3).
