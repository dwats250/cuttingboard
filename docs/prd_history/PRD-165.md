# PRD-165: Candidate-card visual hierarchy, dead-column pruning, RUN SNAPSHOT freshness

Lane: HIGH-RISK / CONSUMER
Status: IN PROGRESS
Author: Dustin
Filed: 2026-06-05

## WHY NOW

The dashboard is the trader-facing decision surface, and three display
weaknesses dull it: (1) each candidate card renders ~10 label+value pairs at
identical visual weight, so ENTRY and INVALIDATION — the only falsifiable,
actionable fields — read no louder than SYMBOL/GRADE/STATE/REASON; (2) the
trend-structure table carries columns that are NOT COMPUTED / INSUFFICIENT
HISTORY for 5–6 of 6 symbols, spending horizontal space on noise; (3) the RUN
SNAPSHOT timestamp renders via the generic `.value` class with no
freshness/staleness token — confirmed on the 2026-06-04 real-run inspection and
a direct PRD-158 §4.1 RUN-SNAPSHOT-freshness contract miss. PRD-162 and PRD-160
(the state-correctness prerequisites) have landed, so hierarchy work is no
longer wasted on a card that shows contradictory state.

## GOAL

Make the candidate card and trend-structure table read at a glance: ENTRY and
INVALIDATION stand out, uniformly-empty trend columns stop consuming space, and
the RUN SNAPSHOT shows whether the data is fresh or stale.

## SCOPE

  label+value rows a distinct visual treatment in high-grade candidate cards.
  Currently both render via the generic `.value` class
  (`dashboard_renderer.py:1482` ENTRY, `:1486` INVALIDATION), identical to
  REASON/PLAY/WATCH (`.value` defined at `:562`).
- **R2 — Prune dead trend-structure columns.** The columns "vs VWAP",
  "vs SMA200", "Alignment", "Entry Context" (headers rendered at
  `dashboard_renderer.py:2006-2010`, cells at `:2026-2030` via `_ts_display`,
  `:77`) are NOT COMPUTED / INSUFFICIENT HISTORY for most symbols. Cut or
  conditionally collapse them (see Design Decision D2).
- **R3 — RUN SNAPSHOT freshness.** Render a relative freshness / staleness
  token on the RUN SNAPSHOT block (`dashboard_renderer.py:1862-1863`) by wiring
  the already-present-but-unused `_compute_timestamp_freshness` (`:222`) and
  `DASHBOARD_STALE_AFTER_SECONDS` (`:167`). Satisfies PRD-158 §4.1.

## REQUIREMENTS

R1 — ENTRY and INVALIDATION are visually emphasized.
In a high-grade candidate card, the ENTRY and INVALIDATION value rows render
with a dedicated CSS class (not the generic `.value` shared by
REASON/PLAY/WATCH), and that class carries a concrete visible property in `_CSS`
(`dashboard_renderer.py:556-648`) — font weight, color, border, or background —
distinct from `.value`.

FAIL: in a rendered high-grade card, the ENTRY/INVALIDATION value elements carry
the same CSS class as the REASON value element, OR the dedicated class's `_CSS`
rule has no visible property different from `.value`.

R2 — Uniformly-empty trend-structure columns conditionally collapse.
D2 is resolved to **conditional collapse** (not unconditional cut): a column is
hidden only when its cells are NOT_COMPUTED / INSUFFICIENT_HISTORY /
DATA_UNAVAILABLE across *every* rendered symbol; a column with at least one
healthy value still renders in full, preserving the PRD-131/132 header set and
column order (`tests/test_dashboard_renderer.py:2577-2580`, `:2769-2778`,
`:2903-2924`) and the healthy-snapshot expectations (`:1138-1155`). The collapse
decision runs *after* the PRD-112 source-valid / all-or-nothing degrade gate and
must not salvage partial rows: a single missing required trend key degrades the
whole section, exactly as today (`tests/test_dashboard_renderer.py:1192-1208`).

FAIL: (a) a healthy snapshot no longer renders populated "Alignment" /
"Entry Context", or SMA Composite / Intraday Context order changes; OR (b) with
"vs VWAP"/"vs SMA200"/"Alignment"/"Entry Context" uniformly unavailable across
all symbols the table still emits a full populated column for each; OR (c)
deleting one required trend key renders a healthy value from another symbol
rather than degrading the whole section.

R3 — RUN SNAPSHOT renders a relative freshness/staleness token.
The RUN SNAPSHOT block renders a relative freshness indicator derived from
`_compute_timestamp_freshness` / `DASHBOARD_STALE_AFTER_SECONDS`. Per PRD-158
§4.1 (`docs/prd_history/PRD-158.md:90-93`, `:205-208`) the contract is *relative
freshness*, so the token **replaces** the bare absolute PT timestamp as the
primary value (an absolute timestamp may remain as secondary detail). This is an
intentional rewrite of the current regression tests
`tests/test_dashboard_renderer.py:626-647`, which today assert the absence of
`STALE` / `minute` — those assertions are inverted by this PRD, and the tests
must inject/freeze the renderer's "now" because `_compute_timestamp_freshness`
calls `datetime.now()` (`dashboard_renderer.py:222-228`).

FAIL: rendering a payload whose `meta.timestamp` is older than
`DASHBOARD_STALE_AFTER_SECONDS` produces a RUN SNAPSHOT block with no staleness
token, OR `tests/test_dashboard_renderer.py:626-647` still assert `STALE` /
`minute` absence after this PRD.
