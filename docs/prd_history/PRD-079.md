# PRD-079 — PRD Review Token Efficiency Guardrails

**Status:** COMPLETE (9e0aca7)
**Author:** Dustin Watson
**Date:** 2026-05-03

---

## GOAL

Reduce token usage during PRD review and implementation by enforcing targeted
verification, canonical field paths, and bounded file reads before code changes
begin.

---

## SCOPE

- Add explicit review rules for Claude/Codex PRD handling
- Prevent unnecessary full-file reads
- Require function-level verification before implementation
- Require uncertain fields to be marked [UNVERIFIED] instead of guessed
- Preserve architecture verification quality

---

## OUT OF SCOPE

- No changes to trading logic
- No changes to runtime behavior
- No dashboard changes
- No pipeline changes
- No reduction in test coverage
- No skipping verification when field paths or injection points are uncertain
- Hook/prompt config changes (addressed separately outside this PRD)

---

## FILES

- `CLAUDE.md`
- `docs/AGENT_WORKFLOW.md`
- `docs/SCHEMA_MAP.md` (new)
- `docs/CALL_SITE_MAP.md` (new)

## REQUIREMENTS

### R1 — Spot-Read First Policy

Before reading any file, identify the exact functions or symbols that need
verification. Prefer function-level or offset/limit reads. Do not read entire
files unless:
- function location is unknown
- module structure is unknown
- symbol lookup fails
- multiple consumers require broad audit

**FAIL:** Full-file read performed before function targets are identified.

---

### R2 — Field Path Verification

PRDs must use verified field paths only. If a field path is uncertain, mark it
as `[UNVERIFIED]`. Do not invent fields without confirming schema. Verified
paths should be added to `docs/SCHEMA_MAP.md`.

**FAIL:** PRD contains an invented or unverified field path without `[UNVERIFIED]`.

---

### R3 — Injection Point Verification

Every PRD that changes behavior must name the exact injection point: function
name, file path, and before/after relationship.

Examples:
- `runtime.py`: after `build_pipeline_output_contract` returns
- `dashboard_renderer.py`: before candidate board rendering
- `payload.py`: after meta assembly

**FAIL:** PRD says "add to runtime" or "update dashboard" without exact location.

---

### R4 — Consumer Impact Check

Before implementation, identify every consumer affected by: new field, renamed
field, new function parameter, enum/constant change, or payload shape change.
Consumer list must be included in PRD or review notes.

**FAIL:** Field or signature changes implemented without consumer audit.

---

`docs/SCHEMA_MAP.md` is created with canonical paths for contract, payload,
market_map, run artifact, and notification-relevant fields. PRD authors must
reference it before proposing new fields.

**FAIL:** New canonical field introduced without updating schema map.

---

### R6 — Call-Site Map Maintenance

`docs/CALL_SITE_MAP.md` is created with known high-value function boundaries.

Minimum initial entries (all verified against source):

| File | Function | Line |
|---|---|---|
| `runtime.py` | `cli_main` | 256 |
| `runtime.py` | `_resolve_effective_mode` | 1879 |
| `runtime.py` | `_run_pipeline` | 634 |
| `contract.py` | `build_pipeline_output_contract` | 52 |
| `delivery/payload.py` | `build_report_payload` | 21 |
| `delivery/dashboard_renderer.py` | `render_dashboard_html` | 511 |
| `output.py` | `build_notification_message` | 811 |

**FAIL:** Reviewer performs broad file scan when target function exists in
`CALL_SITE_MAP.md`.

---

### R7 — Review Output Discipline

PRD review returns only: Strengths, Blockers, Exact fixes, Registry readiness
verdict. No broad summaries of unrelated modules.

**FAIL:** Review includes broad summaries of unrelated modules.

---

### R8 — Implementation Read Budget

Before implementation, state expected read targets (files, functions, reason
for each). If a full-file read is required, explain why.

**FAIL:** Implementation begins with broad exploratory reads.

---

### R9 — Preserve Verification Quality

Token reduction must not remove architecture verification. Expensive reads are
acceptable when they prevent wrong implementation.

**FAIL:** PRD accepted without verifying uncertain field paths or injection
points.
