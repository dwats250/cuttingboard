PRD-104 — Decision Logic and Artifact Flow Audit

STATUS
COMPLETE

GOAL
Create a deterministic documentation audit of Cutting Board's current decision logic,
artifact flow, artifact writers, artifact readers, sidecar boundaries, and mutation
risks so future PRDs can safely add sidecars without accidentally influencing existing
trade logic or artifacts.

SCOPE
This PRD adds documentation only.

Create:
- docs/system_logic_map.md
- docs/artifact_flow_map.md

The audit must document:

1. Decision flow
   - Identify the current runtime path from input data to final outcome.
   - Identify which modules can influence TRADES, NO TRADE, or HALT.
   - Identify which fields are decision-affecting versus display-only.
   - Identify where market regime, qualification, policy, and output materialization occur.

2. Artifact writers
   - Identify every source path that writes current repo artifacts, including at minimum:
     - logs/latest_hourly_contract.json
     - logs/latest_hourly_payload.json  (hourly run write path; runtime.py LATEST_HOURLY_PAYLOAD_PATH)
     - logs/latest_hourly_run.json      (hourly run summary; runtime.py LATEST_HOURLY_RUN_PATH)
     - logs/latest_payload.json         (non-hourly runs; transport._DEFAULT_JSON_PATH)
     - logs/latest_run.json             (non-hourly run summary; runtime.py LATEST_RUN_PATH)
     - logs/market_map.json
     - logs/macro_drivers_snapshot.json
     - logs/audit.jsonl
     - logs/evaluation.jsonl
     - logs/performance_summary.json
     - logs/sector_router_state.json
     - reports/output/report.html       (non-hourly transport default HTML; transport._DEFAULT_HTML_PATH)
     - reports/output/dashboard.html    (dashboard_renderer default local output; dashboard_renderer._OUTPUT_PATH)
     - ui/dashboard.html                (CI hourly only; dashboard_renderer called with --output ui/dashboard.html)
     - ui/index.html                    (CI publish step only; shell copy of ui/dashboard.html, not a Python writer)
   - For each artifact, document:
     - writer module/function where known
     - expected consumers
     - whether the artifact is runtime-critical, dashboard-only, notification-related, or audit-only
     - whether tests must isolate the path to tmp_path
   - Explicitly document the two-path payload relationship:
     - logs/latest_hourly_payload.json is the hourly write path
     - logs/latest_payload.json is the non-hourly write path (transport default) and the
       dashboard renderer's default read path; overridden to the hourly path in the CI workflow
   - Explicitly document the two-path run summary relationship:
     - logs/latest_hourly_run.json is the hourly write path
     - logs/latest_run.json is the non-hourly write path and the dashboard renderer's default
       read path; overridden to the hourly path in the CI workflow
   - Explicitly document the dashboard HTML path resolution:
     - reports/output/dashboard.html is the dashboard_renderer default (local use)
     - ui/dashboard.html is produced only when --output ui/dashboard.html is passed (CI hourly)
     - ui/index.html is a shell copy of ui/dashboard.html made during the CI publish step;
       it is not written by any Python function

3. Artifact readers
   - Identify modules that read existing artifacts.
   - Identify whether each reader is allowed to derive new logic or only render/report
     existing fields.
   - Explicitly document dashboard renderer artifact reads and sidecar reads.

4. Sidecar boundaries
   - Define the current sidecar rule:
     - sidecars may read existing artifacts
     - sidecars may write separate sidecar artifacts
     - sidecars must not mutate contracts, payloads, trade decisions, qualification
       gates, market_map grades, notifications, or existing dashboard artifacts unless
       a future PRD explicitly scopes that change
   - Document current sidecar-like artifacts, including market_map and macro snapshots
     where applicable.

5. Test artifact isolation
   - Document which tests or test helpers are responsible for preventing writes to
     production repo logs.
   - Document the required isolation pattern:
     - monkeypatch artifact paths to tmp_path
     - monkeypatch audit paths to tmp_path
     - monkeypatch transport delivery paths when runtime performs lazy imports
   - Document that tests must not leave modified files under logs/, reports/, or ui/
     after full-suite execution.

6. Future sidecar readiness
   - Add a short section describing how future sidecars should be introduced
     (illustrative, not exhaustive):
     - Sunday report
     - trend structure
     - custom watchlist
     - calendar context
     - news context
     - daily brief / recap
   - This section must be architectural only and must not implement any of those features.

OUT OF SCOPE
- No source code behavior changes.
- No trading decision changes.
- No contract schema changes.
- No payload schema changes.
- No market_map scoring changes.
- No dashboard rendering changes.
- No notification changes.
- No audit writer changes.
- No new sidecar artifacts.
- No new scheduled jobs.
- No implementation of Sunday report, trend structure, custom watchlist, news, calendar,
  daily brief, or recap.
- No cleanup of historical committed log contamination unless separately scoped.
- No changes to TRADES | NO TRADE | HALT outcome semantics.
