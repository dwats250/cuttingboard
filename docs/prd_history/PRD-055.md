# PRD-055 — Signal Forge: Dashboard Upgrade

STATUS
IN PROGRESS

GOAL
Rebuild `dashboard_renderer.py` to render a Signal Forge layout: a true cross-asset
macro tape, a restructured system state block, and a graded candidate visibility
board sourced from `logs/market_map.json`. No execution logic, decision logic,
qualification, or policy enforcement changes.

---

SCOPE
- Rebuild HTML layout in `cuttingboard/delivery/dashboard_renderer.py`
- Add `logs/market_map.json` as an optional third input (absent = explicit placeholder)
- Replace the misnamed `id="macro-tape"` block (currently shows system state fields,
  not cross-asset prices) with a true cross-asset tape using `payload.macro_drivers`
  and `market_map.symbols`
- Add `id="candidate-board"` section rendering graded market_map symbols
- Restructure `id="system-state"` to include outcome, permission, and stay_flat_reason
- Remove `id="decision-summary"`, `id="primary-setup"`, `id="secondary-setups"`,
  and `id="trade-decisions"` — superseded by candidate-board
- Preserve `id="dashboard-header"`, `id="run-delta"`, `id="run-history"`,
  `id="run-health"` with current fields intact
- Update `tests/test_dashboard_renderer.py` to cover new blocks

---

  `qualification.py`, `market_map.py`, or any non-dashboard file

- No broker integration
- No notification changes
- No new data fetches or data sources
- No AI commentary or subjective text generation
- No computed metrics (no arithmetic on prices, scores, or percentages beyond

REQUIREMENTS

R1 — Macro Tape Block

The renderer MUST produce `<div class="block" id="macro-tape">` near the top of
the page, above the system-state block.

The tape renders one labeled slot per data source in this fixed display order:

  Slot 1:  VIX   — source: payload.macro_drivers["volatility"]["change_pct"]
  Slot 2:  DXY   — source: payload.macro_drivers["dollar"]["change_pct"]
  Slot 3:  10Y   — source: payload.macro_drivers["rates"]["change_pct"]
  Slot 4:  BTC   — source: payload.macro_drivers["bitcoin"]["change_pct"]
  Slot 5:  SPY   — source: market_map["symbols"]["SPY"]["trade_framing"]["direction"]
  Slot 6:  QQQ   — source: market_map["symbols"]["QQQ"]["trade_framing"]["direction"]
  Slot 7:  GLD   — source: market_map["symbols"]["GLD"]["trade_framing"]["direction"]
  Slot 8:  SLV   — source: market_map["symbols"]["SLV"]["trade_framing"]["direction"]
  Slot 9:  XLE   — source: market_map["symbols"]["XLE"]["trade_framing"]["direction"]

Arrow derivation — macro_drivers slots (float change_pct):
  change_pct > 0  → "↑"
  change_pct < 0  → "↓"
  change_pct == 0 → "→"

Arrow derivation — market_map slots (string direction):
  "LONG"    → "↑"
  "SHORT"   → "↓"
  all other → "→"

Unavailable slot rules (applied independently per slot):
  - payload.macro_drivers == {} → slots 1–4 render label + "—"
  - market_map argument is None → slots 5–9 render label + "—"
  - Symbol key absent from market_map["symbols"] → that slot renders label + "—"
  - "trade_framing" key absent from symbol entry → that slot renders label + "—"
  - "direction" key absent from trade_framing → that slot renders label + "—"

Rendered format per slot (HTML inline, pipe-separated):
  `<span class="tape-slot">VIX ↑</span>`
  Example row: VIX ↓ | DXY ↑ | 10Y ↑ | BTC ↑ | SPY → | QQQ → | GLD ↑ | SLV ↑ | XLE ↓

The tape block MUST always render regardless of data availability. It MUST NOT
be conditionally omitted.

FAIL:
- id="macro-tape" absent from rendered HTML
- Any slot from 1–4 silently omitted when macro_drivers is non-empty
- Any slot from 5–9 silently omitted when market_map["symbols"] contains that key
- Dashboard raises when macro_drivers == {}
- Dashboard raises when market_map argument is None
- Arrow direction inverted relative to change_pct sign
- Tape block absent from rendered output

R1.1 — Macro Bias Summary

After rendering the macro tape, the renderer MUST compute a directional macro bias
summary using only arrow counts from all available tape slots.

Computation rules:
- Count all "↑" arrows → up_count
- Count all "↓" arrows → down_count
- Ignore "→" and "—" slots

Rendering rules:
- up_count > down_count → render "MACRO BIAS: LONG"
- down_count > up_count → render "MACRO BIAS: SHORT"
- up_count == down_count → render "MACRO BIAS: MIXED"

Output:
Render directly below the macro tape:

  `<div class="macro-bias">MACRO BIAS: LONG</div>`

Constraints:
- No weighting
- No external data
- No arithmetic beyond counting occurrences

FAIL:
- macro-bias element absent from rendered output
- bias derived from anything other than arrow counts
- incorrect classification when counts differ

R2 — System State Block

R2.1 — Action Summary Line

At the top of the system-state block, before all other fields, the renderer MUST
render a single action line.

Rules:
- If run["outcome"] == "STAY_FLAT":
    ACTION: WAIT — NO VALID SETUPS
- Else if run["permission"] == False:
    ACTION: WATCH — SETUPS PRESENT BUT BLOCKED
- Else if run["permission"] == True:
    ACTION: ACTIVE — TRADE CONDITIONS MET
- Else:
    ACTION: MONITOR — SYSTEM ACTIVE

Output:

  `<div class="action-line">ACTION: WATCH — SETUPS PRESENT BUT BLOCKED</div>`

Constraints:
- Must be first child of system-state block
- No additional interpretation logic beyond specified rules

FAIL:
- action-line element absent from rendered output
- incorrect message for given state
- rendered below other fields

The renderer MUST produce `<div class="block" id="system-state">` with exactly
these fields, in this order:

  1. REGIME     — payload.summary.market_regime         (badge with CSS class = value)
  2. POSTURE    — run["posture"]                         (badge with CSS class = value)
  3. CONFIDENCE — run["confidence"]                      (rendered as-is, no rounding)
  4. OUTCOME    — run["outcome"] if key exists else run["status"]
  5. PERMISSION — run["permission"] if key exists and value is non-None; else omit row
  6. STAY FLAT  — payload.sections.validation_halt_detail["reason"] if non-null dict;
                  else omit row

Fields 5 and 6 are omitted (not rendered) when their source value is None or absent.
Fields 1–4 are always rendered.

The existing `id="decision-summary"` block is REMOVED. Its content is superseded
by the system-state block. The block ID "decision-summary" MUST NOT appear in output.

FAIL:
- id="system-state" absent from rendered HTML
- REGIME or POSTURE rendered without badge CSS class matching the value string
- Field 5 or 6 rendered when source is None or key absent
- Any of fields 1–4 absent
- id="decision-summary" present in rendered output

R3 — Candidate Visibility Board

The renderer MUST produce `<div class="block" id="candidate-board">`.

Case A — market_map argument is None (file absent):
  Render the block with exactly:
  `<div class="unavailable">MARKET MAP UNAVAILABLE</div>`
  No other content in the block.

Case B — market_map["symbols"] is an empty dict {}:
  Render the block with exactly:
  `<div class="unavailable">NO SYMBOL DATA</div>`

Case C — market_map["symbols"] is non-empty:
  Render one candidate card per symbol. See R4 for card fields.
  Sort order (deterministic):
    Primary key:   grade enum position (see _GRADE_ORDER in R6)
    Secondary key: symbol name alphabetical ascending
    No other sort key permitted.

R3.1 — Tier Grouping

Candidate cards MUST be grouped into tier sections in the following order:

  A+ → A → B → C → D/F

Each group MUST render as:

  `<div class="tier-group" id="tier-{name}">`
  `  <div class="tier-header">{LABEL}</div>`
  `  ...cards...`
  `</div>`

Exact labels:
  - A+ → "A+ — ACTIONABLE"
  - A  → "A — HIGH QUALITY"
  - B  → "B — DEVELOPING"
  - C  → "C — EARLY"
  - D/F → "D/F — FAILING"

Rules:
- Cards are appended into their respective group
- D and F grade cards both belong to the D/F group (id="tier-df")
- Empty groups MUST NOT render
- Group order MUST remain fixed regardless of content

FAIL:
- cards rendered without tier-group wrapping
- empty groups rendered
- incorrect ordering of groups
- D or F grade cards not in the D/F group

FAIL:
- id="candidate-board" absent from rendered HTML
- Case A does not contain the exact string "MARKET MAP UNAVAILABLE"
- Case B does not contain the exact string "NO SYMBOL DATA"
- Case C symbols not sorted by grade position then alphabetical
- Sort uses any field other than grade position and symbol name
- A symbol present in market_map["symbols"] has no card rendered

R4 — Candidate Card Fields

Each candidate card MUST be wrapped in:
  `<div class="candidate-card grade-{css_class}" id="card-{symbol}">`

Grade to CSS class mapping (exact):
  "A+" → "aplus"
  "A"  → "a"
  "B"  → "b"
  "C"  → "c"
  "D"  → "d"
  "F"  → "f"
  any unrecognized grade → "unknown"

Fields rendered in ALL cards (grades A+ through F):
  - symbol     (market_map["symbols"][s]["symbol"])
  - grade      (market_map["symbols"][s]["grade"])
  - bias       (market_map["symbols"][s]["bias"])
  - structure  (market_map["symbols"][s]["structure"])

Fields rendered ONLY in grade A+, A, B cards (omitted entirely in C, D, F):
  - setup_state          when value is non-null AND value != "DATA_UNAVAILABLE"
  - trade_framing.if_now when non-null                  (label: "IF NOW")
  - trade_framing.entry  when non-null                  (label: "ENTRY")
  - invalidation[0]      when list is non-null and non-empty  (label: "INVALIDATION")
  - trade_framing.downgrade when non-null               (label: "WATCH FOR")
  - reason_for_grade     when non-null

Fields that do not exist in market_map (rr_ratio, stop_price as numeric value,
trigger_level as numeric value) MUST NOT be rendered. Do not substitute
null placeholders for absent fields.

All rendered field values MUST pass through _esc() before insertion into HTML.

R4.1 — Candidate State Emphasis

For A+, A, and B cards:

1. setup_state:
   - MUST be rendered when value is non-null and != "DATA_UNAVAILABLE"
   - Render as:
       `<div class="candidate-state">STATE: {setup_state}</div>`
   - Must appear above all other optional fields

2. trade_framing.downgrade:
   - MUST be rendered when non-null
   - Label MUST be "RISK" (not "WATCH FOR" as listed in R4)
   - Render as:
       `<div class="candidate-risk">RISK: {value}</div>`

3. Ordering within A+, A, B cards:
   - candidate-state appears first (if present)
   - candidate-risk appears after state

Constraints:
- Do NOT render candidate-state or candidate-risk in C, D, or F cards
- Do NOT compute or derive values

FAIL:
- candidate-state rendered in C, D, or F cards
- candidate-risk rendered in C, D, or F cards
- Label "WATCH FOR" used instead of "RISK" for trade_framing.downgrade
- candidate-risk rendered before candidate-state when both present

FAIL:
- Any card missing id="card-{symbol}"
- Any card missing its grade CSS class
- C, D, or F card renders trade_framing, invalidation, or reason_for_grade
- Any field value computed or derived rather than read directly from market_map
- rr_ratio, stop_price, or trigger_level rendered as numeric values

R6 — Sorting Contract

A module-level constant MUST be defined:

  _GRADE_ORDER = {"A+": 0, "A": 1, "B": 2, "C": 3, "D": 4, "F": 5}

Sort key applied to list(market_map["symbols"].keys()):

  key=lambda sym: (_GRADE_ORDER.get(
      market_map["symbols"][sym].get("grade", ""), 6
  ), sym)

No other sort key is used. The sort MUST be stable and deterministic.

FAIL:
- _GRADE_ORDER constant absent from module
- Sort key uses any field not derivable from _GRADE_ORDER.get(grade) and sym
- Two identical market_map["symbols"] inputs produce different card order

R8 — No Computation in Renderer

The renderer MUST NOT compute, infer, or derive any value beyond:
  - String formatting via f-strings and _esc()
  - Arrow derivation from sign comparison (change_pct > 0, < 0, == 0)
  - Grade sort position lookup from _GRADE_ORDER constant
  - None/key-existence checks for optional field rendering
  - HTML escaping

Prohibited in renderer:
  - Arithmetic on level, change_pct, confidence, score, or any numeric field
  - Regime or posture classification logic
  - String construction that combines multiple source fields into a new derived metric

FAIL:
- Any expression `a OP b` where OP is +, -, *, / and a or b are sourced from
  payload or market_map numeric fields (excluding change_pct sign comparison)
- Any conditional that selects regime or posture classification
