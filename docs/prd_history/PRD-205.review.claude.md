## REQUIRED EDITS

1. **R3 was too narrow — RESOLVED in revision.**
   As reviewed at `2118426`, R3 fell back to CI only on "connectivity/egress
   failure." The capability gate checks credential EXISTENCE, not VALIDITY, so a
   stale/expired/revoked `~/.codex/auth.json` or `OPENAI_API_KEY` passes the gate,
   then `codex exec` fails with a 401 AUTH error — which is reachable-but-erroring,
   NOT connectivity. Under the old R3 that case does not fall back, so the command
   fails on a host even though the CI route would succeed — directly falsifying the
   goal "works from any environment."
   RESOLUTION: R3 now falls back on ANY local-ENVIRONMENT failure the CI route
   could satisfy (egress, auth/401, stale-or-revoked credential, missing/broken
   `codex` binary), with an optional exit-code discrimination that surfaces
   invalid-TARGET failures instead of burning a CI dispatch (and documents the
   burns-a-dispatch trade-off when discrimination isn't cheap). DATA FLOW and FAIL
   CONDITIONS were synced to match.
