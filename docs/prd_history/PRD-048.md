# PRD-048 — Trade Decision Visibility in Payload and Dashboard

**Status:** IN PROGRESS
**Commit:** —
**Author:** Dustin Watson
**Date:** 2026-04-28

---

## GOAL

Surface explicit trade decisions and first-failure explanations from the canonical contract into the downstream report payload and slim dashboard.

This is a downstream visibility PRD only. The contract already contains `decision_status`, `block_reason`, and `decision_trace` from PRD-045 and PRD-046. The payload/dashboard path must stop collapsing that information into older generic slices.

---

## SCOPE

- Add a dedicated payload section for trade decision detail, sourced from `contract["trade_candidates"]`
- Render a read-only dashboard block that shows each candidate's decision outcome and deterministic trace
- Preserve existing dashboard summary/primary/secondary behavior
- Add payload and dashboard tests for exact field mapping and no-interpretation behavior

---

## OUT OF SCOPE

- Contract schema changes
- Qualification, options, chain, or runtime logic changes
- Reordering trade candidates
- New trade filtering or ranking logic
- New top-level run fields
- Editing alert/report prose outside the dashboard renderer

---

## REQUIREMENTS

### R1 — Dedicated Payload Surface

`build_report_payload()` must add a new section:

```python
payload["sections"]["trade_decision_detail"]
```

This must be a list derived directly from `contract["trade_candidates"]`, in the same order, with one dict per candidate containing exactly:

```python
{
    "symbol": candidate["symbol"],
    "direction": candidate["direction"],
    "strategy_tag": candidate["strategy_tag"],
    "entry_mode": candidate["entry_mode"],
    "decision_status": candidate["decision_status"],
    "block_reason": candidate["block_reason"],
    "decision_trace": candidate["decision_trace"],
}
```

FAIL: this section is absent.
FAIL: any item is re-sorted, filtered, enriched, or inferred.
FAIL: `decision_trace` is flattened into strings in the payload builder.

### R2 — Payload Validation

`assert_valid_payload()` must require:

- `sections["trade_decision_detail"]` exists and is a list
- each item is a dict with exactly the keys listed in R1
- `symbol`, `direction`, `strategy_tag`, `entry_mode`, `decision_status` are strings
- `block_reason` is `None` or a non-empty string
- `decision_trace` is a dict with exactly keys `stage`, `source`, `reason`
- all `decision_trace` values are non-empty strings

FAIL: malformed decision detail entries pass validation.
FAIL: payload validation allows missing or extra keys in `decision_trace`.

### R3 — Dashboard Trade Decisions Block

`render_dashboard_html()` must render a new read-only block:

```html
<div class="block" id="trade-decisions">
```

Source only:

```python
payload["sections"]["trade_decision_detail"]
```

Per candidate, render exact field values only:

- symbol
- direction
- strategy_tag
- entry_mode
- decision_status
- block_reason when non-null
- decision_trace.stage
- decision_trace.source
- decision_trace.reason

No run-derived substitutions. No explanatory prose. No inferred labels like "good setup", "failed due to liquidity", or "best trade".

FAIL: renderer reads `top_trades` or raw contract data for this block.
FAIL: renderer hides blocked candidates.
FAIL: renderer adds interpretation beyond literal field display.

### R4 — Visibility Rules

- If `trade_decision_detail` is empty, omit the `trade-decisions` block entirely
- If `block_reason` is `None`, do not render a `Block Reason` label for that candidate
- If `block_reason` is present, it must match `decision_trace["reason"]` exactly in the rendered output

FAIL: empty-state filler prose is added.
FAIL: ALLOW_TRADE candidates render a blank `Block Reason` row.

### R5 — Existing Dashboard Blocks Stay Intact

This PRD does not replace:

- `decision-summary`
- `primary-setup`
- `secondary-setups`
- `macro-tape`
- `system-state`
- `run-health`

Those blocks keep their current source rules. The new trade decision block is additive.

FAIL: existing block sourcing changes as part of this PRD.
FAIL: primary/secondary setup logic is rewritten to depend on new payload fields.

---

## DATA FLOW

```text
contract["trade_candidates"][n]
    -> payload["sections"]["trade_decision_detail"][n]
    -> dashboard block #trade-decisions
```

The payload builder is the only transformation point. The dashboard remains read-only.

---

## ACCEPTANCE CRITERIA

**AC1** — Payload contains `sections.trade_decision_detail` with one entry per `trade_candidate`, preserving order.

**AC2** — Each payload entry includes `decision_status`, `block_reason`, and structured `decision_trace`.

**AC3** — Payload validation fails on malformed decision detail or malformed trace keys.

**AC4** — Dashboard renders a `trade-decisions` block when decision detail exists.

**AC5** — Dashboard omits the block entirely when no decision detail exists.

**AC6** — Dashboard shows blocked candidates and their exact first-failure reason.

**AC7** — Dashboard adds no interpretation beyond literal field display.

**AC8** — Existing dashboard summary, primary setup, and secondary setup behavior remain unchanged.
