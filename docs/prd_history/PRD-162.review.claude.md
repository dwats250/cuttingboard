# PRD-162 Claude Review

VERDICT
ACCEPT WITH CHANGES

SUMMARY
PRD-162 (narrowed to actionable-trade reconciliation) correctly identifies a real
class of cross-field contradiction and is tightly scoped after the AC5 split —
the renderer and integrator are correctly excluded, and the non-goals fence out
notification/W-spine/RUN-SNAPSHOT/AC5 sprawl (focus #1, #4, #7: good). The
weaknesses are all in the *outcome-derivation mechanics*: the PRD treats the two
outcome sites and the `top_trades` gate as if a single `TradeDecision`-typed
predicate cleanly serves all three, but `output.py`'s outcome reads the
already-built payload (so it needs no edit and is dict-typed, not
`TradeDecision`-typed), the `runtime.py` outcome site does not currently apply the
NON_TRADABLE exclusion (so AC1 agreement is not guaranteed), and `size_multiplier=0`
is not independently realizable alongside `ALLOW_TRADE`. AC3 and the PRD-161
interaction-risk are also left non-binary. None of these is fatal; they are
specification gaps that would otherwise surface as mid-implementation scope-lock
amendments.

REQUIRED EDITS

1. **Remove `cuttingboard/output.py` from production FILES (or downgrade to
   verify-only).** PRD-162.md:68-69 instructs editing `output.py:478-484` to "replace
   the `top_trades`-non-empty rule with the shared actionable predicate." But
   `output.py:480-484` is inside `render_report_from_payload(payload)` and reads
   `sections = payload.get("sections", {})` (`output.py:465`) — i.e. the
   already-assembled payload. Once `payload.py:40` gates `top_trades` to actionable
   candidates, `output.py`'s existing `elif sections.get("top_trades")` rule is
   correct unchanged; and `output.py` holds candidate **dicts**, not `TradeDecision`
   objects, so applying a `TradeDecision`-typed predicate there is type-incoherent.
   FAIL: `cuttingboard/output.py` remains in PRD-162 production FILES with a
   code-edit instruction.

2. **Require the same actionable definition — including NON_TRADABLE exclusion — at
   the `runtime.py:875` outcome site, or the two outcome sites can disagree.**
   `runtime.py:875` derives outcome via `any(decision.status == ALLOW_TRADE for
   decision in trade_decisions)` — no `size_multiplier` and no NON_TRADABLE check —
   while the `payload.py:40` gate (PRD-162.md:73-76) additionally excludes
   `symbol ∈ NON_TRADABLE_SYMBOLS`. A `^VIX` (∈ `NON_TRADABLE_SYMBOLS`) candidate
   already reaches an `OptionSetup`/decision via the continuation path; if its live
   chain validates to `ALLOW_TRADE`, `runtime.py` outcome = `TRADE` while the gated
   `top_trades` is empty → `output.py` outcome = `NO_TRADE`. FAIL: PRD-162 does not
   state that `runtime.py:875` applies the same actionable definition (incl.
   NON_TRADABLE exclusion) as the `top_trades` gate, so AC1 (PRD-162.md:122-125)
   "agreement" is not guaranteed and is untestable as written.

3. **Specify how one predicate definition serves both a `TradeDecision` and a
   candidate dict.** PRD-162.md:65-67 places the predicate in `trade_decision.py`
   operating on a `TradeDecision` (fields `status`, `size_multiplier` confirmed at
   `trade_decision.py:11-12,42`), but the `top_trades` gate (`payload.py:40`)
   operates on candidate **dicts** with keys `decision_status` / `size_multiplier` /
   `symbol` (`contract.py:332,337`). FAIL: PRD-162 asserts "one definition, two
   sites" without naming the representation bridge (a shared helper over the common
   fields, or an explicit dict adapter), so the single-definition claim is not
   verifiable.

4. **Declare the `size_multiplier > 0` clause defensive, or cite a realistic
   `ALLOW_TRADE`+`size_multiplier=0` path.** Under current `execution_policy`
   routing, every zero-size branch returns `PolicyDecision(False, …)` → `BLOCK_TRADE`
   (`execution_policy.py:217-227`), and `ALLOW_TRADE` requires `confidence ≥ 0.60`
   → `size_multiplier_for_confidence ≥ 0.5` (`execution_policy.py:62-68,215`), so
   `status == ALLOW_TRADE ⟹ size_multiplier > 0`. The `size_multiplier > 0` clause in
   the predicate (PRD-162.md:66) and AC3 (PRD-162.md:129-130) is therefore redundant
   with `status == ALLOW_TRADE` and never independently active today. FAIL: per
   `CLAUDE.md § PRD-author disciplines → Realizability check`, PRD-162 presents
   `size_multiplier=0.0` as an independent actionable-exclusion without either
   declaring it defensive-against-future-routing or citing a current input path that
   yields `ALLOW_TRADE` with `size_multiplier == 0`.

5. **Make AC3 and the PRD-161 re-point target single-valued.** AC3 (PRD-162.md:129-133)
   permits "suppressed from `top_trades`, **or** clearly labeled blocked," and the
   Interaction-risk re-point (PRD-162.md:113-117) lists "`contract.trade_candidates`
   **or** an observational `blocked`/`rejected` section." FAIL: AC3 and the
   Interaction-risk section each admit ≥2 mutually-exclusive implementations, leaving
   the `top_trades` membership of a `BLOCK_TRADE` candidate undefined and the PRD-161
   gate-test re-point target indeterminate (non-binary acceptance).

6. **Add an explicit runtime/output outcome-agreement test to FILES.** The test list
   (PRD-162.md:82-99) covers the predicate, payload gating, contract shape, runtime
   outcome, and the NO_TRADE idle render, but no test pins that the `runtime.py:875`
   outcome and the `output.py:482` (gated-`top_trades`) outcome are **equal** for the
   same decision set. FAIL: no FILES test asserts runtime-path/output-path outcome
   equality, so AC1's agreement claim (PRD-162.md:122-125) has no covering test.

RECOMMENDED EDITS

1. **Restate AC4 data-layer-primary.** AC4 (PRD-162.md:134-136) is phrased on "the
   rendered `top_trades`/decision surface" while no renderer file is in production
   FILES. FAIL-style restatement: "when no actionable candidate exists, the payload
   `sections.top_trades` is empty," with `tests/test_dash_system_state.py` verifying
   the existing renderer shows the idle state. Keeps AC4 testable without implying a
   renderer code change.

2. **Resolve the `contract.py` conditional now.** PRD-162.md:77-80 leaves
   `contract.py` "conditional — only if D1/D2 gating is applied at candidate
   construction." Deciding the single gate location (the `payload.py:40` chokepoint
   vs `_build_trade_candidates` at `contract.py:285`) before implementation avoids
   the mid-implementation FILES amendment pattern (cf. PRD-136 / PRD-158).

3. **Reconcile AC1's `market_map` framing with the actual mechanism.** AC1
   (PRD-162.md:122-125) is titled "verdict/market_map agreement" and references
   `market_map` qualifying candidates, but `cuttingboard/market_map.py` is not in
   FILES and the mechanism centers on `trade_decisions`/`top_trades`. Restate AC1 in
   terms of the actionable `top_trades`/decision set (the surface the FILES actually
   change), or add `market_map` to the agreement definition explicitly.

RATIONALE

- REQUIRED 1 is a FILES-minimality and correctness issue (`CLAUDE.md § Strict scope
  locking`: "A PRD's FILES section is a hard boundary"). Listing `output.py` with an
  edit instruction that is both unnecessary (transitive correctness via the payload
  gate) and type-incoherent (dicts vs `TradeDecision`) would either produce a no-op
  diff outside intent or force a mid-implementation re-think.
- REQUIRED 2 is the core of AC1. The PRD's own evidence (the `^VIX` case from the
  PRD-157 gate report) is precisely a NON_TRADABLE symbol reaching the candidate
  surface; if the actionable definition lives only at `top_trades` and not at the
  decision-level outcome site, the two outcomes diverge exactly on the case the PRD
  exists to fix. This is the `CLAUDE.md § Downstream-consumer audit` discipline
  applied to the outcome emission's two producers.
- REQUIRED 3 and 4 are `CLAUDE.md § PRD-author disciplines` (Downstream-consumer
  audit + Realizability check). A "shared predicate" that cannot be applied uniformly
  to both representations is not actually shared; and an exclusion clause with no
  realizable input path is "dead code with extra steps" per the Realizability rule.
- REQUIRED 5 and 6 are binary-acceptance requirements (`CLAUDE.md § PRD process`:
  acceptance criteria must be observable). AC3's "suppress OR label" fork directly
  determines the PRD-161 interaction outcome; leaving both open means the PRD-161
  gate-test cannot be deterministically updated, which the PRD itself flags as
  in-scope (PRD-162.md:118) but does not resolve.
- Focus-question coverage: #1 scope — tight (renderer/integrator correctly out);
  #2 ACs binary — mostly, except AC3 (REQUIRED 5); #3 predicate defined — partially,
  representation + realizability gaps (REQUIRED 3, 4); #4 no redesign drift — clean;
  #5 PRD-161 interaction — identified but non-determinate (REQUIRED 5); #6 FILES
  accurate/minimal — `output.py` over-included (REQUIRED 1), `contract.py`
  unresolved (RECOMMENDED 2); #7 non-goals — strong; #8 tests — missing the
  outcome-agreement test (REQUIRED 6), otherwise sufficient once REQUIRED 2/5 pin
  the surfaces.
