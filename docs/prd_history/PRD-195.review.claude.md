## Correctness review

- **Retention ordering.** Files are `run_YYYY-MM-DD_HHMMSS.json`; a C-locale
  lexicographic sort equals chronological order, so "keep the newest N, drop the
  oldest k" is correct. The current run's file carries the latest timestamp and
  is therefore always retained — the prune only ever removes strictly-older
  files. (Confirmed by `test_prunes_oldest_beyond_retain` /
  `test_keeps_exactly_retain_count`.)
- **Staging / commit integration.** The prune `git rm -f`s in `$wt`, so deletions
  are staged and join the publish commit. Placed before the no-net-change check, a
  prune-only delta still lands. (`test_prunes_oldest_beyond_retain` asserts the
  staged `D` entries; the hygiene guard asserts the ordering.)
- **Scope of the glob.** Only `logs/run_*.json` matches;
  `audit.jsonl`/`latest_*`/`regime_history`/macro snapshots are never touched.
  (`test_non_run_logs_are_never_pruned`.)
- **Hardening.** Non-numeric retain → default 50; retain floored at 1 so a
  misconfiguration can never wipe the whole history; empty/`<=retain` → clean
  no-op exit 0. The empty-array-under-`set -u` phantom-element trap is avoided by
  sorting only after the `<= retain` early-return.
  (`test_non_numeric_retain_falls_back_to_default`, `test_zero_retain_is_floored
  _to_one`, `test_noop_*`.)
- **Idempotency / retries.** `attempt_publish` rebuilds the worktree from the
  current publish tip on each attempt and re-runs the prune; the prune is a pure
  function of the worktree contents, so retries converge.
- **CWD independence.** The call resolves via `SCRIPT_DIR` (`BASH_SOURCE`), not a
  relative path, so it is correct regardless of the workflow's working directory.
