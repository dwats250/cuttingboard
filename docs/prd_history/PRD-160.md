# PRD-160: Fix macro_bias arrow-counting inversion

Lane: MICRO / CONTRACT
Status: Stage 0 — Active
Estimated size: ~30 LOC + tests
Author: Dustin
Filed: 2026-05-28
Supersedes: PRD-160 candidate stub (docs/PROJECT_STATE.md:57)

## Problem

The macro_bias computation counts a falling arrow on every macro driver (VIX, DXY, Rates/10Y) as bearish uniformly. This is incorrect for contra-cyclical drivers:

- A falling VIX is risk-ON (less fear), not bearish
- A falling DXY is risk-ON (weaker dollar supports risk assets)
- A falling 10Y is generally risk-ON for equities (lower discount rate)

Result: the dashboard renders MACRO BIAS: SHORT while the sub-signal phrases beneath it correctly say "VIX permits longs", "DXY supports risk-on", etc. Two parts of the same block contradict each other on the same screen. Corroborated by 2026-05-28 fixture and live renders.

## Root cause

`macro_bias` arithmetic treats arrow direction as monotonically signed across all drivers, instead of applying per-driver semantics (some drivers are pro-cyclical, some contra-cyclical relative to risk).

## Compensating workaround to unwind

PRD-158 made the dashboard integrator use the SAME flawed arithmetic as macro_bias so Rule 3 (consistency check) would suppress the visible label when the contradiction would otherwise show. This was a deliberate stopgap, not a fix.

PRD-160 MUST unwind this workaround. Otherwise:
- The integrator and macro_bias will both compute correctly post-fix and Rule 3 will never trip, masking any future divergence
- OR the workaround will double-suppress and the label will go missing in cases where it should render

Required: identify the PRD-158 workaround in the integrator, document it inline with a reference to this PRD, and replace it with the corrected per-driver logic. Verify Rule 3 still fires correctly on genuinely divergent states.

## Per-driver semantics (proposed)

Pro-cyclical (rising = risk-on, falling = risk-off): XAU, XAG, BTC, OIL
Contra-cyclical (rising = risk-off, falling = risk-on): VIX, DXY, 10Y

This list lives in config, not hardcoded in the arithmetic, so future additions don't require touching the bias function.

## Acceptance criteria

- macro_bias produces SHORT only when contra-cyclical drivers are rising AND pro-cyclical drivers are falling (or sufficient majority)
- Sub-signal phrases ("VIX permits longs" etc.) and the headline MACRO BIAS label never contradict in any rendered output
- Test: fixture with VIX↓ DXY↓ 10Y↓ XAU↑ XAG↑ → macro_bias = LONG (currently produces SHORT; this is the failing case)
- Test: fixture with VIX↑ DXY↑ 10Y↑ XAU↓ XAG↓ → macro_bias = SHORT
- Test: mixed/conflicting → macro_bias = NEUTRAL with Rule 3 firing
- PRD-158 integrator workaround removed and tagged in commit message
- Full suite passes; baseline 2444 maintained or increased

## Non-goals

- Not adding new drivers
- Not changing the visual rendering of the macro block
- Not touching outcome/market_map logic (that's PRD-162)

### Config-placement decision

PRD text says the cyclicality list "lives in config." It lands in
`cuttingboard/delivery/macro_tape_layout.py` rather than
`cuttingboard/config.py` because the bias arithmetic keys on
payload_key (`volatility` / `dollar` / `rates` / `bitcoin`), which is the
namespace `macro_tape_layout.py` already owns (`TapeSlot.payload_key`).
`config.MACRO_DRIVERS` uses quote symbols (`^VIX`, `DX-Y.NYB`, …), a
different namespace, so co-locating with the tape layout keeps the data
in one key-space and next to the existing macro-tape data definitions.
