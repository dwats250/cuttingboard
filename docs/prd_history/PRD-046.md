# PRD-046 — Decision Trace (First-Failure Explanation)

**Status:** IN PROGRESS
**Commit:** —
**Author:** Dustin Watson
**Date:** 2026-04-28

---

## GOAL

Make every `TradeDecision` explicitly explainable by attaching a structured `decision_trace` that records the deterministic, single-source-of-truth reason a candidate was `ALLOW_TRADE` or `BLOCK_TRADE`.

This is the canonical explanation layer. No new logic, no new gates, no UI.

---

## SCOPE

- Add `decision_trace` field to `TradeDecision` dataclass
- Populate trace in `create_trade_decision()` for all candidates
- Persist trace into `contract["trade_candidates"][n]`
- Extend `assert_valid_contract()` to enforce trace invariants
- Add `decision_trace` to audit output

---

## OUT OF SCOPE

- New signals or indicators
- Changing qualification or chain gate logic
- Post-trade evaluation
- UI or dashboard changes
- Multi-reason explanations (exactly one reason per candidate)
- Reordering or modifying gate logic
- New BLOCK conditions

## REQUIREMENTS

### R1 — decision_trace Field on TradeDecision

Add to the `TradeDecision` dataclass in `trade_decision.py`:

```python
decision_trace: dict  # keys: stage, source, reason — all non-empty strings
```

`TradeDecision.__post_init__` must validate:
- `decision_trace` has exactly the keys: `stage`, `source`, `reason`
- all three values are non-empty strings

FAIL: `decision_trace` absent from `TradeDecision`.
FAIL: `__post_init__` does not validate trace keys and values.

---

### R2 — Trace Generation in create_trade_decision()

Replace the current unstructured `block_reason` derivation with structured trace generation:

```python
if status == ALLOW_TRADE:
    decision_trace = {
        "stage":  "CHAIN_VALIDATION",
        "source": "chain_validation",
        "reason": "TOP_TRADE_VALIDATED",
    }

else:  # BLOCK_TRADE
    decision_trace = {
        "stage":  "CHAIN_VALIDATION",
        "source": "chain_validation",
        "reason": chain.reason or chain.classification,
    }
```

`block_reason` must remain consistent with the trace:

```python
block_reason = None if status == ALLOW_TRADE else decision_trace["reason"]
```

FAIL: any branch other than these two exists in trace generation.
FAIL: `block_reason != decision_trace["reason"]` for any `BLOCK_TRADE` decision.
FAIL: a fallback / UNKNOWN branch is added.
FAIL: `result.rejection_reason` is used as a trace source (unreachable in current pipeline).

---

### R3 — Contract Materialization

`_build_candidates()` in `contract.py` must add `decision_trace` to each candidate dict:

```python
{
    ...
    "decision_status":  decision.status,          # from PRD-045
    "block_reason":     decision.block_reason,    # from PRD-045
    "decision_trace":   decision.decision_trace,  # NEW — plain dict, serializes natively
}
```

FAIL: `decision_trace` absent from any candidate dict.
FAIL: `decision_trace` does not match the `TradeDecision.decision_trace` value exactly.

---

### R4 — Contract Validation

Extend `assert_valid_contract()` for each item in `trade_candidates`:

```
decision_trace exists
decision_trace has exactly keys: stage, source, reason
all values are non-empty strings

if decision_status == BLOCK_TRADE:
    block_reason == decision_trace["reason"]

if decision_status == ALLOW_TRADE:
    block_reason is None
    decision_trace["reason"] == "TOP_TRADE_VALIDATED"
```

The `decision_status` and `block_reason` checks from PRD-045 remain unchanged. Only `decision_trace` checks are added.

FAIL: `assert_valid_contract()` passes a contract with `decision_trace` missing or malformed.
FAIL: `assert_valid_contract()` passes a `BLOCK_TRADE` candidate where `block_reason != decision_trace["reason"]`.

---

### R5 — Audit Output

`audit.py` explicitly maps `TradeDecision` fields per candidate (lines 127–132). Add:

```python
entry["decision_trace"] = decision.decision_trace
```

`decision_trace` is a plain dict of strings — no custom serialization needed.

FAIL: `decision_trace` absent from audit JSONL output.
FAIL: audit `decision_trace` value differs from contract value for the same candidate.

## DATA FLOW

```
chain.classification == VALIDATED
         ↓
create_trade_decision()
         ↓
TradeDecision.decision_trace = {stage, source, reason}
TradeDecision.block_reason   = decision_trace["reason"] or None
         ↓
contract["trade_candidates"][n]["decision_trace"]  ← _build_candidates()
audit JSONL entry["decision_trace"]                ← audit.py
```

---

## FIRST-FAILURE PRINCIPLE

- Exactly one reason per candidate
- Must reflect earliest failing stage
- No aggregation, no lists, no concatenation
- Only two possible stages exist in the current pipeline: CHAIN_VALIDATION (fail) and CHAIN_VALIDATION (pass)
- Any future block source requires a new PRD to extend the trace logic

---

## FAIL CONDITIONS

- `decision_trace` absent from `TradeDecision`, contract candidate, or audit entry
- `block_reason != decision_trace["reason"]` for any `BLOCK_TRADE` candidate
- ALLOW_TRADE candidate has `decision_trace["reason"] != "TOP_TRADE_VALIDATED"`
- More than two trace generation branches in `create_trade_decision()`
- `result.rejection_reason` referenced in trace logic
- UNKNOWN / fallback branch present
- Qualification or chain logic modified
- Existing tests regress

---

## ACCEPTANCE CRITERIA

**AC1** — Every `TradeDecision` has a `decision_trace` with keys `stage`, `source`, `reason`.

**AC2** — Each candidate has exactly one reason string.

**AC3** — `BLOCK_TRADE` reason is always sourced from `chain.reason or chain.classification`.

**AC4** — `block_reason == decision_trace["reason"]` for all `BLOCK_TRADE` decisions.

**AC5** — `ALLOW_TRADE` candidates have `decision_trace["reason"] == "TOP_TRADE_VALIDATED"` and `block_reason == None`.

**AC6** — Same inputs produce identical `decision_trace`.

**AC7** — Qualification and chain outputs unchanged.

**AC8** — `decision_trace` appears in audit JSONL unchanged from contract value.

**AC9** — Full test suite passes with no regressions.
