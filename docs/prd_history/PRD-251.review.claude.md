SUMMARY
PRD-251 gives credit spreads (BULL_PUT_SPREAD, BEAR_CALL_SPREAD) a real
max-loss figure (width minus estimated credit) instead of sizing them off the
debit-proxy value, and threads that single figure from candidate-build time
through Gate 8 into the final `build_option_setups` resize so the two never
independently diverge for the standard qualify_candidate path. R1 and R2 are
implemented correctly and are covered by real, verified-passing tests; the
new `TradeCandidate.max_loss` field is `Optional[float] = None` and every
pre-existing `TradeCandidate(...)` construction site (11 call sites across
`tests/`) is unaffected because both `qualify_candidate` and
`build_option_setups` fall back to `spread_width`-based sizing when
`max_loss` is absent. The one substantive problem: `build_option_setups`'s
`candidate is None` fallback branch (options.py:229-232) now also resolves
the strategy-aware max loss for *any* `QualificationResult` with no matching
`TradeCandidate` — which in production is exactly the shape of a
continuation-path candidate (`_qualify_continuation_candidate`,
qualification.py:642, never populates the `candidates` dict). This silently
changes the final rendered `dollar_risk`/`max_contracts` for continuation
trades that land as a credit strategy, a case the PRD's own sibling proposal
doc (`PRD-251.continuation-path.proposal.md`) identifies as a live, current
routing path — while Gate 8 for that same trade keeps using the untouched
ATR-based proxy. That recreates, in the one path this PRD explicitly
excludes, exactly the Gate-8-vs-final-render divergence R3 exists to
eliminate.

RECOMMENDED EDITS

1. **Rendered `OptionSetup.spread_width` and the value driving
   `OptionSetup.dollar_risk` diverge for the same fallback branch.**
   In `build_option_setups` (options.py), `spread_width` (rendered field,
   line ~204) stays `_estimated_debit(strike_distance)` for the
   `candidate is None` case, while `risk_per_contract` (line 234) now uses
   `_max_loss_for_strategy(strategy, strike_distance)` for the same
   candidate. For a continuation-shaped credit-strategy candidate this
   means the rendered `OptionSetup.spread_width` (e.g. $1.50) and the
   `dollar_risk` that was actually computed from it (now based on $3.50) no
   longer describe the same number — a smaller, output.py-facing version of
   REQUIRED EDIT 1. Worth a one-line comment or a `max_loss`-carrying field
   on `OptionSetup` once the continuation-path scope question above is
   resolved, so a reader of the rendered card isn't shown a stale
   `spread_width` next to a `dollar_risk` it didn't produce.

2. **`_max_loss_for_strategy` duplicates the credit/debit strategy-name
   check `_select_strategy` already encodes.**
   `_max_loss_for_strategy` (options.py:418) re-lists
   `(BULL_PUT_SPREAD, BEAR_CALL_SPREAD)` as the credit set, a second place

   (options.py:272-288) if a third strategy is ever added. Low risk today
   (both are co-located in the same module and covered by the new tests),
