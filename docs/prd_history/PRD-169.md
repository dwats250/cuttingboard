# PRD-169: Persist continuation_audit to logs/audit.jsonl

Lane: STANDARD / SIDECAR
Status: IN PROGRESS
Author: Dustin (Gate-1 direction approved 2026-06-09); Claude Code implementing
Filed: 2026-06-09

## WHY NOW

The EXPANSION-regime continuation-rejection tally
(`QualificationSummary.continuation_audit`, built by
`cuttingboard/qualification.py:_build_continuation_audit` at `:821`) is
populated on the canonical `_run_pipeline` path and emitted to the run
logger via `_log_continuation_audit` (`cuttingboard/runtime.py:1294`),
but it is **not** written to `logs/audit.jsonl`. The canonical pipeline
record built by `cuttingboard/audit.py:_build_record` (`:90`) snapshots
every other decision-time tally — `symbols_qualified`,
`symbols_excluded`, `excluded_symbols`, `trade_decisions`,
`suppressed_candidates`, `regime_short_circuited` — but the EXPANSION
continuation breakdown is the lone decision-time tally missing from the
record. This is the standalone audit-completeness gap recorded under
`docs/PROJECT_STATE.md § Known technical debt` (re-eval 2026-08-15),
pulled forward as the first batched-gate-autonomy target.

## LANE / CLASS NOTE

`LANE: STANDARD / SIDECAR` is the binding classification, not the
`CONTRACT` label used in the initial Gate-1 framing. Per
`docs/audit_doctrine.md:110-113`, field additions to the pipeline-record
shape **do not** require a CONTRACT-class PRD; schema discipline is
governed by the `docs/PRD_PROCESS.md § CLASS Matrix` CONTRACT row "on its
own merits." The CONTRACT row's HIGH-RISK FILES are `cuttingboard/output.py`,
payload-shape definitions, `ui/contract.json`, and any module defining the
`TradeDecision` shape — `cuttingboard/audit.py` is **none** of these, and
the CONTRACT forbidden-mutation surfaces (silent fallback, partial malformed
recovery, threshold→label synthesis) are not touched. The change adds an
observed field to an append-only observational record with targeted
writer/reader tests, which is the SIDECAR shape. `FILES` does not intersect
any SIDECAR HIGH-RISK file (`trend_structure.py`, `evaluation.py`, a new
`*_sidecar.py`), so the lane is STANDARD, not HIGH-RISK. The SIDECAR CLASS
Matrix row still requires **Claude + Codex review** before merge; that gate
is honored regardless of lane.

1 production LOC in `cuttingboard/audit.py` (one key in the record dict),

## GOAL

Persist `continuation_audit` as a field on the existing single pipeline
record in `logs/audit.jsonl`, so the EXPANSION continuation-rejection
breakdown is recoverable from the canonical run log alongside every other
decision-time tally, closing the named audit-completeness gap.

## SCOPE

- **R1 — `continuation_audit` is a field on the pipeline record.** Add the
  key `continuation_audit` to the record dict built by
  `cuttingboard/audit.py:_build_record`, sourced from the
  `qualification_summary` the function already binds as `qual` (`:109`).
  Value is `qual.continuation_audit` when a qualification summary is present,
  else `None`. No signature change, no new write site, no change to
  `write_audit_record`'s threading (it already passes `qualification_summary`
  to `_build_record` at `:71`), and no change to the `runtime.py:1022` call
  site (it already passes `qualification_summary`).

## OUT OF SCOPE

- **Any new audit write site.** The field rides the one canonical
  `_run_pipeline` write site (`runtime.py:1022` → `write_audit_record` →
  `_build_record`); Audit Doctrine Rule 1 (writes only from `_run_pipeline`)
  and Rule 2 (one record per invocation) are preserved unchanged. The
  notify-path `_log_continuation_audit` calls (`runtime.py:511,534`) remain
  log-only and are not given audit-write coverage.
- **The shape of `continuation_audit` itself** — the dict produced by
  `_build_continuation_audit` is persisted verbatim; its keys are not
  renamed, filtered, or reshaped.
- **Any consumer that reads the new field.** No reader is added in this PRD;
  the rationale is completeness/shape-consistency of the record, not a
  decision-feeding consumer. (Permitted: this is a field addition, not a new
  write site, so Rule 1's named-consumer requirement is not triggered.)
- **`runtime.py`, `qualification.py`, the contract/payload, the dashboard,
  notifications** — untouched.

## REQUIREMENTS

R1 — The pipeline audit record carries `continuation_audit`. The dict
returned by `_build_record` (and therefore the JSON appended by
`write_audit_record`) contains a `continuation_audit` key. Its value is the
`continuation_audit` dict from the qualification summary when one is present,
and `None` when `qualification_summary` is `None`.

FAIL (any of):
- `_build_record(...)` returns a dict with no `continuation_audit` key;
- given a `qualification_summary` whose `continuation_audit` is
  `{"total_candidates": 3, "accepted": 1, "NO_BREAKOUT": 2}`, the record's
  `continuation_audit` is not equal to that dict;
- given `qualification_summary=None`, the record's `continuation_audit` is
  not `None`;
- the change alters the record's other keys, the call signatures of
  `_build_record` / `write_audit_record`, or adds a second write site.

## DATA FLOW

`qualify_all` → `QualificationSummary.continuation_audit`

## FAIL CONDITIONS

- The audit record omits `continuation_audit` (R1).
- A populated `continuation_audit` is not persisted verbatim (R1).
- A `None` qualification summary yields a non-`None` value (R1).
- The change introduces a new write site or signature change (Audit Doctrine
  Rule 1 / R1 OUT OF SCOPE).
