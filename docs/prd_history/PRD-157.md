WHY NOW
The prescriptive pre-market report planned in `audits/recon-2026-05-24/next-batch-staging.md` (W4) requires a position size in contracts, computed from account equity, ready to type into Moomoo. Today's pipeline sizes against `config.TARGET_DOLLAR_RISK = 150` (a fixed dollar target), and `dollar_risk` / `max_contracts` are computed in `qualification.py` but dropped at contract assembly. This PRD is the foundational prereq (B4 in the plan); without it, every downstream report unit (W4/W5/W6) is blocked on sizing fields that don't reach the contract.

MAX EXPECTED DELTA
Production LOC: ≤ 50 across `config.py` / `qualification.py` / `options.py` / `contract.py`. Test LOC: new `tests/test_account_equity_sizing.py` + small additions to `tests/test_phase5.py` and `tests/test_contract.py` (≤ 150 LOC total). Doc LOC: ≤ 20 across `docs/trade_qualification.md` and `docs/decision_quality_map.md`. No new modules.

GOAL
Replace the fixed `TARGET_DOLLAR_RISK` / `MAX_DOLLAR_RISK` sizing knobs in `cuttingboard/config.py` with `ACCOUNT_EQUITY` + `MAX_RISK_PCT_PER_TRADE`. Rewrite the two `qualification.py` sizing call sites and the `options.py` correlation-recompute call site to read the new knobs. Pass `position_size`, `dollar_risk`, and `estimated_debit` through `contract._build_trade_candidates` per-candidate dict so the prescriptive reports can consume them. Defaults preserve current sizing behavior exactly.

IMPLEMENTATION WATCHOUTS
- **QualificationResult field semantics must not drift.** The PRD changes how `max_contracts` / `dollar_risk` are computed but not what they mean. The dataclass at qualification.py:100-101 stays the same shape; only the formula and the docstring update. If implementation pressure tempts a meaning change (e.g. renaming `max_contracts` to `position_size` on the dataclass, or making `dollar_risk` represent something other than `max_contracts × spread_cost`), STOP. Downstream consumers in `contract.py` (and `cuttingboard/output.py:326-327`) assume the existing semantics; a field-semantic drift mid-implementation expands scope beyond this PRD.
- **`tests/test_contract.py:614` (`_assert_trade_candidates_valid`) is the canary.** This helper pins the per-candidate dict shape. R8 extends it to require the three new keys. If it starts failing mid-implementation in any way OTHER than the expected "new key missing → add it" path, the contract passthrough has drifted from spec. Stop and reconcile before continuing.
- **Commit per validation step, not in one batch.** R11's full-suite + ruff gate applies at each commit, not just the final one. The COMMIT PLAN steps 3–9 produce ~7 logical commits; batching them defeats `git bisect` if a regression appears between steps. Land step N, run `python3 -m ruff check cuttingboard tests && python3 -m pytest tests -q`, commit, then proceed to step N+1.
- **After B4 lands: PAUSE before B1/B2.** Do not auto-chain into B1 (absolute strikes) or B2 (calendar expiry). If the numbers feel off, revisit `MAX_RISK_PCT_PER_TRADE` default or the GATE_MAX_RISK soft-failure edge case BEFORE B1/B2/W4 layer more contract surface on top. Changing it once it's load-bearing for the report is more expensive.
