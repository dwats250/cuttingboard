CLASS
INFRA

WHY NOW
CI/test hygiene was the next gap: the operational workflow throws two
static-analyzer problems in VS Code, there is no isolated lint+test
merge gate guarding `main`, and the local pre-push hook tests only one
file — so a broken full suite can be pushed unnoticed.

MAX EXPECTED DELTA
<= 30 changed lines across the three target files plus one new workflow
file. Zero Python production-code lines. Exceeding this stops
implementation and re-triggers review.

GOAL
Add an isolated lint+test merge gate, make the local pre-push hook run
the full suite, and resolve the two VS Code "Context access might be
invalid: PUBLISH_READY" problems on the operational workflow without
changing its runtime behavior.

SCOPE
- Add `jobs.pipeline.env.PUBLISH_READY: 'false'` to
  `.github/workflows/cuttingboard.yml` as a static-analysis default
  that is overridden at runtime by the existing `$GITHUB_ENV` writes.
- Add `.github/workflows/ci.yml`: a lint+test gate on `pull_request`
  and `push` to `main`, isolated from the operational pipeline.
- Replace the single-file conditional block in
  `scripts/pre_push_check.sh` with a full-suite `python -m pytest
  tests/ -q` run; keep ruff and git-status steps unchanged.

OUT OF SCOPE
- No change to any operational-pipeline step logic, gating expression,
  run command, schedule, or `$GITHUB_ENV` write in cuttingboard.yml.
- No production code, no test files, no dependency changes.
- No change to the other operational workflows (hourly_alert.yml,
  dashboard_preview.yml, pages.yml, telegram_debug.yml).

REQUIREMENTS

R1 — Static-analysis resolution without runtime change
A job-level `PUBLISH_READY: 'false'` default is added under
`jobs.pipeline.env` so the GitHub Actions VS Code extension resolves
both `env.PUBLISH_READY` references. The default must not alter gating:
GitHub precedence has a prior step's `$GITHUB_ENV` write override a
job-level `env` of the same name for subsequent steps, so the existing
`echo "PUBLISH_READY=true" >> "$GITHUB_ENV"` writes still win.

FAIL: `jobs.pipeline.env.PUBLISH_READY` is absent; OR any of the eight
`PUBLISH_READY=true >> "$GITHUB_ENV"` lines is removed or modified; OR
either gate's `if:` expression diverges from
`success() && env.PUBLISH_READY == 'true'`.

R2 — Isolated CI merge gate
`.github/workflows/ci.yml` triggers on `pull_request` and on `push` to
`main`, checks out, sets up Python 3.11, installs `.[dev]`, runs
`ruff check cuttingboard/ tests/`, then `pytest tests/ -q`. It does not
read secrets, write `$GITHUB_ENV` publish flags, commit artifacts, or
push.

FAIL: ci.yml is absent; OR it lacks a `pull_request` trigger; OR it
lacks a `push: branches: [main]` trigger; OR it references
POLYGON_API_KEY, artifact commit/push, or a PUBLISH_READY `$GITHUB_ENV`
write.

R3 — Pre-push hook runs the full suite
`scripts/pre_push_check.sh` runs `python -m pytest tests/ -q`. The
prior `tests/test_dashboard_renderer.py`-only conditional block is
gone; the ruff preflight and `git status --short` steps are unchanged.

FAIL: the script still contains the
`tests/test_dashboard_renderer.py`-only conditional; OR it does not run
`python -m pytest tests/`; OR the ruff or git-status steps are removed.

R4 — No behavior or production surface touched
No file outside FILES is modified. The Python test baseline is
unchanged.

FAIL: `git diff --name-only` lists any path outside FILES; OR
`python -m pytest tests/ -q` diverges from the 2607-passing / 1-xfailed
baseline.

DATA FLOW
PR or push-to-main -> ci.yml (ruff + pytest) -> merge gate pass/fail
local pre-push -> pre_push_check.sh (ruff + full pytest + git status) -> go/no-go
cuttingboard.yml run step -> $GITHUB_ENV PUBLISH_READY=true -> overrides job-level 'false' -> commit/push gates fire as before

FAIL CONDITIONS
- `jobs.pipeline.env.PUBLISH_READY` absent, or any `$GITHUB_ENV`

- ci.yml missing either trigger, or touching secrets/artifacts/push.
- pre_push_check.sh not running the full suite, or ruff/git-status
  steps removed.
- Any modified path outside FILES, or any test-baseline delta.
