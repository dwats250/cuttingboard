# PRD-195: Publish-branch run_*.json storage cap/prune

Lane: HIGH-RISK / INFRA
Status: IN PROGRESS
Author: Claude Code (scoped per 2026-06-17 morning charge; landed per Dustin direction 2026-06-18)
Filed: 2026-06-18

## Problem

Every pipeline / hourly / Sunday run writes a timestamped `logs/run_<ts>.json`
summary (`cuttingboard/runtime/__init__.py` `_write_summary_files`). Because
`logs/` is gitignored, the publish helper force-adds it
(`tools/ci_push_artifacts.sh`), and the restore step pulls the whole
`logs/run_*.json` glob back from the publish tip before each run. The net effect
is **restore-all → add-this-run → push-all**, so the publish branch accumulates
one new `run_*.json` **forever** — none are ever deleted.

The dashboard renderer already bounds *display*: `dashboard_renderer.py`
(`HISTORY_LIMIT = 5`) globs every `run_*.json`, sorts by timestamp descending,
and slices to the newest 5. So **display is capped but storage is not** — the
publish branch (and every restore/checkout of it) grows without bound at roughly
8–12 files/week. This is the unbounded-accumulation follow-up tracked in
`docs/PROJECT_STATE.md`.

## Approach

Add a **storage cap** that keeps only the newest N `run_*.json` on the publish
branch, where N is comfortably above the renderer's display window so the
dashboard is never starved.

1. **New `tools/prune_run_history.sh <worktree> [retain]`.** Operating inside a
   given worktree, it lists `logs/run_*.json` (shell glob), sorts by filename —
   which is `run_YYYY-MM-DD_HHMMSS.json`, lexicographically == chronologically —
   keeps the newest `retain`, and `git rm -f` the rest so the deletions are
   staged for the caller's commit. Retain defaults to `$CB_RUN_HISTORY_RETAIN`
   or **50**; a non-numeric value falls back to 50 and the value is floored at 1
   so a misconfiguration can never wipe history below the display cap. A count
   `<= retain` (or no run files) is a clean no-op exit 0. Only `logs/run_*.json`
   is touched — `audit.jsonl`, `latest_*`, `regime_history`, and macro snapshots
   are never matched.
2. **One call wired into `tools/ci_push_artifacts.sh`.** Inside `attempt_publish`,
   after the artifacts are staged (the `git add -A` / force-add block) and
   **before** the no-net-change `diff --staged --quiet` check, invoke
   `prune_run_history.sh "$wt"` against the publish worktree. Placing it after
   staging makes the pruned deletions part of *this* publish commit; placing it
   before the no-net-change check lets a prune-only delta still land. The call is
   resolved via the script's own directory (`BASH_SOURCE`), so it is independent
   of the workflow's CWD, and runs on every attempt (each attempt rebuilds the
   worktree from the current publish tip and re-prunes — idempotent).

The current run's `run_<ts>.json` is always the newest timestamp, so it is
always retained; the prune only ever removes strictly-older files.

## Decision surface

- **Retention default = 50** (`CB_RUN_HISTORY_RETAIN` overridable). Rationale:
  must be `>= HISTORY_LIMIT` (5) so the dashboard's "newest 5" is always
  available; 50 ≈ 5–6 weeks of audit history at the observed 8–12/week rate while
  bounding the branch hard. Operators raising the renderer's `HISTORY_LIMIT` must
  keep `CB_RUN_HISTORY_RETAIN >= HISTORY_LIMIT`.
- **Count-based, not age-based.** Deterministic and testable; filename sort is the
  chronological order with no clock parsing.

## Acceptance criteria

- After a publish, the publish branch holds at most `retain` `logs/run_*.json`
  files; the newest `retain` (including the run just published) are kept and only
  strictly-older files are removed.
- The prune is a clean no-op (exit 0, no deletions) when the count is `<= retain`
  or there are no run files.
- A non-numeric or `< 1` retain value can never delete below 1 file; the default
  (50) is `>= HISTORY_LIMIT` so the renderer's display window is never starved.
- Only `logs/run_*.json` is pruned; `audit.jsonl` / `latest_*` / `regime_history`
  / macro snapshots are untouched.
- The deletions land in the same publish commit (they are staged before commit),
  so a single publish both adds this run and drops the over-cap tail.

## Non-goals

- Not changing run-summary creation, naming, or content (`runtime`).
- Not changing the renderer's `HISTORY_LIMIT` or any display logic.
- Not changing the restore step, the audit delta-append, the ui/ sync, or the
  non-fast-forward retry behavior of `ci_push_artifacts.sh`.
- Not age-based retention or compaction of run contents.
- Not rewriting publish-branch history (no force-push / filter-branch); the cap
  applies going forward via ordinary commits.

## FAIL-line observability

- FAIL: `tests/test_prune_run_history.py` — with `retain` run files + k extra on a
  worktree, the prune leaves exactly `retain` (the newest), `git rm`-ing the k
  oldest (today no prune exists → all files remain → fails).
- FAIL: `tests/test_prune_run_history.py` — non-run `logs/` files (`audit.jsonl`)
  survive a prune (guards the glob is scoped to `run_*.json`).
- FAIL: `tests/test_ci_artifact_hygiene.py` — `ci_push_artifacts.sh` invokes
  `prune_run_history.sh` after staging and before the no-net-change check (today
  the call is absent → fails).
