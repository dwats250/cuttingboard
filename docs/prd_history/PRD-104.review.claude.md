## Strengths

- Scope is tightly bounded: documentation only, no source changes, no contract mutations.
- OUT OF SCOPE section is exhaustive and names every protected boundary explicitly.
- Acceptance criteria are predominantly deterministic and grep-able.
- VALIDATION block provides ready-to-run commands that directly verify each heading AC.
- Future sidecar guidance section preemptively documents planned sidecars without implementing them — good architectural constraint capture.
- FILES section cleanly separates "create" from "read", reducing implementer ambiguity.
- The requirement for `frozen=True`-style sidecar boundary rules is correct and load-bearing.

---

## Cohesiveness

GOAL → SCOPE → OUT OF SCOPE → FILES → REQUIREMENTS → FAIL CONDITIONS form a coherent chain with one gap: the SCOPE section's "Artifact Writers" requirement says to identify "every source path that writes current repo artifacts" but the artifact inventory in REQUIREMENTS and AC #5 enumerates only 8 artifacts. The actual system writes at least 5 additional artifacts not covered. An implementer following the AC strictly could produce a document that is technically passing but structurally incomplete — undermining the stated goal of making future PRDs safe to implement.

The REQUIREMENTS/DATA FLOW sections describe what the documentation should contain but do not map directly to a DATA FLOW heading in the PRD template. This is acceptable given the documentation-only nature of the PRD.

## Critical Problems

**Problem 1:** The dashboard renderer default output path does not match the artifact paths listed in the PRD.
- **Root cause:** `dashboard_renderer.py:38` sets `_OUTPUT_PATH = Path("reports/output/dashboard.html")`. In CI, the hourly workflow overrides with `--output ui/dashboard.html` (`.github/workflows/hourly_alert.yml:66`). `ui/index.html` is produced by `cp ui/dashboard.html ui/index.html` in the shell step — it is not written by any Python function. The PRD's artifact inventory and writer documentation will be inaccurate if it names `ui/dashboard.html` without qualifying this CI-only behavior.
- **Fix:** In the artifact flow map, document two distinct renderer output paths: `reports/output/dashboard.html` (local/default render), `ui/dashboard.html` (CI hourly, via `--output` flag), and note that `ui/index.html` is a shell copy artifact, not a Python writer artifact. AC #5 should continue to require both `ui/` paths since they are the live Pages artifacts, but the Artifact Writers section must accurately reflect the path resolution chain.

---

**Problem 2:** Non-hourly payload path `logs/latest_payload.json` is absent from the artifact inventory.
- **Root cause:** `transport.py:16` sets `_DEFAULT_JSON_PATH = "logs/latest_payload.json"`. Non-hourly pipeline runs call `deliver_json(payload)` which writes to this path. The hourly run writes to `logs/latest_hourly_payload.json` explicitly (`runtime.py:1841`). The dashboard renderer defaults to reading `logs/latest_payload.json` (`dashboard_renderer.py:36`). These are different files on disk. The PRD lists only `logs/latest_hourly_payload.json`, leaving the non-hourly path undocumented and the renderer's primary read path misidentified.
- **Fix:** Add `logs/latest_payload.json` to the artifact inventory and document both payload write paths: hourly writes to `logs/latest_hourly_payload.json`, non-hourly writes to `logs/latest_payload.json` via `transport._DEFAULT_JSON_PATH`. Document that the dashboard renderer reads `logs/latest_payload.json` by default and is overridden to `logs/latest_hourly_payload.json` in the hourly workflow.

---

**Problem 3:** Four additional artifacts written by the system are missing from the required inventory.
- **Root cause:** The PRD's AC #5 requires entries for 8 artifacts. The following system-written artifacts are absent: `logs/latest_run.json` (`runtime.py:165`, non-hourly run summary), `logs/evaluation.jsonl` (`evaluation.py:31`, written per signal evaluation), `logs/performance_summary.json` (written by `performance_engine.py`), `logs/sector_router_state.json` (`runtime.py:2055`, written per run in non-prefetch mode). Additionally, `reports/output/report.html` is the transport layer's default HTML output (`transport.py:15`) and is a distinct path from both `reports/output/dashboard.html` and `ui/dashboard.html`.
- **Fix:** Either add these artifacts to AC #5 and the REQUIREMENTS inventory, or add explicit language to the REQUIREMENTS section stating: "The Artifact Inventory must document all artifacts currently written by the system, not only the 8 named in AC #5. The 8 listed are minimum required; implementers must audit the codebase for additional paths."

---

**Problem 4:** AC #10 is broken by a pre-existing uncommitted change.
- **Root cause:** `git status --short` currently shows `M CLAUDE.md`, which predates PRD-104. AC #10 reads "git status after validation shows only the two new documentation files modified or added." This will always fail. The escape hatch ("unless the implementer explicitly reports a justified exception") is non-deterministic and makes the AC non-binary.
- **Fix:** Replace AC #10 with: `git diff --name-only HEAD -- docs/` shows exactly `docs/system_logic_map.md` and `docs/artifact_flow_map.md`. Or scope the git status check: `git status --short -- docs/system_logic_map.md docs/artifact_flow_map.md` shows both as newly added.

---

**Problem 5:** VALIDATION grep for sidecar mutation constraint verifies only one of seven required items.
- **Root cause:** AC #6 requires the document to explicitly state that sidecars must not mutate: contracts, payloads, trade decisions, qualification gates, market_map grades, notifications, and existing dashboard artifacts. The VALIDATION block only greps for `"must not mutate contracts"`. A single phrase match cannot verify that all 7 items are documented.
- **Fix:** Either add individual grep commands for each of the 7 items, or change the requirement to use a single canonical phrase that contains all 7 (e.g., `grep -n "must not mutate" docs/system_logic_map.md` is necessary but not sufficient — the document must contain all 7 items in proximity).
