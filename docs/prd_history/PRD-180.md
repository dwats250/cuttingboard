WHY NOW (ratified decision D-Q2, 2026-06-12 gate recon)
The 2026-06-12 gate recon (docs/audit/gate_recon_2026-06-12.md, flags C1/O1,
open question 2) found a doc-vs-code conflict: docs/system_logic_map.md:63
states runtime "computes _kill_switch which can force HALT", but the code does
not force HALT. _kill_switch (cuttingboard/runtime/__init__.py:1951) only zeroes
the qualified-trade count in the run summary, with a verify-step assertion as
the sole backstop. Terminal HALT comes exclusively from validation
(validation_summary.system_halted -> OUTCOME_HALT). Separately, the three
trigger thresholds are documented nowhere. Dustin ratified the resolution:
make the kill switch force a real HALT, and document the thresholds. This PRD
encodes that decision; it is drafted, not implemented.

MAX EXPECTED DELTA
~50 production LOC, confined to cuttingboard/runtime/__init__.py. Plus
documentation edits (system_logic_map.md, VISION.md) and test additions. No
new production files. If implementation exceeds this, stop and amend.

GOAL
When any kill-switch threshold trips, the run resolves to the same terminal
HALT state that validation already produces - not merely a zeroed
qualified-count - and the three thresholds become named, documented constants.

SCOPE
- On a _kill_switch trip, drive the run to the existing terminal HALT outcome
  (OUTCOME_HALT) with the same downstream effects validation HALT produces:
  trade content suppressed, publish/notification suppression consistent with a
  halted run, and verify_run_summary internally consistent.
- Lift the three trigger thresholds (currently the literals vix_level > 35,
  vix_pct_change > 0.15, abs(spy_pct_change) > 0.03 at
  runtime/__init__.py:1957-1959) into named module-level constants.
- Document the thresholds canonically in docs/system_logic_map.md and resolve
  the C1 conflict text there so doc and code agree.
- Correct the partial kill-switch threshold doc at output.py:204 (it omits the
  vix_pct_change > 0.15 trigger) so it agrees with the canonical doc / code.
- VISION.md "what invalidates" pointer DEFERRED to the declaration workstream
  (recorded in docs/DECISIONS.md); not part of this PRD.

OUT OF SCOPE
- No change to the validation-driven HALT path (validation_summary.system_halted
  and its existing pipeline-skip behavior remain exactly as-is).
- No new or changed threshold VALUES. The three values already in code are
  reused unchanged; only their HALT effect and their named-constant/doc form
  change.
- No automated recovery, cooldown, or hysteresis logic.
- No change to regime, qualification, sizing, candidate ranking, or options
  logic.

REQUIREMENTS

R1 - Threshold trip forces terminal HALT
When _kill_switch returns True for a run, that run resolves to OUTCOME_HALT
with trade content suppressed, identical to a validation-driven HALT, rather
than producing a non-HALT outcome with candidates_qualified forced to 0.

FAIL: A fixture that trips any single threshold produces a run whose recorded
outcome is not OUTCOME_HALT, or whose published/notified content includes trade
candidates.

R2 - Validation HALT path unchanged
A run that is halted by validation (validation_summary.system_halted True) with
the kill switch NOT tripped resolves to HALT exactly as it does today, with no
change in outcome, suppression, or verify behavior.

FAIL: Any existing validation-HALT test changes outcome, suppression, or verify
result under this PRD.

R3 - Thresholds are named constants, values unchanged
The three trigger comparisons read from named module-level constants whose
values equal the current literals (VIX level 35, VIX pct_change 0.15, abs SPY
pct_change 0.03). No threshold value changes.

FAIL: Any threshold literal remains inline at the comparison site, OR any
constant's value differs from the corresponding current literal.

R4 - Thresholds documented; C1 conflict resolved (VISION pointer deferred)
docs/system_logic_map.md names the three thresholds and its _kill_switch line
no longer contradicts the code's HALT behavior. The partial threshold doc at
output.py:204 is corrected (or pointed at system_logic_map.md) so it no longer
omits vix_pct_change > 0.15. The VISION.md "what invalidates" pointer is
DEFERRED to the declaration workstream and tracked in docs/DECISIONS.md; this
PRD does not edit VISION.md.

FAIL: system_logic_map.md still asserts a HALT mechanism that does not match
code, OR the three thresholds appear in no canonical doc, OR output.py:204
still documents an incomplete/inconsistent kill-switch threshold set.

R5 - Publish suppressed under kill-switch HALT
Under a kill-switch-induced HALT, the publish/notification path is suppressed on
the same terms as a validation HALT (no trade alert, halt-banner render).

FAIL: A kill-switch-tripped run publishes or notifies trade content, or renders
a non-HALT report body.

DATA FLOW
regime + normalized_quotes -> _kill_switch (named-constant thresholds) -> on
True: run outcome = OUTCOME_HALT -> trade content suppressed + publish/notify
suppression + verify_run_summary consistent.

FAIL CONDITIONS
- Threshold trip yields any outcome other than OUTCOME_HALT (R1).
- Trade content survives a kill-switch HALT in summary, publish, or notify
  (R1/R5).
- Validation-HALT behavior changes for non-kill-switch runs (R2).
- Any threshold value changes or remains an inline literal (R3).
- Doc conflict (C1) unresolved or thresholds undocumented (R4).

OPEN AT IMPLEMENTATION (review attention)
Mechanism choice (a vs b) is not free. Option (a) - route the late _kill_switch
trip to OUTCOME_HALT - makes the recorded outcome and publish-suppression
correct, but the pipeline has already executed by the time _build_run_summary
evaluates. Option (b) - evaluate the kill switch early (its inputs regime +
normalized_quotes are available early) - shares validation's pipeline-skip and
collapses the two HALT paths into one, reducing future drift. Default to (b)
unless implementation surfaces a concrete reason against. Reviewer must
explicitly check the side-effect / pipeline-skip distinction rather than pick on
convenience.

RATIFIED AT GO (2026-06-13)
- Mechanism (b), TIGHT path. On a _kill_switch trip the run computes regime
  early (regime = compute_regime(validation_summary.valid_quotes);
  normalized_quotes already available), evaluates _kill_switch, sets
  outcome = OUTCOME_HALT, escalates the run to a system halt with
  system_halted = (validation system_halted OR kill-switch tripped), and skips
  the qualification/options/decision block exactly as the validation system_halted
  branch does. candidates_qualified stays 0.
- Implementation strategy: at the trip, the validation HALT carrier
  (validation_summary) is rebuilt as halted (system_halted True + a market-stress
  halt_reason) so every existing downstream consumer (report banner, contract
  status/derive_run_status, contract system_state stay_flat_reason, notification,
  audit record, run summary) treats the market-stress halt identically to a
  validation halt with no per-consumer wiring. R2 is preserved: the validation
  system_halted branch itself is untouched; the kill-switch escalation lives in
  the parallel else path.
- halt_reason is a REQUIREMENT, not a nicety. A kill-switch HALT sets a
  market-stress halt_reason that surfaces in the report banner and notification
  text and reads as a market-stress HALT, NOT as a data/validation/system
  failure (even though status is FAIL).
