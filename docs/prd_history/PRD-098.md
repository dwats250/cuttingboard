# PRD-098 - Candidate Board Visibility and Validation Diagnostics

STATUS
IN PROGRESS
GOAL
Make the Candidate Board a trader visibility layer that renders fresh market_map opportunity context independently of execution permission.

SCOPE
- Render fresh market_map candidates on Candidate Board even when execution permission is NONE, null, or otherwise unavailable.
- Keep A/A+ candidates visible as actionable/high-priority and B candidates visible as watch candidates without promoting B to trades.
- Keep C/D/F and failing candidates hidden by default under collapsed diagnostics sections.
- Render symbol, grade, bias/direction, current_price, reason/summary, failure reasons, and validation requirements for each candidate where source data exists.
- Add deterministic renderer-safe display data named `validation_acceptance` or `validation_requirements` if existing market_map fields do not already provide it.
- Preserve stale market_map protection, deterministic output, and ASCII-only dashboard output.

OUT OF SCOPE
- Do not mutate trade decision logic.
- Do not change candidate grading, qualification semantics, execution permission semantics, or turn B candidates into trades.
- Do not change alerting or notification output unless a missing field makes dashboard rendering impossible.
- Do not change runtime, payload, or market_map producers unless existing fields are missing.
- Do not introduce non-deterministic summaries, generated prose, external data calls, or unrelated dashboard redesigns.
- Renderer MUST NOT synthesize new market opinions, trade recommendations, or generated prose.

REQUIREMENTS
R1 - Candidate visibility separated from execution permission
Candidate Board rendering MUST be driven by fresh market_map availability and candidate data, not by execution permission value.
FAIL: A fresh market_map with candidates renders no Candidate Board candidates solely because permission is NONE, null, or no qualified setup.

R2 - A/A+ actionable rendering preserved
A and A+ candidates MUST remain visible in the high-priority/actionable tier with existing card fields and behavior preserved.
FAIL: Existing A or A+ candidate board assertions regress or A/A+ candidates render only inside diagnostics.

R3 - B watch candidate rendering
B candidates MUST render in the normal Candidate Board flow as watch candidates when market_map is fresh, including when execution permission is NONE.
FAIL: A B candidate from a fresh market_map is absent from the Candidate Board because no trade is permitted.

R4 - Lower-grade diagnostics collapsed by default
C, D, F, and failing candidates MUST render only under collapsed diagnostics by default, preserving operator access without polluting the main board.
FAIL: A lower-grade or failing candidate is visible in the main Candidate Board flow before the diagnostics section is expanded.

R5 - Failure reasons deterministic
Failure reasons MUST use only existing candidate failure, block, or context fields; if none exist, render `No failure reason provided`.
FAIL: The renderer invents a failure reason, omits the deterministic fallback, or emits unstable/non-ASCII output.

R6 - Validation requirements deterministic
Validation requirements MUST use precedence: `candidate.validation_requirements`, then `candidate.validation_acceptance`, then deterministic renderer-derived requirements from existing failure/context fields.
FAIL: A candidate with available validation context renders no validation requirements or uses the wrong source precedence.

R7 - Stale market_map protection preserved
Candidate Board MUST still pause candidate rendering when market_map is genuinely older than the selected run.
FAIL: A stale market_map renders candidates instead of the existing stale protection state.

R8 - Renderer-derived constraints
Renderer-derived requirements MUST use stable templates, stable ordering, ASCII-only output, no external data calls, and no inference beyond existing candidate state.
FAIL: Renderer-derived requirements vary across identical input or contain inferred opinions, recommendations, generated prose, external data, or non-ASCII text.

R9 - Native collapsed diagnostics
Collapsed diagnostics MUST use deterministic native HTML, preferably `details`/`summary`, default collapsed, with no JavaScript dependency.
FAIL: Diagnostics require JavaScript, render expanded by default, or use non-deterministic markup.

DATA FLOW
`render_dashboard_html(payload, run, market_map, ...)` reads market_map freshness first. If fresh, candidate tiers render independently from `run.permission`; A/A+ render as actionable, B renders as watch, and lower-grade/failing candidates render inside native collapsed diagnostics. Validation requirements follow R6 precedence without modifying trade decisions.

FAIL CONDITIONS
- Candidate rendering branches on execution permission instead of fresh market_map candidate availability.
- B candidates disappear when permission is NONE.
- Lower-grade/failing candidates are expanded by default.
- Failure reasons or validation requirements are missing, invented, unstable, out of precedence, or non-ASCII.
- Renderer-derived requirements synthesize opinions, recommendations, generated prose, or external data.
- A/A+ candidate rendering or stale market_map protection regresses.

ACCEPTANCE CRITERIA
- B candidates render from fresh market_map data even when execution permission is NONE.
- Lower-grade and failing candidates are present only inside native collapsed diagnostics by default.
- Failure reasons use only existing fields or `No failure reason provided`.
- Validation requirements follow explicit source precedence and deterministic renderer-derived constraints.
- Existing A/A+ Candidate Board behavior is preserved.
- Candidate Board visibility does not depend on execution permission.
- Genuinely stale market_map data still pauses candidate rendering.
