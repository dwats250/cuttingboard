GOAL
Three small, independent process-hygiene fixes batched into one
micro-PRD. All are tooling or documentation; no production code, no
test changes, no runtime behavior. (1) The `.claude/hooks/prd_eval.sh`
registry-gap check has been emitting a false-positive on
`PRD-138.impl_notes.codex.md` every prompt this session — its
exclusion list already covers `.review.`, `.adjudication.md`, and
`.codex_prompt.md` but not `.impl_notes.`. (2) The `runtime.py` 2100+
LOC monolith was identified as technical debt during PRD-135's
milestone review and again during PRD-141 implementation; record it
in `PROJECT_STATE.md` as a tracked debt item so future PRDs that
touch the file have a natural reference point. (3) Add a one-paragraph
"skill drift" clause to `CLAUDE.md` clarifying when memory-based
discipline application is sufficient versus when explicit `Skill`-tool
invocation is required, to protect against silent drift if skill
definitions are revised.

PRD-142 is IN PROGRESS pending Tuesday autonomous validation but its
implementation commit (`32ef103`) is on `main`; this PRD-143 work is
orthogonal (docs + hook only, zero overlap with the hourly notify
path) and is safe to land in parallel.

`CLAUDE.md` gains a new "Skill invocation discipline" paragraph stating
that (a) memory-based application of established skills (closeout,
scope-lock, dashboard-publish, etc.) is acceptable when the discipline
is concrete and the next step is unambiguous, and (b) when the agent
cannot recall a skill's checklist verbatim, or when the discipline's
output feels uncertain, the agent MUST invoke the skill via the
`Skill` tool rather than reconstruct from memory. The clause sits
immediately after the existing "## session mode" block and before
the "## instrument universe" block.
