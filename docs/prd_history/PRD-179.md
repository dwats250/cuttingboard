PRD-179 — Preview fixture/all-section-state coverage (fast-follow to PRD-178)

Status: IN PROGRESS

LANE
STANDARD (INFRA, additive; no render-logic change, no decision surface, no Codex gate)

CLASS
INFRA

WHY NOW (recorded decision, 2026-06-10)
PRD-178 ships the fresh-data preview loop ONLY, per Dustin's direction at the
PRD-178 direction gate. Fresh data exercises whatever section states the market
happens to produce that hour; it cannot exercise the empty-state forms,
coherence-warning banner, premarket/Sunday banners, or red-folder
populated/missing/malformed branches on demand. That coverage gap is deliberately
deferred to this PRD so it is a recorded decision, not a silent gap.

GOAL
Extend the PRD-178 preview loop with a deterministic fixture harness that renders
the dashboard against synthetic payload sets exercising each conditional section
state, to NON-ui paths only, alongside (never instead of) the fresh-data render.

SCOPE (ratified at opening, 2026-06-17)
- Define a single section-state fixture catalog reusing the existing
  `tests/dash_helpers.py` builders (SEAM 179-A: reuse/extend, approved). Each case
  is (name, payload, run, market_map, render kwargs, expected marker substring).
- Add a local preview harness that renders every case to
  `reports/output/fixture_<name>.html` (SEAM 179-D: non-ui, gitignored).
- Add tests asserting (R1) every enumerated reachable state has a case, (R2) the
  harness output path is non-ui and every case is structurally unpublishable to
  `ui/`, (R4) each render contains its section-state marker.

SEAM DECISIONS (resolved)
- 179-A — reuse/extend `tests/dash_helpers.py` builders. APPROVED (no parallel corpus).
- 179-B — harness home: LOCAL script only this PRD. The optional non-blocking CI
  job in `dashboard_preview.yml` is DEFERRED — `.github/workflows/*` is a
  protected path and the CI job is an explicit bonus (PRD-179.runway §5). Recorded
  as a realizable follow-up, not built here, to keep the warm-up off the
  protected-file path.
- 179-C — coverage set: all reachable states (see REALIZABILITY below).
- 179-D — output `reports/output/fixture_<name>.html` (non-ui, gitignored).

REALIZABILITY (per CLAUDE.md realizability discipline)
Reachable section states covered by a fixture (marker asserted):
- coherence warning (`MIXED_ARTIFACTS`) — three differing generation_ids → MIXED lineage.
- Sunday pre-market banner (`SUNDAY PRE-MARKET CONTEXT`) — coherent + session_type
  SUNDAY_PREMARKET + Sunday-PT timestamp.
- session inactive (`SESSION INACTIVE`) — coherent + SUNDAY_PREMARKET + non-Sunday-PT
  timestamp (sunday_coherent false, inactive_session true).
- macro tape no live data (`NO LIVE MACRO DATA`) — empty macro_drivers + empty snapshot path.
- red folder error (`RED FOLDER UNAVAILABLE`) — red_folder.ok=False.
- red folder empty (`No red-folder events in the next 48 hours.`) — ok, events empty.
- red folder expiring (`Red-folder schedule nearing expiry`) — ok, events present, expiring.
- trend structure awaiting/closed (`MARKET CLOSED — AWAITING INTRADAY DATA`) — coherent,
  fresh snapshot with zero usable rows (AWAITING_DATA active variant).
- trend structure no-data (`no trend structure data`) — coherent active, no snapshot.
- lineage missing / candidate board + trend unavailable (`UNAVAILABLE
  artifact_lineage_state=MISSING`) — market_map absent.
- candidate board no candidates (`NO_CANDIDATES`) — coherent, empty symbols dict.
- healthy baseline (positive control: high-grade card renders, DEMO MODE absent).

Pre-empted / NOT covered (declared dead-by-routing, not faked):
- candidate-board `SOURCE_MISSING` / `PARSE_ERROR` as a *distinct* line requires
  healthy lineage, but a missing/parse-error market_map forces MISSING lineage
  (unhealthy), whose UNAVAILABLE branch pre-empts the distinct status line. So the
  distinct branch is defensive against a routing that does not occur via the
  path-based loader; covered transitively by the MISSING-lineage case.
- trend-structure STALE / INVALID freshness states are time-and-file dependent and
  overlap the lineage STALE path; deferred (not part of the enumerated reachable set).

REQUIREMENTS
- R1 — Coverage. Each enumerated reachable section state (REALIZABILITY above) has a
  catalog case that triggers it.
  FAIL: a listed reachable state lacks a case in SECTION_STATE_CASES.
- R2 — Non-ui only. The harness output path is non-ui, and every catalog case is
  structurally unpublishable to `ui/` (validate_coherent_publish raises for a ui/
  output path).
  FAIL: the harness targets a ui/ path, or any case passes the publish gate to ui/.
- R3 — Additive. No render-logic, gate, or fresh-data-path change.
  FAIL: dashboard_renderer.py render/gate logic or the PRD-178 preview path is modified.
- R4 — Assertion. Tests assert each case render contains its section-state marker.
  FAIL: a case renders without its expected marker and the suite still passes.
