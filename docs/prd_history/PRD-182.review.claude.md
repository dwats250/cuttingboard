## RECOMMENDED EDITS

None blocking. Two optional items, neither in PRD-182 scope:

1. ci.yml sets no `timeout-minutes`, so the merge-gate job inherits the
   360-minute default (cuttingboard.yml caps its job at 20). Harmless for a
   lint+test gate that finishes in ~1 minute; add a cap only if desired.

2. Pre-existing tooling drift surfaced during this PRD (NOT introduced by
   it, NOT fixed here): `docs/PROJECT_STATE.md` writes the active pointer as
   a bulleted `- **Active PRD:**` line, but `scripts/pre_commit_sanity.sh`
   (awk `/^\*\*Active PRD:/`) and `scripts/prd_close.sh` (regex
   `^\*\*Active PRD:\*\*`) both anchor to `**Active PRD:**` at line start.
   Result: the pre-commit scope-lock check silently treats every commit as
   "no active PRD," and prd_close.sh will WARN-and-skip the pointer reset at
   closeout. Confirmed present at HEAD~2 (before PRD-182). Recommend a
   separate micro-PRD to reconcile (de-bullet the line, or relax both
   parsers to tolerate an optional `- ` prefix).

A single key `PUBLISH_READY: 'false'` was appended under the existing
`jobs.pipeline.env` block (alongside POLYGON_API_KEY). The change is
additive-only: all 8 per-mode `echo "PUBLISH_READY=true" >> "$GITHUB_ENV"`
writes are present and unmodified; both publish gates remain
`if: ${{ success() && env.PUBLISH_READY == 'true' }}`; no step, schedule,
or run command changed. Behavior preservation rests on GitHub Actions
precedence: a value written to `$GITHUB_ENV` by a prior step is available to
all subsequent steps in the job and overrides a same-named job-level `env:`
default for those steps. Both gates are STEP-level `if:` (evaluated on the
runner after the run step executes), so they read the runtime `true`; the
well-known "job-level `if:` cannot see $GITHUB_ENV" gotcha does not apply
here. The `'false'` default is load-bearing only for non-publishing modes
(the `noop` else-branch and intraday no-write modes), where it reproduces
the pre-change semantics exactly (an unset var also failed `== 'true'`).
Sources: GitHub Docs - Workflow commands ("all subsequent steps in a job
will have access"); Variables reference (step-scoped env overrides job and
workflow env of the same name); Store-information-in-variables.
