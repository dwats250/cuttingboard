WHY NOW
`watch.py` computes the opening range from the first 5 bars of an already
tail-truncated 120-bar window, so on any full session the ORB is a
mid-afternoon window rather than the 09:30 open.

The mechanism, verified at `15b2819` (three independent truncation/slice
sites, all current):

1. `cuttingboard/ingestion.py::fetch_intraday_bars` fetches `interval="1m"`
   bars, filters to `between_time("09:30", "15:30")`, and returns
   `frame.tail(120)`.
2. `cuttingboard/watch.py::_bars_from_df` truncates again:
   `frame = df.tail(MAX_INTRADAY_BARS)` (`MAX_INTRADAY_BARS = 120`,
   `watch.py:30`).
3. `cuttingboard/watch.py:164` then takes `orb_slice = bars[:N_RANGE]`
   (`N_RANGE = 5`) — the first 5 bars *of the tail window*, not of the
   session.

A full regular session is 361 one-minute bars (09:30-15:30 inclusive).
After truncation to the last 120, `bars[0]` is approximately **13:31 ET**.
The "opening range" is therefore a ~13:31-13:35 window on every session
long enough to exceed the 120-bar cap — i.e. every full session after
roughly 11:30 ET. `orb_high` / `orb_low` (`watch.py:165-166`) are wrong by
construction, silently, with no fail-loud path (PRD-198 invariant 1).

WHY IT MATTERS BEYOND WATCH: ORB is not display-only. It feeds
`cuttingboard/execution_policy.py`'s `orb_inside_range` **BLOCK_TRADE** gate
(`execution_policy.py:231-233,254-273`, per
`docs/audit/gate_recon_2026-06-12.md`), so a wrong opening range can both
block trades that should pass and pass trades that should block. The
downstream-consumer audit is part of this PRD's authoring, not settled here.
