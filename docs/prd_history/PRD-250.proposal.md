## 1. Problem

A stale board presents as **fresh** with every workflow green. During the
2026-07-07 incident the published board read `UPDATED 2026-07-06 09:14:23 PT`
for ~23h while `hourly_alert.yml` runs all reported success — because the publish
tail was *skipped* (slot suppressed, `fresh=false`), not failed. Nothing on the
board, and nothing in CI, said "this read is old."

This is the worst failure class for a discretionary-sizing tool: **a frozen
board can feed a real trade.** The trader opens the page pre-market, reads a
regime/verdict that looks current, and sizes against it — unaware the data is a
day old. The danger is not that the board goes stale (bounded cron latency makes
occasional staleness inevitable); the danger is that **stale is indistinguishable
from fresh at read time.** The goal is to make staleness impossible to miss the
moment the trader looks.

## 2. Load-bearing decision — the freshness verdict is computed CLIENT-SIDE

This is the hinge the whole proposal turns on, and the reason a banner beats
every server-side option:

> The banner's "fresh vs stale" judgment MUST be computed **in the browser at
> view time** — JS embedded in the board reads the embedded `UPDATED` timestamp,
> compares it to the *viewer's* current clock, and renders `STALE · Xh old` past
> threshold. It MUST NOT be baked at publish/render time.

**Why client-side is non-negotiable.** A staleness label computed at render time
is written *into* the artifact — so a board that stops updating **freezes its own
freshness label along with everything else.** A frozen board would proudly
display "fresh" forever, which is exactly the failure we are trying to surface. A
publish-time check cannot detect the pathology (publish not happening) because it
only runs when publish *does* happen. Only a judgment made by the viewer's own
clock, against a timestamp frozen in the page, correctly reports "this page is N
hours old" no matter why the page stopped moving. The staleness verdict has to
live in the one component that keeps running when the pipeline stops: the
reader's browser.

**Minimal render-path dependency (flag for spec, not this pass).** The current
`UPDATED` value is rendered as localized text (`2026-07-06 09:14:23 PT`).
Parsing a localized "PT" string in-browser is fragile (DST, locale). The robust
form is for the renderer to *additionally* emit a machine-readable ISO-8601 UTC
value (e.g. a `data-updated-utc="…Z"` attribute on the `#system-state` block or a
`<meta>` tag) that the client JS reads. That is one small addition to the
presentation layer — **named here, not made here.** To be confirmed against the
render path Dustin will paste before spec.

## 3. Explicitly rejected alternative — widening `max_lag_minutes`

The freeze's proximate trigger was `routine_pt_slot(max_lag_minutes=25)`
suppressing scheduled runs that GitHub delivered 8–46 min late. The tempting
"fix" is to widen the window (25 → 60 → …). **Rejected, on principle:**

- **No fixed window beats unbounded latency.** GitHub scheduled-cron delay is
  best-effort and unbounded — it can and does exceed any constant you pick.
  Widening `max_lag` only chooses a larger number to lose against on the next bad
  day; it converts "often suppressed" into "less often suppressed," never
  "never." The failure recurs at a lower rate, which is *worse* — rare enough to
  stop watching for, common enough to eventually feed a trade.
- **Wrong axis entirely.** Widening treats the *symptom* (the board sometimes
  goes stale). The banner treats the *danger* (stale reads as fresh). Staleness
  is tolerable if it is visible; invisibility is the actual defect. If a wider
  window feels like the answer, that is the signal you are optimizing the wrong
  variable — recorded here so the instinct is not silently re-litigated.

The banner makes the freshness of the read a first-class, always-honest fact,
which is robust to *any* upstream cause (cron latency, suppressed slot, failed
publish, Pages outage) — not just the one that happened this time.

## 4. Threshold policy (to specify)

Freshness is not one number — the board is *expected* to be stale outside the
routine window, so a naive wall-clock age alarm would cry wolf every night.

- **Active window (weekdays, ~06:00–13:00 PT — the `ALLOWED_PT_SLOTS` span plus a
  render/settle margin).** A healthy board updates roughly hourly. Proposed
  threshold: **stale if age > ~90 min** (one missed slot + realistic lag +
  render). Past that during active hours the read is genuinely behind → loud
  `STALE · Xh old` banner at the top of the board.
- **Off-hours / overnight / weekends / holidays.** No slots fire, so staleness is
  *expected and not alarming*. The banner should NOT fire a red alarm here;
  instead show a neutral line ("market closed · last update HH:MM PT") or
  reference age to the last *expected* slot rather than raw wall-clock. Open
  design question for spec: (a) suppress the alarm outside the window vs (b)
  reference-to-expected-slot.
- **Wrinkle to flag:** distinguishing "expected off-hours staleness" from "active
  hours freeze" requires the client to know the session calendar (weekday +
  PT window; US market holidays as an edge). Doing this purely client-side means
  embedding a small session policy in JS. The fixed 06:00–13:00 PT / Mon–Fri
  window makes the common case trivial; holiday handling can be coarse or
  deferred. Confirm the acceptable simplification at spec.

## 5. Where the banner lives (presentation layer)

- Rendered board: `cuttingboard/delivery/dashboard_renderer.py` emits
  `ui/dashboard.html`, copied byte-identical to `ui/index.html` (the Pages-served
  artifact). The banner is a small **inline `<script>` / static JS asset** in the
  renderer's template, plus a banner container element at the top of the board.
  It runs on load, reads the embedded machine-readable timestamp (§2), computes
  age against `Date.now()`, applies the §4 threshold policy, and reveals the
  banner. **No data-path, contract, payload, or aggregation code is touched** —
  the banner reads only what is already in the page and writes nothing back.
- The banner element sits above `#system-state` so it is the first thing read.
- Exact template location and the machine-timestamp emission point: to confirm
  against the render path before spec (charge: do not touch it now).

## 7. OPTIONAL SECONDARY layer — CI / Telegram staleness ping (flagged)

**Client-side banner leads; this is secondary and separable — a real PRD could
ship the banner alone.** A secondary layer could have a scheduled check (or the
existing Telegram path) emit a "board is Xh stale" ping so a freeze surfaces even
when no one is looking at the page.

**Flagged as the part most worth scrutinizing against the no-automation
non-goal.** It does not execute or size trades, and the repo already runs a
Telegram hourly-alert path (NOTIFY_HOURLY), so a staleness ping is arguably
within an existing notification pattern rather than new automation. But it edges
toward "the system monitoring itself" and acting without the human at the seam —
which is exactly the kind of scope creep VISION warns against ("no execution
automation… a human stays at every seam"). **Recommendation: keep it OUT of the
lead proposal; if wanted, spec it as its own follow-up so the no-automation
question gets a clean, isolated decision rather than riding in on the banner.**
