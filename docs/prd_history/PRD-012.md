PRD-012 — Deterministic Payload Delivery Layer

STATUS
COMPLETE
Commit: b90c8938

GOAL
Add a deterministic payload adapter and transport layer that separates contract serialization from delivery, with full test coverage of both layers.

SCOPE
- Add cuttingboard/delivery/__init__.py
- Add cuttingboard/delivery/payload.py: contract → payload adapter
- Add cuttingboard/delivery/transport.py: payload → delivery transport
- Add cuttingboard/delivery/html_renderer.py: HTML payload render
- Update output.py to use delivery layer
- Update runtime.py to route through delivery layer
- Add tests/test_delivery.py and tests/test_payload.py

OUT OF SCOPE
- No changes to contract schema
- No changes to Telegram transport credentials

FILES
A cuttingboard/delivery/__init__.py
A cuttingboard/delivery/html_renderer.py
A cuttingboard/delivery/payload.py
A cuttingboard/delivery/transport.py
M cuttingboard/output.py
M cuttingboard/runtime.py
A tests/test_delivery.py
A tests/test_payload.py

REQUIREMENTS

R1 — Adapter Separation
payload.py MUST accept a contract and produce a delivery-ready payload dict.
payload.py MUST NOT call any transport function.

FAIL: payload.py calls send_telegram or any transport function.

R2 — Transport Separation
transport.py MUST accept a payload dict and deliver it.
transport.py MUST NOT read from the contract directly.

FAIL: transport.py imports or reads from contract.py directly.

R3 — Determinism
Given the same contract input, payload.py MUST produce identical output on every call.

FAIL: Two calls to payload.py with identical input produce different output.

DATA FLOW
contract → payload.py → payload dict → transport.py → delivery

FAIL CONDITIONS
- Adapter calls transport
- Transport reads contract
- Non-deterministic payload output

VALIDATION
Run: pytest tests/test_delivery.py tests/test_payload.py -q
Confirm: 415 + 364 tests passing.
