# Codex Cross-Review - PRD-139

## Strengths

- The PRD correctly triggers the Cross-review gate: it introduces a new sidecar, new artifact path, new schema, new data source set, and source/test changes.
- The FILES list is narrow and matches the stated implementation surface: PRD, isolated tool, isolated tests.
- The sidecar data flow is explicit that no `cuttingboard/` module reads or writes the snapshot, which is consistent with CLAUDE.md sidecar isolation principles.
- The Fed/Treasury/BLS allowlist is appropriate for a no-decision-impact macro sidecar. These are primary-source macro feeds, and the PRD correctly freezes feed changes behind a follow-up PRD.
- The `tmp` + `os.replace(...)` pattern is sufficient to preserve the prior final snapshot byte-identical on normal fetch/LLM/validation failures, provided all validation and failure-prone work occurs before `os.replace` and the final path is never opened for direct writes.

## Cohesiveness

- R2, R3, R4, DATA FLOW, and FAIL CONDITIONS mostly align around a single standalone collector that writes one flat JSON artifact.
- R2's atomic-write requirement aligns with R4's prior-snapshot preservation tests for HTTP and malformed-LLM failures.
- R1's second grep is directionally adequate for normal static imports from `cuttingboard/` into `tools.macro_collector`, though it will still match comments/strings.
- The explicit bypass of `cuttingboard/config.py` is already acknowledged as a CLAUDE.md technical rule #2 exception. However, it is not the only CLAUDE.md technical-rule conflict: technical rule #10 says "No ... ML models", and this PRD requires an Anthropic LLM call. If the sidecar exception is intentional, the PRD should explicitly call out rule #10 as a second accepted exception or remove the LLM dependency.

## Critical Problems

1. **R1 first grep does not fire on all import violations.**
   - Current check: `grep -rEn "^(from|import)\s+cuttingboard" tools/macro_collector.py`
   - This misses indented imports such as `    import cuttingboard.config` inside a function.
   - It also lacks a module boundary after `cuttingboard`.
   - A binary check should allow leading whitespace and require `cuttingboard` as a module token, for example `^\s*(from|import)\s+cuttingboard(\.|$)`.

2. **The PRD undercounts CLAUDE.md technical-rule exceptions.**
   - The config bypass violates technical rule #2, as noted.
   - The required Anthropic API call also conflicts with technical rule #10's "No ... ML models" language.
   - Because the PRD is explicitly sidecar-only, this may be acceptable, but it must be acknowledged as an exception rather than treated as no additional rule conflict.

3. **Atomic preservation needs one missing observable guard.**
   - `tmp` + `os.replace` preserves the prior final snapshot on failures before replacement.
   - The PRD does not explicitly forbid opening `logs/macro_regime_snapshot.json` directly before validation/replacement.
   - Add a FAIL condition that any implementation path writing/truncating the final path before successful validation is a failure. This closes the exact byte-identical guarantee.

4. **Partial RSS failure behavior is specified incompletely.**
   - R2 says fetch the FEEDS allowlist, but FAIL CONDITIONS only cover "All RSS endpoints fail".
   - Missing binary behavior: if one feed fails after 3 attempts and others succeed, should the run fail, proceed with degraded input, or emit a specific warning?
   - Because the PRD says fixed allowlist and no hidden drift, this needs an observable FAIL condition.

5. **Specified behaviors lacking FAIL coverage.**
   - Exact model target `claude-sonnet-4-6` is required but has no FAIL condition or test assertion.
   - FEEDS must include Federal Reserve, US Treasury, and BLS, but there is no FAIL condition that verifies those entries are present.
   - Per-feed 10s timeout and 1s/2s/4s backoff are required, but R4 only asserts failure after 3 retries; it does not require asserting timeout/backoff values.
   - Missing credentials must not initiate any network call, but R4 does not list that as a required test case even though VALIDATION includes a manual check.

6. **Potentially non-observable wording should be tightened.**
   - R2 FAIL says a successful run must "produce or refresh" the snapshot. "Produce" is observable; "refresh" should be defined as final file replacement, changed mtime, or schema-valid final artifact after command exit.
   - R4 FAIL says pytest must not perform "any real network I/O"; that is only binary if tests install an explicit network-blocking sentinel or fully monkeypatch the HTTP/Anthropic boundaries and assert the mocks.
