## 3. DURABLE FINDINGS NOT IN ANY PRD

Two tooling traps and one open-debt caution were promoted to
`docs/DECISIONS.md` (2026-07-23 entries) this handoff — read those directly
for full detail; summarized here for quick scan:

- **`get_status` (GitHub's legacy combined commit-status API) is silently
  wrong for this repo, not just unhelpful.** It returns `pending`/0 checks
  for every commit regardless of true CI state — a completed green run and
  "CI never started" are indistinguishable through that endpoint. Use
  `get_check_runs` and verify against actual job log content when a
  decision depends on it. Cost real time this session before being caught
  and fixed mid-turn (twice — PRs #162 and #163's early status reads were
  both wrong reads via this endpoint, though no false action was taken off
  them).
- **A sandbox/interactive session's git checkout may be shallow, and a
  shallow clone produces false "commit unresolvable" findings** —
  indistinguishable from real ones without checking
  `git rev-parse --is-shallow-repository` first. This is the **third**
  time this exact trap has bitten a session in this repo (two prior
  `.review.claude.md` artifacts hit it in passing, per the DECISIONS.md
  entry) without ever being promoted to a standing check before now.
- **OPEN DEBT (not fixed, no PRD filed):** `scripts/prd_close.sh`'s
  Test-baseline bullet writes `"(CI truth on main; test job for
  <hash>)"` regardless of whether the count came from `--ci-summary` (real
  CI) or a hand-supplied `--tests` value — the wording doesn't distinguish
  the two, so a sandbox-guessed count would silently misrepresent itself
  as CI-verified. Every closeout this session (`#163`, `#164`) manually
  waited for real CI and read the job log before running `--tests`; the
  script itself doesn't enforce that discipline. Candidate for a future
  MICRO/STANDARD PRD.
- **This session's own drift-correction discipline:** three separate times
  during PRD-269's implementation, a "should be done" state turned out to
  be incomplete once verified against the *full* live registry (not just
  the fixture/sample docs) or against a fresh, adversarial review pass —
  each time by widening scope rather than declaring victory early. Worth
  remembering as a pattern for any future "detector/validator" PRD: verify
  against the full corpus before shipping, not just the known instances
  that motivated the PRD.
