# PRD-020 — Engine Doctor Gate System (CI + Runtime Guardrails)

## SUMMARY

Promote engine_doctor from reporting tool to enforcement authority.
CI enforcement via exit-code gating. Optional runtime guardrail, disabled by default.

---

## OBJECTIVE

- Block CI merges when structural integrity is broken
- Optionally prevent runtime execution when system state is invalid
- Deterministic, auditable health decisions

---

## SCOPE

IN SCOPE:
- Remove `continue-on-error: true` from CI workflow (engine_doctor exit codes are the gate)
- Config flag: `runtime_gate_enabled` in `config.toml`
- Runtime guard in `cli_main()` (subprocess, no tests, before execute_run)

OUT OF SCOPE:
- `evaluate_engine_health.py` — dropped; engine_doctor exit codes are sufficient
- Per-category gate thresholds — dropped; engine_doctor severity model is authoritative
- Changes to engine_doctor core logic (PRD-019)

---

## REQUIREMENTS

### 1. CI Gate

Remove `continue-on-error: true` from the Engine health check step in
`.github/workflows/cuttingboard.yml`. No new file or evaluation layer.

Engine_doctor already emits correct exit codes:
- 0: OK
- 1: unexpected test failures
- 2: import failures
- 3: new circular dependencies
- 4: missing required runtime files
- 5: baseline mismatch

### 2. Config

Add to `config.toml`:

```toml
[engine_doctor]
runtime_gate_enabled = false
```

### 3. Config Reader

Add to `cuttingboard/config.py`:

```python
def get_engine_doctor_runtime_gate(_config_path=None) -> bool:
    """Return runtime_gate_enabled from config.toml [engine_doctor] section."""
```

### 4. Runtime Guard

In `cuttingboard/runtime.py`, `cli_main()`, after mode dispatch for VERIFY and PREFETCH,
before `execute_run()`:

```python
_run_engine_health_gate()
```

`_run_engine_health_gate()`:
- Reads `runtime_gate_enabled` via `config.get_engine_doctor_runtime_gate()`
- If False: returns immediately
- If True: runs engine_doctor as subprocess
    - `python3 tools/engine_doctor.py --json --baseline tools/baseline.json`
    - exit 0 → proceed
    - exit ≠ 0 → log failure reason, raise SystemExit with same code

No tests flag (faster). Subprocess approach avoids coupling runtime to tools internals.

## FILES TO MODIFY

- `.github/workflows/cuttingboard.yml` — remove `continue-on-error: true`
- `config.toml` — add `[engine_doctor]` section
- `cuttingboard/config.py` — add `get_engine_doctor_runtime_gate()`
- `cuttingboard/runtime.py` — add `_run_engine_health_gate()`, call from `cli_main()`
- `tests/test_engine_doctor_gate.py` — new

No other files may be modified.

---

## ACCEPTANCE CRITERIA

AC1 — CI import gate: introduce import failure → CI fails
AC2 — CI circular dep gate: introduce new circular dep → CI fails
AC3 — CI baseline gate: baseline mismatch → CI fails
AC4 — Runtime disabled: `runtime_gate_enabled=false` → `cli_main` proceeds normally
AC5 — Runtime enabled + failure: doctor exit≠0 → `cli_main` aborts before `execute_run`
AC6 — Determinism: same doctor exit code → same gate decision every time

---

## REJECTION CRITERIA

- CI passes despite FAIL-level condition
- Runtime halts inconsistently
- Separate evaluation script introduced
- Per-category config thresholds introduced
- Gate logic duplicated across files
