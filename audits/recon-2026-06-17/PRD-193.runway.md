## 1. Recon (manual trace — gitnexus unavailable)

**Two defects from PRD-189 (why prefetch was parked):**
1. **Publish-unsafe.** The prefetch step sets `PUBLISH_READY=true` (`cuttingboard.yml:160-169`), so
   Commit-artifacts renders `ui/dashboard.html`. But `execute_prefetch()`
   (`runtime/__init__.py:224-236`) only fetches/normalizes/validates/derives (L1–L4) and **writes no
   fresh `latest_payload`/`latest_run`**. At 12:50 UTC the checked-in payload is yesterday's (~23-24h
   old); a weekday `session_type` is not in `INACTIVE_SESSION_TYPES` (`dashboard_renderer.py:197` =
   `{"SUNDAY_PREMARKET"}`), so the PRD-119 gate uses the 3h LIVE window
   (`LIVE_SESSION_MAX_AGE_MINUTES=180`, `:361`) and raises `StalePublishError`
   (`:536-541`, gate decision `:394-398`) → the job fails red daily.
2. **No cross-run benefit.** The OHLCV warm-up cache at `config.OHLCV_CACHE_DIR="data/cache"`
   (`config.py:166`), symbol-keyed parquet (`ingestion.py:369-372`,
   `data/cache/<SYM>_ohlcv.parquet`), freshness by last-bar age only (`_is_fresh_ohlcv_cache`,
   `:146-158`). Commit-artifacts (`cuttingboard.yml:251-288`) stages only `reports/`, `logs/`, `ui/`
   — **never `data/cache`** (gitignored). The warmed cache is discarded; the 13:00 live run
   re-fetches cold.

**Interface lock:** `docs/prd_history/INTERFACE_LOCK.md:74` still lists `Prefetch (12:50 UTC M–F)`
as a live scheduled `cuttingboard.yml` invocation — STALE (reachable only via `workflow_dispatch`).

**Shape-aware cache note:** PRD-190 (IN PROGRESS) adds shape-aware cache invalidation. Any persisted
cache MUST carry a shape/version key or a pre-190 (~126-bar) parquet will be served as "fresh" but
fail the post-190 ≥200-bar SMA requirement. This couples PRD-193 to PRD-190's cache key.

## 3. PRD-194 constraint check (CORE RISK)

PRD-194 invariants and how each option relates:
- **Per-file ownership** (one owner per publish artifact; readers restore read-only).
- **Two-tree sync** (logs flow publish→run delta-append; ui flow main→publish full-sync; exactly two
  directions).
- **Read-back completeness** and **fail-loud restore**.

`data/cache` is currently **outside** the publish model (gitignored, discarded each run).

- **SEAM 193-A Option A (commit `data/cache` to publish):** introduces a THIRD sync direction (a
  binary, full-overwrite, shape-versioned artifact that is neither a logs-style delta-append nor a
  ui-style main-authority sync) and a NEW per-file ownership entry (co-owned by every OHLCV-fetching
  writer, with race risk on concurrent parquet writes). **This perturbs the two-tree invariant and
  requires an explicit PRD-194 ownership amendment + HIGH-RISK review.** Builds AGAINST the invariant,
  not ON it.
- **SEAM 193-A Option B (`actions/cache`):** keeps `data/cache` entirely out of the git/publish model.
  Two-tree sync, per-file ownership, read-back, and fail-loud restore are **untouched**. Builds ON the
  invariant by not perturbing it. **Recommended.**

## 4. Proposed path (recommended: Option B + suppress-render)

3. For prefetch, do NOT set `PUBLISH_READY=true` (cache-warm only, no render) — the simplest

## 5. DECISION SEAMS (pre-surfaced)

**SEAM 193-A — cache persistence mechanism (ARCHITECTURAL).**
- *Option A: commit `data/cache` to the publish branch.* Single sync point, auto-restore — but
  perturbs the PRD-194 two-tree invariant, needs an ownership amendment, has parquet-race + size-creep
  risk, and forces HIGH-RISK. **Touches publish machinery → ESCALATE to human regardless of default.**
- *Option B (recommended): `actions/cache` keyed on trading-day + shape/version.* PRD-194 invariants
  untouched; GitHub-managed TTL handles cleanup; keeps the lane STANDARD. Caveats to handle: cache-key
  versioning for the PRD-190 transition, hourly backup-run (06:00/06:05) write-window serialization,
  and the Sunday→Monday boundary (Sunday report doesn't pre-warm Monday).
- **Recommended default: Option B.** Option A only if there's a hard reason to avoid GitHub Actions
  cache.

**SEAM 193-B — publish-safety mode.**
- *Option 1 (recommended): suppress publish for `prefetch`* — cache-warm only, render nothing. Minimal,
  removes the StalePublishError path entirely. **Concrete (review-required):** `cuttingboard.yml:169`
  currently sets `PUBLISH_READY=true` for prefetch — that line must be DELETED or made conditional-false
  for the prefetch mode, or Commit-artifacts still renders and trips the gate. Add a test guard proving
  prefetch enters neither Commit-artifacts nor Push.
- *Option 2: prefetch produces fresh gate-passing artifacts* (write a real `latest_payload`/`latest_run`
  at 12:50). Larger change to `execute_prefetch`; risks double-publishing the day's dashboard from a
  pre-open snapshot. **Recommended default: Option 1.**

**SEAM 193-C — PRD-190 sequencing + cache-version token (review-required: define, don't defer).** The
cache shape/version key (R3) depends on PRD-190's shape-aware cache — which has **NOT landed** (config
still 6, no shape logic; see PRD-190 runway §0). Pick explicitly: (a) **recommended** — implement PRD-193
AFTER PRD-190 and source the version token from PRD-190's window constant (auto-tracks); or (b) if
implementing first, **hard-pin `CACHE_VERSION=1` in the actions/cache key and bump it when PRD-190 lands**.
Do not leave "defensively" undefined. *Recommended default: (a), sequence after PRD-190.*
