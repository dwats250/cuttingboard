## 1. Recon (manual trace — gitnexus unavailable)

**The problem (from PRD-189).** The four intraday slots — `orb_trajectory`, `post_orb`,
`midmorning`, `power_hour` — were removed from the `cuttingboard.yml` schedule because their
real per-slot behavior lives in the runtime notify path, not the full pipeline, and the intraday
window is already covered by `hourly_alert.yml`. They currently resolve to NOOP.

**Where the pieces are:**
- `cuttingboard/runtime/__init__.py:355-360` — `_execute_notify_run(mode, run_date, notify_mode,
  slot_utc=None)`: the lightweight alert path for non-premarket notify modes. Persists `slot_utc`
  via `save_last_slot(slot_utc)` (`:523-525`) for cross-run dedup; passes `slot_utc` to the hourly
  formatter and `_build_hourly_run_summary` (`:520`). **Does NOT write a pipeline audit record**
  (per `docs/audit_doctrine.md` R1 — notify-mode runs bypass `write_audit_record`).
- `cuttingboard/alert_runner.py:81-95` — entrypoint; calls `_execute_notify_run(..., notify_mode=
  NOTIFY_HOURLY, slot_utc=slot_utc)`; current dedup uses file-based `load_last_slot()` /
  `save_last_slot()`, NOT audit records.
- `cuttingboard/audit.py:90-242` — `_build_record`. Fields include `run_at_utc`, `outcome`,
  `regime`, `router_mode` (**= sector regime LONG/SHORT/MIXED, NOT a slot id**), etc. **No slot/mode
  identifier exists.** Per-slot context only via nested `intraday_state_context` (`:147-151`), which
  is SHORT-permission state, not dedup.
- `.github/workflows/hourly_alert.yml:3-18` — intraday crons (PT-anchored 06:00–13:00 PT);
  entrypoint `python -m cuttingboard.alert_runner` (`:95-100`) → `_execute_notify_run(MODE_LIVE,
  NOTIFY_HOURLY, slot_utc)`.
- `.github/workflows/cuttingboard.yml:11-15` — the ORB/intraday crons were removed (PRD-189 comment).
- `scripts/resolve_run_mode.py:46-63` — `_DEDICATED = {"0 13 * * 1-5":"live", "30 23 * * 0":"sunday"}`;
  prefetch + the two intraday crons (`50 13 * * 1-5`, `*/30 14-21 * * 1-5`) are parked → NOOP.
- `docs/prd_history/INTERFACE_LOCK.md:76-80` — STALE rows still list the four slots as live
  `cuttingboard.yml` scheduled invocations (`--mode <slot> --notify-mode <slot>`); they NOOP now.

**Blast radius of adding a slot/mode marker to the audit record** — all downstream readers are
field-agnostic (filter on `"outcome"`+`"run_at_utc"`), so a new field is additive/safe:
- `runtime/__init__.py:1637` `_load_run_history` — filter-safe (LOW).
- `delivery/regime_history.py:118` `aggregate` — filter-safe (LOW).
- `evaluation.py` `load_most_recent_prior_run()` — signature unchanged (LOW).
- `reports/postmarket.py:101` — reads contract, not audit directly (LOW).
- `delivery/dashboard_renderer.py` — reads `latest_run.json`, not `audit.jsonl` (NONE).

**Protected surface:** `cuttingboard/runtime/__init__.py`, `_constants.py`, `_types.py` are the
acknowledged-debt protected package (re-eval 2026-08-15). `audit.py` is not in `runtime/` but its
schema is governed by `audit_doctrine.md` Rule 1 (a new field must cite a named consumer).

**REQUIREMENTS**
- R1 — Correct home. Each of the four slots executes its intended per-slot scan via the notify path
  (`_execute_notify_run`), not a full `cli_main` pipeline at intraday times.
  FAIL: any slot triggers a full live pipeline + generic live notification instead of its slot scan.
- R2 — Slot marker. The run record carries a deterministic slot/mode identifier.
  FAIL: a slot's run record carries no identifier distinguishing it from another slot's record.
- R3 — Dedup keys on marker. Fired-today dedup uses the slot marker, not wall-clock attribution of
  `run_at_utc`.
  FAIL: two distinct slots in the same wall-clock window collide, or a re-run of one slot is treated
  as a different slot.
- R4 — Interface lock reconciled. INTERFACE_LOCK rows match the landed routing. (NOTE: rows 76-80 are
  currently STALE — R4 is failing today — reconcile as part of this PRD.)
  FAIL: an INTERFACE_LOCK row describes a routing the workflows do not implement.
- R5 — Delta-append invariant (review-required). The marker is written at record-creation time, never
  retroactively to existing rows.
  FAIL: any existing `logs/audit.jsonl` row is rewritten to add the marker (breaks the append-only prefix
  PRD-194 delta-append relies on).

## 3. PRD-194 constraint check

Touches the publish/state machinery only via `audit.jsonl`, which is a **publish-authority,
delta-append** artifact (PRD-194 invariant 2). Adding a field to appended records is compatible with
delta-append (the appended rows simply carry an extra key) **provided** the marker is added to the
record at write time, not retroactively to existing rows — a retroactive rewrite would break the
append-only prefix property the delta-append relies on. **This must be an explicit FAIL condition at
Stage 0.** No change to per-file ownership, two-tree sync direction, read-back completeness, or
fail-loud restore.

## 5. DECISION SEAMS (pre-surfaced)

**SEAM 192-A — slot home (ARCHITECTURAL, highest-leverage).**
- *Option 1 (recommended): fold the four slots into `hourly_alert.yml`* with distinct `notify_mode`
  values. The hourly path already covers the intraday window via `alert_runner`/`_execute_notify_run`;
  this avoids reviving the broken `cli_main` full-pipeline path and keeps all intraday notification in
  one workflow. Lower blast radius (no resolver change).
- *Option 2: resurrect the slots in `cuttingboard.yml`* with a queue-delay-tolerant `*/30` resolver
  (late-only, start-sampled, attribution-dedup). Higher complexity, re-touches `resolve_run_mode.py`,
  and risks the overlap PRD-189 explicitly removed.
- **Recommended default: Option 1.** Escalate to human only if there's a product reason the slots must
  run the full pipeline (not just a notification scan).

**SEAM 192-B — dedup state model.** Replace file-based `last_slot.json` with audit-marker dedup, or
keep file-based and *add* the marker for robustness. *Recommended default: add the marker and switch
the read to it, retaining file-based as a same-run fallback* — minimizes behavior change while
delivering R3.

**SEAM 192-C — marker location.** Put the slot id on the audit record (`audit.py`) vs only on the
hourly run-summary (`latest_run.json`). *Recommended default: hourly run-summary* if notify-mode runs
don't write audit records today (they don't) — avoids forcing notify runs to start writing audit rows
(a larger behavior change). Confirm at Stage 0 which artifact the dedup reader can see.
