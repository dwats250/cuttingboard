## 1. Recon (manual trace — gitnexus unavailable)

**The tracked debt (PRD-194 closeout, DECISIONS 2026-06-17, PROJECT_STATE:12):** each live/hourly run
force-adds a `logs/run_<ts>.json` archive to the `publish` branch. The renderer caps what it
**displays** but storage grows **unbounded**.

- `dashboard_renderer.py:172` `HISTORY_LIMIT = 5` — display cap only.
- `dashboard_renderer.py:1391` and `:2605` glob `run_*.json`, sort, take `[:HISTORY_LIMIT]` — reads at
  most 5; the rest sit unused on `publish` forever.
- `tools/ci_push_artifacts.sh:148` writes a fresh `run_<ts>.json` every run.
- `tools/ci_restore_publish_state.sh:36` glob-restores `logs/run_*.json` from `publish` before a run
  (so the renderer can read history). Restore cost grows with the archive.
- Per PRD-194, `run_*.json` are **immutable per file** (no revert in the push path) and are
  **pipeline-owned** (co-written by the pipeline and hourly STATE-WRITERS).

**Why it's not trivial:** the publish path today only **appends/overwrites** owned artifacts. There is
no delete path. Pruning means the worktree publish must `git rm` superseded `run_*.json` and have that
**deletion propagate** to `publish` — a new operation that must respect per-file ownership, the
two-tree sync invariant, and the cross-workflow race handling PRD-194 built.

## 2. Scope (standard PRD format — DRAFT)

**GOAL** — Bound `run_*.json` storage on the `publish` branch with a deterministic cap and
delete-propagation, without perturbing any PRD-194 publish invariant.

**SCOPE**
- Add a prune step that, at publish time, retains the N most-recent `run_*.json` and deletes the rest,
  with the deletion propagating to `publish`.
- Choose the cap policy (see SEAM 195-A) and align it with (or decouple it from) the renderer's
  `HISTORY_LIMIT`.
- Make prune race-safe under the existing cross-workflow retry (two writers publishing near-simultaneously
  must not resurrect or double-delete archives).

**OUT OF SCOPE**

- No change to `audit.jsonl` delta-append, `ui/` sync, or any other owned artifact's lifecycle.
- No pruning of any other `logs/*` artifact.

**CHANGE SURFACE** (INFRA, publish scripts → T0/HIGH-RISK): `ci_push_artifacts.sh` gains a prune that
runs inside the worktree on the publish tip, ordered AFTER the owned-artifact writes and BEFORE commit,
and inside the existing non-fast-forward retry loop so a re-fetched tip is re-pruned deterministically.

**REQUIREMENTS**
- R1 — Bounded storage. After a successful publish, `publish` holds at most N `run_*.json` files.
  FAIL: a publish leaves more than N `run_*.json` on `publish`.
- R2 — Delete-propagation. Superseded archives are removed from `publish`, not just the worktree.
  FAIL: a pruned file remains present on `origin/publish` after the run.
- R3 — Race-safe. Concurrent pipeline + hourly publishes converge to ≤ N without resurrecting a deleted
  archive or deleting a just-written one.
  FAIL: after two interleaved publishes, a current run's archive is missing or count exceeds N.
- R4 — Renderer unaffected. The dashboard still shows the same history it would under `HISTORY_LIMIT`.
  FAIL: pruning removes an archive the renderer would have displayed.

**DATA FLOW**
each run → write `run_<ts>.json` (owned) → prune to newest N (delete-propagate) → commit/push to publish.

## 3. PRD-194 constraint check (CORE RISK)

- **Per-file ownership:** `run_*.json` is pipeline-owned. The prune must be performed only by an owner
  publisher (the same STATE-WRITERS that create them), never by a non-owner restore-reader. Keep the
  delete inside `ci_push_artifacts.sh` (the owner's publish path). ✔ builds ON ownership.
- **Two-tree sync:** `run_*.json` lives in the `logs/` publish-authority tree. Today that tree only
  delta-appends (`audit.jsonl`) or overwrites; **adding a delete is a new lifecycle operation in the
  publish-authority tree.** This is the precise invariant interaction to get right — the delete must be
  expressed against the publish tip (the authority), not against `main`, and must survive the retry loop.
  This is why the PRD must **amend PRD-194's ownership/lifecycle note**, not silently extend it.
- **Read-back completeness:** the bounded restore (if changed) must still give the renderer its newest N.
- **Fail-loud restore:** unchanged; a present-branch checkout error still fails the job.

Because this edits the publish machinery, it inherits **HIGH-RISK** (Codex required). The amendment to
PRD-194.md's ownership/lifecycle note is **STANDARD domain-doc editing, NOT a governance carve-out** —
the CLAUDE.md manual-merge carve-out (CLAUDE.md:64-71) applies only to edits of CLAUDE.md's own policy /
the `prd-review-claude` gate, not to PRD reference docs (confirmed in the PRD-195 fresh-context review).
PRD-195 should note in its body that it does not touch CLAUDE.md, so no carve-out applies.

**Review-required edits (from `PRD-195.review.claude.md`):** (a) the prune must run AFTER the publish tip
is fetched fresh inside `attempt_publish`'s retry loop (`ci_push_artifacts.sh` ~:91/:93), deleting from
the refreshed tip's current `run_*.json` list, never a restore-time cached list; (b) document that
`run_*.json` immutability (PRD-194.md ~:362) makes restore/prune races safe (retain-newest-N cannot
resurrect a deleted file).

## 5. DECISION SEAMS (pre-surfaced)

**SEAM 195-A — retention policy.**
- *Option 1 (recommended): count-based cap of N, with N ≥ `HISTORY_LIMIT` (=5).* Simplest, deterministic,
  directly matches the renderer's need. Recommended N = a small multiple (e.g. 20) to keep a short audit
  tail beyond what's displayed.
- *Option 2: age-based (keep last K trading days).* More semantically meaningful but couples to a calendar
  and complicates the race logic.
- *Option 3: size-based cap.* Hardest to reason about deterministically.
- **Recommended default: Option 1, N = 20** (≥ HISTORY_LIMIT, with audit headroom). Confirm N with Dustin.

**SEAM 195-B — who prunes (ownership under co-ownership).** `run_*.json` is co-written by pipeline +
hourly. *Recommended default: every owner publisher prunes on its own publish* (idempotent: "retain newest
N" converges regardless of which writer runs), relying on PRD-194's retry loop for races. Alternative: a
single designated pruner workflow — rejected as it adds a moving part and a failure mode (pruner down →
unbounded again).

**SEAM 195-C — bound the restore too?** *Recommended default: yes — cap the restore glob to newest N* so
restore cost is bounded and symmetric with storage. Low risk; keeps read-back complete for the renderer.
